## Еще 4 способов запуска команды

До сих пор использовали два основных способа:
1. Простая команда: `grep Nutshell animals.txt`
2. Конвейер: `cut -f1 grades | sort | uniq -c`

Но оболочка предлагает гораздо больше гибкости. Ниже — 4 первые  из одиннадцати способов.

---

### Способ #1. Условные списки (`&&` и `||`)

Команды могут зависеть от результата предыдущих. Это управляется через **коды возврата**:
- `0` → успех  
- `≠0` → ошибка

##### Оператор `&&` — «выполни следующее, если всё хорошо»
```bash
cd dir && touch new.txt
```
> → `touch` запустится **только если** `cd` завершился успешно.

Практические примеры:
```bash
# Резервное копирование + редактирование + удаление бэкапа
cp file.txt file.safe && nano file.txt && rm file.safe

# Git workflow в одной строке
git add . && git commit -m "fix" && git push
```
##### Оператор `||` — «выполни, если что-то пошло не так»
```bash
cd dir || mkdir dir
```
>→ Если `cd` не удалось — создаём каталог.

Часто используется для аварийного выхода:
```bash
cd important_dir || exit 1
# Комбинирование
cd dir || mkdir dir && cd dir || echo "Не удалось войти или создать dir"
```

>Совет: группируйте сложные цепочки скобками, если нужно контролировать порядок:

```bash
(cd dir || mkdir dir) && cd dir
```

### Способ #2. Безусловные списки (`;`)

Команды выполняются **последовательно**, независимо от результата:

```bash
sleep 300; echo "Пора гулять с собакой!" | mail -s "Напоминание" $USER
```

Особенность:

- Код возврата всего списка — это код **последней** команды.
- Все промежуточные коды теряются.

Пример:
```bash
mv a b; mv b c; mv c d
echo $?  # покажет результат только последнего `mv`
```

>Используйте `;`, когда команды логически связаны, но не зависят друг от друга.

### Способ #3. Подстановка команд `$()`

Оболочка **выполняет команду и подставляет её вывод** прямо в строку.

Синтаксис:
```bash
команда $(другая_команда)
```

Аналог с обратными кавычками (устаревший, но рабочий):
```bash
mv `grep -l "Artist: Kansas" *.txt` kansas
```

Предпочтителен `$()` — он лучше читается и поддерживает **вложенность**:
```bash
echo Today is $(echo $(date +%A) | tr a-z A-Z)!
```

##### Сохранение результата в переменную
```bash
files=$(grep -l "Artist: Kansas" *.txt)
echo "$files"   # обязательно в кавычках!
```

>**Важно**: если имена файлов содержат пробелы, `$()` без дополнительной обработки **сломается**.  
В таких случаях лучше использовать `while read` или массивы .

##### Практический пример: открыть последний PDF
```bash
okular $(ls eStmt_*.pdf | tail -n1)
```

### Коды возврата: как проверить успех?

После любой команды проверьте `$?`:
```bash
ls existing.txt
echo $?   # 0 — успех

ls nonexistent.txt
echo $?   # 1 (или другое ≠0) — ошибка
```

Это то, на чём строятся `&&` и `||`.

### Способ #4. Подстановка процесса (`<()`)

Подстановка процесса — это мощный механизм, позволяющий **использовать вывод команды как файл**, не создавая временных файлов на диске. Это особенно ценно в задачах системного администрирования, где часто нужно сравнивать, фильтровать или синхронизировать данные между разными источниками.

Синтаксис:
```bash
<(команда)
```

Оболочка запускает `команда` в подоболочке и предоставляет её вывод через виртуальный файл в `/dev/fd/...`, который можно передать любой утилите, ожидающей путь к файлу.
#### Пример 1: Сравнить список запущенных служб с эталонным

Допустим, у вас есть эталонный список служб, которые **должны быть активны** на сервере (например, `nginx`, `sshd`, `cron`). Вы хотите проверить, не отключилась ли какая-то из них.

Эталонный список хранится в файле:
```bash
cat /etc/allowed-services.txt
# nginx
# ssh
# cron
```

Теперь сравните его с реально запущенными службами:
```bash
diff <(systemctl list-units --type=service --state=active --no-legend | awk '{print $1}' | sed 's/\.service$//') \
     <(sort /etc/allowed-services.txt)
```

Если вывод пуст — всё в порядке. Если есть расхождения — вы сразу видите, что лишнее запущено или что-то пропало.
#### Пример 2: Найти пользователей, у которых нет домашнего каталога

Сравним список пользователей из `/etc/passwd` с реальными каталогами в `/home`:
```bash
diff <(cut -d: -f1,6 /etc/passwd | grep '^.*:/home' | cut -d: -f1 | sort) \
     <(ls /home | sort)
```

- Левая часть: пользователи, у которых домашний каталог указан как `/home/...`
- Правая часть: реальные каталоги в `/home`

Результат покажет:
- `>` — каталог есть, но пользователя нет («мёртвый» домашний каталог)
- `<` — пользователь есть, но каталога нет (возможно, ошибка при создании)

#### Пример 3: Сравнить списки установленных пакетов на двух серверах

Хотите убедиться, что два сервера имеют одинаковый набор пакетов?

На первом сервере сохраните список:
```bash
dpkg --get-selections > /tmp/packages-server1.txt
```
На втором — сравните напрямую без временного файла:
```bash
diff /tmp/packages-server1.txt <(ssh user@server2 "dpkg --get-selections")
```
Или, если оба списка генерируются локально (например, из разных chroot-окружений):
```bash
diff <(chroot /mnt/server1 dpkg --get-selections) \
     <(chroot /mnt/server2 dpkg --get-selections)
```
#### Как это работает под капотом?

Когда вы пишете `<(...)`, ядро Linux создаёт **анонимный pipe** и связывает его с файловым дескриптором в `/dev/fd/N`. Для внешней программы (например, `diff`) это выглядит как обычный файл:
```bash
echo <(whoami)
# → /dev/fd/63
```
>Это позволяет использовать потоковые данные там, где требуется файл — без записи на диск, без риска коллизий имён, без необходимости очистки.


---

#### Ограничения

- Работает только в **Bash** и **Zsh** (не в POSIX-совместимом `/bin/sh`)
- Некоторые утилиты требуют **seekable** файлы (например, `sort -m`), и тогда `<()` не подойдёт — потребуется временный файл
- Не поддерживает запись (для этого есть `>(...)`, но он редко нужен)
