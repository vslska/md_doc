
### **Переменные окружения**

В Linux (и Unix-подобных системах) каждая оболочка (например, `bash`) работает в своём **окружении** — наборе переменных, которые могут влиять на поведение программ.

Есть два типа переменных:

- **Локальные переменные** — существуют только внутри текущей оболочки.  
Пример:
```bash
name="Alice"
echo $name        # Выведет "Alice"
bash              # Запускаем дочернюю оболочку
echo $name        # Ничего не выведет — переменная не передалась!
exit
```

**Переменные окружения** — доступны не только в текущей оболочке, но и во всех её **дочерних процессах** (программах, скриптах, новых оболочках).  
Чтобы сделать переменную «окружением», используйте `export`:
```bash
export name="Alice"
bash
echo $name        # Теперь выведет "Alice" — переменная передалась!
exit
```

>**Важно**: Переменные окружения **копируются**, а не передаются по ссылке. Изменения в дочернем процессе **не влияют** на родительский.

### **Популярные переменные окружения**
| Переменная | Назначение                                                |
| ---------- | --------------------------------------------------------- |
| `HOME`     | Путь к домашней папке (`/home/user`)                      |
| `PWD`      | Текущая рабочая директория                                |
| `USER`     | Имя текущего пользователя                                 |
| `SHELL`    | Какая оболочка используется (`/bin/bash`)                 |
| `EDITOR`   | Предпочитаемый текстовый редактор (`nano`, `vim`, `code`) |
| `LANG`     | Язык и кодировка системы (`en_US.UTF-8`)                  |
**Пример использования**:  
Многие программы (например, `git commit`) вызывают редактор через `$EDITOR`. Если вы установите:
```bash
export EDITOR=nano
```
>— то при коммите в Git будет открываться `nano`, а не `vim`.

### **Как посмотреть переменные окружения?**


```bash
printenv  # - Все переменные окружения
printenv | sort | less  # Или отсортировано
echo $my_var # Локальные переменные **не видны** в `printenv`. Их можно так проверить 
```

### **Ключевая идея**

> **`export` делает переменную «видимой» для всех программ, запущенных из этой оболочки. Без `export` — только для самой оболочки.**

Это особенно важно при написании скриптов или настройке среды разработки (например, указание путей к Python-библиотекам через `PYTHONPATH`).
### **Бонус: временный экспорт**

Можно задать переменную окружения **только для одной команды**:

```bash
EDITOR=vim git commit
```

>Здесь `EDITOR` будет равен `vim` **только для этого вызова `git`**, а потом вернётся к прежнему значению (или исчезнет).

---
Такой подход безопасен: вы можете экспериментировать в дочерней оболочке — всё исчезнет после `exit`

### Миф: «Переменные окружения — это глобальные переменные»

На самом деле — **нет**.  
В Linux **не существует глобальных переменных**, доступных всем процессам напрямую.

### Правда: каждая оболочка имеет **свою копию** переменных

Когда вы видите, что `HOME` или `PATH` одинаковы в разных терминалах — это **не магия**, а следствие двух простых механизмов:

#### 1. **Наследование от родительского процесса**

- При входе в систему запускается **основная (родительская) оболочка** (например, из графического менеджера или `login`).
- Она устанавливает и экспортирует ключевые переменные: `HOME`, `USER`, `PATH` и т.д.
- Все новые терминалы, скрипты и программы — **дочерние процессы** этой оболочки.
- Они **копируют** её окружение при запуске → поэтому значения совпадают.

>  Но если вы измените `PATH` в одном терминале — во втором он **останется прежним**!  
> Потому что процессы не делят память — у каждого своя копия.

#### 2. **Общие файлы конфигурации**

- Каждый новый экземпляр `bash` читает одни и те же файлы:  
    `~/.bashrc`, `~/.profile`, `/etc/environment` и др.
- Даже если переменная **не экспортирована**, но задана в `.bashrc` — она появится в каждом новом терминале.
- Это создаёт **иллюзию** "глобальности", хотя на самом деле каждый раз происходит **независимая инициализация**.
```bash
# Добавим в ~/.bashrc:
my_color=blue

# Откроем два новых терминала:
echo $my_color   # В обоих будет "blue"
printenv my_color  # Но ничего не выведет — переменная локальная!
```

>→ Значение совпадает не потому, что оно "глобальное", а потому, что оба терминала выполнили один и тот же код при запуске.

### Главный вывод

> **`export` не делает переменную глобальной — он делает её _наследуемой_.**  
> Она передаётся **вниз по цепочке потомков**, но **никогда не влияет на другие независимые процессы**.

Это фундаментальный принцип Unix: **процессы изолированы**, и взаимодействие между ними требует явных механизмов (файлы, сокеты, сигналы и т.д.), а не «общей памяти».
### Почему важно это понимать?
Если вы думаете, что `export PATH=...` изменит `PATH` «везде» — вы ошибаетесь.  
Изменение затронет только:

- текущую оболочку,
- и всё, что вы запустите **из неё дальше**.

Чтобы изменить `PATH` **постоянно для всех будущих сессий** — нужно прописать его в `~/.bashrc` или `~/.profile`.

---

Так что да: переменные окружения — **локальные копии с наследованием**, а не глобальные сущности. И в этом — сила и безопасность Unix-систем.

## (Доп.)
### Как на самом деле работает "передача" переменных окружения

1. **Система изолирована**  
Каждый процесс (включая оболочки вроде `bash`, `zsh`) — независим. У него **своя память**, **своё окружение**. Никакой «глобальной доски объявлений» нет.
2. **Переменные окружения передаются _только при запуске_ дочернего процесса**  
Когда ты запускаешь `zsh` из текущей оболочки, он **копирует** её окружение на момент запуска. После этого — полная независимость.
3. **Файлы конфигурации — источник "единообразия"**
- При входе в систему (login) запускается **login-оболочка** (например, `zsh --login`).
- Она читает **глобальные** (`/etc/zshenv`, `/etc/profile`) и **пользовательские** (`~/.zprofile`, `~/.zshrc`) файлы.
- Эти файлы **устанавливают и экспортируют** переменные (`export PATH=...`, `export EDITOR=...`).
- Все последующие процессы (терминалы, скрипты, GUI-приложения через DE) — либо потомки этой login-оболочки, либо сами читают те же конфиги → поэтому всё выглядит "единообразно".
4. **Если ты меняешь переменную "вручную" — эффект локальный**

```bash
  export MY_VAR=test
```

 >— это повлияет только на текущую сессию и всё, что ты запустишь **из неё**.  
После перезапуска терминала — переменная исчезнет, если она не прописана в`.zshrc` или аналоге.

5. **Login vs non-login оболочки**
- `zsh` (обычный терминал) → читает `~/.zshrc`
- `zsh --login` или вход в систему → читает `~/.zprofile` (и иногда `~/.zshrc`, зависит от настроек)  
>→ Поэтому важно понимать, **какой файл за что отвечает**, особенно для `PATH` и других критичных переменных.

### Пример из жизни

Ты установил Python через `pyenv` и добавил в `~/.zshrc`:
```bash
export PATH="$HOME/.pyenv/bin:$PATH"
eval "$(pyenv init -)"
```

>→ Теперь **каждый новый терминал**, запущенный из графической среды (GNOME, KDE и т.д.), прочитает `.zshrc` и получит правильный `PATH`.

Но если ты запустишь терминал **не как потомок графической сессии** (например, через SSH без login-shell), или если DE не наследует окружение правильно — может не сработать.  
Потому что **нет магии** — только цепочка:  
**login → конфиги → экспорт → наследование → новые процессы**.

### Совет

Если хочешь, чтобы переменная была доступна **везде** (в терминале, в IDE, в GUI-приложениях):

- Для **терминала** — пропиши в `~/.zshrc` (или `~/.bashrc`)
- Для **всей графической сессии** — добавь в `~/.profile` (этот файл читается большинством дисплейных менеджеров при входе)
- Или используй системные механизмы: `/etc/environment`, `systemd --user` (для современных систем)
