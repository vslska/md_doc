### Перенаправление ввода-вывода: основы

В Unix/Linux каждая команда работает с тремя стандартными потоками:

|     |        |                                      |
| --- | ------ | ------------------------------------ |
| `0` | stdin  | Стандартный ввод (обычно клавиатура) |
| `1` | stdout | Стандартный вывод (обычно терминал)  |
| `2` | stderr | Стандартный поток ошибок             |
По умолчанию все три связаны с терминалом. Но их можно **перенаправить** в файлы, другие команды или даже друг в друга.

### Основные операторы перенаправления
| Синтаксис     | Действие                                                     |
| ------------- | ------------------------------------------------------------ |
| `>`           | Перезаписать файл stdout (`1>file`)                          |
| `>>`          | Дописать в конец файла stdout                                |
| `<`           | Прочитать stdin из файла                                     |
| `2>`          | Перенаправить stderr в файл                                  |
| `2>&1`        | Направить stderr туда же, куда stdout                        |
| `&>` или `>&` | Перенаправить **оба** потока (stdout + stderr) в файл (Bash) |
> В Bash `&>` — это сокращение от `> file 2>&1`.

### Глубже: как работает `2>&1`
```bash
command > output.txt 2>&1
```

1. `> output.txt` — связывает **stdout** (`1`) с файлом `output.txt`.
2. `2>&1` — говорит: «свяжи **stderr** (`2`) с тем же местом, куда сейчас направлен `1`» → то есть в `output.txt`.

**Порядок критичен!**  
Неправильно:
```bash
command 2>&1 > output.txt
```
>— здесь `2>&1` направит ошибки в **терминал** (потому что `1` ещё не перенаправлен), а потом `> output.txt` перенаправит только stdout.

Правильно:
```bash
command > output.txt 2>&1
# или
command &> output.txt
```

### Практические фишки и приёмы

#### 1. **Полное подавление вывода**
```bash
command > /dev/null 2>&1
# или короче (в Bash):
command &> /dev/null
```

> Полезно в cron-скриптах, чтобы не засорять почту.

#### 2. **Логирование + просмотр в реальном времени**
```bash
long_task > app.log 2>&1 &
tail -f app.log
```

> → Запускаете задачу в фоне и следите за логом.

#### 3. **Раздельное логирование stdout и stderr**
```bash
command > stdout.log 2> stderr.log
```

> → Удобно для отладки: где ошибка, а где просто данные?

#### 4. **Перенаправление stderr в конвейер**
```bash
# Только stdout идёт в grep
command | grep "error"

# Оба потока — в grep
command 2>&1 | grep "error"
```

#### 5. **Запись в файл и вывод на экран одновременно (`tee`)**
```bash
command 2>&1 | tee logfile.txt
```

>→ Всё видно в терминале **и** сохраняется в файл.

Для дописывания:
```bash
command 2>&1 | tee -a logfile.txt
```

#### 6. **Использование `/dev/stdout`, `/dev/stderr`**

```bash
# Пример: отправить лог в stdout, но ошибки — в stderr
echo "INFO: started" > /dev/stdout
echo "ERROR: failed" > /dev/stderr
```

#### 7. **Here-document и Here-string**

- **Here-doc** — многострочный ввод:
```bash
cat << EOF > config.txt
host = localhost
port = 8080
EOF
```

- **Here-string** — одноразовая строка:

``` bash
grep "root" <<< "root:x:0:0:root:/root:/bin/bash"
```

#### 8. **Обработка ошибок в скриптах**
```bash
if ! command > out.log 2>&1; then
    echo "Команда упала! Лог:"
    cat out.log
    exit 1
fi
```

#### 9. **Фон + логирование без мусора**
```bash
{
    echo "[$(date)] Starting backup..."
    rsync -av /data/ user@remote:/backup/
    echo "[$(date)] Done."
} > /var/log/backup.log 2>&1 &
```

> → Группировка команд в блок `{ }` позволяет логировать всё целиком.

| Ошибка                | Почемуплохо                        | Как исправить                  |
| --------------------- | ---------------------------------- | ------------------------------ |
| `command 2>&1 > file` | Ошибки уходят в терминал           | `command > file 2>&1`          |
| `echo error 2> log`   | `echo` пишет в stdout, не в stderr | `echo error >&2 2> log`        |
| `ls *.txt             | xargs rm`                          | Ломается при пробелах в именах |
### Профессиональный совет

> **Всегда логируйте stderr вместе с stdout в скриптах.**  
> Иначе вы пропустите критические ошибки, которые «исчезли в никуда».


Пример шаблона для надёжного скрипта:
```bash
#!/bin/bash
set -euo pipefail
exec > /var/log/myscript.log 2>&1

echo "[$(date)] Script started"
# ... ваш код ...
echo "[$(date)] Script finished"
```

> - `set -e` — завершить при первой ошибке
> - `set -u` — ругаться на неопределённые переменные
> - `set -o pipefail` — учитывать ошибки в конвейерах

### Итог

Перенаправление — не просто «записать в файл». Это **мощный механизм управления потоками данных**, который позволяет:

- отделять логику от вывода,
- автоматизировать отладку,
- строить надёжные скрипты,
- интегрировать команды в сложные пайплайны.

Освоив его — вы перестанете бояться длинных логов, таинственных ошибок и «пропавших» сообщений.
