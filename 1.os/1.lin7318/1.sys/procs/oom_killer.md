#  OOM Killer

---

## 1. Что такое OOM Killer и зачем он нужен?

Когда **вся доступная память исчерпана** (включая swap), а ядро не может выделить страницу памяти новому/текущему процессу — наступает *out-of-memory condition*.

Без OOM Killer:
- Все процессы бы блокировались в `D` (uninterruptible sleep) при попытке `malloc()`,
- Система — **полный локдаун**: даже `Ctrl+Alt+F2` не работает.

С OOM Killer:
- Ядро **выбирает "жертву"** — процесс с наивысшим `oom_score`,
- Убивает его сигналом `SIGKILL`,
- Освобождает его память → система продолжает жить.

>  Это **последняя линия обороны**. Лучше один `firefox` погиб, чем весь сервер уйдёт в hard lock.

---

## 2. Как OOM Killer выбирает жертву?

Алгоритм (упрощённо):

1. **Собирает все процессы**, у которых `oom_score_adj ≥ -1000` (см. ниже).
2. Вычисляет для каждого **`oom_score`** — «цену убийства».
3. Выбирает процесс с **наибольшим `oom_score`**.

### `oom_score` = базовый скор + `oom_score_adj`

| Фактор | Влияние на `oom_score` |
|--------|------------------------|
| **Потребление памяти** | Прямо пропорционально: чем больше RSS + swap + tmpfs, тем выше скор |
| **Приоритет (`oom_score_adj`)** | Пользовательский штраф/бонус (от `-1000` до `+1000`) |
| **root-процессы** | Слегка понижаются (но не защищены!) |
| **Процессы с `CAP_SYS_ADMIN`** | Немного понижаются |
| **Процессы с `memcg`-лимитами** | Учёт ведётся **внутри cgroup** |

>  Формула (ядро ≥ 5.10):  
> ```
> oom_score = (процент памяти, занимаемой процессом) + oom_score_adj
> ```
> Максимум — 1000 (100% памяти + adj=0 → 1000).  
> `oom_score_adj = -1000` → `oom_score = 0` → **процесс никогда не будет убит**.
