# work с atop

### 1.История ~~России~~
Система пишет логи в `/var/log/atop/`.
* `atop -r` — открыть лог за текущие сутки.
* `atop -r /var/log/atop/atop_YYYYMMDD` — открыть конкретный день.

### 2.Навигатор
* **t** — следующий интервал (вперед).
* **T** (Shift + t) — предыдущий интервал (назад).
* **b** — прыгнуть на время (формат HH:MM).

### 3.Отобрази
* **m** — Сортировка по памяти (Memory). Смотрим колонки VSIZE/RSIZE.
* **c** — Сортировка по процессору (CPU).
* **d** — Сортировка по дискам (Disk).
* **v** — Подробный вид (показывает статус процесса S, R, D).
* **y** — Показать потоки (Threads). Важно для БД.

### 4.В цвет
* **Красный** — Ресурс критически перегружен (90%+).
* **Голубой** — Ресурс близок к пределу (70-80%).


# Чек.лст: завис серв?

### 1.Секция SYSTEM (Верхняя панель)
* **SWP | free ≈ 0** — Память кончилась, система "свопится" на диск. Это тормозит всё.
* **PAG | swin/swout** — Если цифры высокие, значит идет активная перекачка RAM <-> Disk.
* **DSK | busy 90-100%** — Диск захлебнулся. Смотри колонку `avio` (норма < 10ms).
* **PSI | memfull / iofull** — Если > 50%, сервер "стоит в очереди" за ресурсом и не отвечает.

### 2.Статусы процессов (Колонка S)
* **Статус "D" (Uninterruptible Sleep)** — Процесс ждет ответа от диска.
  ! КРИТИЧНО: Если в "D" висят системные службы (sshd, syslog, etcd), сервер будет недоступен.
* **Статус "R" (Running)** — Процесс ест CPU.
* **Процесс "kswapd0"** — Если он в топе CPU, значит ядро судорожно пытается спасти память.

### 3.Типовой сценарий "Зависания":
1. RAM забита (Acronis/Kaspersky/DB).
2. Начинается Swap (рост нагрузки на диск).
3. Диск уходит в 100% busy.
4. Системные процессы (SSH, логи) встают в статус "D".
5. Результат: Сервер "колом", помогает только Hard Reset.
