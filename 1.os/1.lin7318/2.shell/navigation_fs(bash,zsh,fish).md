> [!SUMMARY] Цель 
> Освоить механизмы перемещения по файловой системе в различных оболочках: базовые команды `cd`, переменные окружения, функции навигации и стек каталогов с учётом различий между Bash, Zsh и Fish.

> [!OBJECTIVES] Задачи
> 
> -  Понять механику разрешения путей в `cd`: относительные, абсолютные, `~`, `$HOME`, `CDPATH`
> -  Освоить автодополнение (tab completion) и его настройку в Bash/Zsh/Fish.
> -  Создать функции быстрой навигации с поддержкой нескольких оболочек.
> -  Использовать стек каталогов (`pushd`/`popd`/`dirs`) для переключения между контекстами
> -  Настроить `CDPATH` для глобального поиска подкаталогов.


## Базовое поведение `cd`: механика разрешения путей

### Алгоритм обработки аргументов `cd`
```shell
1. Если аргумент не передан:
   → перейти в $HOME (или ~)
2. Если аргумент начинается с /:
   → абсолютный путь, перейти напрямую
3. Если аргумент начинается с ~:
   → ~ = $HOME текущего пользователя
   → ~user = домашний каталог указанного пользователя (из /etc/passwd)
4. Если аргумент содержит / (но не начинается с / или ~):
   → относительный путь от текущего каталога
5. Если аргумент не содержит /:
   → поиск по CDPATH (если задана)
   → если не найдено → ошибка
```
### Примеры и ожидаемое поведение
```shell
# Переход в домашний каталог (без аргумента)
$ cd
$ pwd
/home/user

# Использование тильды
$ cd ~/projects/ansible
$ echo ~other_user
/home/other_user  # Работает, если пользователь существует в системе

# Использование $HOME (явная переменная)
$ cd $HOME/.config
$ echo $HOME
/home/user

# Относительный путь
$ cd ../../etc  # Поднимается на 2 уровня, затем в etc

# Абсолютный путь
$ cd /var/log/journal
```
> [!NOTE] Разница между `~` и `$HOME`
> 
> - `~` обрабатывается оболочкой на этапе **парсинга команды** (до выполнения)
> - `$HOME` — переменная окружения, подставляется при **выполнении**
> - В большинстве сценариев эквивалентны, но `~` работает даже если `$HOME` не задана
> - `~user` работает только если пользователь существует в `/etc/passwd`

## Автодополнение (Tab Completion): сравнение оболочек

### Механика работы
|Оболочка|Механизм|Конфигурация|Особенности|
|---|---|---|---|
|**Bash**|`compgen` + `complete` builtin|`~/.bashrc`, `/etc/bash_completion`|Требует явной настройки для кастомных команд|
|**Zsh**|`compsys` (модуль)|`~/.zshrc`, `autoload -Uz compinit`|Более гибкая, поддерживает контекстные подсказки|
|**Fish**|Встроенная система автодополнения|`~/.config/fish/completions/`|Автодополнение работает «из коробки», на основе истории и мануалов|

### Базовое использование (одинаково во всех оболочках)
```shell
# Ввод начала пути + Tab
$ cd /usr/lo<Tab>      # → /usr/local/
$ cd ~/pro<Tab><Tab>   # Показать все варианты: projects/ profiles/ ...

# Для команд с файлами
$ cat /etc/ssh/sshd_<Tab>  # → sshd_config sshd_config.d/
```
### Настройка автодополнения для кастомной функции

#### Bash: `complete -W`
Флаг `-W` (сокращение от «wordlist» — список слов) принимает строку с вариантами, разделенными пробелами. Когда вы начинаете вводить аргумент для команды и нажимаете `Tab`, Bash ищет совпадения в этом списке.

**Базовый синтаксис:**
```shell
# В ~/.bashrc
complete -W "слово1 слово2 слово3" имя_команды
# Пример
# В ~/.bashrc
complete -W "work pics app" qcd
```
#### Zsh: `compctl` (legacy) или `compdef` (рекомендуется)
```shell
# В ~/.zshrc (legacy, работает в примере пользователя)
compctl -k "(work pics app)" qcd

# Современный способ (compsys)
# ~/.zshrc
autoload -Uz compinit && compinit
_qcd_completion() {
  _arguments '*: :((work pics app))'
}
compdef _qcd_completion qcd
```
#### Fish: `complete -c`
Флаг `-c` (или `--command`) указывает, для какой команды создается правило. Можно добавить описание аргументов`-d "Описание"` и пользователь увидит подсказку прямо в меню выбора.
```shell
# В ~/.config/fish/completions/qcd.fish
complete -c qcd -a "work pics app" -d "Quick directory change"
```
>[!NOTE] Fish: автодополнение без настройки Fish автоматически предлагает аргументы на основе:
>
>- Истории команд
>- Файлов в текущем каталоге
>- Документации (мануалов) 
> Явная настройка нужна только для кастомной валидации аргументов.

## Быстрая навигация: алиасы, переменные, функции

### Сравнение подходов
|Метод|Синтаксис|Область видимости|Гибкость|Недостатки|
|---|---|---|---|---|
|**Алиас**|`alias work="cd ~/proj"`|Текущая сессия + конфиг|Низкая|Не принимает аргументы, конфликт имён|
|**Переменная**|`w=~/proj; cd $w`|Текущая сессия|Средняя|Требует явного `cd`, нет автодополнения|
|**Функция**|`qcd() { cd ...; }`|Текущая сессия + конфиг|Высокая|Сложнее в отладке, требует синтаксиса оболочки|

### Реализация функции `qcd` (кросс-оболочечная)

#### Bash / Zsh (совместимый синтаксис)
```shell
# В ~/.bashrc или ~/.zshrc
qcd () {
  case "$1" in
    work)     cd "$HOME/work/proj./dev" ;;
    app)  cd "$HOME/my/app"        ;;
    pics)  cd "$HOME/unknown/pics"   ;;
    "") 
      echo "Usage: qcd {work|app|pics}"
      return 1
      ;;
    *)
      echo "qcd: unknown key '$1'" >&2
      return 1
      ;;
  esac
  # Вывод подтверждает переход (опционально)
  pwd
}
```
#### Fish (синтаксис функций отличается)
```shell
# В ~/.config/fish/functions/qcd.fish
function qcd
    set -l key $argv[1]
    switch "$key"
        case work
            cd $HOME/work/proj./dev
        case app
            cd $HOME/my/app
        case pics
            cd $HOME/unknown/pics
        case ""
            echo "Usage: qcd {work|app|pics}"
            return 1
        case '*'
            echo "qcd: unknown key '$key'" >&2
            return 1
    end
    # Подтверждение перехода
    pwd
end
```
### Проверка работы функции
```shell
# Тестирование валидных ключей
$ qcd work
/home/user/work/proj./dev

# Тестирование ошибочного ключа
$ qcd unknown
qcd: unknown key 'unknown'
$ echo $?  # Код возврата: 1 (ошибка)

# Проверка автодополнения (после настройки)
$ qcd <Tab><Tab>
app  pics  work
```
>[!WARNING] Конфликт имён 
>Перед созданием алиаса/функции проверьте, не существует ли команда с таким именем:
>```shell
>type qcd      # Bash/Zsh: покажет, если уже определено
functions qcd # Fish: проверка наличия функции
which qcd     # Поиск исполняемого файла в PATH
>```

## CDPATH: глобальный поиск подкаталогов

### Механика работы
```
При выполнении: cd <dirname>
1. Проверить ./<dirname> (относительный путь от текущего каталога)
2. Если не найдено и CDPATH задана:
   → перебрать каталоги из CDPATH по порядку
   → для каждого: проверить <cdpath_entry>/<dirname>
3. Если найдено → выполнить переход и вывести абсолютный путь
4. Если не найдено → ошибка "No such file or directory"
```
### Настройка CDPATH

#### Bash / Zsh
```shell
# В ~/.bashrc или ~/.zshrc
export CDPATH=".:$HOME:$HOME/app:$HOME/projects/ansible:/usr/local"
```
#### Fish
```shell
# В ~/.config/fish/config.fish
set -gx CDPATH . $HOME $HOME/app $HOME/projects/ansible /usr/local
```
### Пример использования
```shell
# Предположим, структура каталогов:
# $HOME/projects/ansible/playbooks
# $HOME/Family/Memories/Photos

$ pwd
/var/log
$ cd Photos        # Ищет: ./Photos → $HOME/Photos → $HOME/unknown/pics/Photos → $HOME/Family/Memories/Photos ✓
/home/user/Family/Memories/Photos  # Выводит абсолютный путь при успешном переходе через CDPATH

$ cd playbooks    # Переходит в $HOME/projects/ansible/playbooks
/home/user/projects/ansible/playbooks
```

>[!NOTE] Вывод абсолютного пути 
При успешном переходе через `CDPATH` (не из текущего каталога) оболочка выводит абсолютный путь. Это поведение можно отключить:
>```bash
># Bash/Zsh: подавить вывод
cd Photos >/dev/null
>
># Fish: вывод подавляется по умолчанию, если не задано иное
>```

### Ограничения CDPATH

- Не работает с абсолютными путями (`cd /etc` игнорирует CDPATH)
- Не работает с путями, начинающимися с `./`, `../`, `~`
- При совпадении в нескольких местах используется первое вхождение в списке
- Может создавать неочевидное поведение: `cd config` может перейти не туда, где ожидается.

## Возврат в предыдущий каталог: `cd -` и `OLDPWD`

### Механика

- При каждом успешном `cd` оболочка сохраняет предыдущий путь в переменную `OLDPWD`
- `cd -` выполняет `cd "$OLDPWD"` и выводит новый текущий путь
- `OLDPWD` доступна для использования в скриптах и командах
### Пример
```shell
$ pwd
/home/user/work
$ cd /etc/nginx
$ pwd
/etc/nginx
$ cd -              # Возврат в предыдущий каталог
/home/user/work
$ echo $OLDPWD
/etc/nginx          # Теперь OLDPWD = предыдущий до текущего
```
### Использование `OLDPWD` в командах
```shell
# Копирование файла из предыдущего каталога
$ cp "$OLDPWD"/config.txt .

# Сравнение содержимого каталогов
$ diff <(ls) <(ls "$OLDPWD")

# Быстрый переход + выполнение команды
$ cd /var/log && some_command && cd -
```
>[!WARNING] Ограничение `cd -` 
>Запоминается только **один** предыдущий каталог. Для переключения между несколькими используйте стек каталогов (`pushd`/`popd`).

## Стек каталогов: `pushd`, `popd`, `dirs`

### Архитектура стека

- Стек — это упорядоченный список каталогов, поддерживаемый оболочкой
- Вершина стека (индекс 0) = текущий рабочий каталог
- Каждая оболочка имеет **независимый** стек (не разделяется между сессиями)
- Стек сохраняется в переменной `DIRSTACK` (Bash/Zsh) или через `dirs` (Fish)

### Команды управления стеком
|                       |                                                                                        |                                        |
| --------------------- | -------------------------------------------------------------------------------------- | -------------------------------------- |
| `pushd <dir>`         | 1. Добавляет `<dir>` в вершину стека  <br>2. Выполняет `cd <dir>`  <br>3. Выводит стек | Новый стек (пробел-разделённый список) |
| `pushd` (без арг.)    | Меняет местами топ-2 элемента стека + `cd` в новый топ                                 | Обновлённый стек                       |
| `pushd +N`            | Ротирует стек: перемещает элемент на позиции N в топ + `cd` в него                     | Обновлённый стек                       |
| `pushd -N`            | Ротирует в обратном направлении                                                        | Обновлённый стек                       |
| `popd`                | 1. Удаляет вершину стека  <br>2. Выполняет `cd` в новую вершину  <br>3. Выводит стек   | Новый стек                             |
| `popd +N` / `popd -N` | Удаляет элемент на позиции N (не обязательно вершину)                                  | Обновлённый стек                       |
| `dirs`                | Выводит стек (пробел-разделённый)                                                      | Список каталогов                       |
| `dirs -v`             | Выводит стек с нумерацией (индексы 0..N)                                               | Нумерованный список                    |
| `dirs -p`             | Выводит стек по одному каталогу на строку                                              | Вертикальный список                    |
### Пример работы со стеком (Bash/Zsh)
```shell
$ pwd
/home/user
$ dirs -v
0  /home/user

$ pushd /var/log
/var/log /home/user
$ dirs -v
0  /var/log
1  /home/user

$ pushd ~/.config
~/.config /var/log /home/user
$ pushd /opt
/opt ~/.config /var/log /home/user

$ dirs -v
0  /opt
1  ~/.config
2  /var/log
3  /home/user

$ popd          # Удаляет /opt, переходит в ~/.config
~/.config /var/log /home/user
$ popd +1       # Удаляет элемент на позиции 1 (/var/log), остаётся в ~/.config
~/.config /home/user
```

### Fish: особенности реализации
```shell
# Синтаксис pushd/popd/dirs в Fish совместим с Bash, но:
# - Вывод стека может отличаться форматированием
# - Переменная $DIRSTACK доступна, но обновляется асинхронно

$ dirs -v
0  /home/user
$ pushd /var/log
$ dirs -v
0  /var/log
1  /home/user
```
### рактические сценарии использования стека

#### Сценарий 1: Переключение между проектом и логами
```shell
# Начало работы
$ pushd ~/projects/webapp    # Стек: [webapp, ~]
$ pushd /var/log/nginx       # Стек: [nginx, webapp, ~]

# Просмотр логов, затем возврат к коду
$ tail -f access.log
$ popd                       # Возврат в webapp, стек: [webapp, ~]

# Быстрая проверка конфигурации
$ pushd /etc/nginx           # Стек: [nginx, webapp, ~]
$ cat nginx.conf
$ popd                       # Возврат в webapp
```
#### Сценарий 2: Ротация стека для доступа к середине
```shell
$ dirs -v
0  /current
1  /first
2  /second
3  /third

# Перейти в /second (индекс 2) без popd
$ pushd +2
/second /third /current /first

# Вернуться в исходный порядок
$ pushd +1
/current /first /second /third
```
>[!NOTE] Сохранение стека между сессиями 
По умолчанию стек не сохраняется после закрытия оболочки. Для персистентности:
>```shell
># Bash: в ~/.bashrc
export SAVEHIST=1000
># + плагин для сохранения DIRSTACK (требует кастомной реализации)
>
># Fish: автоматически сохраняет историю команд, но не стек каталогов
># Для сохранения: записать dirs -p в файл при выходе, загрузить при старте
>```

## Выводы

**Механика `cd`**:

- Обработка путей следует строгому порядку: абсолютные → `~` → относительные → `CDPATH`
- `~user` требует существования пользователя в системе; `$HOME` — всегда доступна
- `cd -` использует `OLDPWD`, которая обновляется после каждого успешного перехода

 **Автодополнение**:

- Базовое завершение путей работает одинаково во всех оболочках
- Кастомное завершение требует настройки: `complete` (Bash), `compctl`/`compdef` (Zsh), `complete -c` (Fish)
- Fish предоставляет наиболее «умное» автодополнение из коробки

 **Функции навигации**:

- Алиасы просты, но негибки; переменные требуют явного `cd`; функции — оптимальный баланс
- Кросс-оболочечные функции требуют учёта синтаксических различий (`case`/`switch`, `function`/`func`)
- Обработка ошибок (`return 1`, вывод в `stderr`) улучшает юзабилити

 **`CDPATH`**:

- Позволяет глобальный поиск подкаталогов, но может создавать неочевидное поведение
- Вывод абсолютного пути при переходе через `CDPATH` — индикатор «непрямого» перехода
- Не работает с путями, содержащими `/`, `~`, `.`

 **Стек каталогов**:

- `pushd`/`popd`/`dirs` обеспечивают навигацию между несколькими контекстами
- Индексация стека начинается с 0 (вершина); ротация через `pushd +N`/`-N`
- Стек изолирован на уровне сессии; для персистентности требуется кастомная реализация
