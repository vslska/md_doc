

## 1.Архитектура backend-стека

###  Сетевой уровень

| Компонент                | Описание                                                                                     |
| ------------------------ | -------------------------------------------------------------------------------------------- |
| **Базовые протоколы**    | `TCP/IP`, `DNS` — фундамент сетевого взаимодействия                                          |
| **Прикладные протоколы** | `HTTP/HTTPS`, `FTP`, `UDP` — для различных видов передачи данных (веб, файлы, потоки и т.д.) |
### Веб-стек

| Уровень             | Технология                  | Комментарий                                                                                                                                                                               |
| ------------------- | --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Фреймворк**       | `Django` или `Flask`        | Django — full-stack с ORM, админкой, аутентификацией «из коробки»; Flask — минималистичный, гибкий                                                                                        |
| **WSGI-сервер**     | `Gunicorn`                  | Стандарт WSGI — интерфейс между Python-приложением и веб-сервером. Gunicorn запускает _воркеров_ (процессы/потоки), обрабатывающих запросы. <br> 📄 [gunicorn.org](https://gunicorn.org/) |
| **Обратный прокси** | `Nginx`                     | Принимает входящие HTTP-запросы, балансирует, кэширует, передаёт в Gunicorn (обычно на `127.0.0.1:8000`)                                                                                  |
| **Развёртывание**   | `Docker` + `docker-compose` | Удобно для локальной разработки и изоляции сервисов. Поддержка _Docker-сетей_ для внутреннего взаимодействия (например, `app ↔ db ↔ redis`)                                               |
###  Хранилища и обработка

| Компонент         | Технология   | Примечание                                                                                                                                                                                          |
| ----------------- | ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **База данных**   | `PostgreSQL` | Надёжная реляционная СУБД. В Django используется через **встроенный ORM**; в Flask — через `SQLAlchemy`/`Peewee`                                                                                    |
| **Кэш / брокер**  | `Redis`      | Используется как: <br> • кэш (ускорение запросов) <br> • брокер сообщений для Celery <br> • временное хранилище сессий и состояний                                                                  |
| **Очередь задач** | `Celery`     | Распределённая очередь: асинхронная обработка тяжёлых/долгих задач (email, PDF, обработка данных). Поддерживает расписание и мониторинг. <br> 📄 [Celery docs](https://docs.celeryq.dev/en/stable/) |

### Язык и инструменты
| Категория                    | Инструмент                                                                            |
| ---------------------------- | ------------------------------------------------------------------------------------- |
| **Язык**                     | `Python`                                                                              |
| **Управление зависимостями** | `pip`, `poetry`, `venv` / `virtualenv`                                                |
| **Контроль версий**          | `Git` (GitHub / GitLab / Codeberg)                                                    |
| **CI/CD**                    | GitLab CI / GitHub Actions / Argo CD — для автоматизации тестирования, сборки, деплоя |

##  2.Принципы backend-разработки

Backend — не набор технологий, а **дисциплина проектирования надёжных, масштабируемых и поддерживаемых систем**. Основные принципы:

| Принцип | Описание | Практическое проявление |
|--------|----------|--------------------------|
| **Идемпотентность** | Повторный вызов операции не меняет результат по сравнению с первым вызовом. | `PUT /users/123` — обновление данных идемпотентно; `POST /orders` — не идемпотентно (создаёт новую сущность) |
| **Statelessness (безсостоятельность)** | Сервер не хранит сессионное состояние между запросами. Вся нужная информация передаётся в запросе (токен, параметры). | RESTful API, JWT-авторизация |
| **Разделение ответственности (SoC)** | Каждый компонент решает одну задачу: DB — хранение, Cache — ускорение, Queue — отложенная обработка. | Redis не заменяет PostgreSQL; Celery не заменяет Nginx |
| **Fault Tolerance (отказоустойчивость)** | Система продолжает работать при частичном отказе (падение воркера, сеть, БД). | Health-checks, retry-логика, dead-letter queues, fallback-кэш |
| **Observability (наблюдаемость)** | Возможность понять *почему* система ведёт себя так, а не иначе — через логи, метрики, трейсы. | Prometheus + Grafana (метрики), ELK / Loki (логи), Jaeger / OpenTelemetry (трейсы) |
| **Security by Design** | Безопасность закладывается на этапе проектирования: минимальные привилегии, валидация, шифрование, аудит. | Параметризованные запросы (защита от SQLi), rate limiting, CSP, HTTPS everywhere |
| **API-first** | Сначала проектируется контракт интерфейса (OpenAPI/Swagger), потом — реализация. Это упрощает интеграции и тестирование. | `openapi.yaml` → генерация клиентов, автодокументация, контрактное тестирование |
| **Loose Coupling (слабая связанность)** | Компоненты взаимодействуют через чёткие интерфейсы (HTTP, очереди), а не напрямую. | Celery worker не зависит от Django — лишь от сигнатуры задачи и сериализуемости аргументов |
| **Graceful Degradation** | При пиковой нагрузке или частичной недоступности система сохраняет частичную функциональность. | Отключение тяжёлых фич (рекомендаций), fallback на кэш при падении БД |

> 💡 Эти принципы реализуются *через* технологии — но они выше их. Например:  
> — Celery позволяет применить *асинхронность* и *отложенную обработку* → поддерживает *fault tolerance* и *scalability*.  
> — Nginx как reverse proxy → обеспечивает *безопасность*, *балансировку*, *rate limiting*.
