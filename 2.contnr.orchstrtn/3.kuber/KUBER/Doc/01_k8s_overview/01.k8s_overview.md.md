# Kubernetes (K8s) — Обзор

> *📌* K8s — это платформа для автоматического управления контейнерами в production. Вы описываете *желаемое состояние*, система приводит к нему *фактическое*.

---

## Что такое Kubernetes

| Параметр | Значение |
|----------|----------|
| **Тип** | Open-source платформа для оркестрации контейнеров |
| **Название** | От греч. «κυβερνήτης» — кормчий, пилот |
| **Аббревиатура** | `K8s` = K + 8 букв + s |
| **Появление** | Открыт Google в 2014, основа — опыт систем Borg/Omega |
| **Философия** | Декларативная конфигурация + автоматизация + расширяемость |

**Ключевая идея**: вы не управляете каждым контейнером вручную. Вы задаёте правила («нужно 3 реплики приложения Х, с 512 МБ ОЗУ, доступных по порту 80»), а Kubernetes постоянно работает, чтобы реальность соответствовала этим правилам.

---

## Зачем нужен Kubernetes

Контейнеры решают проблему «работает на моей машине», но в production возникают **новые задачи:
❓ Что если контейнер упал? 
❓ Как распределить трафик между 10 одинаковыми экземплярами? 
❓ Как обновить приложение без простоя? 
❓ Как эффективно разместить 100 контейнеров на 10 серверах?**

**Kubernetes отвечает на эти вопросы**, предоставляя:
### Основные возможности

| Возможность | Что это даёт |
|-------------|-------------|
| **🔍 Обнаружение сервисов и балансировка** | Контейнеры получают DNS-имена или IP. K8s автоматически распределяет входящий трафик, если нагрузка растёт. |
| **💾 Оркестрация хранилищ** | Автоматическое подключение хранилищ: local, NFS, облачные (EBS, GCE PD, Azure Disk и др.). |
| **🚀 Развёртывание и откат** | Описываете желаемое состояние → K8s плавно меняет фактическое. Поддерживает rolling updates, canary, rollback при ошибках. |
| **📦 Автоматическая упаковка (Bin Packing)** | Вы указываете требования к CPU/RAM для каждого контейнера. K8s оптимально размещает их на нодах, экономя ресурсы. |
| **🩹 Самоисцеление (Self-healing)** | Перезапускает упавшие контейнеры, заменяет недоступные ноды, не направляет трафик на неподготовленные поды (через `readinessProbe`). |
| **🔐 Управление секретами и конфигами** | Хранение паролей, токенов, SSH-ключей в объектах `Secret` и `ConfigMap`. Обновление без пересборки образов. |
| **📊 Поддержка пакетных задач** | Запуск `Job` и `CronJob` для фоновых задач, обработки данных, CI/CD-пайплайнов. |
| **📈 Горизонтальное масштабирование** | Ручное (`kubectl scale`) или автоматическое (HPA) изменение количества реплик на основе CPU, памяти или кастомных метрик. |
| **🌐 Dual-Stack IPv4/IPv6** | Нативная поддержка выделения обоих типов адресов для подов и сервисов. |
| **🔌 Расширяемость** | Добавление новых функций через CRD (Custom Resource Definitions), операторы, вебхуки — без изменения ядра K8s. |

## Чем Kubernetes НЕ является

Важно понимать границы платформы, чтобы не ждать от неё лишнего:

| ❌ Не является | ✅ Что вместо этого |
|---------------|---------------------|
| **Традиционная PaaS** | Не навязывает стек технологий. Даёт примитивы, вы выбираете инструменты (логирование, мониторинг, CI/CD). |
| **Система сборки кода** | Не компилирует и не собирает ваше приложение. Интегрируется с внешними CI/CD (GitLab CI, GitHub Actions, ArgoCD). |
| **Поставщик прикладных сервисов** | Не включает встроенные БД, брокеры, кэши. Но вы можете запустить их *внутри* кластера как обычные контейнеры. |
| **Монолитный оркестратор** | Не выполняет линейные скрипты «сделай A → потом B». Работает через **Reconciliation Loop**: контроллеры постоянно сравнивают текущее и желаемое состояние и устраняют расхождения. |
| **Решение для управления железом** | Не занимается настройкой серверов, прошивками, сетевым оборудованием. Работает поверх готовой инфраструктуры. |

## Эволюция деплоя: почему появились контейнеры и K8s

### 1️. Эпоха физических серверов (Bare Metal)
`Приложения → напрямую на железо`
- **Проблема**: конфликты ресурсов, сложно масштабировать, низкая утилизация, дорого.
- **Пример**: одно «прожорливое» приложение может замедлить все остальные на том же сервере.
### 2️. Эпоха виртуализации (VMs)
`Железо → Гипервизор → ВМ с полной ОС → Приложение`
- **Плюсы**: изоляция, лучшее использование ресурсов, переносимость.
- **Минусы**: каждая ВМ — полноценная ОС → большой оверхед, медленный старт, дублирование библиотек.

### 3️. Эпоха контейнеров
`Железо → ОС → Container Runtime → Контейнеры (shared kernel) → Приложение`
- **Плюсы**:
  - Легковесные: общий ядро ОС, быстрый запуск (~секунды)
  - Переносимые: работает на ноутбуке, в облаке, на любом дистрибутиве
  - Идеальны для микросервисов: слабая связность, независимое масштабирование
  - Поддержка CI/CD: неизменяемые образы, быстрый откат
- **Новая проблема**: как управлять сотнями контейнеров в продакшене? → **Kubernetes**

### 4️. Эпоха оркестрации (Kubernetes)
`Кластер нод → Kubernetes → Автоматическое управление тысячами контейнеров`
- **Результат**: вы фокусируетесь на логике приложения, а инфраструктура адаптируется под нагрузку, сбои и обновления.

## Ключевые преимущества контейнеров (контекст для K8s)

| Преимущество                  | Практическая польза                                            |
| ----------------------------- | -------------------------------------------------------------- |
| 🔄 Гибкость развёртывания     | Образ собирается один раз — запускается где угодно             |
| ⚡ Непрерывная доставка        | Частые релизы, быстрый откат благодаря неизменяемости образов  |
| 👥 Разделение ответственности | Разработчик готовит образ, оператор — разворачивает в кластере |
| 👁️ Наблюдаемость             | Метрики и логи на уровне приложения + инфраструктуры           |
| 🌍 Единообразие сред          | «Работает на ноутбуке» = «работает в продакшене»               |
| 📦 Портативность              | Один образ для Ubuntu, RHEL, AWS, GCP, on-premise              |
| 🧩 Микросервисная архитектура | Приложение = набор независимых, масштабируемых компонентов     |
| 📊 Эффективность ресурсов     | Высокая плотность размещения, предсказуемая производительность |
