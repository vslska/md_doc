## 📚 Теория: Механика постоянного хранилища

### 1. Танец трех объектов (PV → PVC → Pod)
Kubernetes разделяет предоставление и потребление хранилища, чтобы разработчикам не нужно было знать детали инфраструктуры (SSD, NFS, AWS EBS).

1. **Администратор** создает **PersistentVolume (PV)**. Это реальный кусок диска (в нашем случае — директория `/mnt/data` на ноде через `hostPath`). Статус: `Available`.
2. **Разработчик** создает **PersistentVolumeClaim (PVC)**. Это "запрос" на хранилище (например, "мне нужно 3Gi, ReadWriteOnce"). 
3. **Control Plane** находит PV, который удовлетворяет требованиям PVC (совпадают `storageClassName`, `accessModes`, и capacity >= requested). 
4. Происходит **Binding** (Связывание). Статус PV и PVC меняется на `Bound`.
5. **Pod** ссылается **только на PVC**. Для Pod'а PVC выглядит как обычный `volume`.

### 2. hostPath vs Production Storage
- **`hostPath`**: Используется *только* для обучения (Minikube) или специфичных системных демонов (например, доступ к `/var/log` ноды). Данные привязаны к конкретной физической ноде. Если Pod переедет на другую ноду, данные будут потеряны.
- **Production**: Используются сетевые хранилища (NFS, Ceph, EBS, GCE PD), которые могут быть подключены к любой ноде кластера.

### 3. Магия `subPath`
Если смонтировать том в `/etc/nginx`, он **полностью скроет** всё содержимое этой директории из образа Docker (включая дефолтный `nginx.conf`). 
**`subPath`** решает эту проблему: он монтирует не весь том, а конкретный файл или поддиректорию из него, оставляя остальное содержимое образа нетронутым.

## 🚀 Практика: Пошаговое выполнение

### Шаг 1: Подготовка хранилища на ноде (Admin Task)
Поскольку мы используем `hostPath`, нам нужно создать директорию на самой ноде.

1. Подключаемся к ноде Minikube:
   ```bash
   minikube ssh
   ```
2. Создаем директорию и файл с данными (выполняем внутри SSH-сессии):
   ```bash
   sudo mkdir -p /mnt/data
   sudo sh -c "echo 'Hello from Kubernetes storage' > /mnt/data/index.html"
   
   # Проверяем
   cat /mnt/data/index.html
   exit # Выходим из SSH обратно на свой компьютер
   ```

---

### Шаг 2: Создание PersistentVolume (Admin Task)
Создаем объект PV, который указывает на эту директорию.

```yaml
# pv-volume.yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: task-pv-volume
  labels:
    type: local
spec:
  storageClassName: manual       # Важно: класс должен совпасть с PVC
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce              # RWO: чтение/запись одной нодой
  hostPath:
    path: "/mnt/data"
```

Применяем и проверяем:
```bash
kubectl apply -f pv-volume.yaml
kubectl get pv task-pv-volume
```
**Ожидаемый результат:** Статус `Available`. Claim пустой, так как запроса еще нет.

---

### Шаг 3: Создание PersistentVolumeClaim (Dev Task)
Создаем запрос на хранилище.

```yaml
# pv-claim.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: task-pv-claim
spec:
  storageClassName: manual       # Должен совпадать с PV!
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 3Gi               # Запрашиваем меньше, чем есть в PV (это ОК)
```

Применяем и проверяем связывание:
```bash
kubectl apply -f pv-claim.yaml

# Проверяем PVC
kubectl get pvc task-pv-claim
# Проверяем PV снова
kubectl get pv task-pv-volume
```
**Ожидаемый результат:** Статус обоих объектов изменится на **`Bound`**. В поле `CLAIM` у PV появится `default/task-pv-claim`.

---

### Шаг 4: Создание Pod'а и проверка данных
Pod ссылается на PVC, а не на PV напрямую.

```yaml
# pv-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: task-pv-pod
spec:
  volumes:
    - name: task-pv-storage
      persistentVolumeClaim:
        claimName: task-pv-claim # Ссылка на PVC
  containers:
    - name: task-pv-container
      image: nginx
      ports:
        - containerPort: 80
      volumeMounts:
        - mountPath: "/usr/share/nginx/html" # Монтируем ВЕСЬ том сюда
          name: task-pv-storage
```

Применяем и проверяем:
```bash
kubectl apply -f pv-pod.yaml
kubectl get pod task-pv-pod
```

Заходим внутрь и проверяем, что данные с ноды (`/mnt/data/index.html`) доступны в контейнере:
```bash
kubectl exec -it task-pv-pod -- /bin/bash
# Внутри контейнера:
apt update && apt install curl -y
curl http://localhost/
# Должно вывести: Hello from Kubernetes storage
exit
```

---

### Шаг 5: Продвинутый уровень — использование `subPath`
Частая задача: нужно подкинуть только один конфиг-файл, не затирая весь каталог `/etc/nginx`.

1. Готовим файлы на ноде (через `minikube ssh`):
   ```bash
   sudo mkdir -p /mnt/data/html
   sudo mv /mnt/data/index.html /mnt/data/html/
   
   # Создаем кастомный nginx.conf прямо на ноде
   sudo sh -c "echo 'keepalive_timeout  60;' > /mnt/data/nginx.conf"
   exit
   ```

2. Создаем Pod с `subPath`:
```yaml
# pv-duplicate.yaml
apiVersion: v1
kind: Pod
metadata:
  name: test-subpath
spec:
  containers:
    - name: test
      image: nginx
      volumeMounts:
        - name: config
          mountPath: /usr/share/nginx/html
          subPath: html          # Монтируем только директорию html из тома
        - name: config
          mountPath: /etc/nginx/nginx.conf
          subPath: nginx.conf    # Монтируем только этот файл, остальной /etc/nginx остается от образа!
  volumes:
    - name: config
      persistentVolumeClaim:
        claimName: task-pv-claim
```

Применяем и проверяем:
```bash
kubectl apply -f pv-duplicate.yaml
kubectl exec -it test-subpath -- /bin/bash

# Проверяем, что файл доступен
curl http://localhost/

# Проверяем, что подменился только один параметр в конфиге, а остальной nginx.conf на месте
cat /etc/nginx/nginx.conf | grep keepalive_timeout
exit
```

## 🧪 Практические задания (для закрепления)

### Задание 1: Проверка персистентности (Persistency)
1. Удали Pod: `kubectl delete pod task-pv-pod`
2. Создай его заново: `kubectl apply -f pv-pod.yaml`
3. Зайди внутрь и сделай `curl http://localhost/`.
   - **Вопрос**: Данные сохранились? (Ответ: Да, потому что они лежат на ноде в `/mnt/data`, а не внутри эфемерного контейнера).

### Задание 2: Проверка прав доступа (GID Annotation)
Если приложение в контейнере запускается не от `root` (например, от пользователя с UID 1000), оно не сможет писать в том, созданный `root` на хосте.
1. Посмотри, как в теории решается эта проблема через аннотацию `pv.beta.kubernetes.io/gid: "1234"` в PV.
2. Kubelet автоматически применит этот GID к процессам в контейнере, обеспечивая доступ на запись.

## 🐛 Troubleshooting

| Симптом | Возможная причина | Решение |
|---------|-------------------|---------|
| PVC висит в статусе `Pending` | 1. Нет PV с подходящим `storageClassName`. 2. Запрошенный `capacity` больше, чем есть у Available PV. 3. Не совпадают `accessModes`. | Сравни `kubectl get pv` и `kubectl get pvc`. Проверь `kubectl describe pvc <name>` (раздел Events). |
| Pod висит в `ContainerCreating` с ошибкой `MountVolume.SetUp failed` | Директория на хосте (`/mnt/data`) не существует или у kubelet нет прав на чтение/запись в неё. | Зайди через `minikube ssh` и проверь права: `ls -la /mnt/data`. |
| Контейнер падает или не видит дефолтные файлы | Ты смонтировал том в `/etc/nginx` без `subPath`, полностью перекрыв содержимое образа. | Используй `subPath`, как показано в Шаге 5. |
