## 📚 Теория: Разбор механики

### 1. Volume vs Env Var: Почему они ведут себя по-разному?

| Способ подключения | Механика обновления в K8s | Реакция приложения |
|--------------------|---------------------------|---------------------|
| **Volume (Файл)** | Kubelet каждые ~1 минуту опрашивает API. При изменении ConfigMap он **атомарно** (через symlink) обновляет файлы в директории тома. | Приложение увидит изменения **только если** оно умеет перечитывать файлы "на лету" (hot-reload) или читает их при каждом запросе. |
| **Env Var** | Переменные окружения внедряются в процесс ядром Linux **строго в момент запуска** (execve). | Приложение **НИКОГДА** не увидит изменений. ОС не поддерживает динамическую смену Env Var для запущенного процесса. Требуется пересоздание Pod'а (`rollout restart`). |

### 2. Deep Dive: Паттерн Sidecar (Побочный контейнер)

**Проблема:** У нас есть основное приложение (например, Nginx), которое не умеет динамически перечитывать конфигурацию, или нам нужно генерировать конфиг сложным образом (например, скачивать его из Vault, шифровать или форматировать).

**Решение:** Добавить в тот же Pod второй контейнер (Sidecar), который будет заниматься *только* подготовкой конфигурации, а основное приложение будет её просто читать.

#### Архитектура взаимодействия:
```text
┌───────────────────────────────────────────────────────────────────────┐
│                            POD                                        │
│                                                                       │
│  ┌──────────────────────────────┐      ┌──────────────────────────┐   │
│  │   Main Container (Nginx)     │      │  Sidecar Container       │   │
│  │   (Отдает трафик)            │      │  (Генерирует конфиг)     │   │
│  │                              │      │                          │   │
│  │  Читает файл:                │<────>│  Пишет файл:             │   │
│  │  /usr/share/nginx/html/      │      │  /pod-data/index.html    │   │
│  └──────────────┬───────────────┘      └────────────┬─────────────┘   │
│                 │                                   │                 │
│                 └───────────────┬───────────────────┘                 │
│                                 ▼                                     │
│                      ┌─────────────────────┐                          │
│                      │   emptyDir Volume   │ (Живет, пока жив Pod)    │
│                      └──────────┬──────────┘                          │
│                                 │                                     │
│                      ┌──────────┴──────────┐                          │
│                      │   ConfigMap (цвет)  │ (Смонтирован как файл)   │
│                      └─────────────────────┘                          │
└───────────────────────────────────────────────────────────────────────┘
```

#### Эволюция реализации Sidecar в Kubernetes:
1. **Старый способ (Legacy):** Sidecar объявлялся в массиве `containers`. 
   - *Минус:* K8s запускал все контейнеры параллельно. Nginx мог стартовать *раньше*, чем Sidecar успеет создать файл, и упасть с ошибкой 404.
2. **Современный способ (Native Sidecars, K8s 1.28+):** Sidecar объявляется в массиве `initContainers`, но с добавлением **`restartPolicy: Always`**.
   - *Плюс 1:* Гарантированный порядок запуска. Sidecar стартует первым и создает файл.
   - *Плюс 2:* Основной контейнер стартует только после успешного завершения инициализации (или сразу, если sidecar уже работает).
   - *Плюс 3:* Если Sidecar упадет, он перезапустится **независимо** от основного контейнера (основной контейнер продолжит работать, используя последний известный конфиг).


### 3. Immutable (Неизменяемые) ConfigMap

- Помечаются флагом `immutable: true`.
- **Плюс**: Kubelet не тратит ресурсы на периодическую проверку их изменений (экономия CPU и нагрузки на API Server).
- **Минус**: Их нельзя редактировать (`kubectl edit` выдаст ошибку).
- **Workflow обновления**: Создать _новый_ ConfigMap с новым именем → Обновить Deployment, чтобы он ссылался на новый ConfigMap → Дождаться Rolling Update → Удалить старый ConfigMap.

## 🚀 Практика: Пошаговое выполнение

### Часть 1: Базовое обновление (Volume vs Env Var)

### Шаг 1: ConfigMap как Volume (Автоматическое обновление)

1. Создаем ConfigMap:
```bash
kubectl create configmap sport --from-literal=sport=football
```

Ниже приведён пример манифеста развертывания с использованием ConfigMap. `sport`установленный как [volume](https://kubernetes.io/docs/concepts/storage/volumes/) в единственный контейнер Pod.
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: configmap-volume
  labels:
    app.kubernetes.io/name: configmap-volume
spec:
  replicas: 3
  selector:
    matchLabels:
      app.kubernetes.io/name: configmap-volume
  template:
    metadata:
      labels:
        app.kubernetes.io/name: configmap-volume
    spec:
      containers:
        - name: alpine
          image: alpine:3
          command:
            - /bin/sh
            - -c
            - while true; do echo "$(date) My preferred sport is $(cat /etc/config/sport)";
              sleep 10; done;
          ports:
            - containerPort: 80
          volumeMounts:
            - name: config-volume
              mountPath: /etc/config
      volumes:
        - name: config-volume
          configMap:
            name: sport
```

2. Создаем Deployment, который монтирует этот ConfigMap как файл и каждые 10 секунд читает его:
```bash
kubectl apply -f https://k8s.io/examples/deployments/deployment-with-configmap-as-volume.yaml
```

3. Проверяем, что Pod'ы читают старое значение:
```bash
kubectl logs deployment/configmap-volume
# Ожидаемый вывод: "... My preferred sport is football"
```

4. **Обновляем ConfigMap**:
```bash
kubectl edit configmap sport
# Меняем значение 'football' на 'cricket' и сохраняем (:wq)
```

5. Следим за логами (используем `-f` или `--follow`):
```bash
kubectl logs deployment/configmap-volume -f
```
**Наблюдение**: Через 10-60 секунд (время синхронизации kubelet) вывод изменится на `cricket`. Приложение не перезапускалось, оно просто прочитало обновленный файл.

### Шаг 2: ConfigMap как Environment Variable (Требует перезапуска)

1. Создаем новый ConfigMap:
```bash
kubectl create configmap fruits --from-literal=fruits=apples
```

Ниже приведён пример манифеста развертывания с переменной среды, настроенной через ConfigMap. `fruits`.
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: configmap-env-var
  labels:
    app.kubernetes.io/name: configmap-env-var
spec:
  replicas: 3
  selector:
    matchLabels:
      app.kubernetes.io/name: configmap-env-var
  template:
    metadata:
      labels:
        app.kubernetes.io/name: configmap-env-var
    spec:
      containers:
        - name: alpine
          image: alpine:3
          env:
            - name: FRUITS
              valueFrom:
                configMapKeyRef:
                  key: fruits
                  name: fruits
          command:
            - /bin/sh
            - -c
            - while true; do echo "$(date) The basket is full of $FRUITS";
                sleep 10; done;
          ports:
            - containerPort: 80
```


2. Создаем Deployment, использующий его как Env Var:
```bash
kubectl apply -f https://k8s.io/examples/deployments/deployment-with-configmap-as-envvar.yaml
```

3. Проверяем логи:
```bash
kubectl logs deployment/configmap-env-var
# Ожидаемый вывод: "... The basket is full of apples"
```

4. **Обновляем ConfigMap**:
```bash
kubectl edit configmap fruits
# Меняем 'apples' на 'mangoes' и сохраняем
```

5. Следим за логами:
```bash
kubectl logs deployment/configmap-env-var -f
```
**Наблюдение**: Вывод **НЕ ИЗМЕНИТСЯ**. Он продолжит показывать `apples`, потому что процесс внутри контейнера был запущен со старым значением переменной.

6. **Применяем изменения (Rolling Restart)**:
```bash
# Эта команда обновляет annotation в Deployment, что триггерит Rolling Update
kubectl rollout restart deployment configmap-env-var

# Ждем завершения
kubectl rollout status deployment configmap-env-var
```

7. Проверяем логи новых Pod'ов:
```bash
kubectl logs deployment/configmap-env-var
# Теперь вывод: "... The basket is full of mangoes"
```

### Часть 2: Sidecar (Нативный подход)

В современных K8s (1.28+) сайдкары реализуются через `initContainers` с `restartPolicy: Always`. Это гарантирует, что сайдкар запустится _до_ основного контейнера и будет работать параллельно с ним, разделяя том.

#### Шаг 2.1: Развертывание архитектуры

1. Создаем ConfigMap:

```bash
kubectl create configmap color --from-literal=color=red
```

Выглядит след. образом:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: configmap-two-containers
  labels:
    app.kubernetes.io/name: configmap-two-containers
spec:
  replicas: 3
  selector:
    matchLabels:
      app.kubernetes.io/name: configmap-two-containers
  template:
    metadata:
      labels:
        app.kubernetes.io/name: configmap-two-containers
    spec:
      volumes:
        - name: shared-data
          emptyDir: {}
        - name: config-volume
          configMap:
            name: color
      containers:
        - name: nginx
          image: nginx
          volumeMounts:
            - name: shared-data
              mountPath: /usr/share/nginx/html
        - name: alpine
          image: alpine:3
          volumeMounts:
            - name: shared-data
              mountPath: /pod-data
            - name: config-volume
              mountPath: /etc/config
          command:
            - /bin/sh
            - -c
            - while true; do echo "$(date) My preferred color is $(cat /etc/config/color)" > /pod-data/index.html;
              sleep 10; done;

```

_Разбор YAML:_

- `emptyDir` том `shared-data` монтируется и в `nginx`, и в `alpine`.
- `alpine` (sidecar) читает ConfigMap и каждые 10 сек перезаписывает `/pod-data/index.html`.
- `nginx` (main) просто отдает этот файл по HTTP.

2. Применяем Deployment с Sidecar'ом:
```bash
kubectl apply -f https://k8s.io/examples/deployments/deployment-with-configmap-and-sidecar-container.yaml
```

3. Проверяем, что оба контейнера в Pod'е работают (статус `2/2`):
```bash
kubectl get pods -l app.kubernetes.io/name=configmap-sidecar-container
```

#### Шаг 2.2: Исследование взаимодействия контейнеров (Важно!)
Давай заглянем внутрь Pod'а и увидим разницу между контейнерами.

1. Получим имя пода:
```bash
export POD_NAME=$(kubectl get pods -l app.kubernetes.io/name=configmap-sidecar-container -o jsonpath='{.items[0].metadata.name}')
```

2. **Заглянем в Sidecar (alpine):** Он читает ConfigMap и пишет в `emptyDir`.
   ```bash
   kubectl exec -it $POD_NAME -c alpine -- cat /pod-data/index.html
   # Вывод: "Wed ... My preferred color is red"
   ```

3. **Заглянем в Main (nginx):** Он просто читает тот же самый файл из `emptyDir`.
```bash
   kubectl exec -it $POD_NAME -c nginx -- cat /usr/share/nginx/html/index.html
   # Вывод: Тот же самый текст!
```

#### Шаг 2.3: Динамическое обновление через Sidecar
1. Открываем сервис для проверки:
   ```bash
   kubectl expose deployment configmap-sidecar-container --name=sidecar-svc --port=8081 --target-port=80
   kubectl port-forward service/sidecar-svc 8081:8081 &
   curl http://localhost:8081
   ```
2. Обновляем ConfigMap:
   ```bash
   kubectl edit configmap color # меняем 'red' на 'green'
   ```
3. Делаем несколько запросов `curl http://localhost:8081` с интервалом в 10 секунд.
   - **Что происходит:** Kubelet обновляет файл в томе Sidecar'а → Скрипт в Sidecar'е (цикл `while true`) считывает новое значение и перезаписывает `index.html` в `emptyDir` → Nginx при следующем запросе отдает уже обновленный файл. 
   - **Итог:** Nginx даже не знает о существовании ConfigMap, он просто отдает файлы. Вся логика обновления инкапсулирована в Sidecar.

### Часть 3: Immutable (Неизменяемые) ConfigMap

Используются для критичных конфигов, которые не должны меняться (повышает безопасность и производительность, так как Kubelet перестает их опрашивать).

1. Создаем immutable ConfigMap:
```bash
kubectl apply -f https://k8s.io/examples/configmap/immutable-configmap.yaml
```
2. Пытаемся изменить (получим ошибку):
```bash
kubectl edit configmap company-name-20150801
# Ошибка: ConfigMap "company-name-20150801" is invalid: data: Forbidden: field is immutable
```
3. **Правильный workflow обновления:**
   - Создать новый ConfigMap с новым именем:
```bash
kubectl apply -f https://k8s.io/examples/configmap/new-immutable-configmap.yaml
```
   - Обновить Deployment, изменив ссылку в `volumes`:
```bash
kubectl edit deployment immutable-configmap-volume
# Найти секцию volumes и поменять name: company-name-20150801 на company-name-20240312
```
   - K8s автоматически запустит Rolling Update. Старые Pod'ы удалятся, новые создадутся с новым конфигом.
   - Удалить старый ConfigMap:
```bash
kubectl delete configmap company-name-20150801
```

## 🐛 Troubleshooting

| Симптом                                      | Возможная причина                                                                                               | Решение                                                                                                                       |
| -------------------------------------------- | --------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| Файл в томе не обновляется > 2 мин           | 1. Имя ключа в ConfigMap не совпадает с путем. 2. Сбой kubelet.                                                 | Проверь `kubectl describe pod`, секцию `Volumes`.                                                                             |
| Nginx отдает 404 при использовании Sidecar   | Sidecar не успел создать файл до старта Nginx (если не использован `initContainer`), или скрипт в sidecar упал. | Проверь логи sidecar: `kubectl logs <pod> -c alpine`.                                                                         |
| Ошибка `Forbidden: is immutable`             | Попытка изменить `data` в immutable ConfigMap.                                                                  | Используй workflow: Новый ConfigMap → Обновление Deployment → Удаление старого.                                               |
| Приложение игнорирует изменения файла в томе | Приложение не поддерживает hot-reload конфигурации.                                                             | Измени приложение или используй `kubectl rollout restart deployment` для принудительного пересоздания Pod'ов (как с Env Var). |

## 💡 Главный вывод модуля

ConfigMap сам по себе не "магически" обновляет приложения. Он лишь предоставляет данные. **Архитектура твоего Pod'а определяет, как эти данные будут доставлены приложению.** Паттерн Sidecar + `emptyDir` — это золотой стандарт для разделения ответственности: один контейнер отвечает за *актуальность конфигурации*, другой — за *бизнес-логику*, и они не зависят от перезапусков друг друга.
