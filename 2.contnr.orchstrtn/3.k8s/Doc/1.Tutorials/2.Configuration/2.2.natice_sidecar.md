## 📚 Теория: Эволюция и механика Native Sidecars

### 1. Проблема "старых" сайдкаров
Раньше сайдкар создавался просто добавлением второго элемента в массив `containers`. Это порождало три критические проблемы:
1. **Непредсказуемый запуск:** K8s запускал все контейнеры параллельно. Основной контейнер мог стартовать раньше сайдкара и упасть, не найдя нужных файлов или прокси.
2. **Блокировка завершения (Jobs):** При `restartPolicy: OnFailure` или `Never` (в Job), K8s считает Job завершенным, только когда **ВСЕ** контейнеры завершены. Если сайдкар работает в бесконечном цикле (например, сбор логов), Job никогда не завершится (зависнет в статусе `Running`).
3. **Непредсказуемый shutdown:** При удалении Pod'а все контейнеры получают `SIGTERM` одновременно. Сайдкар мог умереть раньше основного приложения, оборвав исходящие соединения или не успев отправить последние логи.

### 2. Решение: Native Sidecar (Kubernetes 1.29+, Stable)
Нативный сайдкар объявляется в массиве **`initContainers`**, но с одним магическим добавлением: **`restartPolicy: Always`**.

#### Механика жизненного цикла (Startup & Shutdown)

**Фаза запуска (Startup):**
1. Выполняются обычные `initContainers` (строго по очереди, до успешного завершения).
2. Запускаются **Native Sidecars** (параллельно друг другу).
3. Запускаются **Main Containers** (основные приложения).
> *Итог:* Сайдкар гарантированно готов к работе до того, как стартует основное приложение.

**Фаза завершения (Shutdown):**
1. Pod получает сигнал на удаление (например, `kubectl delete` или вытеснение).
2. **Main Containers** получают `SIGTERM` и начинают graceful shutdown.
3. K8s ждет завершения основных контейнеров.
4. Только после того, как основные контейнеры умерли, **Native Sidecars** получают `SIGTERM`.
5. Если сайдкар не завершается в течение `terminationGracePeriodSeconds`, он получает `SIGKILL`.
> *Итог:* Сайдкар живет ровно до тех пор, пока нужен основному приложению для корректного завершения.

**Фаза Jobs:**
- Нативные сайдкары **игнорируются** при определении завершения Job. Job считается успешным, когда основные контейнеры завершили работу, даже если сайдкар все еще работает или перезапускается.

## 🚀 Практика: Диагностика и внедрение

### Шаг 1: Проверка включения Feature Gate
Функция `SidecarContainers` включена по умолчанию начиная с v1.29. Но в некоторых управляемых кластерах или старых сборках она может быть отключена.

Проверка на уровне API Server:
```bash
kubectl get --raw /metrics | grep kubernetes_feature_enabled | grep SidecarContainers
```
*Ожидаемый вывод:* `kubernetes_feature_enabled{name="SidecarContainers",stage="STABLE"} 1` (или BETA для 1.28).

### Шаг 2: Создание нативного сайдкара
Создадим простой пример, демонстрирующий порядок запуска.

```yaml
# sidecar-demo.yaml
apiVersion: v1
kind: Pod
metadata:
  name: native-sidecar-demo
spec:
  initContainers:
  - name: setup
    image: alpine:3
    command: ['sh', '-c', 'echo "1. Обычный init-контейнер отработал" && sleep 2']
  - name: log-sidecar
    image: alpine:3
    restartPolicy: Always # <--- МАГИЯ ЗДЕСЬ
    command: ['sh', '-c', 'echo "2. Сайдкар запустился и работает в фоне"; while true; do sleep 10; done']
  containers:
  - name: main-app
    image: alpine:3
    command: ['sh', '-c', 'echo "3. Основное приложение запустилось"; sleep 30; echo "4. Основное приложение завершается"']
```

Применяем и наблюдаем:
```bash
kubectl apply -f sidecar-demo.yaml
kubectl get pod native-sidecar-demo -w
```
**Наблюдение:** Ты увидишь, как статус меняется: `Init:0/2` -> `Init:1/2` -> `PodInitializing` -> `Running`. Сайдкар и main-app запускаются практически одновременно после завершения `setup`.

### Шаг 3: Проверка порядка завершения (Shutdown)
Удалим Pod и посмотрим на логи, чтобы подтвердить порядок.

```bash
kubectl delete pod native-sidecar-demo
# Сразу же смотрим логи сайдкара и основного контейнера
kubectl logs native-sidecar-demo -c log-sidecar --previous
kubectl logs native-sidecar-demo -c main-app --previous
```
**Вывод:** Ты увидишь сообщение "4. Основное приложение завершается" в логах main-app, и только после этого сайдкар получит шанс завершить свой цикл (или будет убит по таймауту).

## 🐛 Troubleshooting: Проблема "исчезающего" restartPolicy

Это самая коварная проблема в продакшене. 

**Сценарий:** Ты пишешь правильный YAML с `restartPolicy: Always` в `initContainers`. Применяешь его. Pod зависает в `Init:CrashLoopBackOff` или ведет себя как старый сайдкар.

**Причина:** В кластере работает **Mutating Admission Webhook** (например, старая версия Istio, Linkerd, или кастомный оператор безопасности), который модифицирует Pod перед сохранением. Если этот веб-хук скомпилирован со старой библиотекой клиента Kubernetes (до v1.28), он не знает о поле `restartPolicy` внутри `initContainers`. 
По правилам API, если веб-хук использует стратегию "заменить весь объект" (а не "слить изменения"), он **молча удаляет** неизвестные поля.

**Диагностика:**
1. Посмотри на *фактически сохраненный* в кластере Pod, а не на свой YAML:
   ```bash
   kubectl get pod <имя_пода> -o yaml | grep -A 5 "initContainers"
   ```
2. Если ты видишь `initContainers`, но **НЕ видишь** там `restartPolicy: Always`, значит, веб-хук "съел" это поле.

**Решение:**
- Обновить mutating webhook до версии, совместимой с K8s 1.29+.
- Временно отключить инъекцию для этого namespace, пока не будет обновлен контроллер.

## 🧠 Продвинутая концепция: Universal Sidecar Injector

Что делать, если у тебя кластер с разными версиями нод (где-то 1.28, где-то 1.30), и ты не можешь гарантировать поддержку фичи? Документация предлагает хитрый паттерн "Универсального инжектора".

**Логика работы:**
1. Веб-хук внедряет **ДВА** контейнера:
   - Нативный сайдкар (с `restartPolicy: Always`).
   - Старый сайдкар (как обычный `container`).
2. Они разделяют `emptyDir` том.
3. **Нативный сайдкар** при старте первым делом создает файл-флаг в `emptyDir` (например, `/shared/native-sidecar-alive`) и продолжает работу.
4. **Старый сайдкар** при старте проверяет наличие этого файла:
   - Если файл **есть**: значит, кластер поддерживает нативные сайдкары, и этот "старый" контейнер просто делает `exit 0` (завершается), чтобы не дублировать работу.
   - Если файла **нет**: значит, фича отключена, и этот "старый" контейнер берет всю работу на себя.

*Это элегантный, хоть и немного "грязный", способ обеспечить обратную совместимость при автоматической инъекции.*

## 💡 Главный вывод модуля

Нативные сайдкары — это не просто синтаксический сахар. Это **гарантия предсказуемости жизненного цикла**. Если ты используешь сервисную сетку (Istio/Linkerd), прокси-логирование или синхронизацию секретов, переход на `initContainers` + `restartPolicy: Always` обязателен для стабильной работы, особенно в Jobs и при деплое с нулевым простоем. Но всегда проверяй, не ломает ли эту конфигурацию твой mutating webhook.
