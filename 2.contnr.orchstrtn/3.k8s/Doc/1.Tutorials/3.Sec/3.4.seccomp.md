## 📚 Теория: Seccomp в Kubernetes "под капотом"

### 1. Что такое Seccomp?
Seccomp (Secure Computing Mode) — это функция ядра Linux, которая позволяет ограничить набор системных вызовов (syscalls), которые процесс может сделать к ядру. Если приложение пытается сделать syscall, не разрешенный профилем, ядро либо логирует это, либо убивает процесс (в зависимости от действия).

### 2. Три типа профилей (API v1.19+)
Профиль задается в `securityContext` (на уровне Pod или Container):
1. **`RuntimeDefault`**: Использует профиль по умолчанию, предоставляемый средой выполнения контейнеров (containerd, CRI-O). **Это современный золотой стандарт безопасности.**
2. **`Localhost`**: Использует кастомный JSON-файл, предварительно загруженный на узел в директорию `/var/lib/kubelet/seccomp/profiles/`.
3. **`Unconfined`**: Seccomp отключен. (Исторически это было значением по умолчанию в K8s, что было опасно).

> ⚠️ **Критическое правило:** Если в `securityContext` контейнера установлено `privileged: true`, **все** профили Seccomp и AppArmor игнорируются, и контейнер всегда запускается как `Unconfined`.

### 3. Методология создания кастомного профиля (3 шага)
Написать профиль seccomp с нуля сложно. Правильный подход:
1. **Audit (`SCMP_ACT_LOG`)**: Разрешить все syscalls, но записывать их в лог ядра (`dmesg` / `syslog`). Собрать статистику, что реально использует приложение.
2. **Violation (`SCMP_ACT_ERRNO`)**: Запретить всё по умолчанию. Убедиться, что приложение падает (это доказывает, что защита работает).
3. **Fine-grained (`SCMP_ACT_ALLOW`)**: Взять список syscalls из шага 1 и явно разрешить только их, оставив `defaultAction: SCMP_ACT_ERRNO`.

## 🚀 Практика: Пошаговое выполнение

### Шаг 1: Подготовка кластера `kind` с профилями
Поскольку Kubelet читает локальные профили с диска узла, мы используем `kind`, чтобы смонтировать нашу локальную папку с профилями прямо в контрольную плоскость.

1. Создай папку и скачай примеры профилей:
```bash
mkdir -p ./profiles
curl -L -o profiles/audit.json https://k8s.io/examples/pods/security/seccomp/profiles/audit.json
curl -L -o profiles/violation.json https://k8s.io/examples/pods/security/seccomp/profiles/violation.json
curl -L -o profiles/fine-grained.json https://k8s.io/examples/pods/security/seccomp/profiles/fine-grained.json
```

2. Создай файл `kind.yaml` для монтирования:
```yaml
apiVersion: kind.x-k8s.io/v1alpha4
kind: Cluster
nodes:
- role: control-plane
  extraMounts:
  - hostPath: "./profiles"
    containerPath: "/var/lib/kubelet/seccomp/profiles" # Путь, который читает kubelet
```

3. Создай кластер:
```bash
kind create cluster --config=kind.yaml
```

4. Проверь, что профили видны внутри узла:
```bash
docker exec -it kind-control-plane ls /var/lib/kubelet/seccomp/profiles
# Должен вывести: audit.json  fine-grained.json  violation.json
```

### Шаг 2: Базовый уровень (`RuntimeDefault`)
Сначала убедимся, что стандартный профиль среды выполнения работает.

```yaml
# default-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: default-pod
spec:
  securityContext:
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: test-container
    image: hashicorp/http-echo:1.0
    args: ["-text=just made some syscalls!"]
    securityContext:
      allowPrivilegeEscalation: false
```
```bash
kubectl apply -f default-pod.yaml
kubectl get pod default-pod # Должен быть в статусе Running
kubectl delete pod default-pod --wait --now
```

---

### Шаг 3: Фаза Аудита (`SCMP_ACT_LOG`)
Применим профиль `audit.json`, который разрешает всё, но логирует каждый syscall.

```yaml
# audit-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: audit-pod
spec:
  securityContext:
    seccompProfile:
      type: Localhost
      localhostProfile: profiles/audit.json # Путь относительно /var/lib/kubelet/seccomp/
  containers:
  - name: test-container
    image: hashicorp/http-echo:1.0
    args: ["-text=just made some syscalls!"]
    securityContext:
      allowPrivilegeEscalation: false
```
```bash
kubectl apply -f audit-pod.yaml
kubectl expose pod audit-pod --type NodePort --port 5678
```

**Снимаем показания:**
1. Узнай NodePort: `kubectl get svc audit-pod` (например, порт `32373`).
2. Сделай запрос изнутри узла: `docker exec -it kind-control-plane curl localhost:32373`
3. **Смотри логи ядра хоста** (в новом терминале на твоей машине):
   ```bash
   sudo tail -f /var/log/syslog | grep 'http-echo'
   # Или dmesg -w | grep 'http-echo'
   ```
**Что ты увидишь:** Строки вида `syscall=51`, `syscall=202`. Это и есть "отпечатки пальцев" приложения. Запиши эти номера или имена syscalls.

```bash
kubectl delete service audit-pod --wait
kubectl delete pod audit-pod --wait --now
```

---

### Шаг 4: Фаза Нарушения (`SCMP_ACT_ERRNO`)
Теперь применим профиль `violation.json`, где `defaultAction: "SCMP_ACT_ERRNO"` (запретить всё).

```bash
kubectl apply -f https://k8s.io/examples/pods/security/seccomp/ga/violation-pod.yaml
kubectl get pod violation-pod
```
**Ожидаемый результат:** Статус `CrashLoopBackOff` или `RunContainerError`. 
Приложение пытается сделать первый же syscall (например, `execve` или `open`), ядро блокирует его и убивает процесс. Seccomp работает!

```bash
kubectl delete pod violation-pod --wait --now
```

---

### Шаг 5: Точная настройка (`Fine-grained`)
Используем `fine-grained.json`. В нем `defaultAction` всё еще `ERRNO`, но добавлен блок `syscalls` с `action: "SCMP_ACT_ALLOW"`, содержащий те самые syscalls, которые мы увидели в Шаге 3.

```bash
kubectl apply -f https://k8s.io/examples/pods/security/seccomp/ga/fine-pod.yaml
kubectl get pod fine-pod
```
**Ожидаемый результат:** Статус `Running`. Приложение работает, потому что мы разрешили ему ровно то, что нужно, и ничего более.

```bash
kubectl delete pod fine-pod --wait --now
```

### Шаг 6: Включение `SeccompDefault` на уровне кластера (Best Practice)
Начиная с K8s 1.27, можно заставить Kubelet использовать `RuntimeDefault` для **всех** Pod'ов, даже если в манифесте ничего не указано (заменяя опасный `Unconfined`).

Для этого нужно передать флаг `--seccomp-default="true"` в Kubelet. В `kind` это делается через патч `kubeadm`:

```yaml
# kind-seccomp-default.yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
    kubeadmConfigPatches:
      - |
        kind: JoinConfiguration
        nodeRegistration:
          kubeletExtraArgs:
            seccomp-default: "true"
```
*(Примечание: Для продакшена этот флаг добавляется в `/var/lib/kubelet/config.yaml` через Ansible или системы управления конфигурациями).*

Если эта функция включена, ты можешь проверить её через `crictl` внутри узла:
```bash
kubectl run --rm -it --restart=Never --image=alpine alpine-test -- sh
# В другом окне:
docker exec -it kind-control-plane bash -c 'crictl inspect $(crictl ps --name=alpine-test -q) | jq .info.runtimeSpec.linux.seccomp'
```
Ты увидишь, что профиль применен, хотя в манифесте `alpine-test` не было никакого `securityContext`.

### Задание 1: Проверка правила `privileged`
Попробуй применить профиль `fine-grained.json` к Pod'у, у которого в `containers[].securityContext` установлено `privileged: true`.
**Вопрос:** Будет ли профиль применен? (Ответ: Нет, Kubelet проигнорирует seccomp, так как привилегированный режим отключает все механизмы изоляции ядра).

### Задание 2: Ошибка в пути к профилю
Измени в манифесте `localhostProfile` на `wrong-path/audit.json`.
**Вопрос:** Что произойдет при `kubectl apply`? (Ответ: Pod будет создан, но застрянет в статусе `Pending` или `Failed`, а в `kubectl describe pod` в Events будет ошибка: `cannot load seccomp profile: no such file or directory`).

## 🐛 Troubleshooting

| Симптом | Возможная причина | Решение |
|---------|-------------------|---------|
| Pod в статусе `Failed` с ошибкой `no such file or directory` | Неверный путь в `localhostProfile`. Помни: путь считается от `/var/lib/kubelet/seccomp/`. | Проверь монтирование в `kind` и имя файла. |
| Приложение падает сразу после старта с `fine-grained` профилем | В профиле не хватает какого-то критического syscall (например, `rt_sigaction` или `futex`). | Вернись к этапу Аудита, сделай больше запросов к приложению, чтобы поймать все syscalls, и добавь их в список `ALLOW`. |
| Профиль не применяется, хотя указан в YAML | Контейнер запущен с `privileged: true` или Feature Gate Seccomp отключен (редко в K8s 1.19+). | Убери `privileged: true`. |

## 💡 Главный вывод модуля

Seccomp — это самый низкоуровневый и надежный механизм защиты в Kubernetes. Однако писать профили вручную для сложных приложений — это боль. **Стратегия для Production:** 
1. Включить `--seccomp-default="true"` на всех узлах (это дает базовую защиту `RuntimeDefault` бесплатно).
2. Для критически важных приложений использовать инструменты типа [Kubernetes Security Profiles Operator (SPO)](https://github.com/kubernetes-sigs/security-profiles-operator), который умеет автоматически генерировать fine-grained профили на основе реального поведения приложения (Recording mode).
