## 📚 Теория: Механика кластерного PSA "под капотом"

### 1. Три режима, три профиля
PSA оценивает каждый создаваемый Pod по трем профилям (`privileged`, `baseline`, `restricted`) и может действовать в трех режимах:
- **`enforce`**: Блокирует создание Pod'а, если он нарушает профиль. (Hard fail)
- **`audit`**: Позволяет создать Pod, но записывает нарушение в аудит-логи API-сервера.
- **`warn`**: Позволяет создать Pod, но возвращает предупреждение (Warning) пользователю в выводе `kubectl`.

> 💡 **Важно:** К одному namespace (или кластеру) можно одновременно применить все три режима к разным профилям. Например: `enforce=baseline`, `warn=restricted`.

### 2. Namespace Labels vs Cluster-Level Admission Configuration
- **Namespace Labels** (`pod-security.kubernetes.io/enforce=...`): Простой способ. Работает "из коробки". Идеально для разработчиков и изоляции конкретных проектов.
- **Cluster-Level Config**: Требует изменения флага `--admission-control-config-file` у API-сервера. Это **административная задача**. Она задает *дефолтное поведение* для всего кластера и позволяет глобально исключать системные компоненты (например, `kube-system`), которые по своей природе требуют привилегий (`hostPath`, `hostNetwork`).

## 🚀 Практика: Пошаговое выполнение

### Шаг 1: Оценка влияния (Dry-Run) — "Семь раз отмерь"
Никогда не применяй `restricted` ко всем namespace'ам в продакшене без проверки. Сначала создадим "чистый" кластер kind.

```bash
# Создаем кластер без PSA
kind create cluster --name psa-wo-cluster-pss
kubectl cluster-info --context kind-psa-wo-cluster-pss
```

Теперь проверим, что будет, если мы применим разные профили ко **всем** namespace'ам, используя `--dry-run=server`. Эта команда отправляет запрос на изменение в API-сервер, но он только валидирует его и возвращает предупреждения, не сохраняя изменения.

**Тест 1: Privileged (Все разрешено)**
```bash
kubectl label --dry-run=server --overwrite ns --all \
  pod-security.kubernetes.io/enforce=privileged
```
*Результат:* Тишина. Никаких предупреждений. Все существующие Pod'ы проходят.

**Тест 2: Baseline (Запрещены известные векторы атак)**
```bash
kubectl label --dry-run=server --overwrite ns --all \
  pod-security.kubernetes.io/enforce=baseline
```
*Результат:* Ты увидишь `Warning` для `kube-system`. Системные Pod'ы (etcd, kube-proxy, kindnet) используют `hostPath`, `hostNetwork` и `privileged` режимы, что нарушает `baseline`.

**Тест 3: Restricted (Максимальная защита)**
```bash
kubectl label --dry-run=server --overwrite ns --all \
  pod-security.kubernetes.io/enforce=restricted
```
*Результат:* Массив предупреждений. Даже `coredns` нарушает `restricted`, так как не имеет `seccompProfile`, `runAsNonRoot` и `drop ALL capabilities`.

**Вывод:** Мы не можем просто включить `enforce=restricted` глобально. Нам нужна конфигурация с исключениями.

---

### Шаг 2: Создание конфигурации Admission Controller

Мы создадим конфиг, который:
1. Принудительно применяет (`enforce`) профиль `baseline` ко всему кластеру.
2. Выдает предупреждения (`warn` и `audit`) о нарушениях профиля `restricted`.
3. **Полностью исключает** namespace `kube-system` из проверок (чтобы системные компоненты могли работать).

Создай файл `/tmp/pss/cluster-level-pss.yaml`:
```bash
mkdir -p /tmp/pss
cat <<EOF > /tmp/pss/cluster-level-pss.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: AdmissionConfiguration
plugins:
- name: PodSecurity
  configuration:
    apiVersion: pod-security.admission.config.k8s.io/v1
    kind: PodSecurityConfiguration
    defaults:
      enforce: "baseline"
      enforce-version: "latest"
      audit: "restricted"
      audit-version: "latest"
      warn: "restricted"
      warn-version: "latest"
    exemptions:
      usernames: []
      runtimeClasses: []
      namespaces: ["kube-system"] # Критически важно для работы кластера!
EOF
```

---

### Шаг 3: Создание кластера Kind с этой конфигурацией

Чтобы API-сервер прочитал этот файл, мы должны передать его внутрь контрольной плоскости kind при создании.

Создай файл `/tmp/pss/cluster-config.yaml`:
```yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
  kubeadmConfigPatches:
  - |
    kind: ClusterConfiguration
    apiServer:
        extraArgs:
          admission-control-config-file: /etc/config/cluster-level-pss.yaml
        extraVolumes:
          - name: accf
            hostPath: /etc/config
            mountPath: /etc/config
            readOnly: false
            pathType: "DirectoryOrCreate"
  extraMounts:
  - hostPath: /tmp/pss
    containerPath: /etc/config
    readOnly: false
    propagation: None
```

> ⚠️ **Важно для macOS (Docker Desktop):** Убедись, что директория `/tmp` добавлена в *Settings > Resources > File sharing* в Docker Desktop, иначе kind не сможет смонтировать файл внутрь контейнера.

Создаем кластер:
```bash
kind create cluster --name psa-with-cluster-pss --config /tmp/pss/cluster-config.yaml
kubectl cluster-info --context kind-psa-with-cluster-pss
```

---

### Шаг 4: Тестирование политик (Enforce vs Warn)

Создадим простой Pod Nginx, который **соответствует** профилю `baseline`, но **нарушает** профиль `restricted`.

```yaml
# example-baseline-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
    - image: nginx
      name: nginx
      ports:
        - containerPort: 80
```

Применяем его:
```bash
kubectl apply -f example-baseline-pod.yaml
```

**Ожидаемый результат:**
```text
Warning: would violate PodSecurity "restricted:latest": allowPrivilegeEscalation != false (container "nginx" must set securityContext.allowPrivilegeEscalation=false), unrestricted capabilities (container "nginx" must set securityContext.capabilities.drop=["ALL"]), runAsNonRoot != true (pod or container "nginx" must set securityContext.runAsNonRoot=true), seccompProfile (pod or container "nginx" must set securityContext.seccompProfile.type to "RuntimeDefault" or "Localhost")
pod/nginx created
```

### 🔍 Разбор результата:
1. **`pod/nginx created`**: Pod успешно создан! Почему? Потому что наш `enforce` профиль установлен в `baseline`, а этот Pod ему соответствует (он не использует запрещенные `hostPath` или `privileged`).
2. **`Warning: would violate PodSecurity "restricted:latest"`**: Сработал режим `warn`. PSA любезно сообщает нам: "Этот Pod создан, но если бы ты включил enforce=restricted, он бы заблокировался из-за отсутствия `seccompProfile`, `runAsNonRoot` и т.д.".

Это идеальный сценарий для миграции: ты включаешь `warn=restricted`, собираешь логи/предупреждения в течение недели, исправляешь манифесты разработчиков, и только потом переключаешь `enforce=restricted`.

## 🧪 Практические задания (для закрепления)

### Задание 1: Проверка исключения (Exemption)
Попробуй создать в namespace `kube-system` Pod, который нарушает даже `baseline` (например, с `hostNetwork: true`).
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: test-privileged
  namespace: kube-system
spec:
  hostNetwork: true
  containers:
    - name: test
      image: alpine
      command: ["sleep", "3600"]
```
**Вопрос:** Будет ли он создан? (Ответ: Да, без предупреждений, потому что `kube-system` находится в списке `exemptions` нашего AdmissionConfiguration).

### Задание 2: Эскалация до Enforce Restricted
1. Измени файл `/tmp/pss/cluster-level-pss.yaml`, установив `enforce: "restricted"`.
2. Пересоздай кластер kind с новой конфигурацией.
3. Попробуй снова применить `example-baseline-pod.yaml`.
**Ожидаемый результат:** `Error from server (Forbidden): pods "nginx" is forbidden: violates PodSecurity "restricted:latest"...` (Pod не будет создан).

## 🐛 Troubleshooting

| Симптом | Возможная причина | Решение |
|---------|-------------------|---------|
| Ошибка монтирования при `kind create cluster` | Docker Desktop (macOS/Windows) не имеет доступа к `/tmp/pss`. | Добавь `/tmp` в File Sharing в настройках Docker Desktop или используй директорию внутри `/Users/` или `C:\Users\`. |
| Предупреждения не появляются при `apply` | PSA плагин не загрузился. | Проверь логи API-сервера: `kubectl logs -n kube-system kube-apiserver-...`. Ищи ошибки парсинга `cluster-level-pss.yaml`. |
| Системные Pod'ы (`kube-proxy`) не запускаются | Ты применил `enforce=baseline` или `restricted` глобально, забыв добавить `kube-system` в `exemptions`. | Пересоздай кластер с правильным `AdmissionConfiguration`. |

## 💡 Главный вывод модуля

Настройка PSA на уровне кластера — это мощный инструмент для установления "гигиенического минимума" безопасности для всех новых рабочих нагрузок. Однако, слепое применение `restricted` ко всему кластеру сломает системные компоненты. Правильный путь миграции: **Dry-Run → Warn (сбор нарушений) → Исправление манифестов → Enforce**. И всегда исключай `kube-system` на глобальном уровне, управляя его безопасностью через строгий RBAC.
