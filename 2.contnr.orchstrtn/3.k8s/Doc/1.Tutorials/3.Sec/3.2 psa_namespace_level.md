## 📚 Теория: Механика Namespace-Level PSA

### 1. Как это работает "под капотом"
Когда ты отправляешь запрос на создание Pod'а, **Pod Security Admission (PSA) контроллер** делает следующее:
1. Проверяет, есть ли у целевого namespace метки, начинающиеся с `pod-security.kubernetes.io/`.
2. Если метки есть, он извлекает профиль (`privileged`, `baseline`, `restricted`) и версию (`latest` или конкретную, например, `v1.25`).
3. Сравнивает `securityContext` создаваемого Pod'а с правилами выбранного профиля.
4. Применяет действие в зависимости от режима:
   - **`enforce`**: Отклоняет запрос (HTTP 403 Forbidden), если есть нарушения.
   - **`warn`**: Пропускает запрос, но добавляет предупреждение в ответ API-сервера (виден в выводе `kubectl`).
   - **`audit`**: Пропускает запрос, но записывает событие нарушения в аудит-лог API-сервера (и в события namespace, если настроено).

### 2. Иерархия и приоритет
Если в кластере настроены **оба** метода (глобальный `AdmissionConfiguration` и метки namespace), **метки namespace имеют приоритет** для этого конкретного namespace. Это позволяет администратору задать безопасные дефолтные значения для всего кластера, но дать отдельной команде разработчиков возможность смягчить или ужесточить правила для своего проекта.

### 3. Синтаксис меток
Полный набор меток для одного профиля выглядит так:
```yaml
pod-security.kubernetes.io/<режим>: <профиль>
pod-security.kubernetes.io/<режим>-version: <версия> # например, "latest" или "v1.25"
```
Где `<режим>` = `enforce`, `audit` или `warn`.

## 🚀 Практика: Пошаговое выполнение

### Шаг 1: Создание чистого кластера и Namespace

Создаем изолированную среду для экспериментов:
```bash
kind create cluster --name psa-ns-level
kubectl cluster-info --context kind-psa-ns-level

# Создаем целевое пространство имен
kubectl create ns example
```

### Шаг 2: Применение политик безопасности через метки

Мы настроим namespace `example` по "золотому стандарту" для постепенной миграции:
- **`enforce=baseline`**: Жестко блокируем создание откровенно опасных Pod'ов (с `hostPath`, `privileged` и т.д.).
- **`warn=restricted`** и **`audit=restricted`**: Мягко предупреждаем разработчиков, что их Pod не соответствует самым строгим стандартам, но пока не блокируем их, давая время на исправление.

Выполняем одну команду для установки всех меток:
```bash
kubectl label --overwrite ns example \
  pod-security.kubernetes.io/enforce=baseline \
  pod-security.kubernetes.io/enforce-version=latest \
  pod-security.kubernetes.io/warn=restricted \
  pod-security.kubernetes.io/warn-version=latest \
  pod-security.kubernetes.io/audit=restricted \
  pod-security.kubernetes.io/audit-version=latest
```

Проверяем, что метки применились:
```bash
kubectl get ns example --show-labels
```

### Шаг 3: Тестирование в защищенном Namespace (`example`)

Используем тот же манифест `example-baseline-pod.yaml` (простой Nginx без `securityContext`), который мы видели в прошлом модуле. Он соответствует `baseline`, но нарушает `restricted`.

```bash
kubectl apply -n example -f https://k8s.io/examples/security/example-baseline-pod.yaml
```

**Ожидаемый результат:**
```text
Warning: would violate PodSecurity "restricted:latest": allowPrivilegeEscalation != false ...
pod/nginx created
```
**Анализ:** Pod создан успешно, потому что он прошел проверку `enforce=baseline`. Но мы получили `Warning`, потому что сработал режим `warn=restricted`.

### Шаг 4: Тестирование изоляции (Namespace `default`)

Теперь попробуем создать **тот же самый Pod** в namespace `default`, который мы **не метили**.

```bash
kubectl apply -n default -f https://k8s.io/examples/security/example-baseline-pod.yaml
```

**Ожидаемый результат:**
```text
pod/nginx created
```
**Анализ:** Никаких предупреждений! PSA полностью игнорирует namespace `default`, потому что у него нет меток `pod-security.kubernetes.io/`. Это доказывает, что политики применяются точечно.

## 🧪 Практические задания (для глубокого понимания)

### Задание 1: Проверка режима `enforce` (Hard Fail)
Мы проверили `warn`, но действительно ли `enforce=baseline` блокирует нарушения? Попробуем создать откровенно опасный Pod в namespace `example`.

Создай файл `privileged-pod.yaml`:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: privileged-pod
  namespace: example
spec:
  containers:
    - name: priv
      image: alpine
      command: ["sleep", "3600"]
      securityContext:
        privileged: true # Нарушает профиль baseline!
```

Примени его:
```bash
kubectl apply -f privileged-pod.yaml
```
**Ожидаемый результат:**
```text
Error from server (Forbidden): error when creating "privileged-pod.yaml": pods "privileged-pod" is forbidden: violates PodSecurity "baseline:latest": privileged (container "priv" must not set securityContext.privileged=true)
```
*Вывод:* Режим `enforce` работает корректно и защищает кластер.

### Задание 2: Проверка режима `audit`
Режим `audit` записывает нарушения в события. Посмотрим, записал ли K8s наше предыдущее предупреждение о нарушении `restricted`.

```bash
kubectl get events -n example --field-selector reason=PodSecurityPolicyViolation
```
*(Примечание: в некоторых версиях K8s или дистрибутивах аудит может записываться с другим `reason`, например, просто в общие события namespace. Можно выполнить `kubectl get events -n example --sort-by='.metadata.creationTimestamp'` и найти записи от `pod-security-admission`).*

## 🐛 Troubleshooting

| Симптом | Возможная причина | Решение |
|---------|-------------------|---------|
| Предупреждения не появляются при `apply` в `example` | Опечатка в ключе метки (например, `pod-security` вместо `pod-security.kubernetes.io`). | Проверь метки: `kubectl get ns example -o jsonpath='{.metadata.labels}'`. |
| Pod создается без ошибок, даже с `privileged: true` | PSA отключен на уровне кластера (Feature Gate `PodSecurity=false`), что редкость для K8s 1.25+. | Проверь флаги API-сервера или используй `kind`, как в инструкции. |
| Ошибка `namespace "example" not found` | Попытка применить ресурс до создания namespace. | Убедись, что `kubectl create ns example` выполнился успешно. |

## 💡 Главный вывод модуля

Метки namespace — это **самый гибкий и рекомендуемый способ** управления безопасностью в Kubernetes. Он позволяет реализовать принцип "безопасных умолчаний" (через глобальный конфиг), но дает командам автономию: команда может пометить свой namespace как `restricted`, чтобы заставить себя писать безопасный код, или временно использовать `baseline`, пока идет рефакторинг. Главное правило: **нет метки — нет проверки**.
