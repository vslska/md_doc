## 📚 Теория: AppArmor в Kubernetes "под капотом"

### 1. Что такое AppArmor?
Это модуль ядра Linux, реализующий **Mandatory Access Control (MAC)**. В отличие от стандартных прав Linux (DAC), которые может изменить `root` внутри контейнера, правила AppArmor "пришиты" к исполняемому файлу на уровне ядра и не могут быть обойдены даже пользователем root.

### 2. Эволюция API в Kubernetes (Важно!)
- **До v1.30**: Профили задавались через *аннотации* (`container.apparmor.security.beta.kubernetes.io/<container_name>: localhost/<profile>`). Это было неудобно и не валидировалось схемой.
- **Начиная с v1.31 (Stable)**: Профили задаются нативно в `securityContext` на уровне Pod'а или Контейнера.
  - `type: RuntimeDefault` — использовать профиль по умолчанию, который предоставляет containerd/CRI-O (рекомендуется для большинства случаев).
  - `type: Localhost` — использовать кастомный профиль, предварительно загруженный в ядро узла.
  - `type: Unconfined` — отключить AppArmor для этого контейнера (опасно!).

> 💡 **Приоритет**: Если профиль задан и на уровне Pod'а, и на уровне Контейнера, профиль Контейнера имеет приоритет.

### 3. Главная проблема: "Слепое пятно" Планировщика (Scheduler Blind Spot)
Kubernetes Scheduler **не знает**, какие AppArmor профили загружены на какие узлы. 
Если ты запросишь `localhostProfile: my-custom-profile`, Scheduler может отправить Pod на любую ноду. Если на этой конкретной ноде профиль не загружен, **Kubelet отклонит запуск контейнера**, и Pod застрянет в состоянии ошибки.

**Решение в Production**: 
1. Использовать **Kubernetes Security Profiles Operator (SPO)** для декларативного управления профилями.
2. Или помечать узлы метками (например, `apparmor-profiles: my-custom-profile`) и использовать `nodeSelector` / `nodeAffinity` в Pod'е, чтобы гарантировать его попадание только на подходящие ноды.

## 🚀 Практика: Пошаговое выполнение

*Предполагается, что у тебя есть кластер (например, kind или minikube) на базе Linux (Ubuntu/Debian), где AppArmor включен по умолчанию.*

### Шаг 1: Проверка поддержки на узле

Подключись к узлу (для minikube: `minikube ssh`, для kind: `docker exec -it <kind-control-plane> bash`) и проверь ядро:

```bash
cat /sys/module/apparmor/parameters/enabled
```
**Ожидаемый результат**: `Y`

Проверь загруженные профили:
```bash
cat /sys/kernel/security/apparmor/profiles | sort
```
*Ты увидишь стандартные профили, такие как `docker-default` или `cri-containerd.apparmor.d`.*

---

### Шаг 2: Загрузка кастомного профиля на узел

В реальном кластере это делает DaemonSet или Security Profiles Operator. Для обучения загрузим профиль вручную через SSH/Shell на узел.

Этот профиль разрешает чтение файлов, но **строго запрещает любую запись** (`deny /** w`).

```bash
# Выполняй это ВНУТРИ ноды (с правами sudo)
sudo apparmor_parser -q <<EOF
#include <tunables/global>

profile k8s-apparmor-example-deny-write flags=(attach_disconnected) {
  #include <abstractions/base>
  file,
  # Запретить все операции записи в файлы
  deny /** w,
}
EOF
```

Проверь, что профиль появился в списке:
```bash
cat /sys/kernel/security/apparmor/profiles | grep k8s-apparmor-example-deny-write
```
**Ожидаемый результат**: `k8s-apparmor-example-deny-write (enforce)`

---

### Шаг 3: Применение профиля к Pod'у (Современный API)

Создай манифест `hello-apparmor.yaml`, используя новый синтаксис `securityContext`:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: hello-apparmor
spec:
  securityContext:
    appArmorProfile:
      type: Localhost
      localhostProfile: k8s-apparmor-example-deny-write
  containers:
  - name: hello
    image: busybox:1.28
    command: [ "sh", "-c", "echo 'Hello AppArmor!' && sleep 1h" ]
```

Примени его:
```bash
kubectl apply -f hello-apparmor.yaml
```

---

### Шаг 4: Верификация применения профиля

**Способ А: Проверка через `/proc` (Самый надежный)**
Каждый процесс в Linux имеет файл атрибутов AppArmor. Проверим процесс с PID 1 (главный процесс контейнера):

```bash
kubectl exec hello-apparmor -- cat /proc/1/attr/current
```
**Ожидаемый результат**: `k8s-apparmor-example-deny-write (enforce)`
*(Если видишь `unconfined` или `cri-containerd.apparmor.d`, значит, твой профиль не применился).*

**Способ Б: Тест на нарушение (Violation Test)**
Попробуем сделать то, что запрещает профиль: создать файл.

```bash
kubectl exec hello-apparmor -- touch /tmp/test
```
**Ожидаемый результат**:
```text
touch: /tmp/test: Permission denied
command terminated with exit code 1
```
*Магия работает! Даже будучи `root` внутри busybox, ядро Linux заблокировало системный вызов записи.*

---

### Шаг 5: Тестирование сценария сбоя (Незагруженный профиль)

Что будет, если мы запросим профиль, которого нет на ноде?

```bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: hello-apparmor-fail
spec:
  securityContext:
    appArmorProfile:
      type: Localhost
      localhostProfile: k8s-apparmor-example-allow-write # Такого профиля НЕТ
  containers:
  - name: hello
    image: busybox:1.28
    command: [ "sh", "-c", "sleep 1h" ]
EOF
```

Проверим статус Pod'а:
```bash
kubectl get pod hello-apparmor-fail
# Скорее всего, будет статус Pending или Error

kubectl describe pod hello-apparmor-fail
```

**Ищи в секции `Events`**:
```text
Warning  Failed  ...  kubelet  Error: failed to get container spec opts: failed to generate apparmor spec opts: apparmor profile not found "k8s-apparmor-example-allow-write"
```
**Вывод**: Kubelet валидирует наличие профиля *перед* запуском контейнера. Если профиля нет, Pod не запустится, предотвращая работу в небезопасном состоянии.

## 🧪 Практические задания (для закрепления)

### Задание 1: Переопределение на уровне контейнера
1. Создай Pod с двумя контейнерами.
2. Задай `appArmorProfile: type: Unconfined` на уровне **Pod** (`spec.securityContext`).
3. Задай `appArmorProfile: type: Localhost, localhostProfile: k8s-apparmor-example-deny-write` на уровне **одного из контейнеров**.
4. Проверь через `/proc/1/attr/current` в каждом контейнере, что приоритет контейнера сработал.

### Задание 2: Имитация Production-подхода (Node Affinity)
1. Повесь метку на ноду, где ты загрузил профиль: `kubectl label nodes <node-name> apparmor-custom=deny-write`.
2. Добавь в `hello-apparmor.yaml`:
   ```yaml
   spec:
     nodeSelector:
       apparmor-custom: deny-write
   ```
3. Это гарантирует, что Pod попадет только туда, где профиль точно есть.

## 🐛 Troubleshooting

| Симптом | Возможная причина | Решение |
|---------|-------------------|---------|
| Pod висит в `Pending` или `Failed` с ошибкой `apparmor profile not found` | Профиль не загружен на ту ноду, куда Scheduler отправил Pod. | Проверь ноду через `kubectl get pod -o wide`, зайди на неё и проверь `/sys/kernel/security/apparmor/profiles`. Используй `nodeSelector`. |
| Команда `touch` проходит успешно (нет Permission denied) | Профиль не применился. Возможно, использован старый синтаксис аннотаций в K8s 1.31+, или опечатка в имени профиля. | Проверь `/proc/1/attr/current`. Убедись, что используешь `securityContext.appArmorProfile`, а не аннотации. |
| Ошибка `apparmor_parser: command not found` | На узле не установлены утилиты AppArmor. | Установи их: `apt-get install apparmor-utils` (Ubuntu/Debian). |

## 💡 Главный вывод модуля

AppArmor — это мощный инструмент защиты "в глубину" (Defense in Depth). Даже если злоумышленник получит `root` внутри контейнера, AppArmor не даст ему изменить критические файлы или выйти за пределы разрешенного. Однако, из-за "слепого пятна" планировщика, управление профилями через голый `localhost` в больших кластерах ненадежно. **Всегда используй `nodeSelector` или Kubernetes Security Profiles Operator** для привязки Pod'ов к узлам с нужными профилями.
