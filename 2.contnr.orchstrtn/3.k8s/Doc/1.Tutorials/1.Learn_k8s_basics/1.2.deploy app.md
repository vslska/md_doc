## 📚 Теория: Deployment и ReplicaSet

### Иерархия объектов

```
┌─────────────────────────────────────────────────────────┐
│                    Deployment                           │
│  "Хочу 3 копии приложения my-app с образом my-app:v1"   │
│  Отвечает за: обновления, rollback, стратегии           │
└────────────────────┬────────────────────────────────────┘
                     │ создаёт и управляет
                     ▼
┌─────────────────────────────────────────────────────────┐
│                  ReplicaSet                             │
│  "Поддерживаю 3 реплики Pod'ов"                         │
│  Отвечает за: создание/удаление Pod'ов,самовосстановление│
└────────────────────┬────────────────────────────────────┘
                     │ создаёт и управляет
                     ▼
┌─────────────────────────────────────────────────────────┐
│                     Pods                                │
│  ┌─────┐  ┌─────┐  ┌─────┐                              │
│  │Pod 1│  │Pod 2│  │Pod 3│                              │
│  └─────┘  └─────┘  └─────┘                              │
│  Отвечают за: запуск контейнеров                        │
└─────────────────────────────────────────────────────────┘
```

### Что делает каждый компонент?

#### Deployment
**Роль**: менеджер ReplicaSet'ов

**Что делает**:
- ✅ Создаёт ReplicaSet
- ✅ Управляет rolling updates (создаёт новый ReplicaSet, постепенно переключает трафик)
- ✅ Управляет rollback (возвращается к предыдущему ReplicaSet)
- ✅ Управляет стратегиями обновлений (RollingUpdate, Recreate)

**Что НЕ делает**:
- ❌ Не создаёт Pod'ы напрямую
- ❌ Не следит за количеством Pod'ов

#### ReplicaSet
**Роль**: менеджер Pod'ов

**Что делает**:
- ✅ Создаёт Pod'ы
- ✅ Удаляет Pod'ы
- ✅ Поддерживает desired replicas (если Pod упал → создаёт новый)
- ✅ Самовосстановление (если нода упала → создаёт Pod на другой ноде)

**Что НЕ делает**:
- ❌ Не управляет обновлениями (это делает Deployment)
- ❌ Не управляет стратегиями обновлений

#### Pod
**Роль**: единица деплоя

**Что делает**:
- ✅ Запускает контейнеры
- ✅ Предоставляет сеть и хранилище для контейнеров

### Аналогия с рестораном

**Deployment** — это **директор ресторана**:
- Говорит: "Мне нужна команда из 3 поваров, которые готовят пиццу"
- Нанимает **менеджера смены** (ReplicaSet)
- Если нужно изменить меню (обновление) — нанимает нового менеджера смены с новыми инструкциями
- Если что-то пошло не так — откатывается к предыдущему менеджеру

**ReplicaSet** — это **менеджер смены**:
- Получает инструкцию: "Поддерживай 3 повара"
- Нанимает поваров (создаёт Pod'ы)
- Если повар заболел (Pod упал) — нанимает нового
- Если повар уволился (Pod удалён) — нанимает нового
- Не думает о стратегиях обновления — просто поддерживает нужное количество

**Pod** — это **повар**:
- Готовит пиццу (запускает контейнер)
- Если повар не может работать — менеджер смены нанимает нового

### Что происходит при различных сценариях?

#### Сценарий 1: Создание Deployment

```
1. Ты: kubectl create deployment my-app --image=my-app:v1 --replicas=3
   ↓
2. API Server: создаёт объект Deployment в etcd
   ↓
3. Deployment Controller: видит новый Deployment
   └─> Создаёт ReplicaSet с desired replicas = 3
   ↓
4. ReplicaSet Controller: видит новый ReplicaSet
   └─> Создаёт 3 Pod'а
   ↓
5. Scheduler: назначает Pod'ы на ноды
   ↓
6. Kubelet: запускает контейнеры в Pod'ах
   ↓
7. Результат: 3 Pod'а Running
```

#### Сценарий 2: Pod упал

```
1. Контейнер в Pod'е crashed
   ↓
2. Kubelet: сообщает API Server, что контейнер в статусе Terminated
   ↓
3. Pod переходит в статус Failed
   ↓
4. ReplicaSet Controller: видит, что ready replicas = 2 (должно быть 3)
   └─> Создаёт новый Pod
   ↓
5. Scheduler: назначает новый Pod на ноду
   ↓
6. Kubelet: запускает контейнер
   ↓
7. Результат: снова 3 Pod'а Running
```

#### Сценарий 3: Rolling update

```
1. Ты: kubectl set image deployment/my-app my-app=my-app:v2
   ↓
2. API Server: обновляет Deployment spec (новый образ)
   ↓
3. Deployment Controller: видит изменение
   └─> Создаёт новый ReplicaSet (ReplicaSet-v2) с образом v2
   └─> Уменьшает replicas в старом ReplicaSet (ReplicaSet-v1)
   └─> Увеличивает replicas в новом ReplicaSet (ReplicaSet-v2)
   ↓
4. ReplicaSet-v1 Controller: удаляет Pod'ы (по одному)
   ↓
5. ReplicaSet-v2 Controller: создаёт новые Pod'ы (по одному)
   ↓
6. Результат: все Pod'ы с образом v2, старый ReplicaSet имеет replicas = 0
```

#### Сценарий 4: Rollback

```
1. Ты: kubectl rollout undo deployment/my-app
   ↓
2. API Server: обновляет Deployment spec (возврат к предыдущей версии)
   ↓
3. Deployment Controller: видит изменение
   └─> Уменьшает replicas в текущем ReplicaSet (ReplicaSet-v2)
   └─> Увеличивает replicas в предыдущем ReplicaSet (ReplicaSet-v1)
   ↓
4. ReplicaSet-v2 Controller: удаляет Pod'ы
   ↓
5. ReplicaSet-v1 Controller: создаёт Pod'ы с образом v1
   ↓
6. Результат: все Pod'ы с образом v1
```

### Контроллеры в Controller Manager

Каждым объектом управляет свой контроллер:

| Контроллер | Что контролирует | Что делает |
|------------|------------------|------------|
| **Deployment Controller** | Deployment | Создаёт/обновляет ReplicaSet'ы |
| **ReplicaSet Controller** | ReplicaSet | Создаёт/удаляет Pod'ы |
| **Node Controller** | Nodes | Следит за здоровьем нод |
| **Job Controller** | Job | Создаёт Pod'ы для задач |
| **EndpointSlice Controller** | Services | Обновляет Endpoints |

Все эти контроллеры работают в **kube-controller-manager** — компоненте Control Plane.

## 🛠️ Основы kubectl

### Синтаксис kubectl

```bash
kubectl <action> <resource> [flags]
```

| Часть | Что означает | Пример |
|-------|--------------|--------|
| `kubectl` | CLI инструмент для K8s | `kubectl` |
| `<action>` | Что сделать | `get`, `create`, `delete`, `describe` |
| `<resource>` | Над каким ресурсом | `pods`, `deployments`, `replicasets` |
| `[flags]` | Дополнительные параметры | `-o yaml`, `-n kube-system` |

### Основные команды

| Команда | Что делает |
|---------|------------|
| `kubectl get <resource>` | Показать ресурсы |
| `kubectl describe <resource> <name>` | Детальная информация |
| `kubectl create <resource>` | Создать ресурс (императивно) |
| `kubectl apply -f <file>` | Создать/обновить ресурс (декларативно) |
| `kubectl delete <resource> <name>` | Удалить ресурс |
| `kubectl logs <pod-name>` | Посмотреть логи Pod'а |
| `kubectl exec -it <pod-name> -- /bin/sh` | Войти в контейнер |

### Получение справки

```bash
# Общая справка
kubectl --help

# Справка по команде
kubectl get --help

# Справка по ресурсу
kubectl explain deployment
kubectl explain deployment.spec
```

### Что такое `kubernetes-bootcamp`?

Это тестовое приложение от Google, которое:
- Запускает веб-сервер на порту 8080
- Отдаёт информацию о Pod'е (hostname, IP)
- Используется в официальных туториалах

### Что происходит под капотом?

```
1. kubectl отправляет запрос в API Server
   └─> POST /apis/apps/v1/namespaces/default/deployments
   └─> Body: {
         "metadata": {"name": "kubernetes-bootcamp"},
         "spec": {
           "replicas": 1,
           "selector": {"matchLabels": {"app": "kubernetes-bootcamp"}},
           "template": {
             "metadata": {"labels": {"app": "kubernetes-bootcamp"}},
             "spec": {
               "containers": [{
                 "name": "kubernetes-bootcamp",
                 "image": "gcr.io/google-samples/kubernetes-bootcamp:v1"
               }]
             }
           }
         }
       }

2. API Server валидирует запрос
   └─> Проверка прав доступа (RBAC)
   └─> Проверка схемы (OpenAPI)
   └─> Admission controllers (mutating, validating)

3. API Server сохраняет Deployment в etcd
   └─> Deployment создан, но ReplicaSet и Pod'ов ещё нет

4. Deployment Controller (в Controller Manager) видит новый Deployment
   └─> Создаёт ReplicaSet с desired replicas = 1

5. ReplicaSet Controller видит новый ReplicaSet
   └─> Создаёт Pod

6. Scheduler видит новый Pod (статус: Pending)
   └─> Выбирает ноду (у нас только одна — minikube)
   └─> Назначает Pod на ноду (статус: Scheduled)

7. Kubelet на ноде видит назначенный Pod
   └─> Запрашивает container runtime запустить контейнер
   └─> Container runtime (containerd) загружает образ
   └─> Запускает контейнер

8. Kubelet сообщает API Server, что Pod запущен
   └─> Статус Pod'а: Running

9. ReplicaSet Controller проверяет, что Pod готов
   └─> Проверяет readiness probe (если есть)
   └─> Обновляет статус ReplicaSet: ready replicas = 1

10. Deployment Controller проверяет статус ReplicaSet
    └─> Обновляет статус Deployment: READY 1/1
```

### Ожидаемый результат

```
deployment.apps/kubernetes-bootcamp created
```

## 🌐 Доступ к приложению через API Proxy

### Проблема

Pod'ы работают в изолированной сети кластера. Они имеют внутренние IP (например, `10.244.0.5`), которые недоступны извне.

**Как получить доступ к приложению?**

Есть несколько способов:
1. **kubectl proxy** — проксирует запросы к API Server (для отладки)
2. **kubectl port-forward** — перенаправляет порт с localhost на Pod
3. **NodePort Service** — открывает порт на ноде
4. **LoadBalancer Service** — создаёт внешний load balancer (в облаке)
5. **Ingress** — L7 маршрутизация (HTTP/HTTPS)

В этом туториале используем **kubectl proxy**.

---

### Шаг 8: Запуск kubectl proxy

**Открой второе окно терминала** и выполни:

```bash
kubectl proxy
```

**Ожидаемый результат**:
```
Starting to serve on 127.0.0.1:8001
```

### Что делает kubectl proxy?

```
┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
│   curl / browser│         │  kubectl proxy  │         │   API Server    │
│  (localhost)    │────────>│  (localhost:8001)│────────>│  (127.0.0.1:8443)│
└─────────────────┘         └─────────────────┘         └─────────────────┘
                                     │
                                     │ проксирует запросы
                                     ▼
                            ┌─────────────────┐
                            │      Pods       │
                            │  (через API)    │
                            └─────────────────┘
```

**kubectl proxy**:
- Запускает HTTP-сервер на `localhost:8001`
- Проксирует все запросы к API Server
- Аутентифицируется автоматически (использует credentials из kubeconfig)
- Позволяет обращаться к Pod'ам через API

> 💡 **Важно**: это **не production-ready** способ. Используется только для отладки.

```bash
curl http://localhost:8001/api/v1/namespaces/default/pods
```

### Шаг 10: Получение имени Pod'а

```bash
export POD_NAME=$(kubectl get pods -o go-template --template '{{range .items}}{{.metadata.name}}{{"\n"}}{{end}}')
echo "Name of the Pod: $POD_NAME"
```

**Ожидаемый результат**:
```
Name of the Pod: kubernetes-bootcamp-7d4f8b7b9c-x2y4z
```

### Разбор команды

| Часть | Что делает |
|-------|------------|
| `kubectl get pods` | Получает список Pod'ов |
| `-o go-template` | Использует Go template для форматирования вывода |
| `{{range .items}}` | Итерирует по всем элементам (Pod'ам) |
| `{{.metadata.name}}` | Извлекает имя Pod'а |
| `{{"\n"}}` | Добавляет новую строку |
| `{{end}}` | Заканчивает цикл |
| `export POD_NAME=...` | Сохраняет результат в переменную окружения |

### Альтернативный способ (проще)

```bash
export POD_NAME=$(kubectl get pods -o jsonpath='{.items[0].metadata.name}')
echo "Name of the Pod: $POD_NAME"
```

**Разбор**:
- `jsonpath` — более простой синтаксис для извлечения данных
- `{.items[0]}` — первый элемент списка
- `{.metadata.name}` — имя Pod'а

### Шаг 11: Доступ к Pod через proxy

```bash
curl http://localhost:8001/api/v1/namespaces/default/pods/$POD_NAME:8080/proxy/
```

**Ожидаемый результат**:
```
Hello Kubernetes bootcamp! | Running on: kubernetes-bootcamp-7d4f8b7b9c-x2y4z | v=1
```

### Разбор URL

```
http://localhost:8001/api/v1/namespaces/default/pods/$POD_NAME:8080/proxy/
│                     │    │         │       │      │         │    │
│                     │    │         │       │      │         │    └─ endpoint для proxy
│                     │    │         │       │      │         └─ порт контейнера
│                     │    │         │       │      └─ имя Pod'а
│                     │    │         │       └─ namespace
│                     │    │         └─ ресурс (pods)
│                     │    └─ API версия
│                     └─ API группа (core)
└─ proxy (localhost:8001)
```

### Как это работает?

1. `curl` отправляет запрос на `localhost:8001`
2. `kubectl proxy` получает запрос
3. Proxy понимает, что это запрос к Pod через API
4. Proxy перенаправляет запрос в API Server
5. API Server перенаправляет запрос в kubelet на ноде
6. Kubelet перенаправляет запрос в контейнер (на порт 8080)
7. Контейнер возвращает ответ
8. Ответ проходит обратно через всю цепочку

> 💡 **Важно**: это работает, потому что kubelet имеет доступ к Pod'ам через container runtime. API Server не имеет прямого доступа к Pod'ам.

## 🐛 Troubleshooting

### Проблема: Pod в статусе `ImagePullBackOff`

**Причина**: не удалось загрузить образ

**Решение**:
```bash
# Посмотреть события
kubectl describe pod $POD_NAME

# Проверить имя образа
kubectl get deployment kubernetes-bootcamp -o jsonpath='{.spec.template.spec.containers[0].image}'

# Если образ приватный, создать imagePullSecret
kubectl create secret docker-registry my-registry-key \
  --docker-server=<server> \
  --docker-username=<username> \
  --docker-password=<password>

# Привязать secret к ServiceAccount
kubectl patch deployment kubernetes-bootcamp -p '{"spec":{"template":{"spec":{"imagePullSecrets":[{"name":"my-registry-key"}]}}}}'
```

### Проблема: Pod в статусе `CrashLoopBackOff`

**Причина**: контейнер падает сразу после запуска

**Решение**:
```bash
# Посмотреть логи
kubectl logs $POD_NAME

# Посмотреть логи предыдущего контейнера
kubectl logs $POD_NAME --previous

# Войти в контейнер (если возможно)
kubectl exec -it $POD_NAME -- /bin/sh
```

### Проблема: `kubectl proxy` не работает

**Решение**:
```bash
# Проверить, что proxy запущен
ps aux | grep "kubectl proxy"

# Перезапустить proxy
# Ctrl+C в терминале с proxy
kubectl proxy

# Проверить порт
curl http://localhost:8001/version
```
