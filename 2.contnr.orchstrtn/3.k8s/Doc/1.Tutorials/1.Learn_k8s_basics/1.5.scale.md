## 📚 Теория: Как работает масштабирование "под капотом"

### 1. Цепочка масштабирования (Кто за что отвечает)
Когда ты выполняешь команду масштабирования, происходит чёткая последовательность действий:

```text
1. Ты: kubectl scale deployment my-app --replicas=4
   ↓
2. API Server: обновляет поле spec.replicas объекта Deployment на 4.
   ↓
3. Deployment Controller: видит изменение. Он НЕ создаёт Pod'ы сам. 
   Он обновляет поле spec.replicas у управляемого им ReplicaSet на 4.
   ↓
4. ReplicaSet Controller: видит, что CURRENT (2) < DESIRED (4).
   ↓
5. ReplicaSet Controller: создаёт 2 новых объекта Pod и отправляет их в API Server.
   ↓
6. Scheduler: назначает новые Pod'ы на доступные ноды.
   ↓
7. Kubelet: запускает контейнеры в новых Pod'ах.
   ↓
8. Service Controller: замечает новые Pod'ы с нужными метками (labels).
   ↓
9. Service Controller: добавляет IP-адреса новых Pod'ов в объект Endpoints (или EndpointSlice).
   ↓
10. kube-proxy на нодах: обновляет правила iptables/IPVS, чтобы трафик мог идти на новые IP.
```
> 💡 **Ключевой вывод**: Deployment управляет ReplicaSet'ом. ReplicaSet управляет Pod'ами. Service слепо читает Endpoints. Все компоненты работают асинхронно через API Server.

### 2. Балансировка нагрузки в Service
Service не является "умным" прокси-сервером уровня приложения (как Nginx). На уровне ноды `kube-proxy` настраивает правила сетевой маршрутизации (обычно через `iptables` в режиме random или `IPVS` в режиме round-robin). Когда запрос приходит на `ClusterIP:Port` или `NodeIP:NodePort`, ядро Linux случайным образом выбирает один из IP-адресов, записанных в **Endpoints**, и перенаправляет пакет туда.

### 3. Уменьшение масштаба (Scale Down)
Когда ты уменьшаешь количество реплик (например, с 4 до 2):
1. Deployment обновляет ReplicaSet.
2. ReplicaSet выбирает "лишние" Pod'ы для удаления.
3. ReplicaSet отправляет запрос на удаление этих Pod'ов в API Server.
4. Pod переходит в статус `Terminating`.
5. Service Controller **мгновенно** удаляет IP этого Pod'а из Endpoints, чтобы новый трафик на него не шёл.
6. Kubelet отправляет сигнал `SIGTERM` контейнеру (даёт время на graceful shutdown, по умолчанию 30 сек).
7. Если контейнер не завершился, отправляется `SIGKILL`.

## 🚀 Практика: Масштабирование и тестирование

*Предполагается, что у тебя запущен Deployment `kubernetes-bootcamp` и создан Service (NodePort или LoadBalancer) из предыдущих модулей.*

### Шаг 1: Оценка текущего состояния

Проверим текущую конфигурацию Deployment и ReplicaSet.

```bash
kubectl get deployments
kubectl get rs
```

**Ожидаемый результат** (если масштаб был 1):
```text
NAME                  READY   UP-TO-DATE   AVAILABLE   AGE
kubernetes-bootcamp   1/1     1            1           20m

NAME                             DESIRED   CURRENT   READY   AGE
kubernetes-bootcamp-7d4f8b7b9c   1         1         1       20m
```
> 💡 Обрати внимание: имя ReplicaSet (`kubernetes-bootcamp-7d4f8b7b9c`) состоит из имени Deployment и хеша. Этот хеш генерируется на основе `pod template`.

---

### Шаг 2: Масштабирование до 4 реплик

Используем императивную команду `scale`:

```bash
kubectl scale deployment kubernetes-bootcamp --replicas=4
```

**Что только что произошло:** API Server обновил `spec.replicas` в Deployment, а Deployment передал это значение в ReplicaSet.

Проверим результат:

```bash
kubectl get deployments
kubectl get rs
kubectl get pods -o wide
```

**Ожидаемый результат:**
```bash
# Deployment
NAME                  READY   UP-TO-DATE   AVAILABLE   AGE
kubernetes-bootcamp   4/4     4            4           22m

# ReplicaSet
NAME                             DESIRED   CURRENT   READY   AGE
kubernetes-bootcamp-7d4f8b7b9c   4         4         4       22m

# Pods (будет 4 строки с разными IP и NODE)
NAME                                   READY   STATUS    RESTARTS   AGE   IP           NODE
kubernetes-bootcamp-7d4f8b7b9c-a1b2c   1/1     Running   0          22m   10.244.0.5   minikube
kubernetes-bootcamp-7d4f8b7b9c-d3e4f   1/1     Running   0          10s   10.244.0.6   minikube
kubernetes-bootcamp-7d4f8b7b9c-g5h6i   1/1     Running   0          10s   10.244.0.7   minikube
kubernetes-bootcamp-7d4f8b7b9c-j7k8l   1/1     Running   0          10s   10.244.0.8   minikube
```

Проверим события Deployment, чтобы увидеть действие контроллера:
```bash
kubectl describe deployment kubernetes-bootcamp
```
*Ищи в разделе Events строку: `Scaled up replica set kubernetes-bootcamp-7d4f8b7b9c to 4`.*

### Шаг 3: Проверка балансировки нагрузки

Теперь у нас 4 Pod'а с разными IP. Service должен автоматически распределять трафик между ними.

1. Получим порт Service (если ещё не получен):
```bash
export NODE_PORT=$(kubectl get services/kubernetes-bootcamp -o go-template='{{(index .spec.ports 0).nodePort}}')
echo "NODE_PORT=$NODE_PORT"
```

2. Если ты на macOS/Windows с Docker Desktop, запусти в **отдельном окне**:
```bash
minikube service kubernetes-bootcamp --url
```
*(Скопируй полученный URL, например `http://127.0.0.1:XXXXX`)*

3. Выполним несколько запросов подряд. Для наглядности используем цикл в bash:
```bash
# Если используешь minikube service URL, замени команду ниже на:
for i in (seq 1 6); curl http://127.0.0.1:XXXXX; echo ""; end

# Если прямой доступ работает (Linux):
for i in {1..6}; do curl http://"$(minikube ip):$NODE_PORT"; echo ""; done
```

**Ожидаемый результат:**
Ты должен увидеть, что поле `Running on:` меняется, указывая на разные имена Pod'ов:
```text
Hello Kubernetes bootcamp! | Running on: kubernetes-bootcamp-7d4f8b7b9c-a1b2c | v=1
Hello Kubernetes bootcamp! | Running on: kubernetes-bootcamp-7d4f8b7b9c-g5h6i | v=1
Hello Kubernetes bootcamp! | Running on: kubernetes-bootcamp-7d4f8b7b9c-a1b2c | v=1
Hello Kubernetes bootcamp! | Running on: kubernetes-bootcamp-7d4f8b7b9c-j7k8l | v=1
...
```
> 💡 **Вывод**: Service успешно балансирует нагрузку между всеми 4 доступными Pod'ами, читая их IP из Endpoints.

---

### Шаг 4: Уменьшение масштаба (Scale Down)

Сократим количество реплик до 2.

```bash
kubectl scale deployment kubernetes-bootcamp --replicas=2
```

Сразу же запустим команду с флагом `-w` (watch), чтобы наблюдать за процессом в реальном времени:

```bash
kubectl get pods -w
```

**Что ты увидишь:**
1. Два Pod'а изменят статус на `Terminating`.
2. Через несколько секунд они исчезнут из списка.
3. Останутся только 2 Pod'а в статусе `Running`.
*(Нажми `Ctrl+C`, чтобы выйти из режима watch)*

Проверим итоговое состояние:
```bash
kubectl get deployments
kubectl get rs
```
*Обрати внимание: поле `DESIRED` в ReplicaSet теперь равно 2. Deployment и ReplicaSet синхронизированы.*

## 🐛 Troubleshooting

| Симптом | Возможная причина | Команда для диагностики |
|---------|-------------------|--------------------------|
| При `scale up` Pod'ы висят в `Pending` | На ноде (в minikube) закончились ресурсы (CPU/RAM), указанные в `requests`. | `kubectl describe pod <pending-pod-name>` (смотри Events: `Insufficient cpu/memory`) |
| Трафик идёт только на 1 Pod, хотя их несколько | 1. Ошибка в readiness probe (Pod не Ready, не добавляется в Endpoints). 2. Включена `sessionAffinity: ClientIP` на Service. | `kubectl get pods` (проверь колонку READY), `kubectl get svc -o yaml` |
| `scale down` зависает на `Terminating` | Приложение игнорирует сигнал `SIGTERM` или есть finalizers, блокирующие удаление. | `kubectl get pod <name> -o yaml` (ищи раздел `finalizers`), проверь логи приложения. |

Масштабирование в Kubernetes — это **декларативное изменение желаемого состояния (desired state)**. Ты не говоришь системе "запусти ещё 3 контейнера". Ты говоришь: "Мне нужно 4 реплики". А уже контроллеры (Deployment → ReplicaSet) и сетевые компоненты (Service → Endpoints → kube-proxy) самостоятельно и асинхронно приводят систему к этому состоянию.
