## 📚 Теория: Механика Rolling Update "под капотом"

### 1. Что такое Rolling Update?
Это стратегия обновления по умолчанию для Deployment. Её цель — обновить приложение **без простоя** (zero-downtime). 

**Как это работает на уровне контроллеров:**
Когда ты меняешь образ (или любую другую часть `pod template` в Deployment), хеш шаблона изменяется.
1. **Deployment Controller** видит изменение и создаёт **новый ReplicaSet** (с новым хешем в имени), желаемое количество реплик которого равно 0.
2. Он начинает **масштабировать новый ReplicaSet вверх** и **старый ReplicaSet вниз** одновременно.
3. Этот процесс контролируется двумя параметрами в `strategy.rollingUpdate`:
   - `maxUnavailable`: Максимальное количество Pod'ов, которые *могут быть недоступны* во время обновления (по умолчанию 25%).
   - `maxSurge`: Максимальное количество Pod'ов, которые *могут быть созданы сверх* желаемого количества (по умолчанию 25%).
4. **Service** продолжает работать. Благодаря **readiness probe**, он добавляет новые Pod'ы в Endpoints *только после* того, как они полностью запустятся и будут готовы принимать трафик. Старые Pod'ы удаляются из Endpoints перед своим уничтожением.

### 2. Механика отката (Rollback)
Команда `kubectl rollout undo` **не является "магической кнопкой отмены"**. 
Когда ты делаешь откат, Deployment Controller просто создаёт **новую ревизию** (новую запись в истории), которая является копией предыдущей стабильной конфигурации, и запускает **ещё один Rolling Update** к этому состоянию.

## 🚀 Практика: Обновление и откат приложения

*Предполагается, что у тебя запущен Deployment `kubernetes-bootcamp` (например, с 2-4 репликами после предыдущего модуля).*

### Шаг 1: Подготовка и проверка текущего состояния

Убедимся, что Service существует (если удаляли в прошлом модуле, создадим заново):
```bash
kubectl expose deployment kubernetes-bootcamp --type="NodePort" --port 8080
```

Проверим текущую версию образа и статус:
```bash
kubectl get deployments
kubectl get pods
# Посмотрим текущий образ
kubectl get deployment kubernetes-bootcamp -o jsonpath='{.spec.template.spec.containers[0].image}'
```

---

### Шаг 2: Инициирование обновления (v1 → v2)

Обновим образ приложения на версию `v2` с помощью императивной команды:

```bash
kubectl set image deployment/kubernetes-bootcamp kubernetes-bootcamp=docker.io/jocatalin/kubernetes-bootcamp:v2
```

**Что произошло:** API Server обновил `spec.template.spec.containers[0].image` в Deployment. Deployment Controller заметил изменение хеша и начал процесс Rolling Update.

Наблюдаем за процессом в реальном времени:
```bash
kubectl get pods -w
```
*Ты увидишь, как старые Pod'ы переходят в `Terminating`, а новые с суффиксом нового хеша создаются и переходят в `Running`.* (Нажми `Ctrl+C` для выхода).

Проверим статус обновления:
```bash
kubectl rollout status deployment/kubernetes-bootcamp
```
*Ожидаемый результат: `deployment "kubernetes-bootcamp" successfully rolled out`*

---

### Шаг 3: Проверка обновления

1. **Проверка через Service (балансировка):**
```bash
export NODE_PORT=$(kubectl get services/kubernetes-bootcamp -o go-template='{{(index .spec.ports 0).nodePort}}')
# Для macOS/Windows: minikube service kubernetes-bootcamp --url
for i in (seq 1 5}; do curl http://"$(minikube ip):$NODE_PORT"; echo ""; end
```
*Обрати внимание: в ответе теперь должно быть `v=2`.*

2. **Проверка истории ревизий:**
Deployment хранит историю изменений. Посмотрим её:
```bash
kubectl rollout history deployment/kubernetes-bootcamp
```
*Ты увидишь как минимум 2 ревизии: 1 (исходная v1) и 2 (обновление до v2).*

Посмотрим детали конкретной ревизии:
```bash
kubectl rollout history deployment/kubernetes-bootcamp --revision=2
```

---

### Шаг 4: Симуляция сбоя и откат (Rollback)

Попробуем обновиться на несуществующую или сломанную версию (например, `v10`).

```bash
kubectl set image deployment/kubernetes-bootcamp kubernetes-bootcamp=gcr.io/google-samples/kubernetes-bootcamp:v10
```

Сразу проверим статус Pod'ов:
```bash
kubectl get pods
```
**Ожидаемый результат:** Ты увидишь Pod'ы в статусе `ImagePullBackOff` или `ErrImagePull`. 

Посмотрим, почему это произошло:
```bash
# Возьми имя любого нового "сломанного" пода
kubectl describe pod <имя_нового_пода>
```
*В секции `Events` будет видно: `Failed to pull image ... v10: not found`.*

**Важный момент:** В этот момент твоё приложение *всё ещё работает* на старых Pod'ах (версии v2), потому что новые не прошли readiness check, и Deployment приостановил откат старых реплик (в зависимости от настроек `maxUnavailable`). Но процесс обновления "застрял".

**Решение: Откат к последней стабильной версии**
```bash
kubectl rollout undo deployment/kubernetes-bootcamp
```

Проверим, что всё вернулось к версии v2:
```bash
kubectl rollout status deployment/kubernetes-bootcamp
kubectl get pods
kubectl get deployment kubernetes-bootcamp -o jsonpath='{.spec.template.spec.containers[0].image}'
```

---

### Шаг 5: Очистка

```bash
kubectl delete deployment kubernetes-bootcamp
kubectl delete service kubernetes-bootcamp
```

## 🧪 Практические задания (для закрепления)

### Задание 1: Наблюдение за двумя ReplicaSet'ами
Во время выполнения Rolling Update (Шаг 2) открой второе окно терминала и выполняй:
```bash
kubectl get rs -w
```
**Наблюдение:** Ты увидишь, как `DESIRED` старого ReplicaSet уменьшается до 0, а `DESIRED` нового ReplicaSet увеличивается до целевого значения. Deployment никогда не управляет Pod'ами напрямую, он управляет *двумя* ReplicaSet'ами одновременно.

### Задание 2: Откат к конкретной ревизии
1. Сделай обновление до `v2`.
2. Сделай фиктивное обновление (например, добавь аннотацию, чтобы создать ревизию 3).
3. Откати не на 1 шаг назад, а сразу к ревизии 1 (оригинальная v1):
   ```bash
   kubectl rollout undo deployment/kubernetes-bootcamp --to-revision=1
   ```

## 🐛 Troubleshooting

| Симптом | Возможная причина | Команда для диагностики |
|---------|-------------------|--------------------------|
| Rollout "завис" и не завершается | Новые Pod'ы не становятся `Ready` (ошибка в коде, нет readiness probe, `ImagePullBackOff`). | `kubectl get pods`, `kubectl describe pod`, `kubectl rollout status` |
| Приложение недоступно во время обновления | `maxUnavailable` установлено в 100% или readiness probe настроен неверно и отклоняет все Pod'ы. | `kubectl get deployment -o yaml` (проверить `strategy`) |
| `rollout undo` не помогает | Проблема не в образе, а, например, в ConfigMap или внешних зависимостях. | Проверить логи приложения: `kubectl logs <pod-name>` |

## 💡 Главный вывод модуля

Обновление в Kubernetes — это не замена файлов на сервере. Это **плавная замена одних управляемых объектов (Pod'ов старого ReplicaSet) на другие (Pod'ы нового ReplicaSet)**, контролируемая правилами доступности. Если что-то идёт не так, система останавливается, не ломая работающее приложение, и ждёт вмешательства (или отката).
