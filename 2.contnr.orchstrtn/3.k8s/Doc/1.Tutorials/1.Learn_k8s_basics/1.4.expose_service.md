## 📚 Теория: Как на самом деле работает Service

### Проблема: Pod'ы смертны (Ephemeral)
Pod'ы создаются и уничтожаются динамически (например, при сбое ноды или масштабировании ReplicaSet). 
- Каждый новый Pod получает **новый, уникальный IP-адрес**.
- Если фронтенд-приложение жестко закодирует IP-адрес бэкенд-Pod'а, при перезапуске бэкенда связь сломается.

### Решение: Service
**Service** — это сетевая абстракция, которая определяет **логический набор Pod'ов** и политику доступа к ним. Он предоставляет **стабильный IP-адрес (ClusterIP) и DNS-имя**, которые не меняются, даже если underlying Pod'ы полностью заменяются.

### Механизм работы

```text
1. Ты создаёшь Service с selector: app=kubernetes-bootcamp
   ↓
2. Service Controller (в Control Plane) постоянно сканирует все Pod'ы в кластере.
   ↓
3. Он находит Pod'ы, у которых в metadata.labels есть app=kubernetes-bootcamp.
   ↓
4. Controller автоматически создаёт/обновляет объект Endpoints (или EndpointSlice), 
   в котором записаны текущие IP-адреса этих Pod'ов.
   ↓
5. kube-proxy на КАЖДОЙ ноде кластера следит за изменениями в Endpoints.
   ↓
6. При изменении Endpoints, kube-proxy перепрограммирует сетевые правила на ноде 
   (через iptables или IPVS), чтобы трафик на ClusterIP:Port перенаправлялся 
   на один из актуальных IP-адресов Pod'ов (Load Balancing).
```

> 💡 **Важно**: Service **не знает** о существовании Deployment или ReplicaSet. Он знает **только** о метках (Labels). Если ты вручную создашь Pod с нужной меткой, Service немедленно начнёт направлять на него трафик.

### Типы Service (Spec.type)

| Тип | Описание | Доступ |
|-----|----------|--------|
| **ClusterIP** (default) | Внутренний виртуальный IP. | Только изнутри кластера. |
| **NodePort** | Открывает статический порт (30000-32767) на **каждой** ноде кластера. Трафик с `NodeIP:NodePort` перенаправляется на ClusterIP. | Извне кластера (через IP любой ноды). |
| **LoadBalancer** | Запрашивает у облачного провайдера (AWS, GCP) внешний балансировщик нагрузки, который направляет трафик на NodePort. | Из интернета (в облаке). В minikube эмулируется через `minikube tunnel`. |
| **ExternalName** | Не имеет селектора и не создаёт Endpoints. Просто возвращает CNAME-запись при DNS-запросе. | Для маппинга на внешние сервисы (например, RDS в AWS). |

## 🚀 Практика: Создание и проверка Service

*Предполагается, что Deployment `kubernetes-bootcamp` из Практики 1.2 всё ещё запущен.*

### Шаг 1: Создание Service типа NodePort

Мы можем создать Service декларативно (через YAML), но `kubectl` предоставляет удобную императивную команду `expose`, которая сама считывает селектор и порт из Deployment.

```bash
kubectl expose deployment kubernetes-bootcamp --type="NodePort" --port 8080
```

**Разбор команды:**
- `expose deployment`: создаёт Service для указанного Deployment.
- `--type="NodePort"`: делает сервис доступным через порт на ноде.
- `--port 8080`: порт, на котором Service будет принимать трафик (и перенаправлять его на targetPort контейнера, который в данном образе тоже 8080).

**Проверка создания:**
```bash
kubectl get services
```
**Ожидаемый результат:**
```text
NAME                  TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)          AGE
kubernetes            ClusterIP   10.96.0.1       <none>        443/TCP          20m
kubernetes-bootcamp   NodePort    10.103.150.20   <none>        8080:31234/TCP   10s
```
> 💡 Обрати внимание на `PORT(S)`: `8080:31234/TCP`. Это означает: Service слушает порт 8080 внутри кластера, и пробрасывает его на порт **31234** (случайный из диапазона 30000-32767) на всех нодах.

---

### Шаг 2: Глубокий осмотр Service

Посмотрим, как Service связан с Pod'ами.

```bash
kubectl describe service kubernetes-bootcamp
```

**На что смотреть в выводе:**
1. **Selector**: `app=kubernetes-bootcamp`. Именно по этой метке Service ищет цели.
2. **Endpoints**: `10.244.0.5:8080`. Это **самая важная строка**. Она показывает, что Controller успешно нашел Pod с такой меткой и записал его IP в Endpoints. Если здесь пусто (`<none>`), значит, ни один Pod не соответствует селектору, или Pod не прошел readiness probe.
3. **Port / NodePort**: подтверждение проброса портов.

---

### Шаг 3: Доступ к приложению извне кластера

Получим значение NodePort в переменную:
```bash
export NODE_PORT=$(kubectl get services/kubernetes-bootcamp -o go-template='{{(index .spec.ports 0).nodePort}}')
echo "NODE_PORT=$NODE_PORT"
```

#### Вариант А: Прямой запрос к ноде (работает на Linux / некоторых конфигурациях)
```bash
curl http://"$(minikube ip):$NODE_PORT"
```

#### Вариант Б: Через minikube (Обязательно для macOS/Windows с Docker driver)
Из-за изоляции сети в Docker Desktop на Mac/Windows, прямой доступ к IP ноды может не работать. Minikube предоставляет специальную команду для проброса:

*Открой новое окно терминала и запусти:*
```bash
minikube service kubernetes-bootcamp --url
```
*Он выдаст URL вида `http://127.0.0.1:XXXXX`. Скопируй его и выполни в основном терминале:*
```bash
curl http://127.0.0.1:XXXXX  # подставь свой порт
```
**Ожидаемый результат:** `Hello Kubernetes bootcamp! | Running on: kubernetes-bootcamp-... | v=1`

---

### Шаг 4: Работа с метками (Labels)

Service работает благодаря меткам. Давай посмотрим, как они работают.

**1. Посмотрим метки Deployment (откуда они берутся у Pod'ов):**
```bash
kubectl describe deployment kubernetes-bootcamp
```
*Ищи секцию `Labels` и `Selector`. Там будет `app=kubernetes-bootcamp`.*

**2. Запросим Pod'ы по этой метке:**
```bash
kubectl get pods -l app=kubernetes-bootcamp
```
*Флаг `-l` (или `--selector`) фильтрует вывод только по совпадающим меткам.*

**3. Добавим новую метку к конкретному Pod'у:**
```bash
export POD_NAME=$(kubectl get pods -o jsonpath='{.items[0].metadata.name}')
kubectl label pods "$POD_NAME" version=v1
```

**4. Проверим, что метка добавилась:**
```bash
kubectl get pods -l version=v1
```
*Теперь этот Pod можно выбрать двумя способами: по `app` или по `version`.*

> ⚠️ **Внимание**: Добавление метки `version=v1` **не сломает** текущий Service, потому что Service ищет по `app=kubernetes-bootcamp`, а мы эту метку не удаляли. Метки можно комбинировать: `-l app=kubernetes-bootcamp,version=v1`.

---

### Шаг 5: Удаление Service и проверка изоляции

Удалим Service, используя селектор (это безопаснее, чем удалять по имени, если их много):

```bash
kubectl delete service -l app=kubernetes-bootcamp
```

**Проверка удаления:**
```bash
kubectl get services
```
*Сервиса `kubernetes-bootcamp` больше нет.*

**Проверка доступности извне (должно упасть):**
```bash
curl http://"$(minikube ip):$NODE_PORT"  # или через minikube service URL
```
*Должно быть: `curl: (7) Failed to connect to ... Connection refused`.*

**Проверка доступности изнутри (должно работать!):**
```bash
kubectl exec -it "$POD_NAME" -- curl http://localhost:8080
```
**Ожидаемый результат:** `Hello Kubernetes bootcamp!...`

**Вывод**: Удаление Service разорвало внешний доступ, но **не убило** Pod и не остановило приложение внутри него. Управление жизненным циклом Pod'а по-прежнему лежит на ReplicaSet/Deployment.

## 🧪 Практические задания (для закрепления)

### Задание 1: Проверка EndpointSlice (современная альтернатива Endpoints)
В новых версиях K8s вместо `Endpoints` используется `EndpointSlice` (масштабируется лучше). Посмотри, как K8s видит цели твоего (теперь уже удаленного, создай его снова на время задания) сервиса:
```bash
# Сначала создай сервис заново
kubectl expose deployment kubernetes-bootcamp --type="NodePort" --port 8080

# Посмотри EndpointSlice
kubectl get endpointslices
kubectl describe endpointslices -l kubernetes.io/service-name=kubernetes-bootcamp
```
*Увидишь список IP-адресов Pod'ов, которые K8s считает валидными целями для этого сервиса.*

### Задание 2: Ошибка в селекторе
1. Измени метку у Pod'а, удалив старую: `kubectl label pods "$POD_NAME" app-` (дефис в конце удаляет метку).
2. Посмотри на `kubectl get endpointslices` или `kubectl describe service`.
3. **Вопрос**: Что произошло с полем Endpoints? Почему? (Ответ: Service потерял цель, трафик некуда направлять).
4. Верни метку назад: `kubectl label pods "$POD_NAME" app=kubernetes-bootcamp`.

## 🐛 Troubleshooting

| Симптом | Возможная причина | Команда для диагностики |
|---------|-------------------|--------------------------|
| `curl` по NodePort выдает `Connection refused` | 1. Неверный порт. 2. Брандмауэр. 3. На Mac/Win нужен `minikube service` или `minikube tunnel`. | `kubectl get svc`, `minikube service <name> --url` |
| Поле `Endpoints` в `describe service` пустое (`<none>`) | 1. Ни один Pod не имеет меток, совпадающих с `selector` сервиса. 2. Pod'ы есть, но они не в статусе `Ready` (не прошли readiness probe). | `kubectl get pods --show-labels`, `kubectl describe pod` |
| Трафик идет только на один Pod, хотя их несколько | Проблема в `sessionAffinity` (по умолчанию None, но могла быть включена) или сбой в kube-proxy. | `kubectl get svc -o yaml`, проверить логи `kube-proxy` на ноде. |

## 💡 Главный вывод модуля

**Deployment** управляет тем, *что* запущено (Pod'ы).  
**Service** управляет тем, *как* к этому получить доступ (сеть), и делает это слепо, ориентируясь только на метки (Labels). Это обеспечивает полную развязку (decoupling) между развертыванием и сетью.
