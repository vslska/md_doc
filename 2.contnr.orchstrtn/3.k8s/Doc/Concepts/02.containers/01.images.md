# Container Images — образы контейнеров в Kubernetes

> 📌 Образ контейнера — исполняемый пакет с приложением и зависимостями. **Имя** = `[registry/]name[:tag|@digest]`. **imagePullPolicy** управляет, когда скачивать: `Always`, `IfNotPresent`, `Never`. Для **приватных реестров** — `imagePullSecrets` (рекомендуется) или credential provider. **Best practice**: используй **digest** вместо `:latest` для production, избегай `ImagePullBackOff` через правильные credentials.

---

## Именование образов

### 🎯 Формат

```
[<registry>/][<namespace>/]<name>[:<tag>|@<digest>]
```

### 📝 Примеры

| Пример | Что означает |
|--------|--------------|
| `busybox` | `docker.io/library/busybox:latest` (публичный Docker Hub, тег latest) |
| `busybox:1.32.0` | `docker.io/library/busybox:1.32.0` |
| `registry.k8s.io/pause:3.5` | Кастомный registry + тег |
| `registry.k8s.io/pause@sha256:1ff6c18...` | По **digest** (неизменяемый идентификатор) |
| `registry.k8s.io/pause:3.5@sha256:1ff6c18...` | Тег + digest (используется digest) |
| `fictional.registry.example:10443/imagename` | Registry с портом |

### 🎯 Tag vs Digest

| Характеристика | Tag | Digest |
|----------------|-----|--------|
| **Формат** | `:v1.42.0`, `:latest` | `@sha256:1ff6c18...` |
| **Изменяемость** | ✅ Может меняться (перемещаться) | ❌ Неизменяемый |
| **Идентификация** | Версия (логическая) | Конкретное содержимое (хеш) |
| **Production** | ⚠️ Рискованно (кроме семантических версий) | ✅ **Рекомендуется** |
| **Откат** | Сложно | Точно знаешь, на что откатываешься |

### 🎯 Правила для тегов

- Макс 128 символов
- Паттерн: `[a-zA-Z0-9_][a-zA-Z0-9._-]{0,127}`
- Если тег не указан → подразумевается `:latest`

## imagePullPolicy — политика загрузки

### 🎯 Три режима

| Значение | Поведение | Когда использовать |
|----------|-----------|-------------------|
| **`IfNotPresent`** | Загружать, только если образа нет локально | Production (по умолчанию для тегов ≠ latest) |
| **`Always`** | Всегда проверять registry (даже если образ есть) | Разработка, `:latest`, частые обновления |
| **`Never`** | Никогда не загружать, только локальный образ | Offline, preloaded images |

### 🎯 Автоматическая установка imagePullPolicy

Если `imagePullPolicy` **не указан**, kubelet устанавливает его автоматически:

| Условие | Автоматическое значение |
|---------|-------------------------|
| Указан **digest** (`@sha256:...`) | `IfNotPresent` |
| Тег **`:latest`** | `Always` |
| Тег **не указан** | `Always` |
| Тег **не `:latest`** (например, `:v1.42.0`) | `IfNotPresent` |

> ⚠️ **Важно**: `imagePullPolicy` устанавливается **один раз** при создании объекта. Если потом изменить тег на `:latest`, `imagePullPolicy` **не изменится** автоматически.

### 📝 Пример

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: my-registry.example.com/my-app:v1.42.0
    imagePullPolicy: IfNotPresent    # ← можно не указывать (установится автоматически)
```

## ImagePullBackOff — ошибки загрузки

### 🎯 Что это

Статус контейнера, когда kubelet **не может загрузить образ**.

### 🔍 Причины

| Причина | Решение |
|---------|---------|
| **Неверное имя образа** | Проверить registry, имя, тег |
| **Приватный registry без credentials** | Добавить `imagePullSecrets` |
| **Registry недоступен** | Проверить сеть, DNS, firewall |
| **Образ не существует** | Проверить тег/digest в registry |
| **Неверные credentials** | Обновить secret |

### 🔍 Retry логика

- Kubernetes увеличивает задержку между попытками (exponential backoff)
- Максимальная задержка: **300 секунд (5 минут)**

### 🔍 Отладка

```bash
# Посмотреть статус пода
kubectl describe pod my-app | grep -A10 'Events:'
# Warning  Failed     ...  Failed to pull image "my-registry.example.com/my-app:v1.42.0": 
# rpc error: code = Unknown desc = Error response from daemon: unauthorized

# Посмотреть статус контейнера
kubectl get pod my-app -o jsonpath='{.status.containerStatuses[0].state}'
# waiting:
#   reason: ImagePullBackOff
#   message: Back-off pulling image "my-registry.example.com/my-app:v1.42.0"

# Проверить imagePullSecrets
kubectl get pod my-app -o jsonpath='{.spec.imagePullSecrets}'

# Проверить secret
kubectl get secret my-registry-secret -o yaml
```

## Параллельная загрузка образов

### 🎯 По умолчанию: последовательная

```
kubelet загружает образы по одному:
1. Образ 1 → ждёт завершения
2. Образ 2 → ждёт завершения
3. ...
```

### ⚙️ Включение параллельной загрузки

```yaml
# KubeletConfiguration
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
serializeImagePulls: false              # ← включить параллельную загрузку
maxParallelImagePulls: 5                # ← макс 5 параллельных загрузок (v1.35+, stable)
```

### 🎯 Ограничения

| Ограничение | Описание |
|-------------|----------|
| **В пределах одного Pod** | Образы для init + app контейнеров **не** параллелятся |
| **Между разными Pod'ами** | ✅ Параллелятся (если `serializeImagePulls: false`) |
| **maxParallelImagePulls ≥ 2** | Требует `serializeImagePulls: false` |

### 🎯 Когда использовать

- **Быстрые сети** — ускорить запуск подов
- **Много образов** — уменьшить общее время загрузки
- **Осторожно**: может перегрузить сеть/диск

## Multi-arch images (image index)

> Один образ для **разных архитектур** (amd64, arm64, etc.).

### 🎯 Как работает

```
registry.k8s.io/pause:3.5 (image index)
├── pause-amd64:3.5
├── pause-arm64:3.5
├── pause-arm:3.5
└── pause-ppc64le:3.5
```

Kubelet автоматически выбирает правильный манифест для архитектуры ноды.

### 📝 Пример

```yaml
# Один и тот же манифест работает на amd64 и arm64
containers:
- name: pause
  image: registry.k8s.io/pause:3.5    # ← автоматически выберет правильную архитектуру
```

### 🎯 Обратная совместимость

- `pause` → multi-arch (image index)
- `pause-amd64` → только amd64 (для старых YAML)

## Private Registries — аутентификация

### 🎯 5 способов аутентификации

| Способ | Scope | Когда использовать |
|--------|-------|-------------------|
| **1. imagePullSecrets** (рекомендуется) | Pod-level | Стандартный подход, гибкость |
| **2. Node-level credentials** | Node-level | Все поды на ноде имеют доступ |
| **3. Credential provider plugin** | Node-level | Динамическое получение credentials |
| **4. Preloaded images** | Node-level | Образы уже на ноде |
| **5. Cloud-specific extensions** | Node-level | AWS ECR, GCP GCR, Azure ACR |

---

## 1. imagePullSecrets (рекомендуется)

### 🚀 Создание secret

```bash
# Способ 1: из CLI
kubectl create secret docker-registry my-registry-secret \
  --docker-server=my-registry.example.com \
  --docker-username=myuser \
  --docker-password=mypassword \
  --docker-email=myemail@example.com \
  --namespace=my-namespace

# Способ 2: из существующего .docker/config.json
kubectl create secret docker-registry my-registry-secret \
  --from-file=.dockerconfigjson=/home/user/.docker/config.json \
  --namespace=my-namespace
```

### 📝 Использование в Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  namespace: my-namespace
spec:
  containers:
  - name: app
    image: my-registry.example.com/my-app:v1.42.0
  imagePullSecrets:
  - name: my-registry-secret    # ← ссылка на secret в том же namespace
```

### 🎯 Автоматизация через ServiceAccount

```yaml
# Добавить imagePullSecrets в ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: default
  namespace: my-namespace
imagePullSecrets:
- name: my-registry-secret

# Теперь все поды в namespace будут использовать этот secret автоматически
```

### ⚠️ Ограничения

- Secret должен быть в **том же namespace**, что и Pod
- Каждый Pod должен ссылаться на secret явно (или через ServiceAccount)

## 2. Node-level credentials

### 🎯 Настройка на ноде

```bash
# Войти в registry на ноде
docker login my-registry.example.com

# Создать/обновить ~/.docker/config.json
cat ~/.docker/config.json
# {
#   "auths": {
#     "my-registry.example.com": {
#       "auth": "base64-encoded-credentials"
#     }
#   }
# }

# Скопировать на все ноды
scp ~/.docker/config.json user@node1:~/.docker/config.json
scp ~/.docker/config.json user@node2:~/.docker/config.json
```

### 🎯 Преимущества

- Все поды на ноде имеют доступ
- Не нужно указывать `imagePullSecrets` в каждом Pod

### ⚠️ Недостатки

- Требует root-доступ к нодам
- Не работает с auto-scaling (новые ноды не настроены)

## 3. Credential Provider Plugin

> Динамическое получение credentials от облачного провайдера (AWS ECR, GCP GCR, Azure ACR).

### 🎯 Как работает

```
1. Kubelet получает запрос на загрузку образа
2. Вызывает credential provider plugin
3. Plugin получает credentials от облака (через IAM, metadata API)
4. Kubelet использует credentials для загрузки
```

### ⚙️ Настройка kubelet

```yaml
# KubeletConfiguration
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
imageCredentialProviderConfig:
  - registryMatch: "*.dkr.ecr.*.amazonaws.com"
    credentialProvider: /usr/local/bin/ecr-credential-provider
  - registryMatch: "gcr.io"
    credentialProvider: /usr/local/bin/gcr-credential-provider
```

### 🎯 Преимущества

- Автоматическая ротация credentials
- Работает с auto-scaling
- Не нужно хранить credentials в K8s

---

## 4. Preloaded Images

> Образы **уже загружены** на ноду.

### 🎯 Как использовать

```bash
# Загрузить образ на ноду
docker pull my-registry.example.com/my-app:v1.42.0

# Установить imagePullPolicy: IfNotPresent или Never
```

```yaml
spec:
  containers:
  - name: app
    image: my-registry.example.com/my-app:v1.42.0
    imagePullPolicy: IfNotPresent    # ← использовать локальный образ
```

### ⚠️ Ограничения

- Нужно загрузить на **все ноды**
- Не работает с auto-scaling (новые ноды не имеют образов)
- Требует root-доступ

## KubeletEnsureSecretPulledImages (v1.35+, beta)

> Проверка credentials для **preloaded images**.

### 🎯 Проблема

Если образ уже на ноде (preloaded), kubelet **не проверяет** credentials → потенциальная уязвимость.

### ⚙️ Решение

```yaml
# KubeletConfiguration
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
imagePullCredentialsVerificationPolicy: AlwaysVerify    # ← всегда проверять credentials
```

### 🎯 Политики

| Политика | Поведение |
|----------|-----------|
| **`NeverVerify`** | Не проверять credentials для локальных образов |
| **`NeverVerifyPreloadedImages`** (по умолчанию) | Не проверять preloaded, но проверять остальные |
| **`NeverVerifyAllowListedImages`** | Не проверять preloaded + из allowlist |
| **`AlwaysVerify`** | Всегда проверять credentials |

## config.json — интерпретация

### 🎯 Отличие от Docker

Kubernetes поддерживает **wildcard patterns** в `config.json`:

```json
{
  "auths": {
    "my-registry.example.com": { "auth": "..." },
    "*.my-registry.example.com": { "auth": "..." },
    "*.kubernetes.io": { "auth": "..." }
  }
}
```

### 🎯 Правила matching

| Pattern | Match | No Match |
|---------|-------|----------|
| `*.kubernetes.io` | `abc.kubernetes.io` | `kubernetes.io` |
| `*.*.kubernetes.io` | `abc.def.kubernetes.io` | `abc.kubernetes.io` |
| `prefix.*.io` | `prefix.kubernetes.io` | — |
| `*-good.kubernetes.io` | `prefix-good.kubernetes.io` | — |

> ⚠️ Wildcard `*` должен соответствовать **одному** поддомену (точка обязательна).

## AlwaysPullImages Admission Controller

> Принудительно устанавливает `imagePullPolicy: Always` для всех подов.

### 🎯 Когда использовать

- **Multi-tenant кластер** — изоляция образов между tenants
- **Безопасность** — гарантировать, что поды используют актуальные образы

### ⚙️ Включение

```bash
# kube-apiserver flags
--enable-admission-plugins=AlwaysPullImages
```

### ⚠️ Последствия

- Все поды будут загружать образы при каждом запуске
- Увеличивается нагрузка на registry
- Увеличивается время запуска подов

## 🔹 Best Practices

### ✅ Делай

1. **Используй digest вместо `:latest`** в production:
   ```yaml
   image: my-registry.example.com/my-app@sha256:1ff6c18...
   ```

2. **Используй семантические версии** для тегов:
   ```yaml
   image: my-registry.example.com/my-app:v1.42.0
   ```

3. **Используй imagePullSecrets** для приватных реестров:
   ```yaml
   imagePullSecrets:
   - name: my-registry-secret
   ```

4. **Автоматизируй через ServiceAccount**:
   ```yaml
   apiVersion: v1
   kind: ServiceAccount
   imagePullSecrets:
   - name: my-registry-secret
   ```

5. **Используй credential provider** для облачных реестров (ECR, GCR, ACR).

6. **Включи параллельную загрузку** для ускорения:
   ```yaml
   serializeImagePulls: false
   maxParallelImagePulls: 5
   ```

7. **Мониторь ImagePullBackOff**:
   ```bash
   kubectl get pods --field-selector=status.phase!=Running | grep ImagePullBackOff
   ```

8. **Используй multi-arch images** для поддержки разных архитектур.

9. **Кэшируй образы** на нодах для ускорения (если возможно).

10. **Ротируй credentials** регулярно.

### ❌ Не делай

```yaml
# ❌ НЕ используй :latest в production
image: my-app:latest    # ← невозможно откатиться, непредсказуемо

# ❌ НЕ храни credentials в Git
# Используй imagePullSecrets или credential provider

# ❌ НЕ забывай указывать registry
image: my-app:v1.42.0    # ← будет искать в Docker Hub, а не в твоём registry

# ❌ НЕ используй imagePullPolicy: Never без preloaded images
# Под не запустится, если образа нет локально

# ❌ НЕ игнорируй ImagePullBackOff
# Это означает, что поды не запускаются

# ❌ НЕ используй AlwaysPullImages без необходимости
# Увеличивает нагрузку на registry и время запуска

# ❌ НЕ загружай образы вручную на ноды без автоматизации
# Новые ноды не будут иметь образов

# ❌ НЕ используй один secret для всех namespaces
# Secret должен быть в том же namespace, что и Pod
```

## Troubleshooting

### 🔍 Проблема 1: ImagePullBackOff

```bash
# 1. Проверить имя образа
kubectl describe pod my-app | grep Image
# Image: my-registry.example.com/my-app:v1.42.0

# 2. Проверить imagePullSecrets
kubectl get pod my-app -o jsonpath='{.spec.imagePullSecrets}'
# [{name: my-registry-secret}]

# 3. Проверить secret
kubectl get secret my-registry-secret -o yaml
# Проверить, что secret существует и содержит правильные credentials

# 4. Проверить доступность registry
kubectl exec my-app -- nslookup my-registry.example.com
# Или с ноды:
curl -I https://my-registry.example.com/v2/

# 5. Проверить логи kubelet
journalctl -u kubelet | grep -i "pull image"
```

### 🔍 Проблема 2: Unauthorized

```bash
# 1. Проверить credentials в secret
kubectl get secret my-registry-secret -o jsonpath='{.data.\.dockerconfigjson}' | base64 -d | jq

# 2. Пересоздать secret
kubectl delete secret my-registry-secret
kubectl create secret docker-registry my-registry-secret \
  --docker-server=my-registry.example.com \
  --docker-username=myuser \
  --docker-password=mypassword

# 3. Пересоздать под
kubectl delete pod my-app
kubectl apply -f pod.yaml
```

### 🔍 Проблема 3: Медленная загрузка образов

```bash
# 1. Включить параллельную загрузку
kubectl edit node <node-name>
# Или через KubeletConfiguration:
# serializeImagePulls: false
# maxParallelImagePulls: 5

# 2. Проверить сеть
kubectl exec my-app -- ping my-registry.example.com

# 3. Кэшировать образы на нодах
# Использовать preloaded images или image cache

# 4. Использовать CDN для registry
# Или зеркало registry ближе к кластеру
```

### 🔍 Проблема 4: Multi-arch не работает

```bash
# 1. Проверить, что образ поддерживает архитектуру ноды
docker manifest inspect my-registry.example.com/my-app:v1.42.0
# Должны быть манифесты для amd64, arm64, etc.

# 2. Проверить архитектуру ноды
kubectl get node <node-name> -o jsonpath='{.status.nodeInfo.architecture}'
# amd64 или arm64

# 3. Использовать multi-arch образ
# Или указать конкретную архитектуру:
image: my-registry.example.com/my-app-amd64:v1.42.0
```

## Ключевые выводы

1. **Имя образа** = `[registry/]name[:tag|@digest]`. По умолчанию — Docker Hub.
2. **Tag** — логическая версия (может меняться). **Digest** — хеш содержимого (неизменяемый).
3. **imagePullPolicy**: `Always`, `IfNotPresent`, `Never`. Автоматически устанавливается в зависимости от тега.
4. **ImagePullBackOff** — ошибка загрузки образа. Причины: неверное имя, нет credentials, registry недоступен.
5. **Параллельная загрузка** (`serializeImagePulls: false`, `maxParallelImagePulls: 5`) — ускоряет запуск подов.
6. **Multi-arch images** — один образ для разных архитектур (amd64, arm64).
7. **Private registries**: 5 способов аутентификации. **imagePullSecrets** — рекомендуемый.
8. **Credential provider** — динамическое получение credentials от облака (ECR, GCR, ACR).
9. **Preloaded images** — образы уже на ноде. Требует `imagePullPolicy: IfNotPresent` или `Never`.
10. **KubeletEnsureSecretPulledImages** (v1.35+) — проверка credentials для preloaded images.
11. **config.json** — Kubernetes поддерживает wildcard patterns (в отличие от Docker).
12. **AlwaysPullImages** — admission controller для принудительной загрузки образов (для multi-tenant).
13. **Best practices**: digest вместо `:latest`, imagePullSecrets, credential provider, параллельная загрузка, мониторинг.
14. **Troubleshooting**: проверять имя образа, credentials, доступность registry, логи kubelet.
15. **Автоматизация**: ServiceAccount с imagePullSecrets, CI/CD для сборки образов, ротация credentials.
