>Метки и селекторы — это «клей», который связывает объекты в Kubernetes в единую систему. Без них невозможно масштабирование, сетевое взаимодействие и управление жизненным циклом.

# Метки (Labels) и селекторы в Kubernetes

> 📌 **Метки** = пары `ключ=значение` для организации объектов. 
> **Селекторы** = фильтры для выбора объектов по меткам. 
> Это основной механизм группировки в K8s: от `Service` → `Pod` до `Deployment` → `ReplicaSet`.

---

## 🔹 Что такое метки (Labels)

| Аспект | Описание |
|--------|----------|
| **Определение** | Пары ключ/значение, прикрепляемые к объектам (`Pod`, `Node`, `Service` и др.) |
| **Назначение** | Организация, фильтрация, выбор подмножеств объектов по пользовательским критериям |
| **Семантика** | Не несут смысла для ядра K8s — это «теги» для людей и инструментов |
| **Гибкость** | Можно добавлять при создании и изменять в любое время |

### 📋 Пример метаданных с метками
```yaml
metadata:
  name: my-app
  labels:
    app.kubernetes.io/name: my-app      # стандартное имя приложения
    app.kubernetes.io/instance: prod-1  # конкретный экземпляр
    environment: production             # среда
    tier: frontend                      # уровень архитектуры
    track: stable                       # трек релиза
```

> 💡 **Метки ≠ Аннотации**: 
> - **Метки** — для выбора объектов (`labelSelector`), должны быть краткими и структурированными
> - **Аннотации** — для хранения произвольных метаданных (описания, контакты, ссылки), не используются в селекторах

## 🔹 Синтаксис и правила именования меток

### Структура ключа: `[префикс/]имя`

| Часть | Правила | Пример |
|-------|---------|--------|
| **Префикс** *(опционально)* | • Поддомен DNS (RFC 1123)<br>• Макс. 253 символа<br>• Завершается `/` | `app.kubernetes.io/`, `example.com/` |
| **Имя** *(обязательно)* | • Макс. 63 символа<br>• `[a-z0-9A-Z]`, `-`, `_`, `.`<br>• Начинается и заканчивается буквенно-цифровым | `name`, `version`, `environment` |
| **Значение** *(опционально)* | • Макс. 63 символа (может быть пустым)<br>• Те же символы, что и для имени | `production`, `v1.2.3`, `us-east-1` |

### 🚫 Зарезервированные префиксы
```
kubernetes.io/*   → ядро Kubernetes
k8s.io/*          → ядро Kubernetes
```
> ⚠️ Не используй эти префиксы для своих меток — они могут быть перезаписаны системой.

### ✅ Примеры валидных меток
```yaml
labels:
  app.kubernetes.io/name: payment-service    # стандарт K8s
  example.com/owner: alice                   # свой префикс
  environment: production                    # без префикса = личная метка
  version: v2.1.0
  canary: "true"                             # значение как строка
```

### ❌ Примеры невалидных меток
```yaml
labels:
  My-App: nginx           # ❌ заглавные буквы в ключе
  app/name: backend       # ❌ косая черта в имени (только в префиксе)
  very-long-key-name-that-exceeds-sixty-three-characters-limit: value  # ❌ >63 символа
```

## 🔹 Селекторы меток (Label Selectors)

> Селектор = фильтр для выбора объектов по меткам. Основной механизм группировки в K8s.

### Два типа селекторов

| Тип | Синтаксис | Пример | Описание |
|-----|-----------|--------|----------|
| **🔹 На основе равенства** | `key=value`, `key!=value` | `environment=production` | Выбор по точному совпадению ключа и значения |
| **🔹 На основе множеств** | `key in (v1,v2)`, `key notin (v1)`, `key`, `!key` | `tier in (frontend,backend)` | Выбор по принадлежности значения к набору |

### 🔗 Комбинирование требований
- Запятая `,` = логическое **И** (`AND`)
- Логическое **ИЛИ** (`OR`) внутри одного ключа — только через `in (v1,v2)`
- ❌ Нет оператора `||` между разными ключами

```bash
# ✅ Выбрать: environment=production И tier!=frontend
kubectl get pods -l environment=production,tier!=frontend

# ✅ Выбрать: environment в (production,qa) ИЛИ-условие по значениям
kubectl get pods -l 'environment in (production,qa)'

# ✅ Выбрать: есть метка partition И environment не в (dev,staging)
kubectl get pods -l 'partition,environment notin (dev,staging)'

# ❌ Нельзя: (environment=prod) ИЛИ (environment=qa) как отдельные условия
# → используй: environment in (prod,qa)
```

## 🔹 Использование селекторов в объектах API

### 🔸 matchLabels (равенство) — поддерживается везде
```yaml
# Service: выбрать поды для балансировки
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app: my-app          # ← эквивалентно: app=my-app
    tier: frontend
  ports:
  - port: 80
```

### 🔸 matchExpressions (множества) — поддерживается в новых ресурсах
```yaml
# Deployment: более гибкий селектор
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-deployment
spec:
  selector:
    matchLabels:
      app: my-app
    matchExpressions:
      - { key: tier, operator: In, values: [frontend, backend] }
      - { key: environment, operator: NotIn, values: [dev] }
      - { key: canary, operator: Exists }        # есть любая метка canary
      - { key: debug, operator: DoesNotExist }   # нет метки debug
  template:
    # ... pod template
```

> ⚠️ **Важно**: все требования из `matchLabels` и `matchExpressions` объединяются через **И**.

## 🔹 Практика: работа с метками через kubectl

### 🔍 Выбор объектов по меткам
```bash
# Базовый синтаксис: -l или --selector
kubectl get pods -l app=nginx
kubectl get pods -l 'environment in (prod,qa)'

# Несколько меток (логическое И)
kubectl get pods -l app=nginx,environment=production

# Отрицание
kubectl get pods -l app!=nginx

# Показать значения меток в виде столбцов
kubectl get pods -L app -L environment -L tier
# NAME     READY  STATUS  AGE  APP    ENVIRONMENT  TIER
# pod-1    1/1    Running 10m  nginx  production   frontend

# Комбинация: выбор + отображение меток
kubectl get pods -l app=nginx -L tier
```

### 🏷️ Управление метками
```bash
# Добавить/изменить метку у одного объекта
kubectl label pod my-pod environment=production

# Добавить метку нескольким объектам по селектору
kubectl label pods -l app=nginx tier=frontend

# Удалить метку (добавь "-" в конце значения)
kubectl label pod my-pod environment-

# Переименовать метку (удалить старую + добавить новую)
kubectl label pod my-pod old-key- new-key=value

# Просмотреть метки объекта
kubectl get pod my-pod --show-labels
kubectl get pod my-pod -o jsonpath='{.metadata.labels}'
```

### 🧪 Валидация селекторов
```bash
# Проверить, какие поды выберет Service
kubectl get endpoints my-service  # покажет выбранные поды

# Протестировать селектор без применения изменений
kubectl get pods -l 'app=my-app,tier in (frontend,backend)' --dry-run=client

# Посмотреть, как контроллер «видит» поды (для отладки Deployment)
kubectl describe deployment my-app | grep -A5 'Selector:'
```

## 🔹 Best Practices: эффективное использование меток

### 🏗️ Рекомендуемая схема меток (по стандартам K8s)
```yaml
labels:
  # Обязательные (app.kubernetes.io/*) — для инструментов и автоматизации
  app.kubernetes.io/name: my-app          # имя приложения
  app.kubernetes.io/instance: prod-1      # уникальный экземпляр
  app.kubernetes.io/version: v1.2.3       # версия
  app.kubernetes.io/component: backend    # компонент внутри приложения
  app.kubernetes.io/part-of: my-ecosystem # группа связанных приложений
  app.kubernetes.io/managed-by: helm      # инструмент управления

  # Дополнительные (пользовательские) — для твоих нужд
  environment: production
  tier: backend
  team: platform
  cost-center: engineering
```

> 📚 Подробнее: [Kubernetes Label Recommendations](https://kubernetes.io/docs/concepts/overview/working-with-objects/common-labels/)

### 🎯 Стратегии группировки

| Сценарий | Рекомендуемые метки | Пример селектора |
|----------|-------------------|-----------------|
| **Мультитенант** | `tenant=customer-a`, `env=prod` | `tenant=customer-a,env=prod` |
| **Canary-релизы** | `app=api`, `track=canary`, `version=v2` | `app=api,track=canary` |
| **Микросервисы** | `app.kubernetes.io/name=payment`, `tier=backend` | `app.kubernetes.io/name=payment` |
| **Гео-распределение** | `region=eu-west-1`, `zone=a` | `region=eu-west-1` |
| **Отладка/тесты** | `debug=true`, `test-run=2024-06-05` | `debug=true` |

### ⚠️ Частые ошибки и как их избежать

```yaml
# ❌ Слишком общие метки — сложно фильтровать
labels:
  type: important  # ← что это значит? для кого?

# ✅ Конкретные, структурированные метки
labels:
  app.kubernetes.io/name: payment-service
  sla-tier: critical
  team: payments

# ❌ Использование имени объекта как метки (избыточно)
labels:
  pod-name: my-pod-123  # ← имя уже есть в metadata.name

# ✅ Метки для группировки, имя — для уникальности
metadata:
  name: my-pod-123
  labels:
    app: my-app
    instance-id: "123"  # если действительно нужно

# ❌ Изменение меток, на которые есть ссылки (Service, NetworkPolicy)
# → может «отвязать» под от сервиса!

# ✅ Перед изменением: проверь, кто использует селектор
kubectl get service,deployment,networkpolicy -A --selector app=my-app
```

## 🔹 Чек-лист: работа с метками и селекторами

```bash
# ✅ При создании ресурса: сразу добавляй структурированные метки
kubectl apply -f deployment.yaml  # где в metadata.labels есть app.kubernetes.io/*

# ✅ Перед изменением меток: проверь зависимости
kubectl get service,deployment,networkpolicy -l app=my-app --all-namespaces

# ✅ Для отладки селекторов: используй --show-labels и -L
kubectl get pods -l app=my-app --show-labels
kubectl get pods -l app=my-app -L tier,environment

# ✅ При написании манифестов: используй matchExpressions для гибкости
# (особенно в Deployment, StatefulSet, DaemonSet)

# ✅ Для массовых операций: тестируй селектор перед применением
kubectl get pods -l 'env=prod,tier=frontend'  # сначала посмотри, что попадёт
# потом:
kubectl label pods -l 'env=prod,tier=frontend' maintenance=true

# ✅ В CI/CD: добавляй метки для трассировки
kubectl label pod my-pod \
  ci-run-id=${GITHUB_RUN_ID} \
  deployed-by=argocd \
  deploy-time=$(date -u +%Y%m%dT%H%M%SZ)

# ✅ Документируй принятые стандарты меток в команде
# (файл `LABELS_CONVENTION.md` в репозитории)
```

> 💡 **Совет для конспекта**:
> 1. Создай файл `00_labels_convention.md` с принятыми в твоих проектах стандартами меток.
> 2. Добавь блок «Шаблоны селекторов»: частые комбинации, которые ты используешь (`app=*,tier=frontend` и т.д.).
> 3. Веди заметку «Зависимости по меткам»: какие сервисы/деплои/политики ссылаются на какие метки — чтобы не сломать при рефакторинге.

## 🔹 Ключевые выводы

1. **Метки — это теги, а не идентификаторы**: они не гарантируют уникальность, но идеальны для группировки.
2. **Селекторы — основа связности**: `Service` → `Pod`, `Deployment` → `ReplicaSet`, `NetworkPolicy` → трафик.
3. **Стандартизируй**: используй `app.kubernetes.io/*` для совместимости с инструментами (Helm, Kustomize, ArgoCD).
4. **Тестируй селекторы**: перед массовым применением меток всегда проверяй, какие объекты попадут под фильтр.
5. **Не меняй метки «на лету»**, если на них есть ссылки — это может разорвать связь между объектами.
