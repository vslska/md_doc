# Kubernetes — Архитектура и компоненты кластера

> *📌* Кластер K8s = **Control Plane** (мозг: принимает решения) + **Worker Nodes** (рабочие: запускают контейнеры) + **Add-ons** (расширения: DNS, сеть, мониторинг). Вы описываете *желаемое состояние*, система через API приводит к нему *фактическое*.

---

## Высокоуровневая схема
```
                ┌─────────────────────────┐
                │     Control Plane       │
                │  (управление кластером) │
                ├─────────────────────────┤
                │ • kube-apiserver        │
                │ • etcd                  │
                │ • kube-scheduler        │
                │ • kube-controller-mgr   │
                │ • cloud-controller-mgr* │
                └──────────┬──────────────┘
                           │ HTTPS/API
    ┌──────────────────────┼──────────────────────┐
    ▼                      ▼                      ▼
│   Node 1    │    │   Node 2    │    │   Node N    │
├─────────────┤    ├─────────────┤    ├─────────────┤
│ • kubelet   │    │ • kubelet   │    │ • kubelet   │
│ • kube-proxy│    │ • kube-proxy│    │ • kube-proxy│
│ • Runtime   │    │ • Runtime   │    │ • Runtime   │
│ • Pods ─────┼───►  • Pods ─────┼───►  • Pods ─────┤
│   [app A]   │    │   [app B]   │    │   [app C]   │
└─────────────┘    └─────────────┘    └─────────────┘
```

> \* `cloud-controller-manager` — только при работе в облаке (AWS, GCP, Azure и др.).

## Контрольная плоскость (Control Plane)

> Принимает глобальные решения: что запустить, где, как масштабировать, как реагировать на сбои.

| Компонент | Роль | Ключевые особенности |
|-----------|------|---------------------|
| **`kube-apiserver`** | 🚪 Единая точка входа в API | • Обрабатывает REST-запросы (`kubectl`, UI, контроллеры)<br>• Аутентификация, авторизация, валидация, admission-контроль<br>• Масштабируется горизонтально (несколько инстансов + LB)<br>• **Единственный** компонент, пишущий напрямую в `etcd` |
| **`etcd`** | 💾 Надёжное KV-хранилище | • Хранит **всё** состояние кластера: объекты, конфиги, секреты, статусы<br>• Распределённая система на основе Raft-консенсуса<br>• **Критичен для HA**: рекомендуется 3/5 нод, регулярные бэкапы (`etcdctl snapshot`) |
| **`kube-scheduler`** | 🗓️ Планировщик подов | • Находит подходящую ноду для новых подов (без привязки)<br>• Учитывает: CPU/RAM, affinity/anti-affinity, taints/tolerations, политики, locality<br>• Не запускает контейнеры — только принимает решение «куда» |
| **`kube-controller-manager`** | 🎮 Мозг бизнес-логики | • Запускает контроллеры: `Node`, `Replication`, `Job`, `EndpointSlice`, `ServiceAccount` и др.<br>• Реализует **Reconciliation Loop**: непрерывно сравнивает желаемое и фактическое состояние |
| **`cloud-controller-manager`** *(опционально)* | ☁️ Интеграция с облаком | • Выносит облачно-специфичную логику: `Node` (проверка удаления ВМ), `Route` (маршруты), `Service` (балансировщики)<br>• Позволяет развивать K8s и облачные провайдеры независимо |

> 💡 **Размещение Control Plane**:
> - **Dev/learning**: все компоненты на одной машине (`minikube`, `kind`)
> - **Production**: компоненты распределены по нескольким нодам для отказоустойчивости (HA)
> - **Managed (GKE/EKS/AKS)**: провайдер управляет Control Plane за вас

## Компоненты рабочего узла (Node Components)

> Запускаются на каждой рабочей ноде, обеспечивают выполнение подов и сетевое взаимодействие.

| Компонент                                          | Роль                  | Важные детали                                                                                                                                                                                                                                                 |
| -------------------------------------------------- | --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`kubelet`**                                      | 🤖 Агент узла         | • Гарантирует запуск контейнеров в подах согласно `PodSpec`<br>• Отправляет метрики и статусы в API Server (`Node`, `Pod` статусы)<br>• Выполняет `liveness`/`readiness`/`startup` пробы<br>• **Не управляет** контейнерами, созданными не через K8s          |
| **`kube-proxy`** *(опционально, но рекомендуется)* | 🌐 Сетевой прокси     | • Поддерживает правила `iptables`/`IPVS` для реализации сервисов (ClusterIP, NodePort, LoadBalancer)<br>• Обеспечивает балансировку трафика внутри кластера<br>• Можно заменить CNI-плагином с аналогичной функциональностью (например, Cilium в eBPF-режиме) |
| **`Container Runtime`**                            | 📦 Запуск контейнеров | • Управляет жизненным циклом контейнеров: запуск, остановка, удаление, логи, метрики<br>• Поддерживаемые: `containerd`, `CRI-O`, `Docker Engine` (через CRI-shim — устарело)<br>• Должен соответствовать **CRI** (Container Runtime Interface)                |

> ⚙️ **Дополнительно на ноде**:
> - `systemd`/`init` — управление процессами компонентов
> - `cni-plugins` — сетевые плагины (Calico, Cilium, Flannel)
> - `csi-drivers` — драйверы хранилищ (при использовании облачных/сетевых дисков)
> - `device-plugins` — доступ к специфичному железу (GPU, FPGA)

## Add-ons: расширения кластера

> Не входят в ядро K8s, но критичны для полноценной работы в production. Разворачиваются как обычные ресурсы (Deployment, DaemonSet) в неймспейсе `kube-system`.

| Add-on | Назначение | Популярные реализации |
|--------|------------|----------------------|
| **🔤 Cluster DNS** | DNS-разрешение для сервисов и подов внутри кластера | `CoreDNS` (стандарт), `kube-dns` (устарел) |
| **🖥️ Dashboard / UI** | Веб-интерфейс для управления и отладки | `Kubernetes Dashboard`, `Lens`, `Octant`, `K9s` (CLI) |
| **📊 Resource Monitoring** | Сбор метрик с нод, подов, контроллеров | `Metrics Server` (для HPA), `Prometheus` + `Grafana`, `VictoriaMetrics` |
| **🗃️ Cluster Logging** | Централизованный сбор и поиск логов контейнеров | `Fluentd`/`Fluent Bit` + `Elasticsearch`, `Loki` + `Grafana` |
| **🔌 CNI Plugins** | Реализация сетевой модели: IP-адресация, политики, шифрование | `Calico`, `Cilium`, `Flannel`, `Weave Net`, `Antrea` |
| **🔐 Ingress Controller** | Управление внешним HTTP/HTTPS-трафиком (L7) | `nginx-ingress`, `Traefik`, `HAProxy`, облачные LB (ALB, GLB) |
| **🔑 CSI Drivers** | Динамическое подключение хранилищ (PV/PVC) | `aws-ebs-csi`, `gcp-pd-csi`, `ceph-csi`, `longhorn` |

> 📦 **Установка**: большинство add-on разворачиваются через `kubectl apply -f manifest.yaml` или Helm-чарты.

## Гибкость архитектуры: варианты развёртывания

### 🎛️ Deployment Control Plane

| Вариант | Описание | Когда использовать |
|---------|----------|-------------------|
| **Traditional** | Компоненты как `systemd`-сервисы на выделенных ВМ/железе | Полный контроль, on-premise, кастомные требования безопасности |
| **Static Pods** | Компоненты как статические поды, управляемые `kubelet` на мастер-нодах | `kubeadm`, баланс между контролем и простотой (де-факто стандарт для self-hosted) |
| **Self-hosted** | Control Plane работает как обычные поды внутри кластера (управляется K8s) | Эксперименты, динамическое обновление (сложно в production, требует bootstrap) |
| **Managed** | Control Plane управляется провайдером (GKE, EKS, AKS, Yandex Managed Kubernetes) | Быстрый старт, минимальные операционные затраты, фокус на приложениях |

### Размещение рабочих нагрузок

| Сценарий | Архитектура | Плюсы / Минусы |
|----------|-------------|----------------|
| **Dev / Small cluster** | Control Plane + workloads на одних нодах | ✅ Простота, экономия ресурсов / ❌ Меньше изоляции, риск конкуренции за ресурсы |
| **Production** | Выделенные ноды для Control Plane, отдельные — для workloads | ✅ Изоляция, безопасность, стабильность / ❌ Больше ресурсов, сложнее управление |
| **Critical add-ons** | Мониторинг/логирование/сеть на нодах Control Plane | ✅ Надёжность сбора метрик / ❌ Риск влияния на стабильность управления |

## Гибкость и расширяемость: как адаптировать под свои нужды

| Возможность | Как реализовать | Пример использования |
|-------------|----------------|---------------------|
| **Кастомный планировщик** | Запустить альтернативный `scheduler` рядом с дефолтным или вместо него | Гео-аффинити, стоимость, специфичные политики размещения |
| **Расширение API** | `CustomResourceDefinitions` (CRD) + `API Aggregation` + Webhooks | Операторы (Patroni, Cert-Manager), декларативное управление приложением |
| **Глубокая интеграция с облаком** | `cloud-controller-manager` + специфичные контроллеры | Авто-создание балансировщиков, управление томами, авто-ремедиация нод |
| **Кастомная сеть/хранение** | Выбор CNI/CSI-плагинов, настройка политик | Шифрование трафика, мульти-кластерная сеть, compliance-требования |
| **Policy as Code** | Admission controllers (OPA/Gatekeeper, Kyverno) | Валидация/мутация ресурсов, безопасность, аудит |

> 💡 **Принцип**: K8s предоставляет примитивы, вы собираете из них решение под свою задачу. Не нужно использовать всё сразу — начинайте с минимума и добавляйте по мере роста сложности.

## Чек-лист: быстрая диагностика кластера

```bash
# 1. Статус нод и распределение подов
kubectl get nodes -o wide
kubectl get pods -A -o wide | grep -v Running

# 2. Компоненты Control Plane (если есть доступ к kube-system)
kubectl get pods -n kube-system | grep -E 'apiserver|etcd|scheduler|controller'

# 3. Runtime и версия kubelet на нодах
kubectl get nodes -o jsonpath='{.items[*].status.nodeInfo.containerRuntimeVersion}'
kubectl get nodes -o jsonpath='{.items[*].status.nodeInfo.kubeletVersion}'

# 4. Сетевые и DNS-аддоны
kubectl get pods -n kube-system -l k8s-app=kube-dns           # DNS
kubectl get pods -n kube-system -l tier=node | grep -E 'calico|cilium|flannel'  # CNI

# 5. Cloud-integration (если используется)
kubectl get pods -n kube-system | grep cloud-controller

# 6. Критичные аддоны: мониторинг, логирование, ingress
kubectl get deployments,daemonsets -n kube-system | grep -E 'metrics|prometheus|loki|ingress'

# 7. Проверка API: какие ресурсы доступны
kubectl api-resources --verbs=list --namespaced -o name | head -20

# 8. Dry-run валидации манифеста (без применения)
kubectl apply -f deploy.yaml --dry-run=server -o yaml
```
