>Сборка мусора (Garbage Collection) — это набор механизмов Kubernetes для автоматической очистки ресурсов, которые больше не нужны. Критически важно для поддержания здоровья кластера и эффективного использования ресурсов.

# Сборка мусора (Garbage Collection) в Kubernetes

> 📌 **GC в K8s** = два уровня очистки: (1) **объекты API** (через ownerReferences + финализаторы) и (2) **контейнеры/образы на нодах** (через kubelet). Понимание каскадного удаления и политик GC критично для управления ресурсами и предотвращения «утечек».

---

## 🔹 Что очищает сборщик мусора в Kubernetes

| Тип ресурса | Когда удаляется | Механизм |
|-------------|----------------|----------|
| **🧱 Завершённые поды** | После завершения работы (успех/ошибка) | `kubectl delete` или автоматическая политика (TTLController) |
| **✅ Завершённые задания (Job)** | После успешного выполнения или по `ttlSecondsAfterFinished` | Job Controller + GC |
| **🔗 Объекты без ownerReferences** | Если нет владельца и не защищены финализаторами | Garbage Collector (контроллер в Control Plane) |
| **🗑️ Неиспользуемые контейнеры** | После завершения, старше `minAge` | kubelet на ноде |
| **🖼️ Неиспользуемые образы** | При нехватке места или старше `imageMaximumGCAge` | kubelet + image manager |
| **💾 Динамические PV с политикой `Delete`** | После удаления PVC | PV Controller + CSI-драйвер |
| **📜 Устаревшие CSR** | После истечения срока или обработки | CSR Controller |
| **🖥️ Удалённые узлы** | Когда нода удалена из облака/инфраструктуры | Node Controller + CCM |
| **🔑 Lease-объекты нод** | После удаления ноды или истечения срока | Координационный контроллер |

> 💡 **Ключевая идея**: в K8s **два независимых уровня GC**:
> 1. **Control Plane**: удаляет объекты API через `ownerReferences` и финализаторы
> 2. **Node (kubelet)**: удаляет контейнеры и образы на диске для освобождения места

## 🔹 Уровень 1: Сборка мусора объектов API

### 🔗 OwnerReferences: связь «владелец → зависимый»

```yaml
# Пример: Pod, созданный ReplicaSet
apiVersion: v1
kind: Pod
metadata:
  name: my-pod-abc123
  ownerReferences:
  - apiVersion: apps/v1
    kind: ReplicaSet
    name: my-app-xyz789
    uid: a1b2c3d4-e5f6-7890
    controller: true
    blockOwnerDeletion: true  # ← может блокировать удаление владельца
```

| Поле | Описание |
|------|----------|
| `apiVersion`, `kind`, `name`, `uid` | Идентификатор владельца (обязательны) |
| `controller: true` | Этот владелец — основной контроллер (может быть только один) |
| `blockOwnerDeletion: true` | Зависимый объект может блокировать удаление владельца до своей очистки |

### ⚠️ Правила области видимости (namespace scoping)

| Зависимый объект | Владелец | Допустимо? | Последствия |
|-----------------|----------|-----------|-------------|
| **Namespaced** (Pod) | Namespaced, тот же NS | ✅ Да | Стандартный случай |
| **Namespaced** | Namespaced, другой NS | ❌ Нет | Ссылка игнорируется, зависимый может быть удалён |
| **Namespaced** | Cluster-scoped (Node) | ✅ Да | Разрешено, но редко используется |
| **Cluster-scoped** (Node) | Namespaced (любой) | ❌ Нет (с K8s 1.20+) | Событие `OwnerRefInvalidNamespace`, объект не удаляется через GC |

```bash
# Проверить события о невалидных ownerReferences
kubectl get events -A --field-selector reason=OwnerRefInvalidNamespace
```

## 🔹 Каскадное удаление (Cascading Deletion)

При удалении владельца можно управлять судьбой зависимых объектов.

### 🎛️ Три политики удаления

| Политика | Флаг `kubectl` | Поведение | Когда использовать |
|----------|---------------|-----------|-------------------|
| **🗑️ Foreground** | `--cascade=foreground` | Владелец ждёт, пока **все** зависимые удалятся (через финализаторы) | Когда нужна гарантия полной очистки перед удалением владельца |
| **🔄 Background** | `--cascade=background` *(по умолчанию)* | Владелец удаляется сразу, зависимые удаляются асинхронно | Стандартный случай: баланс скорости и надёжности |
| **🪶 Orphan** | `--cascade=orphan` | Зависимые **не удаляются**, становятся «сиротами» | Когда нужно сохранить данные/ресурсы при удалении владельца |

### 🔄 Как работает Foreground удаление

```mermaid
graph LR
    A[Пользователь: delete --cascade=foreground] --> B[API Server]
    B --> C[Устанавливает:\n• deletionTimestamp\n• finalizers: [foregroundDeletion]]
    C --> D[Объект в состоянии Terminating]
    D --> E[Garbage Collector видит зависимые с blockOwnerDeletion: true]
    E --> F[Удаляет зависимые объекты]
    F --> G{Все зависимые удалены?}
    G -->|Да| H[Удаляет финализатор с владельца]
    H --> I[API Server удаляет владельца из etcd]
    G -->|Нет| D
```

### 💻 Примеры использования

```bash
# Foreground: ждать полной очистки
kubectl delete deployment my-app --cascade=foreground

# Background: удалить владельца, зависимые удалятся позже (по умолчанию)
kubectl delete deployment my-app

# Orphan: удалить владельца, но сохранить поды/ReplicaSet
kubectl delete deployment my-app --cascade=orphan

# Проверить, что поды остались, но без владельца
kubectl get pods -l app=my-app -o jsonpath='{.items[*].metadata.ownerReferences}'
# → пустой вывод: владелец отвязан
```

> 💡 **Совет**: используй `--dry-run=server` для проверки, что будет удалено:
> ```bash
> kubectl delete deployment my-app --cascade=foreground --dry-run=server -v=6
> ```

## 🔹 Финализаторы и сборка мусора

Финализаторы позволяют выполнить «чистую уборку» перед удалением объекта.

### 🎯 Сценарий: защита от случайного удаления

```yaml
# PersistentVolume с финализатором
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
  finalizers:
  - kubernetes.io/pv-protection  # ← не даст удалить, пока том используется
spec:
  # ...
```

```
Сценарий:
1. Пользователь: kubectl delete pv my-pv
2. PV переходит в Terminating, но не удаляется (есть финализатор)
3. Контроллер проверяет: том привязан к PVC? используется подом?
4. Если свободен → удаляет финализатор → PV удаляется физически
5. Если занят → ждёт освобождения
```

### 🔧 Ручное управление финализаторами (осторожно!)

```bash
# Посмотреть финализаторы объекта
kubectl get pv my-pv -o jsonpath='{.metadata.finalizers}'

# Удалить финализатор (только если понимаешь последствия!)
kubectl patch pv my-pv --type json -p '[{"op": "remove", "path": "/metadata/finalizers/0"}]'

# Отладка «зависшего» объекта
kubectl get pv my-pv -o yaml | grep -E 'deletionTimestamp|finalizers'
kubectl get events --field-selector involvedObject.name=my-pv
```

> ⚠️ **Важно**: не удаляй финализаторы вручную без понимания их назначения — можно потерять внешние ресурсы или оставить «висячие» ссылки.

## 🔹 Уровень 2: Сборка мусора на нодах (контейнеры и образы)

Управляется **kubelet** на каждой ноде, независимо от Control Plane.

### 🗑️ Что и когда удаляет kubelet

| Тип | Критерии удаления | Частота проверки |
|-----|------------------|-----------------|
| **📦 Завершённые контейнеры** | • Старше `minAge` (по умолчанию 0)<br>• Превышено `maxPerPodContainer`<br>• Превышено `maxContainers` глобально | Каждую минуту |
| **🖼️ Неиспользуемые образы** | • Использование диска > `HighThresholdPercent` (по умолчанию 85%)<br>• Образ не использовался > `imageMaximumGCAge` (по умолчанию 0 = отключено) | Каждые 5 минут |

### ⚙️ Настройка GC через KubeletConfiguration

```yaml
# /var/lib/kubelet/config.yaml
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration

# Пороги использования диска для очистки образов
imageGCHighThresholdPercent: 85    # Начать очистку при 85% занято
imageGCLowThresholdPercent: 80     # Остановить, когда освободится до 80%

# Максимальный возраст образа без использования
imageMaximumGCAge: 24h             # Удалять образы, не использовавшиеся > 24 часов

# Параметры для завершённых контейнеров
containerGCHighThresholdPercent: 85
containerGCLowThresholdPercent: 80
minimumContainerAge: 0             # Минимальный возраст контейнера перед удалением
maxPerPodContainer: 1              # Макс. завершённых контейнеров на под
maxContainers: -1                  # Макс. завершённых контейнеров на ноде (-1 = без лимита)
```

### 🔄 Жизненный цикл образа: как kubelet принимает решение

```
1. Образ скачан и использован для запуска контейнера
2. Контейнер завершён → образ помечен как «кандидат на удаление»
3. Каждые 5 минут kubelet проверяет:
   • Использование диска > HighThresholdPercent?
   • Образ не использовался > imageMaximumGCAge?
4. Если да → удаляет образы в порядке LRU (least recently used)
5. Останавливается, когда использование диска ≤ LowThresholdPercent
```

> ⚠️ **Важно**: `imageMaximumGCAge` сбрасывается при перезапуске kubelet — образ снова считается «новым». Это может задержать очистку.

## 🔹 Практика: управление и отладка сборки мусора

### 🔍 Мониторинг состояния

```bash
# Проверить, какие образы занимают место на ноде
# (требуется доступ к ноде или crictl)
crictl images | head -20

# Посмотреть завершённые контейнеры
crictl ps -a --state exited | head -10

# Проверить использование диска на ноде
kubectl top node <node-name>
# или через SSH:
ssh <node> df -h /var/lib/containerd

# Проверить события, связанные с удалением объектов
kubectl get events -A --field-selector reason=SuccessfulDelete | head -20
```

### 🧹 Ручная очистка (если нужно срочно освободить место)

```bash
# Удалить завершённые поды вручную (если нет автоматической политики)
kubectl delete pod --field-selector=status.phase=Succeeded -A
kubectl delete pod --field-selector=status.phase=Failed -A

# Очистить старые образы на ноде (через crictl)
ssh <node> 'crictl rmi --prune'

# Удалить завершённые контейнеры
ssh <node> 'crictl rm $(crictl ps -a --state exited -q)'

# ⚠️ Осторожно: не удаляй образы/контейнеры, которые могут понадобиться!
```

### ⚙️ Настройка автоматической очистки завершённых подов

```yaml
# TTLController для Jobs (встроенная функция)
apiVersion: batch/v1
kind: Job
metadata: { name: my-job }
spec:
  ttlSecondsAfterFinished: 3600  # Удалить объект через 1 час после завершения
  template:
    spec:
      containers:
      - name: task
        image: busybox
        command: ["echo", "done"]
      restartPolicy: Never

# Для подов без контроллера: использовать сторонние операторы
# (например, kube-janitor, odera) или CronJob для периодической очистки
```

### 🔧 Отладка проблем с очисткой

```bash
# 1. Образ не удаляется, хотя место закончилось:
# • Проверить пороги в конфиге kubelet
cat /var/lib/kubelet/config.yaml | grep -i threshold

# • Проверить, когда образ последний раз использовался
crictl inspect <image-id> | grep -i lastused

# 2. Контейнеры не очищаются:
# • Проверить параметры GC в kubelet
journalctl -u kubelet | grep -i "garbage\|containerGC"

# • Убедиться, что контейнеры действительно завершены
crictl ps -a --state exited

# 3. Объект API не удаляется:
# • Проверить финализаторы
kubectl get <kind> <name> -o jsonpath='{.metadata.finalizers}'

# • Проверить ownerReferences и блокирующие зависимости
kubectl get <kind> <name> -o yaml | grep -A10 'ownerReferences'

# • Посмотреть события
kubectl describe <kind> <name> | grep -A10 Events
```

## 🔹 Чек-лист: настройка надёжной сборки мусора

### ✅ Для объектов API
```bash
# • Используй ownerReferences для автоматической очистки зависимых ресурсов
#   (Deployment → ReplicaSet → Pod, Job → Pod)

# • Для критичных ресурсов: добавляй финализаторы для «чистой» уборки
#   (удаление внешних балансировщиков, сброс блокировок, бэкап метаданных)

# • Настрой TTL для временных ресурсов (Jobs, CronJob):
#   ttlSecondsAfterFinished: 3600

# • Регулярно проверяй «осиротевшие» объекты:
#   кubectl get all -A --field-selector metadata.ownerReferences=null

# • Документируй политики удаления для каждого типа ресурса в команде
```

### ✅ Для контейнеров и образов на нодах
```bash
# • Настрой imageGCHighThresholdPercent и imageGCLowThresholdPercent
#   под свою инфраструктуру (обычно 85%/80% — хороший старт)

# • Для CI/CD-нод с частыми сборками: увеличь imageMaximumGCAge
#   или используй отдельный реестр для кэширования образов

# • Регулярно мониторь использование диска на нодах
#   (Prometheus-метрики: node_filesystem_avail_bytes, kubelet_volume_stats_used_bytes)

# • Не используй внешние инструменты очистки, если не понимаешь, что делает kubelet
#   → риск удалить нужные контейнеры или образы

# • Для stateful-приложений: убедись, что образы не удаляются, пока нужны
#   (через imageMaximumGCAge или явное удержание через imagePullPolicy: IfNotPresent)
```

### ✅ Для отладки и мониторинга
```bash
# • Алерт на использование диска > 90% на нодах
#   (чтобы успеть до агрессивной очистки kubelet)

# • Алерт на «зависшие» объекты в Terminating
#   (признак проблемы с финализаторами или зависимостями)

# • Логируй события удаления критичных ресурсов
#   (для аудита и восстановления при ошибках)

# • Тестируй политики GC в staging перед применением в production
#   (особенно imageMaximumGCAge и пороги дискового пространства)
```

### ❌ Чего избегать
```bash
# ❌ Не отключай GC kubelet полностью (imageGCHighThresholdPercent: 100)
#   → диски быстро заполнятся, ноды станут NotReady

# ❌ Не устанавливай слишком агрессивные пороги (HighThresholdPercent < 70)
#   → постоянная очистка образов может замедлить запуск новых подов

# ❌ Не удаляй финализаторы вручную без понимания последствий
#   → можно потерять внешние ресурсы или сломать аудит

# ❌ Не полагайся только на «фоновое» удаление для критичных ресурсов
#   → используй `foreground` или финализаторы для гарантированной очистки

# ❌ Не игнорируй события `OwnerRefInvalidNamespace`
#   → признак некорректной конфигурации, которая может привести к «утечке» объектов
```

## 🔹 Ключевые выводы

1. **Два уровня GC**: Control Plane (объекты API) и Node (контейнеры/образы) работают независимо.
2. **ownerReferences + финализаторы** = мощный механизм для каскадной и «чистой» уборки ресурсов.
3. **Три политики удаления**: `foreground` (ждать), `background` (по умолчанию), `orphan` (сохранить зависимые).
4. **kubelet GC** управляется порогами дискового пространства и возрастом ресурсов — настраивай под свою нагрузку.
5. **Мониторь и алерти**: сборка мусора не отменяет необходимости наблюдать за использованием ресурсов.
6. **Тестируй в staging**: агрессивные настройки GC могут иметь непредсказуемые последствия в production.

> 💡 **Финальный совет**: сборка мусора — это не «настроил и забыл». Это баланс между экономией ресурсов и доступностью. Проектируй приложения и инфраструктуру так, чтобы они корректно реагировали на удаление зависимостей и очистку образов.
