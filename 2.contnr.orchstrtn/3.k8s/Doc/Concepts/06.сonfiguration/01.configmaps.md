# ConfigMap — Управление конфигурацией в Kubernetes

> 📌 `ConfigMap` — это объект API для хранения **неконфиденциальных** данных в формате ключ-значение. Позволяет отделить конфигурацию от кода приложения, делая образы контейнеров переносимыми между средами (dev, staging, prod). **Лимит размера: 1 МиБ**.

---

## 🔹 Зачем нужен ConfigMap

| Проблема | Решение с ConfigMap |
|----------|---------------------|
| Жёстко зашитые конфиги в Docker-образе | Конфиг вынесен во внешний объект K8s |
| Разные конфиги для dev/prod | Один образ, разные ConfigMap для разных сред |
| Сложность обновления настроек | Обновление ConfigMap (без пересборки образа) |

> ⚠️ **ВАЖНО**: ConfigMap **не шифрует** данные и не обеспечивает конфиденциальность. Для паролей, токенов и ключей всегда используй **Secret**.

---

## 🔹 Структура объекта ConfigMap

ConfigMap состоит из двух необязательных полей для данных. Имя объекта должно быть валидным DNS-поддоменом.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: game-demo
  namespace: default
data:
  # 1. Простые ключи (property-like)
  player_initial_lives: "3"
  ui_properties_file_name: "user-interface.properties"

  # 2. Ключи, имитирующие файлы (file-like, используется синтаксис |)
  game.properties: |
    enemy.types=aliens,monsters
    player.maximum-lives=5
  user-interface.properties: |
    color.good=purple
    color.bad=yellow
binaryData:
  # 3. Двоичные данные (должны быть закодированы в Base64)
  logo.png: iVBORw0KGgoAAAANSUhEUgAA...
```

**Правила для ключей:**
- Только буквенно-цифровые символы, `-`, `_` или `.`
- Ключи в `data` и `binaryData` не должны пересекаться.

## 🔹 4 способа использования ConfigMap в Pod

Kubelet использует данные ConfigMap при запуске контейнера.

| Способ | Как работает | Когда использовать |
|--------|--------------|-------------------|
| **1. Переменные окружения** | K8s инжектирует значения как `ENV` переменные | Простые настройки, URL, флаги |
| **2. Аргументы командной строки** | Передача значений в `command` или `args` контейнера | Запуск утилит с динамическими параметрами |
| **3. Том (Volume Mount)** | Монтирование как файлов в файловую систему пода | Сложные конфиги (JSON, YAML, .properties, Nginx conf) |
| **4. Чтение через K8s API** | Приложение само запрашивает ConfigMap через API | Динамическая перезагрузка конфига без рестарта пода |

## 🔹 Практика: Способы подключения

### Способ А: Переменные окружения

#### Вариант 1: Импорт всех ключей (`envFrom`)
Создаёт переменные окружения с именами, совпадающими с ключами ConfigMap.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: env-all-demo
spec:
  containers:
  - name: app
    image: busybox:latest
    command: ["printenv"]
    envFrom:
    - configMapRef:
        name: game-demo
```
> ⚠️ **Ограничение**: Имена ключей в ConfigMap должны быть валидными именами переменных окружения (только буквы, цифры и `_`, не могут начинаться с цифры). Невалидные ключи будут проигнорированы.

#### Вариант 2: Выборочный импорт (`env.valueFrom`)
Позволяет переименовать переменную в поде и выбрать конкретный ключ.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: env-specific-demo
spec:
  containers:
  - name: app
    image: nginx
    env:
    - name: MY_CUSTOM_LIVES_VAR      # Имя переменной в контейнере
      valueFrom:
        configMapKeyRef:
          name: game-demo            # Имя ConfigMap
          key: player_initial_lives  # Ключ внутри ConfigMap
```

### Способ Б: Монтирование как тома (Volume Mount)

Самый частый сценарий для файлов конфигурации.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: volume-demo
spec:
  containers:
  - name: app
    image: redis
    volumeMounts:
    - name: config-volume
      mountPath: "/etc/config"
      readOnly: true
  volumes:
  - name: config-volume
    configMap:
      name: game-demo
      # Опционально: монтировать только конкретные ключи как файлы
      items:
      - key: "game.properties"
        path: "game.properties"       # Будет создан файл /etc/config/game.properties
      - key: "user-interface.properties"
        path: "ui/ui.properties"      # Можно изменить имя файла при монтировании
```
> 💡 **Если не указать `items`**: K8s создаст по файлу для **каждого** ключа в `data`, где имя файла = имя ключа.

## 🔹 Поведение при обновлении (Важные нюансы!)

| Способ использования | Реагирует на обновление ConfigMap? | Детали |
|----------------------|------------------------------------|--------|
| **Том (Volume)** | ✅ **Да** (автоматически) | Kubelet периодически синхронизирует файлы. Задержка зависит от `kubelet-sync-period` и типа кэша (watch по умолчанию). |
| **Переменные окружения** | ❌ **Нет** | Переменные задаются только при старте контейнера. Требуется **рестарт пода** (например, через `kubectl rollout restart`). |
| **Том с `subPath`** | ❌ **Нет** | Если смонтирован конкретный файл через `subPath`, автоматические обновления **не работают**. |

> 💡 **Совет**: Если приложению нужна горячая перезагрузка конфига, монтируй ConfigMap как том (без `subPath`) и убедись, что приложение умеет перечитывать файлы (или используй sidecar-контейнер для отправки сигнала SIGHUP).

## 🔹 Неизменяемые ConfigMap (Immutable ConfigMaps)

> **Стабильно с v1.21**. Позволяет пометить ConfigMap как доступный только для чтения.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: immutable-config
data:
  app.conf: "setting=value"
immutable: true  # ← Ключевое поле
```

**Преимущества:**
1. **Безопасность**: Защита от случайного изменения конфигов, что может вызвать сбой приложений.
2. **Производительность**: Kubelet и API-сервер не тратят ресурсы на отслеживание изменений (watch) для этого объекта. Критично для кластеров с десятками тысяч подов.

**Ограничения:**
- После создания с `immutable: true` изменить `data` или `binaryData` **невозможно**.
- Чтобы изменить конфиг, нужно:
  1. Создать новый ConfigMap с новым именем.
  2. Обновить Pod/Deployment, чтобы он ссылался на новый ConfigMap.
  3. Удалить старый ConfigMap.

## 🔹 Шпаргалка: Работа с ConfigMap через kubectl

```bash
# 1. Создать ConfigMap из literal (ключ-значение)
kubectl create configmap my-config --from-literal=db_host=localhost --from-literal=db_port=5432

# 2. Создать ConfigMap из файла
kubectl create configmap my-config --from-file=app.properties

# 3. Создать ConfigMap из всей директории (каждый файл станет ключом)
kubectl create configmap my-config --from-file=/path/to/dir/

# 4. Посмотреть список ConfigMap
kubectl get configmaps   # или kubectl get cm

# 5. Посмотреть содержимое конкретного ConfigMap
kubectl get cm game-demo -o yaml
kubectl describe cm game-demo

# 6. Обновить существующий ConfigMap из файла (перезапишет данные)
kubectl create configmap game-demo --from-file=game.properties --dry-run=client -o yaml | kubectl replace -f -

# 7. Проверить, как под видит переменные окружения
kubectl exec <pod-name> -- printenv | grep CONFIG
```

## 🔹 Чек-лист: Best Practices для ConfigMap

```text
[ ] Данные не являются чувствительными (нет паролей, ключей API). Если есть → используй Secret.
[ ] Размер данных не превышает 1 МиБ. Если больше → используй Persistent Volume или внешний сервис.
[ ] Имена ключей валидны для выбранного способа использования (особенно для ENV переменных).
[ ] Для файловых конфигов используется монтирование томом (Volume Mount), а не ENV переменными.
[ ] Если приложение не умеет горячо перечитывать конфиги, запланирован механизм рестарта пода при изменении ConfigMap (например, через аннотации в Deployment).
[ ] Для стабильных, редко меняющихся конфигов рассматривается использование immutable: true.
```

> 💡 **Совет для Obsidian**: 
> - Создай шаблон `00_configmap_template.yaml` с базовой структурой для быстрого копирования.
> - Добавь внутреннюю ссылку на этот файл из заметок про `Deployment` и `Secret`, так как они часто используются вместе.

## 🔹 Ключевые выводы

1. **Разделение ответственности**: ConfigMap хранит конфигурацию, образ контейнера хранит код.
2. **Гибкость**: Одни и те же данные можно инжектировать как ENV переменные или как файлы.
3. **Обновления**: Тома обновляются автоматически (с задержкой), ENV переменные — только при рестарте пода.
4. **Immutable**: Используй `immutable: true` для повышения безопасности и производительности кластера.
5. **Лимиты**: Максимум 1 МиБ на один объект ConfigMap.
