>Пространства имён (Namespaces) — это механизм логической изоляции и организации ресурсов в кластере. Критически важная тема для работы с multi-tenant средами и разграничения доступа.

# Пространства имён (Namespaces) в Kubernetes

> 📌 **Namespace** = виртуальный кластер внутри реального. Имена ресурсов уникальны **в рамках** неймспейса, но не между ними. Используй для изоляции команд/проектов/сред, но не для разделения версий одного приложения (для этого есть метки).

---

## 🔹 Что такое Namespace

| Аспект | Описание |
|--------|----------|
| **Определение** | Механизм логической изоляции групп ресурсов в рамках одного физического кластера |
| **Уникальность** | Имя ресурса должно быть уникально **внутри** неймспейса, но может повторяться в разных |
| **Область действия** | Применимо только к namespaced-ресурсам (`Pod`, `Deployment`, `Service`), но не к кластерным (`Node`, `PV`, `StorageClass`) |
| **Иерархия** | ❌ Не могут быть вложены друг в друга. Каждый ресурс принадлежит ровно одному неймспейсу |

```mermaid
graph TD
    Cluster[Kubernetes Cluster] --> NS1[namespace: dev]
    Cluster --> NS2[namespace: staging]
    Cluster --> NS3[namespace: production]
    
    NS1 --> Pod1[Pod: api-v1]
    NS1 --> Svc1[Service: api]
    
    NS2 --> Pod2[Pod: api-v2]
    NS2 --> Svc2[Service: api]  # ← то же имя, другой неймспейс
    
    NS3 --> Pod3[Pod: api-v3]
    NS3 --> Svc3[Service: api]
```

> 💡 **Ключевая идея**: неймспейсы — это не про безопасность (хотя RBAC можно привязать), а про **организацию** и **квотирование ресурсов**.

---

## 🔹 Когда использовать несколько неймспейсов

### ✅ Рекомендуемые сценарии

| Сценарий | Пример | Почему неймспейс, а не метки? |
|----------|--------|------------------------------|
| **👥 Разные команды/проекты** | `team-alpha`, `team-beta` | Изоляция прав доступа (RBAC), квот, лимитов |
| **🌍 Разные среды** | `dev`, `staging`, `production` | Разные политики, ресурсы, доступ к внешним сервисам |
| **🏢 Мультитенант** | `customer-a`, `customer-b` | Изоляция данных, квотирование, аудит по тенанту |
| **📊 Квоты и лимиты** | `ResourceQuota`, `LimitRange` | Эти объекты работают **только** в рамках неймспейса |

### ❌ Когда НЕ использовать неймспейсы

| Сценарий | Лучше использовать | Почему |
|----------|-------------------|--------|
| **Разные версии одного приложения** | Метки: `version=v1`, `version=v2` | Проще управлять, не нужно дублировать конфиги |
| **Небольшие проекты (1-2 человека)** | Один неймспейс `default` или `app-name` | Излишняя сложность без выгоды |
| **Разделение по компонентам** | Метки: `tier=frontend`, `tier=backend` | Компоненты всё равно связаны сетью, проще в одном месте |

> ⚠️ **Важно**: для продакшена **не рекомендуется** использовать `default`. Создавай явные неймспейсы (`prod`, `app-name`) — это упрощает аудит и управление.

## 🔹 Системные неймспейсы (из коробки)

| Неймспейс | Назначение | Можно ли использовать? |
|-----------|------------|----------------------|
| **`default`** | Стартовый неймспейс, если не указан другой | ⚠️ Только для тестов, не для production |
| **`kube-system`** | Системные компоненты K8s (`kube-dns`, `kube-proxy`, metrics-server) | ❌ Не размещай свои приложения здесь |
| **`kube-public`** | Публичные ресурсы, доступные для чтения всем (даже неаутентифицированным) | ⚠️ Только если действительно нужно публичное чтение |
| **`kube-node-lease`** | Объекты `Lease` для heartbeat'ов нод (внутренняя механика) | ❌ Только для системы |

> 🚫 **Правило**: не создавай неймспейсы с префиксом `kube-` — они зарезервированы для системы.

## 🔹 Работа с неймспейсами: kubectl

### 👁️ Просмотр и создание
```bash
# Список всех неймспейсов
kubectl get namespaces
kubectl get ns  # короткая форма

# Детальная информация
kubectl describe namespace production

# Создать новый неймспейс
kubectl create namespace my-app
# или декларативно:
# kubectl apply -f namespace.yaml

# Удалить неймспейс (осторожно: удалит все ресурсы внутри!)
kubectl delete namespace my-app
```

### 🎯 Переключение контекста и неймспейса
```bash
# Разовая команда в конкретном неймспейсе
kubectl get pods --namespace=production
kubectl get pods -n production  # короткая форма

# Установить неймспейс по умолчанию для текущего контекста
kubectl config set-context --current --namespace=production

# Проверить текущий контекст и неймспейс
kubectl config view --minify | grep namespace:

# Временный неймспейс для одной сессии (без изменения конфига)
kubectl config set-context --current --namespace=production && \
  kubectl get pods && \
  kubectl config set-context --current --namespace=default
```

### 📦 Манифест неймспейса
```yaml
# namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    environment: production
    team: platform
  annotations:
    description: "Production environment for customer-facing apps"
```

## 🔹 Namespaces и DNS

При создании `Service` Kubernetes автоматически создаёт DNS-запись:

```
<service-name>.<namespace>.svc.cluster.local
```

### 🌐 Как это работает на практике

```yaml
# В неймспейсе `dev` создан Service: api
# Из пода в том же неймспейсе:
curl http://api          # ✅ работает: короткий имя, тот же неймспейс

# Из пода в неймспейсе `production`:
curl http://api          # ❌ не найдёт: ищет api.production.svc.cluster.local
curl http://api.dev      # ✅ работает: <service>.<namespace>
curl http://api.dev.svc.cluster.local  # ✅ FQDN, всегда работает
```

### ⚠️ Предупреждение про DNS-коллизии

```bash
# Если создать неймспейс с именем публичного домена:
kubectl create namespace com

# И сервис с коротким именем:
kubectl create service clusterip google --tcp=80:80 -n com

# Теперь из любого пода:
curl http://google  # → попадёт в сервис, а не на google.com!
```

> 🛡️ **Решение**: ограничь право создания неймспейсов через RBAC + используй Admission Webhooks для блокировки зарезервированных имён (`com`, `org`, `local` и т.д.).

## 🔹 Какие ресурсы НЕ в неймспейсах

Не все объекты Kubernetes принадлежат неймспейсу. Некоторые — кластерные (cluster-scoped).

### 🔍 Как проверить
```bash
# Ресурсы, которые находятся в неймспейсах
kubectl api-resources --namespaced=true

# Ресурсы, которые НЕ в неймспейсах (кластерные)
kubectl api-resources --namespaced=false
```

### 📋 Примеры cluster-scoped ресурсов

| Ресурс | Почему не в неймспейсе |
|--------|----------------------|
| `Node` | Физическая/виртуальная машина — одна на весь кластер |
| `PersistentVolume` (PV) | Хранилище может использоваться разными неймспейсами |
| `StorageClass` | Шаблон для динамического создания PV — глобальный |
| `Namespace` | Сам является контейнером, не может быть внутри себя |
| `ClusterRole` / `ClusterRoleBinding` | Права доступа на уровне всего кластера |
| `CustomResourceDefinition` (CRD) | Определение нового типа ресурса — глобальное |

> 💡 **Практика**: при написании операторов или политик всегда проверяй, namespaced ли ресурс, с которым работаешь.

## 🔹 Автоматическая маркировка неймспейсов

Начиная с **Kubernetes 1.22**, система автоматически добавляет неизменяемую метку ко всем неймспейсам:

```yaml
metadata:
  labels:
    kubernetes.io/metadata.name: <имя-неймспейса>  # ← автоматически
```

### 🎯 Зачем это нужно
- Упрощает написание `NetworkPolicy`, которые ссылаются на неймспейс по имени
- Позволяет использовать селекторы по имени неймспейса без хардкода
- Надёжно: метка неизменяема, нельзя случайно сломать

```yaml
# Пример NetworkPolicy: разрешить трафик только из неймспейса "monitoring"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-monitoring
spec:
  podSelector: {}
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: monitoring
```

## 🔹 Чек-лист: работа с неймспейсами

```bash
# ✅ При создании ресурсов: явно указывай неймспейс (не надейся на default)
kubectl apply -f deployment.yaml -n production

# ✅ В манифестах: всегда указывай metadata.namespace, если ресурс не cluster-scoped
# Это делает деплой идемпотентным и предсказуемым

# ✅ Перед удалением неймспейса: проверь, что внутри
kubectl get all -n my-app
kubectl delete namespace my-app --dry-run=server  # сначала тест

# ✅ Для отладки: убедись, в каком неймспейсе ты работаешь
kubectl config view --minify -o jsonpath='{..namespace}'

# ✅ При написании скриптов: передавай неймспейс как параметр, а не хардкодь
NS="${NAMESPACE:-default}"
kubectl get pods -n "$NS"

# ✅ Для multi-namespace операций: используй циклы или инструменты типа `kubens`
for ns in dev staging prod; do
  echo "=== $ns ==="
  kubectl get pods -n "$ns" --no-headers | wc -l
done

# ✅ При настройке RBAC: привязывай роли к конкретным неймспейсам, если не нужен кластерный доступ
```

## 🔹 Ключевые выводы

1. **Изоляция ≠ безопасность**: неймспейсы организуют ресурсы, но для реальной изоляции нужен RBAC + NetworkPolicy.
2. **Не переусложняй**: начинай с одного неймспейса, добавляй новые только когда есть реальная потребность (команды, квоты, среды).
3. **DNS-имена зависят от неймспейса**: помни про `<service>.<namespace>.svc.cluster.local` для кросс-неймспейс общения.
4. **Не все ресурсы в неймспейсах**: `Node`, `PV`, `StorageClass` — кластерные, работают везде.
5. **`default` — для тестов**: в production используй явные имена (`prod`, `app-name`) для лучшего аудита.
