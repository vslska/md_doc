# Управление объектами Kubernetes — kubectl: подходы и практика

> *📌* Есть **3 способа** управлять объектами в K8s: императивные команды (быстро, для отладки), императивная конфигурация (файлы + команды), декларативная конфигурация (`apply` — стандарт для production). **Важно**: не смешивай подходы для одного объекта.

---

## Три подхода к управлению: сравнение

| Подход | Что делаешь | Где файлы | Для чего | Сложность |
|--------|-------------|-----------|----------|-----------|
| **⚡ Императивные команды** | `kubectl create deployment ...` | Не нужны | Быстрые задачи, отладка, обучение | 🟢 Низкая |
| **📄 Императивная конфигурация** | `kubectl create/replace -f file.yaml` | Отдельные YAML-файлы | Простые продакшен-сценарии, контроль версий | 🟡 Средняя |
| **🗂️ Декларативная конфигурация** | `kubectl apply -f dir/` | Каталог манифестов (Git) | **Production**, GitOps, командная работа | 🔴 Высокая (но окупается) |

> ⚠️ **Критичное правило**: **Не смешивать подходы для одного объекта**. Если создал через `kubectl run` — не пытайся потом обновлять через `apply`. Это приведёт к конфликтам и неопределённому поведению.

---

## 1. Императивные команды (Imperative Commands)

> *«Сделай это прямо сейчас»* — и система выполняет.

При использовании императивных команд ты работаешь непосредственно с активными (текущими) объектами в кластере.
### Как работает:
```bash
# Создать деплоймент
kubectl create deployment nginx --image=nginx:1.25

# Масштабировать
kubectl scale deployment nginx --replicas=3

# Удалить
kubectl delete deployment nginx
```
### ✅ Плюсы:

- 🚀 Быстро: одна команда = результат
- 🧪 Идеально для: обучения, отладки, разовых задач
- 📉 Низкий порог входа: не нужно писать YAML

### ❌ Минусы:

- 🗑️ Нет истории изменений (не видно, что и когда менялось)
- 🔍 Нет аудита: сложно отследить, кто и что сделал
- 📦 Нет шаблона: нельзя просто «взять и повторить» на другом кластере
- ⚠️ Легко допустить ошибку: опечатка в команде = изменение в продакшене

> 💡 **Используй, когда**: тестируешь идею, чинишь что-то срочно, учишься.
>    **Не используй** для управления важными ресурсами в production.

## 2. Императивная конфигурация объектов (Imperative Object Configuration)

> _«Возьми этот файл и примени команду к его содержимому»_.

### Как работает:
```bash
# Создать объекты из файла
kubectl create -f nginx.yaml

# Обновить (полная замена!)
kubectl replace -f nginx.yaml

# Удалить
kubectl delete -f nginx.yaml
```

### ⚠️ Важное про `replace`:
```yaml
# Было в кластере:
spec:
  replicas: 3
  image: nginx:1.24
  env: { DEBUG: "true" }  # ← добавлено вручную через kubectl edit

# В файле nginx.yaml:
spec:
  replicas: 2
  image: nginx:1.25
  # ← поле env отсутствует

# Результат `kubectl replace -f nginx.yaml`:
# → env: DEBUG удалится! Потому что replace заменяет объект целиком.
```

### ✅ Плюсы:

- 📁 Конфиги можно хранить в Git (контроль версий)
- 🔐 Можно встроить в CI/CD: валидация, ревью, аудит
- 🧱 Есть шаблон для повторного использования

### ❌ Минусы:

- 📝 Нужно писать YAML (дополнительный шаг)
- 🔄 `replace` опасен: стирает изменения, сделанные вне файла
- 📂 Неудобно для работы с каталогами: нужно перечислять файлы вручную

> 💡 **Используй, когда**: у тебя простые сценарии, ты контролируешь все изменения через файлы, и не ожидаешь ручных правок в кластере.

## 3. Декларативная конфигурация объектов (Declarative Object Configuration) — ✅ Стандарт для Production

> _«Сделай так, чтобы реальность соответствовала этим файлам»_ — и K8s сам решает, что нужно создать, обновить или удалить.

### Как работает:
```bash
# Применить все конфиги из каталога (создать/обновить)
kubectl apply -f configs/

# Рекурсивно обработать подкаталоги
kubectl apply -R -f configs/

# Посмотреть, что изменится, до применения (безопасно!)
kubectl diff -f configs/
```

### Ключевая особенность: `apply` использует **трёхстороннее слияние**
```
1. Последняя применённая конфигурация (аннотация в объекте)
2. Текущее состояние объекта в кластере
3. Новая конфигурация из файла

→ K8s вычисляет минимальные изменения (patch), а не заменяет объект целиком.
```

### Что это даёт:
```yaml
# В файле:
spec:
  replicas: 3

# Кто-то вручную изменил в кластере:
spec:
  replicas: 5  # ← через kubectl scale

# Ты снова делаешь `apply -f file.yaml`:
# → replicas останется 5! Потому что поле не менялось в файле.
# → Но если в файле поменяешь на replicas: 4 — применится 4.
```

### ✅ Плюсы:

- 🛡️ Безопасно: сохраняет изменения, сделанные вне файлов (если они не конфликтуют)
- 📁 Удобно для каталогов: одна команда — всё обновилось
- 🔄 Автоматическое определение операции: create / update / delete
- 🤝 Идеально для GitOps: кластер = отражение репозитория

### ❌ Минусы:

- 🧠 Сложнее отлаживать: не всегда очевидно, почему `apply` сработал именно так
- 🧩 Требует понимания: аннотации [`kubectl.kubernetes.io/last-applied-configuration`](https://kubernetes.io/docs/reference/kubectl/generated/kubectl_apply/kubectl_apply_set-last-applied/), merge-стратегии 
- 📚 Выше порог входа: нужно изучить работу `apply`, `diff`, `prune`

> 💡 **Используй всегда для production**. Это единственный подход, который масштабируется, безопасен и совместим с современными практиками (GitOps, ArgoCD, Flux).

## 4. Сравнение команд: шпаргалка

|Задача|Императивная команда|Императивная конф-я|Декларативная конф-я|
|---|---|---|---|
|**Создать**|`kubectl create deployment ...`|`kubectl create -f file.yaml`|`kubectl apply -f file.yaml`|
|**Обновить**|`kubectl scale ...`, `kubectl set image ...`|`kubectl replace -f file.yaml` ⚠️|`kubectl apply -f file.yaml` ✅|
|**Удалить**|`kubectl delete deployment ...`|`kubectl delete -f file.yaml`|`kubectl apply -f file.yaml --prune` / `delete -f`|
|**Посмотреть изменения**|—|—|`kubectl diff -f file.yaml` ✅|
|**Применить каталог**|❌ Неудобно|❌ Нужно перечислять файлы|✅ `apply -R -f dir/`|
|**Откатить**|❌ Сложно|⚠️ Через `replace` старым файлом|✅ `kubectl rollout undo` + Git history|

## 5. Практика: как начать использовать декларативный подход

### 📁 Структура репозитория (пример):
```
k8s-configs/
├── bases/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── configmap.yaml
├── overlays/
│   ├── dev/
│   │   └── patch-replicas.yaml  # replicas: 1
│   └── prod/
│       └── patch-replicas.yaml  # replicas: 5
└── Apply.sh  # опционально: скрипт для деплоя
```

### 🚀 Базовый рабочий процесс:
```bash
# 1. Локально проверить, что изменится
kubectl diff -f bases/

# 2. Применить изменения
kubectl apply -f bases/

# 3. Проверить результат
kubectl get all -l app=myapp

# 4. Закоммитить изменения в Git
git add bases/deployment.yaml
git commit -m "feat: scale nginx to 3 replicas"
git push
```

### 🔄 Если нужно «переопределить» поле (осторожно!):
```bash
# Принудительно перезаписать объект (удалит внешние изменения!)
kubectl replace --force -f file.yaml

# Или лучше: отредактировать файл и сделать `apply`
kubectl edit deployment nginx  # откроет в редакторе → после сохранения автоматически применит
```

## 6. Чек-лист: безопасное управление объектами
``` bash
# ✅ Всегда перед apply: посмотри, что изменится
kubectl diff -f manifest.yaml

# ✅ Используй dry-run для проверки валидации и admission-политик
kubectl apply -f manifest.yaml --dry-run=server -o yaml

# ✅ Фиксируй изменения в Git перед применением
git add . && git commit -m "..."

# ✅ Для отката используй встроенные механизмы, а не ручные правки
kubectl rollout undo deployment/nginx          # откат деплоймента
kubectl rollout history deployment/nginx       # история изменений

# ✅ Если объект «сломался» после apply — проверь аннотации
kubectl get deployment nginx -o jsonpath='{.metadata.annotations}' | jq

# ✅ Никогда не миксуй: если начал с apply — продолжай с apply
# ❌ Не делай: kubectl apply -f ... && kubectl edit ... && kubectl replace -f ...

# ✅ Для массовых операций используй лейблы и селекторы
kubectl apply -k bases/                        # Kustomize
kubectl apply -l tier=backend -f configs/      # по лейблу
```

## Ключевые выводы

1. **Выбирай подход осознанно**: императивные команды — для отладки, декларативные — для всего остального.
2. **`apply` > `replace`**: декларативный подход безопаснее, потому что использует patch, а не полную замену.
3. **Git — источник истины**: храни манифесты в репозитории, делай ревью изменений, веди историю.
4. **`diff` — твой друг**: всегда смотри, что изменится, перед применением.
5. **Не смешивай**: один объект — один подход. Иначе получишь рассинхрон и головную боль.
