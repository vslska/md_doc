# Имена и идентификаторы объектов Kubernetes

> 📌 Каждый объект в K8s имеет **имя** (читаемое человеком, уникальное в рамках типа+неймспейса) и **UID** (уникальный системный идентификатор). Имена должны соответствовать строгим правилам (DNS-стандарты). Для массового создания используй `generateName`.

---

## 🔹 Как K8s уникально идентифицирует объект

Объект однозначно определяется **комбинацией из 4 атрибутов**:

```
[Группа API] + [Тип ресурса] + [Пространство имён] + [Имя]
```

| Атрибут | Пример | Пояснение |
|---------|--------|-----------|
| **Группа API** | `apps`, `networking.k8s.io`, `""` (core) | Логическая группировка ресурсов |
| **Тип ресурса** | `deployments`, `pods`, `services` | Kind объекта (в нижнем регистре, множественное число) |
| **Пространство имён** | `default`, `kube-system`, `my-app` | Изоляция: ресурсы с одинаковым именем могут существовать в разных неймспейсах |
| **Имя** | `nginx-deployment`, `my-pod-abc123` | Уникальное имя в рамках типа+неймспейса |

### 🎯 Примеры уникальности

```yaml
# ✅ Можно: разные типы ресурсов с одинаковым именем в одном неймспейсе
- kind: Pod, name: myapp, namespace: default
- kind: Deployment, name: myapp, namespace: default

# ✅ Можно: одинаковые типы в разных неймспейсах
- kind: Pod, name: myapp, namespace: dev
- kind: Pod, name: myapp, namespace: prod

# ❌ Нельзя: одинаковый тип + имя + неймспейс
- kind: Pod, name: myapp, namespace: default  # ← конфликт!
```

> ⚠️ **Важно**: версия API (`v1`, `v1beta1`) **не входит** в уникальный идентификатор. Объект `apps/v1/Deployment` и `apps/v1beta1/Deployment` с одинаковым именем — это один и тот же объект, просто разные представления.

## 🔹 Имя (Name) vs UID

| Характеристика | **Name** | **UID** |
|---------------|----------|---------|
| **Кто задаёт** | 👤 Ты (или `generateName`) | 🤖 Система (автоматически) |
| **Формат** | Строка, соответствует правилам именования | UUID (например: `a1b2c3d4-e5f6-7890-abcd-ef1234567890`) |
| **Уникальность** | Уникально в рамках: тип ресурса + неймспейс | Уникально во всём кластере, на всём времени жизни |
| **Изменяемость** | ❌ Нельзя изменить после создания | ❌ Никогда не меняется |
| **Назначение** | Человекочитаемый адрес для API, URL, CLI | Внутренняя ссылка, аудит, отслеживание истории |

### 🔄 Почему нужен UID?

```yaml
# Сценарий: удалил и создал под с тем же именем
1. kubectl delete pod myapp
2. kubectl create pod myapp --image=nginx

# Визуально: тот же name, но:
- Старый pod: UID = abc123...
- Новый pod: UID = xyz789...

# Зачем: 
• Контроллеры отслеживают объекты по UID, а не по имени
• Аудит: можно отличить исторические события разных объектов с одинаковым именем
• Сборка мусора: система понимает, что старый объект действительно удалён
```

> 💡 **Практика**: при отладке сложных сценариев (операторы, GC) смотри UID:  
> `kubectl get pod myapp -o jsonpath='{.metadata.uid}'`

## 🔹 Правила именования: 4 основных типа ограничений

Kubernetes применяет разные правила в зависимости от типа ресурса.

### 📏 Сравнение стандартов

| Стандарт | Макс. длина | Допустимые символы | Начало | Конец | Где применяется |
|----------|-------------|-------------------|--------|-------|----------------|
| **🌐 DNS Subdomain** (RFC 1123) | 253 | `a-z`, `0-9`, `-`, `.` | буквенно-цифровой | буквенно-цифровой | Большинство ресурсов: `Pod`, `Deployment`, `Namespace`, `ServiceAccount` |
| **🏷️ Label Name** (RFC 1123) | 63 | `a-z`, `0-9`, `-` | **буква** | буквенно-цифровой | Ключи и значения лейблов, аннотаций |
| **🔖 Label Name** (RFC 1035) | 63 | `a-z`, `0-9`, `-` | **буква** | буквенно-цифровой | Некоторые ресурсы: `Service` (по умолчанию), `Secret` |
| **🛣️ Path Segment** | — | Любые, кроме `.`, `..`, `/`, `%` | — | — | Ресурсы, используемые в URL-путях |

### 🧪 Примеры валидных и невалидных имён

```yaml
# ✅ DNS Subdomain (253 символа, можно точки и дефисы)
my-app.example.com
app-v1.2.3
nginx-deployment-prod

# ❌ DNS Subdomain — ошибки
MyApp          # ❌ заглавные буквы
.app           # ❌ начинается с точки
app-           # ❌ заканчивается на дефис
a{254}         # ❌ больше 253 символов

# ✅ RFC 1123 Label (63 символа, только буквы в начале/конце)
myapp
app-v1
nginx123

# ❌ RFC 1123 Label — ошибки
123-app        # ❌ начинается с цифры*
app_1          # ❌ подчёркивание не разрешено
my.app         # ❌ точка не разрешена

# *Исключение: если включён фича-гейт RelaxedServiceNameValidation, 
# Service может начинаться с цифры.
```

> 💡 **Проверка имени перед созданием**:
> ```bash
> # Быстрая валидация через dry-run
> kubectl create pod test-APP --image=nginx --dry-run=server
> # → Error: invalid metadata.name: [a DNS-1123 subdomain must consist of lower case...]
> ```

## 🔹 Генерация имён: `generateName`

Когда нужно создать несколько объектов с уникальными именами (например, в цикле, в `Job`, при тестировании).

### Как работает:
```yaml
# Манифест с generateName
apiVersion: v1
kind: Pod
metadata:
  generateName: test-pod-  # ← префикс
spec:
  containers:
  - name: nginx
    image: nginx:1.25
```

```bash
# Применение: сервер добавит случайный суффикс
kubectl apply -f pod.yaml

# Результат:
# → Pod создан с именем: test-pod-abc12
# → Следующий запуск: test-pod-xyz89
```

### ⚙️ Поведение сервера (K8s ≥ 1.31):
- Сервер делает **до 8 попыток** сгенерировать уникальное имя (префикс + случайный суффикс)
- Если все попытки конфликтуют с существующими именами → возвращается `HTTP 409 Conflict`
- Суффикс: 5 случайных символов в base36 (0-9, a-z)

### ✅ Когда использовать `generateName`:
- `Job` / `CronJob`: каждый запуск — новый под
- Тестовые сценарии: не нужно придумывать уникальные имена вручную
- Операторы: создание дочерних ресурсов с предсказуемым префиксом

### ❌ Когда не использовать:
- Ресурсы, на которые ссылаются другие объекты (например, `Service` → `selector` по имени)
- Когда имя используется в конфигурации приложения (hardcoded в коде)

> 💡 **Совет**: если используешь `generateName`, добавляй лейблы для последующего поиска:
> ```yaml
> metadata:
>   generateName: test-pod-
>   labels:
>     test-run: "2024-06-05"
>     purpose: debug
> ```

## 🔹 Практика: работа с именами и UID

### 🔍 Просмотр имён и UID
```bash
# Базовый вывод
kubectl get pods
# NAME       READY   STATUS    AGE
# myapp-1    1/1     Running   10m

# С UID
kubectl get pods -o jsonpath='{.items[*].metadata.name}{"\n"}{.items[*].metadata.uid}'

# Полная информация в YAML
kubectl get pod myapp -o yaml | grep -A2 'name:\|uid:'

# Только имя и неймспейс
kubectl get pod myapp -o jsonpath='{.metadata.name}{"."}{.metadata.namespace}'
```

### 🧹 Очистка объектов по префиксу (после `generateName`)
```bash
# Удалить все поды с префиксом test-pod-
kubectl get pods -o name | grep '^pod/test-pod-' | xargs -r kubectl delete

# Или через лейблы (если добавлял)
kubectl delete pods -l test-run=2024-06-05
```

### 🧪 Валидация имени в скрипте (bash)
```bash
validate_dns1123() {
  local name="$1"
  [[ ${#name} -le 253 ]] || return 1
  [[ "$name" =~ ^[a-z0-9]([a-z0-9.-]*[a-z0-9])?$ ]] || return 1
  return 0
}

if validate_dns1123 "my-app.prod"; then
  echo "✓ Имя валидно"
else
  echo "✗ Имя не соответствует RFC 1123"
fi
```

## 🔹 Чек-лист: избегание проблем с именами

```bash
# ✅ Перед созданием: проверь, не занят ли имя
kubectl get <kind> <name> -n <namespace> 2>/dev/null && echo "⚠️ Уже существует"

# ✅ Используй лейблы для группировки, а не имя
# ❌ Избегать: myapp-dev, myapp-prod, myapp-test (сложно выбрать все)
# ✅ Лучше: name: myapp + labels: env: dev/prod/test

# ✅ При генерации имён — добавляй аннотации/лейблы для трассировки
kubectl create pod --generate-name=job- \
  --labels="job-id=${JOB_ID}" \
  --annotations="created-by=ci-pipeline"

# ✅ Для ресурсов, на которые есть ссылки (Service, ConfigMap), 
# используй статичные, предсказуемые имена

# ✅ При отладке: сравнивай UID, если объект «ведёт себя странно»
kubectl get pod old-pod -o jsonpath='{.metadata.uid}'
kubectl get pod new-pod -o jsonpath='{.metadata.uid}'
# Если UID разные — это разные объекты, даже если имя одинаковое

# ✅ При написании операторов: используй `controllerutil.SetControllerReference` 
# для установки ownerReferences по UID, а не по имени
```

> 💡 **Совет для конспекта**:
> 1. Создай файл `00_naming_conventions.md` с краткой памяткой: «Какие правила для каких ресурсов».
> 2. Добавь блок «Частые ошибки»: например, «использовал заглавные буквы в имени пода», «забыл, что точка разрешена в поддомене, но не в лейбле».
> 3. Веди список «Шаблоны имён»: как ты называешь ресурсы в своих проектах (например, `{app}-{env}-{role}-{index}`).

## 🔹 Ключевые выводы

1. **Уникальность = 4 атрибута**: группа + тип + неймспейс + имя. Запомни эту формулу.
2. **Name ≠ UID**: имя — для людей, UID — для системы. При удалении/создании с тем же именем — это новый объект.
3. **Соблюдай стандарты**: DNS-правила не прихоть, а необходимость для интеграции с сетью, DNS, балансировщиками.
4. **`generateName` — мощный инструмент**: для тестов, джоб, операторов, но не для ресурсов, на которые есть внешние ссылки.
5. **Лейблы > имя для группировки**: имя должно быть уникальным идентификатором, а не носителем метаданных.
