>Эфемерные контейнеры (Ephemeral Containers) — это инструмент для отладки, который позволяет временно запустить контейнер в существующем поде.

# Эфемерные контейнеры (Ephemeral Containers) — Отладка в Kubernetes

> 📌 **Ephemeral Container** = временный контейнер для отладки, который можно добавить в **уже запущенный под**. Не имеет гарантий ресурсов, не перезапускается, не имеет портов. Используется через `kubectl debug`, а не через манифесты.

---

## 🔹 Зачем нужны эфемерные контейнеры

| Проблема | Решение |
|----------|---------|
| 🐛 Нужно отладить упавший контейнер, но в образе нет `sh`/`curl`/`ps` | Запустить эфемерный контейнер с отладочными утилитами в том же поде |
| 🔍 Нужно посмотреть процессы/сеть/файлы в работающем поде | Подключиться с общим пространством процессов (`--target`) |
| 🧩 Образ приложения минималистичный (distroless) и не подходит для отладки | Временно добавить полноценный образ (busybox, ubuntu) для инспекции |

> 💡 **Ключевая идея**: эфемерные контейнеры — это «инструмент в кармане» для админа, а не часть приложения. Они не влияют на работу пода и исчезают при его удалении.

---

## 🔹 Ограничения эфемерных контейнеров

| Запрещено | Почему |
|-----------|--------|
| **Порты (`ports`)** | Не должны конфликтовать с основным приложением |
| **Зонды (`livenessProbe`, `readinessProbe`)** | Не предназначены для долгой работы и проверки здоровья |
| **Ресурсы (`resources`)** | Spec пода неизменяем после создания |
| **`restartPolicy`** | Никогда не перезапускаются — если упал, то всё |
| **Изменение/удаление после создания** | Добавил → работает → завершился → удалился вместе с подом |

> ⚠️ **Важно**: эфемерные контейнеры добавляются через специальный subresource `/ephemeralcontainers`, а не через редактирование `pod.spec`. Поэтому `kubectl edit` не подойдёт.

## 🔹 Быстрый старт: `kubectl debug`

```bash
# Базовый случай: добавить контейнер для отладки
kubectl debug -it my-pod --image=busybox:1.36

# Подключиться к целевому контейнеру (общие процессы)
kubectl debug -it my-pod --image=busybox:1.36 --target=my-app

# Скопировать конфигурацию существующего контейнера (переменные, тома)
kubectl debug -it my-pod --image=busybox:1.36 --copy-to=my-app-debug --target=my-app

# Запустить в другом неймспейсе
kubectl debug -it my-pod -n production --image=ubuntu:22.04
```

### 🔍 Что происходит «под капотом»

```bash
# kubectl debug создаёт объект типа EphemeralContainer:
{
  "name": "debugger-abc123",
  "image": "busybox:1.36",
  "targetContainerName": "my-app",  # ← общие PID/сеть с этим контейнером
  "stdin": true,
  "tty": true
}

# И добавляет его в под через API:
POST /api/v1/namespaces/default/pods/my-pod/ephemeralcontainers
```

## 🔹 Практические сценарии

### 🐛 Отладка distroless-образа

```bash
# Приложение в distroless-образе упало, но нет оболочки для отладки
kubectl debug -it my-pod --image=busybox:1.36 --target=my-app

# Внутри эфемерного контейнера:
/ # ps aux                    # ← увидеть процессы основного приложения
/ # cat /proc/1/environ     # ← посмотреть переменные окружения
/ # netstat -tlnp           # ← проверить слушающие порты
/ # ls -la /var/log         # ← посмотреть логи (если том общий)
```

### 🔄 Инспекция без прерывания работы

```bash
# Добавить эфемерный контейнер, который просто читает файлы
kubectl debug my-pod --image=alpine:3.19 --target=my-app -- \
  sh -c 'cat /etc/app/config.yaml && echo "Config inspected"'

# Контейнер выполнит команду и завершится, не мешая основному приложению
```

### 🧹 Очистка эфемерных контейнеров

```bash
# Эфемерные контейнеры нельзя удалить вручную — они живут, пока жив под
# Но можно удалить весь под (если это допустимо):
kubectl delete pod my-pod

# Или дождаться, пока эфемерный контейнер завершится сам
```

## 🔹 Чек-лист: использование эфемерных контейнеров

### ✅ Когда использовать
```bash
• Нужен шелл/утилиты в поде, где их нет
• Нужно посмотреть процессы/сеть/файлы в контексте другого контейнера
• Отладка «на лету» без перезапуска пода
• Временная диагностика, не влияющая на работу приложения
```

### ❌ Когда НЕ использовать
```bash
• Для запуска долгосрочных вспомогательных сервисов → используй sidecar
• Для добавления функциональности приложению → пересобирай образ
• Для продакшен-логики → эфемерные контейнеры не имеют гарантий
```

### 🔍 Отладка проблем
```bash
# Эфемерный контейнер не запускается:
kubectl describe pod my-pod | grep -A10 'EphemeralContainers:'

# Нет доступа к процессам целевого контейнера:
# Убедись, что указан --target и включено общее пространство процессов

# Контейнер сразу завершается:
# Проверь команду: если она завершается быстро, контейнер тоже завершится
# Для интерактива добавляй -it и долгую команду (sleep, sh)
```

---

## 🔹 Ключевые выводы

1. **Эфемерные контейнеры = отладка**: временные, без гарантий, не для продакшен-логики.
2. **`kubectl debug` — основной интерфейс**: не редактируй манифесты вручную.
3. **`--target` для общего контекста**: доступ к процессам и сети другого контейнера.
4. **Ограничения — это фича**: нет портов/зондов/ресурсов, чтобы не ломать под.
5. **Не удаляются вручную**: живут до завершения пода или своего завершения.
