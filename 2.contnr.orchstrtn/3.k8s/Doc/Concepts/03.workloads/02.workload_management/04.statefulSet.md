# StatefulSet в Kubernetes — Управление stateful-приложениями

> 📌 **StatefulSet** — контроллер для приложений с состоянием (БД, очереди, кластеры). В отличие от Deployment, поды имеют **«липкую» идентичность**: уникальное имя (порядковый номер), стабильный сетевой идентификатор (DNS) и персональное постоянное хранилище (PVC). Требует **Headless Service**.

---

## 🔹 Главные отличия от Deployment

| Характеристика | **Deployment** | **StatefulSet** |
|----------------|---------------|-----------------|
| **Идентичность подов** | Случайные имена (`app-5f8b9c-xyz`), взаимозаменяемы | Строгие имена (`app-0`, `app-1`), уникальны |
| **Сеть (DNS)** | Общий ClusterIP для всех реплик | Уникальный DNS для каждого пода: `pod-0.svc.ns...` |
| **Хранилище** | Общий том для всех подов (или без него) | Персональный PVC для каждого пода (1:1) |
| **Порядок запуска** | Параллельно | Строго по порядку: `0` → `1` → `2` |
| **Порядок удаления** | Случайно / параллельно | Строго в обратном порядке: `2` → `1` → `0` |
| **Типичные нагрузки** | Веб-серверы, API, микросервисы | PostgreSQL, Kafka, Elasticsearch, Redis Cluster |

---

## 🔹 Анатомия StatefulSet

StatefulSet **всегда** требует связки с **Headless Service** (сервис без ClusterIP) для управления сетевым доменом подов.

```yaml
# 1. Headless Service (Обязательно!)
apiVersion: v1
kind: Service
metadata:
  name: postgres-headless
spec:
  clusterIP: None                # ← Ключевое: делает сервис "бесголовым"
  selector:
    app: postgres
  ports:
  - port: 5432

---
# 2. StatefulSet
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres-headless # ← Должен совпадать с именем Service!
  replicas: 3
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
      - name: postgres
        image: postgres:15
        volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data
  # 3. Шаблон для создания персональных томов
  volumeClaimTemplates:          
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOncePod"] # Рекомендуется для StatefulSet
      storageClassName: fast-ssd
      resources:
        requests:
          storage: 10Gi
```

---

## 🔹 Идентичность пода: Имена и Сеть

Каждый под получает стабильный идентификатор, который сохраняется даже при переезде на другую ноду.

### 📛 Именование и DNS
Для StatefulSet `web` с `replicas: 3` и Headless Service `nginx`:

| Индекс | Имя пода | Hostname | Полный DNS (FQDN) |
|--------|----------|----------|-------------------|
| 0 | `web-0` | `web-0` | `web-0.nginx.default.svc.cluster.local` |
| 1 | `web-1` | `web-1` | `web-1.nginx.default.svc.cluster.local` |
| 2 | `web-2` | `web-2` | `web-2.nginx.default.svc.cluster.local` |

> 💡 **Зачем это нужно**: Приложения (например, etcd или RabbitMQ) могут использовать эти DNS-имена для обнаружения друг друга и формирования кластера. `web-0` всегда будет `web-0`, даже если нода, на которой он работал, сгорела.

### 🏷️ Автоматические метки
StatefulSet добавляет к подам полезные метки:
* `statefulset.kubernetes.io/pod-name`: точное имя пода (например, `web-0`).
* `apps.kubernetes.io/pod-index`: порядковый номер (например, `0`). Удобно для фильтрации в Prometheus/Grafana.

## 🔹 Гарантии порядка (Ordering Guarantees)

StatefulSet гарантирует строгий порядок операций. Это критично для баз данных, где `node-1` должен подключиться к `node-0`.

### 🟢 Развертывание и Масштабирование (Up)
```
1. Создается web-0. Ждем, пока он станет Ready.
2. Создается web-1. Ждем, пока он станет Ready.
3. Создается web-2. Ждем, пока он станет Ready.
```
*Если `web-0` упадет, `web-1` и `web-2` не будут созданы, пока `web-0` не поднимется.*

### 🔴 Удаление и Уменьшение масштаба (Down)
```
1. Завершается web-2. Ждем полного завершения.
2. Завершается web-1. Ждем полного завершения.
3. Завершается web-0.
```

### ⚙️ Политики управления подами (`podManagementPolicy`)
* **`OrderedReady`** *(по умолчанию)*: Строгий порядок, описанный выше.
* **`Parallel`**: Все поды создаются/удаляются одновременно. Игнорирует готовность. Полезно для быстрого старта больших кластеров, где порядок инициализации не важен.

## 🔹 Стратегии обновлений (Update Strategies)

### 🔄 RollingUpdate (по умолчанию)
Обновление идет **в обратном порядке**: от старшего индекса к младшему (`web-2` → `web-1` → `web-0`).
1. Удаляется `web-2`, создается новый `web-2` с новым образом.
2. Ждем, пока новый `web-2` станет Ready.
3. Переходим к `web-1` и так далее.

**Полезные параметры `rollingUpdate`:**
* **`partition`**: Обновляются только поды с индексом `>= partition`.
  * *Пример*: `replicas: 3`, `partition: 2`. Обновится только `web-2`. Отлично для Canary-развертывания!
* **`maxUnavailable`** *(beta)*: Позволяет обновлять несколько подов параллельно (например, `maxUnavailable: 2` обновит `web-2` и `web-1` одновременно).

### 🛑 OnDelete
StatefulSet не обновляет поды автоматически. Чтобы применить новый образ, ты должен **вручную удалить** под (`kubectl delete pod web-0`). StatefulSet пересоздаст его с новым шаблоном.

## 🔹 Управление жизненным циклом PVC (Retention Policy)

> 🧩 **Статус**: Stable с K8s 1.32. Раньше PVC жили вечно, теперь можно управлять их удалением!

Поле `.spec.persistentVolumeClaimRetentionPolicy` определяет, что делать с томами.

| Событие | Политика `Retain` *(по умолчанию)* | Политика `Delete` |
|---------|-----------------------------------|-------------------|
| **Удаление StatefulSet** | PVC и данные **сохраняются** | PVC **удаляются** (данные теряются!) |
| **Scale Down** (уменьшение replicas) | PVC **сохраняются** | PVC удаляются **только для "лишних" подов** |
| **Сбой ноды / Пересоздание пода** | PVC **сохраняются** (подключаются к новому поду) | PVC **сохраняются** (политика не срабатывает при сбоях) |

```yaml
spec:
  persistentVolumeClaimRetentionPolicy:
    whenDeleted: Delete    # Удалить все PVC при `kubectl delete sts`
    whenScaled: Retain     # Оставить PVC при `kubectl scale sts --replicas=1`
```

> ⚠️ **Важно**: Если ты удаляешь StatefulSet для пересоздания (например, для смены `volumeClaimTemplates`), данные **не удалятся**, если политика `Retain`. Тебе придется чистить PVC вручную.

## 🔹 История ревизий и Откаты

StatefulSet хранит историю изменений в объектах **`ControllerRevision`**.

```bash
# Посмотреть историю изменений
kubectl rollout history statefulset/postgres

# Откатиться к предыдущей версии
kubectl rollout undo statefulset/postgres

# Откатиться к конкретной ревизии
kubectl rollout undo statefulset/postgres --to-revision=2

# Посмотреть сами ревизии (внутренняя кухня K8s)
kubectl get controllerrevisions -l app=postgres
```

## 🔹 ⚠️ Критические «подводные камни» и Ограничения

### 🚨 1. Проблема «Зависшего» обновления (Stuck Rollout)
**Симптом**: Ты обновил образ на битый (например, опечатка в конфиге). Под `web-2` ушел в `CrashLoopBackOff`. StatefulSet завис и не обновляет `web-1` и `web-0`.
**Почему**: Политика `OrderedReady` ждет, пока `web-2` станет Ready.
**Решение**:
1. Откати шаблон StatefulSet (`kubectl rollout undo ...` или `kubectl apply` старого манифеста).
2. **Вручную удали** зависший под: `kubectl delete pod web-2`.
3. StatefulSet пересоздаст `web-2` с откатанным (рабочим) шаблоном, и процесс пойдет дальше.

### 🚨 2. Headless Service обязателен
Без `clusterIP: None` и привязки через `serviceName` поды не получат стабильных DNS-имен. StatefulSet не пройдет валидацию или не будет работать как ожидается.

### 🚨 3. PVC не удаляются автоматически (без политики)
Если ты просто сделаешь `kubectl delete statefulset postgres`, поды удалятся, а PVC (и твои гигабайты данных) останутся в кластере навсегда, занимая место.
**Решение**: Либо используй `persistentVolumeClaimRetentionPolicy`, либо удаляй PVC вручную.

### 🚨 4. Ручное масштабирование vs Манифест
Если ты сделал `kubectl scale statefulset postgres --replicas=5`, а потом применил манифест, где `replicas: 3`, **манифест перезапишет твое ручное масштабирование**. Если используешь HPA — убери `replicas` из манифеста.

## 🔹 Чек-лист: работа со StatefulSet

### ✅ При проектировании
```bash
# • Убедись, что приложение действительно stateful (нужны стабильные имена/данные)
# • Создай Headless Service (clusterIP: None) ПЕРЕД созданием StatefulSet
# • Используй ReadWriteOncePod (RWOP) для volumeClaimTemplates, если версия K8s ≥ 1.29
# • Настрой terminationGracePeriodSeconds (БД нужно время на сброс кэша/wal)
```

### ✅ При обновлении и отладке
```bash
# • Используй partition для Canary-развертывания БД
#   (обнови только web-2, проверь, затем сдвигай partition вниз)

# • Если StatefulSet завис на обновлении:
#   1. kubectl rollout undo sts <name>
#   2. kubectl delete pod <name>-<broken-index>  # Принудительный сдвиг с мертвой точки

# • Проверяй, к каким PVC привязаны поды:
kubectl get pods -l app=postgres -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.volumes[?(@.persistentVolumeClaim)].persistentVolumeClaim.claimName}{"\n"}{end}'
```

### ✅ При удалении
```bash
# • Всегда проверяй persistentVolumeClaimRetentionPolicy перед удалением!
# • Если политика Retain, не забудь удалить PVC вручную:
kubectl delete pvc -l app=postgres

# • Для безопасного завершения (чтобы БД успела закрыть соединения):
#   Сначала scale down до 0, потом удаляй сам StatefulSet.
kubectl scale sts postgres --replicas=0
kubectl delete sts postgres
```

### ❌ Чего избегать
```bash
# ❌ Не используй StatefulSet для stateless-приложений
#   → Это усложнит жизнь (нужен Headless Service, строгий порядок).

# ❌ Не ставь terminationGracePeriodSeconds: 0
#   → БД или кластерное ПО не успеет корректно закрыться, данные могут повредиться.

# ❌ Не редактируй volumeClaimTemplates у существующего StatefulSet
#   → Это поле неизменяемо! Если нужно изменить размер/класс диска — это отдельная сложная операция.

# ❌ Не удаляй поды StatefulSet вручную без причины
#   → При удалении web-0, пока web-1 и web-2 работают, кластер может развалиться (потеря кворума).
```

## Ключевые выводы

1. **StatefulSet = Deployment + Идентичность**: Имя, DNS, Хранилище привязаны к индексу пода и не меняются при переездах.
2. **Headless Service — это закон**: Без `clusterIP: None` стабильной сети не будет.
3. **Строгий порядок**: `0` → `N-1` при старте, `N-1` → `0` при остановке. Это фича, а не баг (нужно для кворумов).
4. **volumeClaimTemplates**: Создает персональный PVC для каждого пода. По умолчанию PVC **никогда не удаляются** при удалении StatefulSet (если не настроена RetentionPolicy).
5. **Откаты и Зависания**: StatefulSet не умеет сам откатываться, если под ушел в CrashLoop. Нужно вручную удалять зависший под после отката шаблона.

> 💡 **Финальный совет**: StatefulSet — это не «волшебная таблетка» для запуска БД в Kubernetes. Запуск production-БД в K8s требует глубокого понимания storage (CSI, снапшоты, бэкапы). StatefulSet лишь дает правильный API для оркестрации.
