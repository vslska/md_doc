# Управление приложениями в Kubernetes — Практика

> 📌  После деплоя приложения нужно **обновлять**, **масштабировать** и **управлять конфигурациями**. K8s предоставляет: массовые операции (`apply -f`, `--recursive`), rolling updates (`rollout`), канареечные релизы (через лейблы), автоматическое масштабирование (HPA). Для сложных сценариев — Helm и Kustomize.

---

## 🔹 Организация манифестов

### 📋 Несколько ресурсов в одном файле

```yaml
# nginx-app.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-nginx-svc
  labels:
    app: nginx
spec:
  type: LoadBalancer
  ports:
  - port: 80
  selector:
    app: nginx
---                              # ← разделитель ресурсов
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-nginx
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
        ports:
        - containerPort: 80
```

```bash
# Применить все ресурсы из файла
kubectl apply -f nginx-app.yaml
# service/my-nginx-svc created
# deployment.apps/my-nginx created
```

> 💡 **Порядок важен**: ресурсы создаются в порядке объявления. Лучше сначала Service, потом Deployment — планировщик сразу свяжет поды с сервисом.

### 📁 Структура проекта

```
project/
├── k8s/
│   ├── base/
│   │   ├── deployment.yaml
│   │   ├── service.yaml
│   │   └── configmap.yaml
│   ├── overlays/
│   │   ├── dev/
│   │   │   └── kustomization.yaml
│   │   └── prod/
│   │       └── kustomization.yaml
│   └── common/
│       ├── namespace.yaml
│       └── networkpolicy.yaml
└── README.md
```

### 🔄 Рекурсивное применение

```bash
# ❌ Не работает: обрабатывает только первый уровень
kubectl apply -f project/k8s/development
# error: you must provide one or more resources

# ✅ Работает: рекурсивно обрабатывает подкаталоги
kubectl apply -f project/k8s/development --recursive
# или
kubectl apply -R -f project/k8s/development
# configmap/my-config created
# deployment.apps/my-deployment created
# persistentvolumeclaim/my-pvc created

# ✅ Работает с несколькими -f
kubectl apply -R -f project/k8s/namespaces -f project/k8s/development
```

> 💡 **`--recursive` / `-R`** работает с любой командой, принимающей `-f`: `apply`, `create`, `get`, `delete`, `describe`, `rollout`.

## 🔹 Массовые операции в kubectl

### 🗑️ Удаление нескольких ресурсов

```bash
# Удалить по файлу
kubectl delete -f nginx-app.yaml
# deployment.apps "my-nginx" deleted
# service "my-nginx-svc" deleted

# Удалить по именам (через пробел)
kubectl delete deployment/my-nginx service/my-nginx-svc

# Удалить по селектору (лейблам)
kubectl delete deployment,service -l app=nginx
# deployment.apps "my-nginx" deleted
# service "my-nginx-svc" deleted

# Удалить все ресурсы в неймспейсе
kubectl delete all --all -n default
# ⚠️ Осторожно: удалит ВСЁ в неймспейсе
```

### 🔗 Цепочки команд

```bash
# Создать ресурсы и показать только Services
kubectl create -f nginx-app.yaml -o name | grep service/ | xargs -i kubectl get '{}'
# NAME           TYPE           CLUSTER-IP   EXTERNAL-IP   PORT(S)      AGE
# my-nginx-svc   LoadBalancer   10.0.0.208   <pending>     80/TCP       0s

# Найти все поды, созданные определённым Deployment
kubectl get pods --selector=app=nginx -o name | xargs -i kubectl describe '{}'

# Удалить все завершенные Job
kubectl get jobs --field-selector=status.successful=1 -o name | xargs -r kubectl delete
```

### 📊 Фильтрация и форматирование

```bash
# JSONPath: извлечь имена подов
kubectl get pods -o jsonpath='{.items[*].metadata.name}'

# JSONPath: извлечь IP и статус
kubectl get pods -o jsonpath='{range .items[*]}{.status.podIP}{"\t"}{.status.phase}{"\n"}{end}'

# Custom columns: кастомные колонки
kubectl get pods -o custom-columns='NAME:.metadata.name,IP:.status.podIP,STATUS:.status.phase'

# Sort by: сортировка по полю
kubectl get pods --sort-by='.metadata.creationTimestamp'
kubectl get pods --sort-by='.status.startTime'
```

## 🔹 Обновления без сбоев (Rolling Updates)

### 🎯 Принцип работы

```
Deployment с replicas: 3, maxSurge: 25%, maxUnavailable: 25%

Шаг 1: Создать 1 новый под (3 + 1 = 4, maxSurge = 25% от 3)
Шаг 2: Дождаться готовности нового пода (readiness probe)
Шаг 3: Удалить 1 старый под (4 - 1 = 3, maxUnavailable = 25% от 3)
Шаг 4: Повторять, пока все поды не обновятся

Результат: всегда доступно минимум 2 пода (3 - 25% = 2.25 → 2)
```

### 📝 Пример: обновление образа

```bash
# 1. Создать Deployment
kubectl create deployment my-nginx --image=nginx:1.14.2
kubectl scale deployment my-nginx --replicas=3

# 2. Обновить образ (rolling update)
kubectl set image deployment/my-nginx nginx=nginx:1.16.1
# или
kubectl edit deployment/my-nginx  # ← изменить image вручную

# 3. Проверить статус обновления
kubectl rollout status deployment/my-nginx --timeout=10m
# deployment "my-nginx" successfully rolled out

# 4. Проверить историю ревизий
kubectl rollout history deployment/my-nginx
# REVISION  CHANGE-CAUSE
# 1         <none>
# 2         kubectl set image deployment/my-nginx nginx=nginx:1.16.1

# 5. Откатиться к предыдущей версии
kubectl rollout undo deployment/my-nginx
# или к конкретной ревизии
kubectl rollout undo deployment/my-nginx --to-revision=1
```

### ⚙️ Стратегии обновления

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-nginx
spec:
  replicas: 3
  strategy:
    type: RollingUpdate        # ← RollingUpdate (по умолчанию) или Recreate
    rollingUpdate:
      maxSurge: 25%            # ← макс. кол-во подов сверх replicas (25% или 1)
      maxUnavailable: 25%      # ← макс. кол-во недоступных подов (25% или 1)
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.16.1
```

| Стратегия | Поведение | Когда использовать |
|-----------|-----------|-------------------|
| **`RollingUpdate`** | Постепенная замена подов | Stateless-приложения, zero-downtime |
| **`Recreate`** | Сначала удалить все старые, потом создать новые | Stateful-приложения, когда нельзя параллельно работать |

### ⏸️ Пауза и возобновление

```bash
# Приостановить обновление (можно внести несколько изменений)
kubectl rollout pause deployment/my-nginx

# Внести изменения
kubectl set resources deployment/my-nginx --limits=cpu=200m,memory=512Mi
kubectl set image deployment/my-nginx nginx=nginx:1.17.0

# Возобновить обновление (применит все изменения за один rollout)
kubectl rollout resume deployment/my-nginx

# Проверить статус
kubectl rollout status deployment/my-nginx
```

## 🔹 Канареечные релизы (Canary Deployments)

### 🎯 Принцип

```
Stable (90% трафика)          Canary (10% трафика)
┌─────────────────┐           ┌─────────────────┐
│ Deployment v3   │           │ Deployment v4   │
│ replicas: 9     │           │ replicas: 1     │
│ track: stable   │           │ track: canary   │
└────────┬────────┘           └────────┬────────┘
         │                             │
         └──────────┬──────────────────┘
                    ▼
            ┌───────────────┐
            │   Service     │
            │ selector:     │
            │  app: nginx   │
            │  tier: front  │
            │  (без track)  │
            └───────────────┘
```

### 📝 Пример манифестов

```yaml
# stable-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend-stable
spec:
  replicas: 9
  selector:
    matchLabels:
      app: guestbook
      tier: frontend
      track: stable
  template:
    metadata:
      labels:
        app: guestbook
        tier: frontend
        track: stable      # ← лейбл для区分
    spec:
      containers:
      - name: frontend
        image: gb-frontend:v3
---
# canary-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend-canary
spec:
  replicas: 1
  selector:
    matchLabels:
      app: guestbook
      tier: frontend
      track: canary
  template:
    metadata:
      labels:
        app: guestbook
        tier: frontend
        track: canary      # ← другой лейбл
    spec:
      containers:
      - name: frontend
        image: gb-frontend:v4
---
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: frontend
spec:
  selector:
    app: guestbook
    tier: frontend
    # ← БЕЗ track! Сервис выберет оба: stable и canary
  ports:
  - port: 80
```

### 🔄 Процесс канареечного релиза

```bash
# 1. Развернуть stable (9 подов)
kubectl apply -f stable-deployment.yaml

# 2. Развернуть canary (1 под)
kubectl apply -f canary-deployment.yaml

# 3. Создать Service (без track в selector)
kubectl apply -f service.yaml

# 4. Проверить трафик (10% на canary)
kubectl get endpoints frontend
# NAME       ENDPOINTS   AGE
# frontend   10.0.0.1:80,10.0.0.2:80,...,10.0.0.10:80   1m

# 5. Если canary работает хорошо — обновить stable
kubectl set image deployment/frontend-stable frontend=gb-frontend:v4

# 6. Удалить canary
kubectl delete deployment frontend-canary
```

> 💡 **Совет**: для более сложного управления трафиком (50/50, A/B тесты) используй **Service Mesh** (Istio, Linkerd) или **Ingress Controller** с поддержкой canary (nginx-ingress).

## 🔹 Масштабирование

### 📈 Ручное масштабирование

```bash
# Масштабировать до 5 реплик
kubectl scale deployment/my-nginx --replicas=5
# deployment.apps/my-nginx scaled

# Проверить
kubectl get deployment my-nginx
# NAME       READY   UP-TO-DATE   AVAILABLE   AGE
# my-nginx   5/5     5            5           2m

# Масштабировать до 0 (полная остановка)
kubectl scale deployment/my-nginx --replicas=0
```

### 🤖 Автоматическое масштабирование (HPA)

```bash
# Создать HPA: 1-10 реплик, целевая загрузка CPU 80%
kubectl autoscale deployment/my-nginx --min=1 --max=10 --cpu-percent=80
# horizontalpodautoscaler.autoscaling/my-nginx autoscaled

# Проверить статус HPA
kubectl get hpa
# NAME       REFERENCE             TARGETS   MINPODS   MAXPODS   REPLICAS   AGE
# my-nginx   Deployment/my-nginx   45%/80%   1         10        3          1m

# Детальная информация
kubectl describe hpa my-nginx

# Удалить HPA
kubectl delete hpa my-nginx
```

### 📝 Манифест HPA

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-nginx
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 80    # ← целевая загрузка CPU
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80    # ← целевая загрузка памяти
```

> ⚠️ **Важно**: для работы HPA нужно установить **Metrics Server**:
> ```bash
> kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
> ```

## 🔹 Обновление ресурсов

### 📝 `kubectl apply` (рекомендуется)

```bash
# Применить изменения из файла (декларативно)
kubectl apply -f deployment.yaml
# deployment.apps/my-nginx configured

# Dry-run: проверить, что изменится, без применения
kubectl apply -f deployment.yaml --dry-run=server -o yaml

# Применить с валидацией
kubectl apply -f deployment.yaml --validate=strict
```

> 💡 **`apply`** сравнивает текущее состояние с файлом и применяет только изменения. Не перезаписывает поля, которые ты не указал.

### ✏️ `kubectl edit` (быстрые правки)

```bash
# Открыть ресурс в редакторе
kubectl edit deployment/my-nginx
# Откроется в $EDITOR (vi, nano, vscode)

# После сохранения — автоматически применит изменения
# deployment.apps/my-nginx configured
```

> 💡 **Указать редактор**:
> ```bash
> export EDITOR=vim
> # или
> export KUBE_EDITOR="code --wait"  # VS Code
> ```

### 🔧 `kubectl patch` (точечные изменения)

```bash
# JSON Patch (RFC 6902)
kubectl patch deployment my-nginx --type='json' -p='[
  {"op": "replace", "path": "/spec/replicas", "value": 5}
]'

# JSON Merge Patch (RFC 7386)
kubectl patch deployment my-nginx --type='merge' -p='{"spec":{"replicas":5}}'

# Strategic Merge Patch (Kubernetes-specific, понимает списки)
kubectl patch deployment my-nginx --type='strategic' -p='{"spec":{"template":{"spec":{"containers":[{"name":"nginx","image":"nginx:1.17.0"}]}}}}'
```

| Тип патча | Когда использовать |
|-----------|-------------------|
| **`strategic`** (по умолчанию) | Для большинства ресурсов K8s (понимает `patchStrategy` в API) |
| **`merge`** | Для CRD и ресурсов, не поддерживающих strategic merge |
| **`json`** | Для точечных изменений (добавить/удалить/заменить конкретное поле) |

### 🔄 `kubectl replace --force` (пересоздание)

```bash
# Пересоздать ресурс (удалить и создать заново)
kubectl replace -f deployment.yaml --force
# deployment.apps/my-nginx deleted
# deployment.apps/my-nginx replaced
```

> ⚠️ **Когда использовать**:
> - Поле нельзя обновить после создания (например, `spec.selector`)
> - Нужно сбросить состояние ресурса
> - Экстренное исправление сломанных подов

> ⚠️ **Осторожно**: `--force` удалит ресурс и создаст заново — будет downtime!

## 🔹 Внешние инструменты

### 🛠️ Helm — пакетный менеджер

```bash
# Установить Helm
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

# Добавить репозиторий
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Установить чарт
helm install my-release bitnami/nginx

# Список релизов
helm list

# Обновить релиз
helm upgrade my-release bitnami/nginx --set replicaCount=5

# Откатить
helm rollback my-release 1

# Удалить
helm uninstall my-release
```

> 💡 **Helm Charts** — это шаблоны манифестов с переменными. Идеально для сложных приложений (БД, мониторинг, service mesh).

### 🎨 Kustomize — кастомизация без шаблонов

```bash
# kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
- deployment.yaml
- service.yaml
patches:
- path: patch-replicas.yaml
configMapGenerator:
- name: my-config
  literals:
  - KEY=value
```

```bash
# Применить с Kustomize (встроено в kubectl)
kubectl apply -k ./overlays/prod/

# Или через kustomize CLI
kustomize build ./overlays/prod/ | kubectl apply -f -
```

> 💡 **Kustomize vs Helm**:
> - **Kustomize**: оверлеи, патчи, генераторы. Нет шаблонов, чистый YAML.
> - **Helm**: шаблоны с переменными, репозитории чартов. Более мощный, но сложнее.

## 🔹 Чек-лист: управление приложением

```bash
# ✅ 1. Организация манифестов
#    - Группировать связанные ресурсы в один файл (через ---)
#    - Использовать структуру: base/overlays или по неймспейсам
#    - Хранить в Git (Infrastructure as Code)

# ✅ 2. Применение изменений
kubectl apply -f deployment.yaml --dry-run=server -o yaml  # тест
kubectl apply -f deployment.yaml                           # применение
kubectl apply -R -f ./k8s/                                 # рекурсивно

# ✅ 3. Обновления
kubectl set image deployment/my-app app=my-app:v2          # обновить образ
kubectl rollout status deployment/my-app                   # проверить статус
kubectl rollout history deployment/my-app                  # история
kubectl rollout undo deployment/my-app                     # откат

# ✅ 4. Масштабирование
kubectl scale deployment/my-app --replicas=5               # вручную
kubectl autoscale deployment/my-app --min=1 --max=10 --cpu-percent=80  # HPA

# ✅ 5. Отладка
kubectl get pods -l app=my-app -o wide                     # статус подов
kubectl logs deployment/my-app --tail=100                  # логи
kubectl describe deployment/my-app                         # детали
kubectl exec -it deployment/my-app -- sh                   # войти в под

# ✅ 6. Очистка
kubectl delete -f deployment.yaml                          # по файлу
kubectl delete all -l app=my-app                           # по лейблу
kubectl delete all --all -n default                        # всё в неймспейсе (осторожно!)

# ✅ 7. Мониторинг
kubectl get hpa                                            # статус HPA
kubectl top pods -l app=my-app                             # использование ресурсов
kubectl get events --sort-by='.lastTimestamp'              # события
```

> 💡 **Совет для конспекта**:
> 1. Создай файл `00_app_management_cheatsheet.md` с шпаргалкой по частым командам.
> 2. Добавь блок «Частые ошибки»: например, «забыл `--recursive` для подкаталогов», «использовал `replace` вместо `apply`».
> 3. Веди список «Какие инструменты используем»: Helm charts, Kustomize overlays, внешние операторы.

## 🔹 Ключевые выводы

1. **Организация манифестов**: группируй ресурсы в один файл (через `---`), используй `--recursive` для каталогов.
2. **Массовые операции**: `apply -f`, `delete -l`, цепочки через `xargs`, фильтрация через `-o jsonpath`.
3. **Rolling updates**: `kubectl set image` + `rollout status/undo`. Стратегии: `RollingUpdate` (по умолчанию) или `Recreate`.
4. **Канареечные релизы**: два Deployment с разными лейблами (`track: stable/canary`), один Service без лейбла `track`.
5. **Масштабирование**: ручное (`scale`) или автоматическое (HPA с Metrics Server).
6. **Обновление ресурсов**: `apply` (декларативно), `edit` (быстро), `patch` (точечно), `replace --force` (пересоздание).
7. **Внешние инструменты**: Helm (пакетный менеджер), Kustomize (кастомизация без шаблонов).
