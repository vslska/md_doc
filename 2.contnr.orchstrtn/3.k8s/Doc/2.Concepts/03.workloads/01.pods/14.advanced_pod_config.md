>Расширенная конфигурация подов охватывает критически важные механизмы: приоритеты, среды выполнения, безопасность и планирование.

# Расширенная конфигурация подов в Kubernetes

> *📌* Расширенная конфигурация подов включает: 
> **PriorityClass** (приоритет при нехватке ресурсов), 
> **RuntimeClass** (выбор runtime: runc, kata, gvisor),
> **SecurityContext** (безопасность: user, capabilities, seccomp), **Affinity/Tolerations** (где запускать),
> **Pod Overhead** (учёт ресурсов runtime). Всё это позволяет тонко настраивать поведение подов под конкретные требования.

---

## 1. PriorityClass: приоритеты подов

### 🎯 Что это и зачем

| Аспект | Описание |
|--------|----------|
| **Определение** | Cluster-scoped объект, задающий числовой приоритет для подов |
| **Назначение** | Определяет порядок вытеснения (preemption) при нехватке ресурсов |
| **Механизм** | Чем выше `value`, тем выше приоритет. Scheduler может вытеснить поды с меньшим приоритетом |
| **Область** | Cluster-scoped (один на весь кластер) |

### 📋 Создание PriorityClass

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 1000000              # ← Чем больше, тем выше приоритет
globalDefault: false        # ← false: не делать дефолтным для всех подов
preemptionPolicy: PreemptLowerPriority  # ← Можно вытеснять поды с меньшим приоритетом
description: "Priority class for critical workloads"
```

### 📋 Использование в поде

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: critical-app
spec:
  priorityClassName: high-priority  # ← Ссылка на PriorityClass
  containers:
  - name: app
    image: my-app:1.0
```

### 🏆 Встроенные PriorityClass

| Класс | Value | Назначение |
|-------|-------|-----------|
| **`system-node-critical`** | 2000001000 | Критичные для ноды компоненты (например, kube-proxy) |
| **`system-cluster-critical`** | 2000000000 | Критичные для кластера компоненты (например, CoreDNS) |

> ⚠️ **Важно**: не используй встроенные классы для своих приложений. Создавай свои PriorityClass с разумными значениями (например, 1000, 10000, 100000).

### 🔄 Механизм preemption

```
Сценарий: кластер перегружен, нет ресурсов для high-priority пода

1. Scheduler видит: нужен под с priority=1000000
2. Проверяет: есть ли свободные ресурсы? → Нет
3. Ищет поды с меньшим приоритетом (например, priority=1000)
4. Вытесняет (preempts) достаточно подов с низким приоритетом
5. Освобождает ресурсы → запускает high-priority под

Результат: критичное приложение получило ресурсы, менее критичное — потеряло поды
```

### 🛠️ Практика

```bash
# Создать PriorityClass
kubectl apply -f priority-class.yaml

# Посмотреть все PriorityClass
kubectl get priorityclasses

# Проверить приоритет пода
kubectl get pod my-pod -o jsonpath='{.spec.priority}{"\n"}{.spec.priorityClassName}'

# Посмотреть, какие поды были вытеснены
kubectl get events --field-selector reason=Preempted
```

## 🔹 RuntimeClass: выбор среды выполнения

### 🎯 Что это и зачем

| Аспект | Описание |
|--------|----------|
| **Определение** | Cluster-scoped объект, определяющий runtime контейнеров (runc, kata, gvisor и др.) |
| **Назначение** | Выбор уровня изоляции и возможностей runtime для разных подов |
| **Примеры** | `runc` (стандарт), `kata` (VM-изоляция), `gvisor` (sandbox), `kvisor` (security) |
| **Область** | Cluster-scoped (настраивается админом кластера) |

### 📋 Создание RuntimeClass

```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: kata-containers
handler: kata              # ← Имя handler'а в containerd/CRI-O
overhead:                  # ← Накладные расходы runtime (см. ниже)
  podFixed:
    memory: "128Mi"
    cpu: "250m"
scheduling:                # ← На каких нодах доступен этот runtime
  nodeSelector:
    runtime: kata
```

### 📋 Использование в поде

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
spec:
  runtimeClassName: kata-containers  # ← Ссылка на RuntimeClass
  containers:
  - name: app
    image: my-app:1.0
```

### 🔍 Когда использовать разные runtime

| Runtime | Изоляция | Производительность | Когда использовать |
|---------|----------|-------------------|-------------------|
| **runc** | Namespaces + cgroups | Высокая | Стандартные приложения, большинство случаев |
| **kata** | Виртуальная машина | Средняя | Multi-tenant, недоверенный код, compliance |
| **gvisor** | User-space kernel | Средняя | Serverless, FaaS, sandboxing |
| **kvisor** | Security-focused | Высокая | Enhanced security, eBPF-based monitoring |

### 🛠️ Практика

```bash
# Посмотреть все RuntimeClass
kubectl get runtimeclasses

# Проверить, какой runtime используется подом
kubectl get pod my-pod -o jsonpath='{.spec.runtimeClassName}'

# Посмотреть детали RuntimeClass
kubectl describe runtimeclass kata-containers
```

## 🔹 SecurityContext: безопасность подов и контейнеров

### 🎯 Что это и зачем

| Аспект | Описание |
|--------|----------|
| **Определение** | Поле в spec пода/контейнера для настройки параметров безопасности |
| **Уровни** | Pod-level (применяется ко всем контейнерам) и Container-level (переопределяет pod-level) |
| **Назначение** | Контроль UID/GID, capabilities, seccomp, SELinux, AppArmor, privileged mode |

### 📋 Pod-level SecurityContext

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: security-demo
spec:
  securityContext:           # ← Применяется ко всем контейнерам
    runAsUser: 1000          # ← UID для всех процессов
    runAsGroup: 3000         # ← GID для всех процессов
    fsGroup: 2000            # ← GID для файлов в томах
    runAsNonRoot: true       # ← Запретить запуск от root
    seccompProfile:
      type: RuntimeDefault   # ← Seccomp профиль
  containers:
  - name: app
    image: my-app:1.0
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    emptyDir: {}
```

### 📋 Container-level SecurityContext

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: container-security-demo
spec:
  containers:
  - name: app
    image: my-app:1.0
    securityContext:         # ← Только для этого контейнера
      allowPrivilegeEscalation: false  # ← Запретить повышение привилегий
      runAsNonRoot: true
      runAsUser: 1000
      capabilities:
        drop: ["ALL"]        # ← Убрать все Linux capabilities
        add: ["NET_BIND_SERVICE"]  # ← Добавить только нужные
      seccompProfile:
        type: Localhost
        localhostProfile: profiles/my-app.json  # ← Кастомный seccomp профиль
      readOnlyRootFilesystem: true  # ← Root FS только для чтения
```

### 🔐 Основные параметры SecurityContext

| Параметр | Уровень | Описание | Рекомендация |
|----------|---------|----------|-------------|
| **`runAsUser`** | Pod/Container | UID для запуска процессов | Не 0 (не root) |
| **`runAsGroup`** | Pod/Container | GID для запуска процессов | Не 0 |
| **`fsGroup`** | Pod | GID для файлов в томах | Для совместимости с томами |
| **`runAsNonRoot`** | Pod/Container | Запретить запуск от root | ✅ `true` |
| **`allowPrivilegeEscalation`** | Container | Разрешить повышение привилегий (setuid) | ❌ `false` |
| **`capabilities.drop`** | Container | Убрать Linux capabilities | ✅ `["ALL"]` |
| **`capabilities.add`** | Container | Добавить специфичные capabilities | Только если необходимо |
| **`readOnlyRootFilesystem`** | Container | Root FS только для чтения | ✅ `true` |
| **`seccompProfile.type`** | Pod/Container | Seccomp профиль | ✅ `RuntimeDefault` |
| **`privileged`** | Container | Privileged mode (полный доступ к хосту) | ❌ Избегать! |

### 🛠️ Практика

```bash
# Проверить, от какого пользователя запущен процесс
kubectl exec my-pod -- id
# → uid=1000(app) gid=3000(app) groups=3000(app),2000

# Проверить capabilities
kubectl exec my-pod -- cat /proc/1/status | grep Cap
# → CapEff: 0000000000000000  (нет capabilities)

# Проверить, root filesystem read-only
kubectl exec my-pod -- touch /test
# → touch: /test: Read-only file system

# Проверить seccomp профиль
kubectl exec my-pod -- cat /proc/1/status | grep Seccomp
# → Seccomp: 2  (2 = SECCOMP_MODE_FILTER)
```

## 🔹 Планирование: где запускать поды

### 🎯 Механизмы управления размещением

| Механизм | Назначение | Сложность | Когда использовать |
|----------|-----------|-----------|-------------------|
| **nodeSelector** | Простая выборка нод по меткам | Низкая | Простые требования (например, только SSD) |
| **nodeAffinity** | Гибкие правила для нод (In, NotIn, Exists) | Средняя | Сложные требования, soft/hard constraints |
| **podAffinity** | Размещать поды рядом с другими подами | Средняя | Кэши, базы данных, связанные сервисы |
| **podAntiAffinity** | Размещать поды подальше от других подов | Средняя | Высокая доступность, распределение по зонам |
| **tolerations** | Разрешить запуск на нодах с taints | Средняя | Специальные ноды (GPU, spot instances) |

### 📋 nodeSelector: простой выбор

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-ssd
spec:
  nodeSelector:
    disktype: ssd            # ← Нода должна иметь метку disktype=ssd
  containers:
  - name: nginx
    image: nginx:1.25
```

```bash
# Добавить метку к ноде
kubectl label nodes node-1 disktype=ssd

# Проверить, на какой ноде запущен под
kubectl get pod nginx-ssd -o wide
```

### 📋 nodeAffinity: гибкие правила

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: affinity-demo
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:   # ← Hard constraint
        nodeSelectorTerms:
        - matchExpressions:
          - key: topology.kubernetes.io/zone
            operator: In
            values:
            - us-east-1a
            - us-east-1b
      preferredDuringSchedulingIgnoredDuringExecution:  # ← Soft constraint
      - weight: 80
        preference:
          matchExpressions:
          - key: node-type
            operator: In
            values:
            - high-performance
  containers:
  - name: app
    image: my-app:1.0
```

### 📋 podAffinity: размещение рядом

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: cache
spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - web-server        # ← Размещать рядом с подами app=web-server
        topologyKey: topology.kubernetes.io/zone  # ← В той же зоне
  containers:
  - name: redis
    image: redis:7.0
```

### 📋 podAntiAffinity: распределение

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-server
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - web-server        # ← Не размещать рядом с другими web-server
        topologyKey: kubernetes.io/hostname  # ← На разных нодах
  containers:
  - name: nginx
    image: nginx:1.25
```

### 📋 tolerations: запуск на tainted нодах

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: gpu-app
spec:
  tolerations:
  - key: "nvidia.com/gpu"
    operator: "Exists"        # ← Принять любой taint с ключом nvidia.com/gpu
    effect: "NoSchedule"      # ← Игнорировать эффект NoSchedule
  containers:
  - name: ml-training
    image: tensorflow:latest
    resources:
      limits:
        nvidia.com/gpu: 1
```

```bash
# Добавить taint к ноде
kubectl taint nodes node-1 nvidia.com/gpu=true:NoSchedule

# Проверить taints ноды
kubectl describe node node-1 | grep -A5 'Taints:'
```

## 🔹 Pod Overhead: учёт накладных расходов

### 🎯 Что это и зачем

| Аспект | Описание |
|--------|----------|
| **Определение** | Дополнительные ресурсы, потребляемые runtime контейнеров (помимо контейнеров) |
| **Назначение** | Точный учёт ресурсов при планировании (scheduler учитывает overhead) |
| **Пример** | Kata Containers добавляет ~128Mi RAM и 250m CPU на каждый под |
| **Настройка** | В RuntimeClass.overhead |

### 📋 Пример RuntimeClass с overhead

```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: kata-containers
handler: kata
overhead:
  podFixed:                # ← Фиксированные накладные расходы на под
    memory: "128Mi"
    cpu: "250m"
```

### 📊 Как учитывается overhead

```
Под с runtimeClass: kata-containers
Контейнер: requests={memory: 256Mi, cpu: 500m}

Scheduler видит:
• Контейнер: 256Mi RAM, 500m CPU
• Overhead: 128Mi RAM, 250m CPU
• ИТОГО: 384Mi RAM, 750m CPU

Планировщик ищет ноду с ≥384Mi RAM и ≥750m CPU
```

### 🛠️ Практика

```bash
# Проверить overhead для RuntimeClass
kubectl get runtimeclass kata-containers -o jsonpath='{.overhead}'

# Посмотреть реальные ресурсы, зарезервированные для пода
kubectl describe pod my-pod | grep -A10 'Allocated resources'
```

## 🔹 Чек-лист: расширенная конфигурация подов

### ✅ PriorityClass
```bash
# • Создавай свои PriorityClass с разумными значениями (1000, 10000, 100000)
# • Не используй system-cluster-critical/system-node-critical для приложений
# • Указывай priorityClassName в spec пода для критичных workload'ов
# • Тестируй preemption в staging перед production
```

### ✅ RuntimeClass
```bash
# • Используй runc для стандартных приложений (по умолчанию)
# • Используй kata/gvisor для multi-tenant или недоверенного кода
# • Настраивай overhead для точного учёта ресурсов
# • Убедись, что ноды имеют нужный runtime и метки для scheduling
```

### ✅ SecurityContext
```bash
# • Всегда задавай runAsNonRoot: true
# • Всегда drop'ай все capabilities: capabilities.drop: ["ALL"]
# • Добавляй только необходимые capabilities (например, NET_BIND_SERVICE)
# • Используй readOnlyRootFilesystem: true где возможно
# • Применяй seccompProfile: type: RuntimeDefault
# • Избегай privileged: true (полный доступ к хосту)
```

### ✅ Планирование (Affinity/Tolerations)
```bash
# • Используй nodeSelector для простых требований (одна метка)
# • Используй nodeAffinity для сложных правил (несколько меток, операторы)
# • Используй podAntiAffinity для высокой доступности (разные ноды/зоны)
# • Используй tolerations для специальных нод (GPU, spot instances)
# • Комбинируй hard (required) и soft (preferred) constraints
```

### ✅ Pod Overhead
```bash
# • Настраивай overhead в RuntimeClass для kata/gvisor
# • Проверяй, что scheduler учитывает overhead при планировании
# • Мониторь реальное использование ресурсов vs запрошенное
```

### ❌ Чего избегать
```bash
# ❌ Не используй privileged: true без веской причины
# ❌ Не запускай контейнеры от root (runAsUser: 0)
# ❌ Не оставляй capabilities по умолчанию (всегда drop ALL)
# ❌ Не игнорируй overhead для kata/gvisor (scheduler будет ошибаться)
# ❌ Не используй system-cluster-critical для своих приложений
# ❌ Не размещай все реплики на одной ноде (используй podAntiAffinity)
```

## 🔹 Ключевые выводы

1. **PriorityClass**: определяет приоритет подов при нехватке ресурсов. Высокий приоритет может вытеснить низкий.
2. **RuntimeClass**: выбор runtime (runc, kata, gvisor) для разного уровня изоляции и безопасности.
3. **SecurityContext**: критичен для безопасности. Всегда non-root, drop capabilities, seccomp, readOnlyRootFilesystem.
4. **Affinity/Tolerations**: гибкое управление размещением подов на нодах и относительно других подов.
5. **Pod Overhead**: учёт ресурсов runtime при планировании. Критично для kata/gvisor.
6. **Комбинируй механизмы**: PriorityClass + SecurityContext + Affinity = полный контроль над поведением пода.

> 💡 **Финальный совет**: начинай с базовой конфигурации (SecurityContext с non-root и drop capabilities), добавляй сложность (PriorityClass, Affinity, RuntimeClass) только когда есть реальная потребность. Всегда тестируй в staging перед production, особенно preemption и overhead.
