>Жизненный цикл пода (Pod Lifecycle) — это фундамент понимания того, как Kubernetes управляет рабочими нагрузками: от создания до завершения и утилизации.

# Жизненный цикл пода (Pod Lifecycle) в Kubernetes

> 📌 Под проходит фазы: `Pending` → `Running` → `Succeeded`/`Failed`. Контейнеры внутри имеют состояния: `Waiting` → `Running` → `Terminated`. Kubernetes управляет перезапусками через `restartPolicy`, проверяет здоровье через **зонды**, и корректно завершает работу через `terminationGracePeriod`. Поды эфемерны: их не «чинят», а заменяют.

---

## 🔹 Фазы пода (Pod Phases)

`status.phase` — высокоуровневое описание состояния пода.

| Фаза | Описание | Когда возникает |
|------|----------|---------------|
| **`Pending`** | Под принят кластером, но контейнеры ещё не запущены | Ожидание планирования, скачивание образов, инициализация томов |
| **`Running`** | Под привязан к ноде, ≥1 контейнер запущен или перезапускается | Основное состояние работающего приложения |
| **`Succeeded`** | Все контейнеры успешно завершены (код 0) и не будут перезапущены | Завершённые `Job`, одноразовые задачи |
| **`Failed`** | Все контейнеры завершены, ≥1 завершился с ошибкой (код ≠0) | Сбой приложения, исчерпание попыток перезапуска |
| **`Unknown`** | Не удалось получить статус пода (проблема связи с нодой) | Сбой ноды, сетевое разделение, `kubelet` не отвечает |

```bash
# Проверить фазу пода
kubectl get pod my-pod -o jsonpath='{.status.phase}'

# Подробная информация о состоянии
kubectl describe pod my-pod | grep -A10 'Status:'
```

> ⚠️ **Важно**: `phase` — не исчерпывающее состояние. Для деталей смотри `status.conditions` и `containerStatuses`.

---

## 🔹 Состояния контейнеров (Container States)

Каждый контейнер в поде имеет одно из трёх состояний:

| Состояние | Описание | Частые причины |
|-----------|----------|---------------|
| **`Waiting`** | Контейнер ещё не запущен, выполняется подготовка | Скачивание образа, ожидание секретов, `ImagePullBackOff`, `ErrImagePull` |
| **`Running`** | Контейнер выполняется без проблем | Нормальная работа приложения |
| **`Terminated`** | Контейнер завершил работу (успешно или с ошибкой) | Код выхода 0 (успех) или ≠0 (ошибка), `OOMKilled`, `Evicted` |

```bash
# Посмотреть состояние контейнеров в поде
kubectl describe pod my-pod | grep -A5 'State:'

# Проверить код выхода завершённого контейнера
kubectl get pod my-pod -o jsonpath='{.status.containerStatuses[*].lastState.terminated.exitCode}'
```

### 🔍 Частые статусы в `kubectl get pods`

| Статус | Значение | Причина |
|--------|----------|---------|
| **`CrashLoopBackOff`** | Контейнер падает → перезапускается → снова падает | Ошибка приложения, неверная конфигурация, нехватка ресурсов |
| **`ImagePullBackOff`** | Не удалось скачать образ | Неверное имя образа, нет доступа к реестру, сетевая проблема |
| **`ErrImagePull`** | Ошибка при скачивании образа (предшествует `ImagePullBackOff`) | Временная проблема сети или реестра |
| **`OOMKilled`** | Контейнер превысил лимит памяти и был убит ядром | Слишком низкий `limits.memory`, утечка памяти в приложении |
| **`Evicted`** | Под удалён из-за нехватки ресурсов на ноде | Давление на диск/память, нода перегружена |

## 🔹 Политика перезапуска (restartPolicy)

Определяет, как Kubernetes реагирует на завершение контейнеров.

### 📋 Уровни применения

| Уровень | Поле | Применимость |
|---------|------|-------------|
| **Пода** | `spec.restartPolicy` | Применяется ко всем контейнерам, кроме sidecar |
| **Контейнера** | `spec.containers[*].restartPolicy` | Переопределяет политику пода для конкретного контейнера (β с 1.35) |
| **Sidecar** | `initContainers[*].restartPolicy: Always` | Sidecar игнорирует политику пода, всегда перезапускается |

### 🔄 Три значения политики

| Значение | Поведение при выходе 0 (успех) | Поведение при выходе ≠0 (ошибка) | Когда использовать |
|----------|-------------------------------|---------------------------------|-------------------|
| **`Always`** (по умолчанию) | Перезапускает | Перезапускает | Долгоживущие сервисы (Deployment, StatefulSet) |
| **`OnFailure`** | Не перезапускает | Перезапускает | Пакетные задачи (Job), где нужно повторить при сбое |
| **`Never`** | Не перезапускает | Не перезапускает | Одноразовые задачи, отладка, миграции |

### 🧪 Примеры

```yaml
# Deployment: всегда перезапускать (единственное допустимое значение)
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      restartPolicy: Always  # ← обязательно для Deployment
      containers:
      - name: app
        image: my-app:1.0

# Job: перезапускать только при ошибке
apiVersion: batch/v1
kind: Job
spec:
  template:
    spec:
      restartPolicy: OnFailure  # ← типично для задач
      containers:
      - name: task
        image: busybox
        command: ['sh', '-c', 'echo "Processing..."; exit 0']

# Sidecar: всегда работает, независимо от основного контейнера
spec:
  restartPolicy: OnFailure  # ← для основного контейнера
  initContainers:
  - name: log-shipper
    image: fluentd:latest
    restartPolicy: Always   # ← sidecar игнорирует политику пода!
```

> 💡 **Sidecar-контейнеры**: это `initContainers` с `restartPolicy: Always`. Они запускаются до основных контейнеров, но работают параллельно с ними на протяжении всей жизни пода.

## 🔹 Экспоненциальная задержка перезапуска (CrashLoopBackOff)

При повторяющихся сбоях Kubernetes применяет задержку перед перезапуском, чтобы не перегружать систему.

### 📊 Алгоритм задержки (по умолчанию)

```
Попытка 1: 10 сек
Попытка 2: 20 сек
Попытка 3: 40 сек
...
Максимум: 300 сек (5 минут)

Если контейнер работает стабильно ≥10 минут → счётчик сбрасывается
```

### ⚙️ Настройка задержки (на уровне ноды)

```yaml
# /var/lib/kubelet/config.yaml
kind: KubeletConfiguration
crashLoopBackOff:
  maxContainerRestartPeriod: "60s"  # Макс. задержка вместо 300с
```

> 🧩 **Фича-гейты**:
> - `ReduceDefaultCrashLoopBackOffDecay` (α): начать с 1с, максимум 60с
> - `KubeletCrashLoopBackOffMax` (β): настраиваемый максимум на ноду

### 🔍 Отладка CrashLoopBackOff

```bash
# 1. Посмотреть логи контейнера
kubectl logs my-pod --previous  # предыдущая попытка
kubectl logs my-pod -c app      # текущий контейнер

# 2. Проверить события пода
kubectl describe pod my-pod | grep -A10 'Events:'

# 3. Проверить ресурсы (нехватка памяти/ЦП?)
kubectl top pod my-pod
kubectl describe pod my-pod | grep -A5 'Limits'

# 4. Проверить зонды (не слишком ли агрессивные?)
kubectl describe pod my-pod | grep -A10 'Liveness\|Readiness'

# 5. Протестировать образ локально
docker run --rm my-app:1.0  # или через podman/nerdctl
```

## 🔹 Зонды (Probes): проверка здоровья

| Тип | Назначение | Действие при провале | Когда проверять |
|-----|-----------|---------------------|----------------|
| **`startupProbe`** | Приложение запустилось? | Ждать, не убивать | Только при старте, пока не пройдёт |
| **`livenessProbe`** | Приложение живо и не зависло? | Перезапустить контейнер | Периодически на протяжении жизни |
| **`readinessProbe`** | Контейнер готов принимать трафик? | Исключить из балансировки (удалить из Endpoints) | Периодически, особенно после старта |

### ⚙️ Пример настройки всех трёх

```yaml
spec:
  containers:
  - name: web-app
    image: my-app:1.0
    ports:
    - containerPort: 8080
    
    # Даём 60 секунд на старт (6 проверок * 10с)
    startupProbe:
      httpGet: { path: /startup, port: 8080 }
      periodSeconds: 10
      failureThreshold: 6
    
    # Перезапускать при зависании (3 провала * 10с)
    livenessProbe:
      httpGet: { path: /healthz, port: 8080 }
      periodSeconds: 10
      failureThreshold: 3
    
    # Исключать из балансировки, если не готов (2 провала * 5с)
    readinessProbe:
      httpGet: { path: /ready, port: 8080 }
      periodSeconds: 5
      failureThreshold: 2
```

### 🔍 Типы проверок

| Механизм | Пример | Когда использовать |
|----------|--------|-------------------|
| **`httpGet`** | `httpGet: { path: /health, port: 8080 }` | Веб-приложения с HTTP-эндпоинтом |
| **`tcpSocket`** | `tcpSocket: { port: 5432 }` | Базы данных, сервисы без HTTP |
| **`exec`** | `exec: { command: ['pg_isready'] }` | Проверка через команду внутри контейнера |
| **`grpc`** | `grpc: { port: 9090, service: "health" }` | gRPC-сервисы (β с 1.24) |

> 💡 **Совет**: 
> - `startupProbe` — для медленных приложений (загрузка кэша, миграции БД)
> - `livenessProbe` — не делай слишком агрессивным, чтобы временные задержки не вызывали перезапуски
> - `readinessProbe` — обязателен для сервисов, чтобы не направлять трафик на неготовые поды

## 🔹 Завершение работы пода (Termination)

### 🔄 Процесс корректного завершения

```
1. Пользователь: kubectl delete pod my-pod (или контроллер инициирует удаление)
2. API Server:
   • Устанавливает .metadata.deletionTimestamp
   • Запускает таймер .spec.terminationGracePeriodSeconds (по умолчанию 30с)
3. Kubelet на ноде:
   • Если есть preStop hook → выполняет его
   • Отправляет SIGTERM процессу 1 в каждом контейнере
   • Ждёт завершения процессов (до истечения grace period)
4. Если процессы не завершились за grace period:
   • Отправляет SIGKILL всем оставшимся процессам
5. Kubelet:
   • Очищает ресурсы (сеть, тома)
   • Удаляет объект пода из API Server
```

### ⚙️ Настройка завершения

```yaml
spec:
  terminationGracePeriodSeconds: 60  # ← увеличить, если приложению нужно больше времени
  
  containers:
  - name: app
    image: my-app:1.0
    lifecycle:
      preStop:
        exec:
          command: ["/bin/sh", "-c", "nginx -s quit"]  # ← корректно остановить nginx
```

### 🪶 Принудительное удаление (осторожно!)

```bash
# Удалить с кастомным grace period
kubectl delete pod my-pod --grace-period=10

# Принудительно удалить (немедленно, без ожидания завершения)
kubectl delete pod my-pod --force --grace-period=0

# ⚠️ Warning: под может продолжить работать на ноде, пока kubelet не очистит его
```

> ⚠️ **Риск**: `--force` не гарантирует, что приложение корректно завершилось. Используй только для «зависших» подов, которые не удаляются обычным способом.

## 🔹 Условия пода (Pod Conditions)

`status.conditions` — детализированное состояние пода.

| Условие | Значение | Описание |
|---------|----------|----------|
| `PodScheduled` | `True`/`False` | Под успешно запланирован на ноду |
| `PodReadyToStartContainers` | `True`/`False` | Песочница создана, сеть настроена, тома смонтированы |
| `Initialized` | `True`/`False` | Все init-контейнеры успешно завершены |
| `ContainersReady` | `True`/`False` | Все контейнеры готовы (прошли readiness) |
| `Ready` | `True`/`False` | Под готов принимать трафик (суммарный статус) |
| `DisruptionTarget` | `True`/`False` | Под будет удалён из-за прерывания (дрейн, эвикшн) |
| `PodResizePending` / `InProgress` | `True`/`False` | Запрошено/выполняется изменение ресурсов пода |

```bash
# Посмотреть условия пода
kubectl get pod my-pod -o jsonpath='{.status.conditions}'

# Проверить, готов ли под к трафику
kubectl get pod my-pod -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}'
```

### 🔧 Пользовательские условия готовности (Readiness Gates)

```yaml
spec:
  readinessGates:
  - conditionType: "www.example.com/custom-check"  # ← кастомное условие

# Приложение должно само обновлять это условие через PATCH /status
# Под считается Ready только если:
# • Все контейнеры готовы (ContainersReady=True)
# • Все readinessGates выполнены (conditionType=True)
```

## 🔹 Изменение размера пода (Pod Resize)

> 🧩 **Статус**: стабильно с 1.35 (изменение ресурсов ЦП/памяти)

### 🔄 Два подхода

| Подход | Описание | Когда использовать |
|--------|----------|-------------------|
| **На месте (In-place)** | Обновить `.spec.containers[*].resources` → kubelet применяет изменения без пересоздания пода | Критичные приложения, где пересоздание дорого; быстрое масштабирование |
| **Через замену (Recreate)** | Обновить шаблон в Deployment/StatefulSet → контроллер создаёт новые поды с новыми ресурсами | Стандартный подход, работает во всех версиях, позволяет менять любые поля |

### ⚙️ Изменение размера на месте

```bash
# Запросить изменение ресурсов через subresource /resize
kubectl patch pod my-pod --subresource=resize --type=merge -p '{
  "spec": {
    "containers": [{
      "name": "app",
      "resources": {
        "requests": { "memory": "512Mi", "cpu": "500m" },
        "limits": { "memory": "1Gi", "cpu": "1" }
      }
    }]
  }
}'

# Проверить статус операции
kubectl get pod my-pod -o jsonpath='{.status.conditions[?(@.type=="PodResizeInProgress")]}'
```

> ⚠️ **Ограничения**:
> - Нельзя изменить QoS-класс пода (определяется при создании)
> - Для некоторых изменений может потребоваться перезапуск контейнера (настраивается через `resizePolicy`)

## 🔹 Сборка мусора подов (Pod GC)

Завершённые поды не удаляются автоматически — нужен контроллер или ручное вмешательство.

### 🗑️ Когда поды удаляются автоматически

| Сценарий | Механизм | Настройка |
|----------|----------|-----------|
| **Завершённые поды** | `PodGC` контроллер удаляет при превышении порога | `--terminated-pod-gc-threshold` в `kube-controller-manager` |
| **Осиротевшие поды** | Под привязан к несуществующей ноде | Автоматически, при обнаружении отсутствия ноды |
| **Поды на нездоровых нодах** | Нода `NotReady` + тейнт `out-of-service` | Автоматически, при очистке ноды |

### 🧹 Ручная очистка

```bash
# Удалить все завершённые поды
kubectl delete pod --field-selector=status.phase=Succeeded -A
kubectl delete pod --field-selector=status.phase=Failed -A

# Удалить поды старше 24 часов (через jsonpath + xargs)
kubectl get pods -A --field-selector=status.phase=Failed \
  -o jsonpath='{range .items[*]}{.metadata.namespace}{" "}{.metadata.name}{"\n"}{end}' | \
  xargs -n2 kubectl delete pod -n

# Настроить автоматическую очистку через TTLController (для Job)
apiVersion: batch/v1
kind: Job
spec:
  ttlSecondsAfterFinished: 3600  # ← удалить объект через 1 час после завершения
```

## 🔹 Чек-лист: отладка жизненного цикла пода

### 🚨 Под не запускается
```bash
# 1. Проверить события
kubectl describe pod my-pod | grep -A10 'Events:'

# 2. Проверить состояние контейнеров
kubectl get pod my-pod -o jsonpath='{.status.containerStatuses[*]}' | jq

# 3. Посмотреть логи (включая предыдущие попытки)
kubectl logs my-pod --previous
kubectl logs my-pod -c init-container  # если есть init-контейнеры

# 4. Проверить ресурсы и лимиты
kubectl describe pod my-pod | grep -A5 'Limits\|Requests'
kubectl top pod my-pod  # если установлен Metrics Server

# 5. Проверить зонды
kubectl describe pod my-pod | grep -A10 'Liveness\|Readiness\|Startup'
```

### 🔄 Под в CrashLoopBackOff
```bash
# 1. Логи приложения — первая точка отладки
kubectl logs my-pod --tail=100

# 2. Проверить переменные окружения и конфиги
kubectl exec my-pod -- env
kubectl describe pod my-pod | grep -A5 'Mounts'

# 3. Проверить, не убивается ли контейнер из-за памяти
kubectl describe pod my-pod | grep -i 'oom\|killed'

# 4. Протестировать образ локально
docker run --rm -it my-app:1.0 /bin/sh
```

### 🛑 Под не удаляется / завис в Terminating
```bash
# 1. Проверить финализаторы
kubectl get pod my-pod -o jsonpath='{.metadata.finalizers}'

# 2. Проверить, не блокирует ли удаление зависимый ресурс
kubectl describe pod my-pod | grep -A5 'Events:' | grep -i 'finalizer\|deletion'

# 3. Если финализатор «завис» — удалить вручную (осторожно!)
kubectl patch pod my-pod --type json -p '[{"op": "remove", "path": "/metadata/finalizers"}]'

# 4. Принудительно удалить (последнее средство)
kubectl delete pod my-pod --force --grace-period=0
```

### 📊 Мониторинг жизненного цикла
```bash
# Алерт на частые перезапуски (Prometheus)
sum by (pod) (rate(kube_pod_container_status_restarts_total[5m])) > 0.5

# Алерт на поды в CrashLoopBackOff
kube_pod_status_phase{phase="Running"} * on(pod) group_right() kube_pod_container_status_waiting_reason{reason="CrashLoopBackOff"}

# Алерт на долгое завершение (терминация > 2 минут)
time() - kube_pod_deletion_timestamp{...} > 120
```

## 🔹 Ключевые выводы

1. **Фазы и состояния**: `phase` — высокоуровнево, `conditions` и `containerStatuses` — детально.
2. **RestartPolicy**: `Always` для сервисов, `OnFailure` для задач, `Never` для одноразовых операций.
3. **Зонды — основа надёжности**: `startup` для долгого старта, `liveness` от зависаний, `readiness` для балансировки.
4. **Graceful shutdown**: настраивай `terminationGracePeriodSeconds` и `preStop` для корректного завершения.
5. **CrashLoopBackOff — не баг, а фича**: это механизм защиты, но требует отладки корневой причины.
6. **Поды эфемерны**: не «чини» упавший под — замени его через контроллер с исправленной конфигурацией.

> 💡 **Финальный совет**: проектируй приложения так, чтобы они корректно реагировали на перезапуски, не зависели от локального состояния и быстро стартовали. Это упростит жизнь и тебе, и Kubernetes.
