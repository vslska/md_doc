# Secret — Управление конфиденциальными данными в Kubernetes

> 📌 `Secret` — объект API для хранения **конфиденциальных** данных (пароли, токены, ключи). Похож на ConfigMap, но с дополнительной защитой. **Лимит: 1 МиБ**. По умолчанию хранится в **незашифрованном виде** в etcd — обязательно включай шифрование at rest и настраивай RBAC.

---

## 🔹 Зачем нужен Secret

| Проблема | Решение с Secret |
|----------|------------------|
| Пароли и токены зашиты в образе контейнера | Вынесены во внешний объект K8s |
| Конфиденциальные данные видны в `docker inspect` | Secret хранится отдельно от образа |
| Разные креды для dev/prod | Один образ, разные Secret для разных сред |

### ⚠️ Критичные предупреждения по безопасности

**Secret — НЕ панацея!** По умолчанию:

1. ❌ Данные хранятся в **незашифрованном виде** в etcd (base64 — это не шифрование!)
2. ❌ Любой с доступом к etcd может прочитать все секреты
3. ❌ Любой, кто может создать Pod в namespace, может прочитать **все** секреты этого namespace (через volume mount)
4. ❌ Secret виден в логах `kubectl describe pod` (если используется как env var)

### 🛡️ Minimum Security Checklist

```text
[ ] Включено шифрование Secret at rest (EncryptionConfiguration)
[ ] Настроены RBAC-правила с минимальными привилегиями
[ ] Ограничен доступ к Secret только нужным контейнерам
[ ] Рассмотрено использование внешних хранилищ (Vault, AWS Secrets Manager, etc.)
```

## 🔹 Структура объекта Secret

Secret состоит из двух полей для данных. Имя должно быть валидным DNS-поддоменом.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-secret
  namespace: default
type: Opaque              # ← тип секрета (по умолчанию Opaque)
data:
  # Данные в Base64 (обязательно для data)
  username: YWRtaW4=      # admin
  password: cDRzc3cwcmQ=  # p4ssw0rd
stringData:
  # Данные в открытом виде (K8s сам закодирует в base64 при сохранении)
  api-key: super-secret-key-12345
immutable: true           # ← опционально: запрет на изменение (v1.21+)
```

**Правила:**
- `data` — значения должны быть в **Base64**
- `stringData` — значения в **открытом виде** (удобно для манифестов)
- Если ключ есть и в `data`, и в `stringData` — приоритет у `stringData`
- Ключи: только буквенно-цифровые, `-`, `_`, `.`

## 🔹 Типы Secrets

Kubernetes предоставляет встроенные типы для распространённых сценариев.

| Тип | Назначение | Обязательные ключи |
|-----|------------|-------------------|
| **`Opaque`** (по умолчанию) | Произвольные данные | — |
| **`kubernetes.io/tls`** | TLS-сертификаты | `tls.crt`, `tls.key` |
| **`kubernetes.io/dockerconfigjson`** | Креды для Docker Registry | `.dockerconfigjson` |
| **`kubernetes.io/basic-auth`** | Basic Auth | `username`, `password` |
| **`kubernetes.io/ssh-auth`** | SSH-ключи | `ssh-privatekey` |
| **`kubernetes.io/service-account-token`** | Токен ServiceAccount | (заполняется автоматически) |
| **`bootstrap.kubernetes.io/token`** | Bootstrap токены для kubeadm | `token-id`, `token-secret` |

> 💡 **Зачем типы?** K8s проверяет наличие обязательных ключей и помогает инструментам (Ingress, kubelet) понимать структуру секрета.

## 🔹 Практика: Создание Secrets

### Способ 1: Через kubectl (быстро)

```bash
# 1. Opaque из literal
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password='p4ssw0rd'

# 2. Opaque из файла
kubectl create secret generic tls-cert \
  --from-file=tls.crt=./cert.pem \
  --from-file=tls.key=./key.pem

# 3. TLS-секрет (специальная команда)
kubectl create secret tls my-tls \
  --cert=./cert.pem \
  --key=./key.pem

# 4. Docker Registry credentials
kubectl create secret docker-registry regcred \
  --docker-server=myregistry.example.com \
  --docker-username=user \
  --docker-password=pass \
  --docker-email=user@example.com
```

### Способ 2: Через YAML-манифест (рекомендуется для GitOps)

```yaml
# TLS Secret
apiVersion: v1
kind: Secret
metadata:
  name: my-tls-secret
type: kubernetes.io/tls
stringData:
  tls.crt: |
    -----BEGIN CERTIFICATE-----
    MIICpDCCAYwCCQDU+pQ4pHgSpDANBgkqhkiG9w0BAQsFADAUMRIwEAYDVQQDDAls
    ...
    -----END CERTIFICATE-----
  tls.key: |
    -----BEGIN PRIVATE KEY-----
    MIIEvgIBADANBgkqhkiG9w0BAQEFAASCBKgwggSkAgEAAoIBAQC7...
    ...
    -----END PRIVATE KEY-----
```

```yaml
# Basic Auth Secret
apiVersion: v1
kind: Secret
metadata:
  name: basic-auth-secret
type: kubernetes.io/basic-auth
stringData:
  username: admin
  password: t0p-Secret
```

> ⚠️ **Важно**: `stringData` **плохо работает** с Server-Side Apply. Для GitOps лучше использовать `data` с base64-значениями или Kustomize с `secretGenerator`.

## 🔹 3 способа использования Secret в Pod

| Способ | Как работает | Когда использовать |
|--------|--------------|-------------------|
| **Переменные окружения** | K8s инжектирует значения как `ENV` | Простые креды, токены |
| **Том (Volume Mount)** | Монтирование как файлов в ФС пода | TLS-сертификаты, SSH-ключи, сложные конфиги |
| **imagePullSecrets** | Kubelet использует для pull из private registry | Приватные Docker-реестры |

### Способ А: Переменные окружения

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: env-secret-demo
spec:
  containers:
  - name: app
    image: nginx
    env:
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: db-creds
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-creds
          key: password
    # Или импортировать все ключи сразу
    envFrom:
    - secretRef:
        name: db-creds
```

> ⚠️ **Проблема**: Значения env vars видны через `kubectl describe pod` и в `/proc/1/environ`. Не используй для высокочувствительных данных.

### Способ Б: Монтирование как тома (рекомендуется)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: volume-secret-demo
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: tls-certs
      mountPath: "/etc/tls"
      readOnly: true
  volumes:
  - name: tls-certs
    secret:
      secretName: my-tls-secret
      # Опционально: задать права доступа к файлам
      defaultMode: 0400  # ← только владелец может читать
      # Опционально: монтировать только конкретные ключи
      items:
      - key: tls.crt
        path: server.crt
      - key: tls.key
        path: server.key
```

**Преимущества:**
- ✅ Данные хранятся в `tmpfs` (RAM), не на диске
- ✅ Файлы удаляются при удалении пода
- ✅ Можно настроить права доступа (`defaultMode`)
- ✅ Автоматически обновляются при изменении Secret

### Способ В: imagePullSecrets (для private registry)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: private-image-demo
spec:
  containers:
  - name: app
    image: myregistry.example.com/myapp:v1
  imagePullSecrets:
  - name: regcred
```

> 💡 **Совет**: Чтобы не указывать `imagePullSecrets` в каждом Pod, добавь его в ServiceAccount (см. документацию по ServiceAccounts).

## 🔹 Поведение при обновлении

| Способ использования | Реагирует на обновление Secret? | Детали |
|----------------------|--------------------------------|--------|
| **Том (Volume)** | ✅ **Да** (автоматически) | Kubelet синхронизирует файлы. Задержка зависит от стратегии кэша (watch по умолчанию). |
| **Переменные окружения** | ❌ **Нет** | Требуется **рестарт пода**. |
| **Том с `subPath`** | ❌ **Нет** | Если смонтирован конкретный файл через `subPath`, обновления не работают. |

> 💡 **Совет**: Для горячего обновления TLS-сертификатов используй том (без `subPath`) и убедись, что приложение умеет перечитывать файлы (или отправь SIGHUP через sidecar).

## 🔹 Optional Secrets

По умолчанию, если Secret не существует, Pod **не запустится**. Можно сделать Secret опциональным:

```yaml
volumes:
- name: optional-secret
  secret:
    secretName: maybe-exists
    optional: true  # ← если Secret нет, Pod всё равно запустится
```

## 🔹 Immutable Secrets

> **Стабильно с v1.21**. Помечает Secret как доступный только для чтения.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: immutable-secret
type: Opaque
data:
  api-key: c3VwZXItc2VjcmV0
immutable: true
```

**Преимущества:**
1. **Безопасность**: Защита от случайного изменения критичных кредов.
2. **Производительность**: Kubelet не отслеживает изменения (watch) для immutable Secrets. Критично для кластеров с десятками тысяч секретов.

**Ограничения:**
- После установки `immutable: true` изменить `data` **невозможно**.
- Чтобы изменить — создать новый Secret и обновить Pod.

## 🔹 Dotfiles (скрытые файлы)

Можно создать ключ, начинающийся с точки — он станет скрытым файлом:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: dotfile-secret
data:
  .secret-file: dmFsdWUtMg==
```

При монтировании как тома файл будет называться `.secret-file` (скрытый в `ls`, виден в `ls -la`).

## 🔹 Разделение Secret между контейнерами в Pod

Если в Pod несколько контейнеров, можно дать доступ к Secret только одному из них:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: multi-container-secret
spec:
  containers:
  - name: frontend
    image: myapp
    # Нет доступа к secret-volume!
  - name: signer
    image: signer
    volumeMounts:
    - name: secret-volume
      mountPath: "/etc/secrets"
      readOnly: true
  volumes:
  - name: secret-volume
    secret:
      secretName: signing-key
```

**Зачем**: Если frontend скомпрометирован, злоумышленник не получит доступ к секретному ключу подписи.

## 🔹 Шпаргалка: Работа с Secrets через kubectl

```bash
# 1. Посмотреть список Secrets
kubectl get secrets
kubectl get secrets -o wide  # с типом

# 2. Посмотреть детали (без значений!)
kubectl describe secret my-secret

# 3. Декодировать значение
kubectl get secret my-secret -o jsonpath='{.data.username}' | base64 -d

# 4. Декодировать все значения
kubectl get secret my-secret -o json | jq -r '.data | to_entries[] | "\(.key): \(.value | @base64d)"'

# 5. Создать Secret из файла
kubectl create secret generic my-secret --from-file=key=./file.txt

# 6. Редактировать Secret (откроется в редакторе)
kubectl edit secret my-secret

# 7. Удалить Secret
kubectl delete secret my-secret

# 8. Проверить, какие Secrets используются в Pod
kubectl get pod my-pod -o jsonpath='{.spec.volumes[*].secret.secretName}'
kubectl get pod my-pod -o jsonpath='{.spec.containers[*].env[*].valueFrom.secretKeyRef.name}'
```

## 🔹 Чек-лист: Best Practices для Secrets

```text
[ ] Включено шифрование Secret at rest (EncryptionConfiguration в kube-apiserver)
[ ] Настроены RBAC-правила: доступ к Secret только для нужных ServiceAccounts
[ ] Используется тип Secret, соответствующий назначению (tls, dockerconfigjson, etc.)
[ ] Для TLS-сертификатов используется тип kubernetes.io/tls
[ ] Для private registry используется imagePullSecrets
[ ] Критичные секреты помечены как immutable: true
[ ] Секреты монтируются как тома (не env vars) для высокочувствительных данных
[ ] Установлены правильные права доступа (defaultMode: 0400)
[ ] Не используются Secrets в статических Pod (не поддерживается)
[ ] Рассмотрено использование внешних хранилищ (Vault, AWS Secrets Manager) для production
```

## 🔹 Альтернативы Secret

Для production рассмотри более безопасные решения:

| Решение | Описание | Когда использовать |
|---------|----------|-------------------|
| **HashiCorp Vault** | Внешнее хранилище секретов с динамической генерацией | Production, enterprise |
| **AWS Secrets Manager** | Managed-сервис для секретов | AWS EKS |
| **Azure Key Vault** | Managed-сервис для секретов | Azure AKS |
| **External Secrets Operator** | Синхронизация внешних секретов в K8s | Мультикластер, гибридные среды |
| **ServiceAccount Tokens** | Встроенные токены для Pod-to-Pod аутентификации | Внутренняя аутентификация в кластере |

## 🔹 Ключевые выводы

1. **Secret ≠ ConfigMap**: Secret для конфиденциальных данных, ConfigMap для конфигов.
2. **Безопасность**: По умолчанию Secret **не зашифрован** в etcd. Обязательно включай EncryptionConfiguration.
3. **Типы**: Используй встроенные типы (`tls`, `dockerconfigjson`) для валидации и совместимости с инструментами.
4. **Способы использования**: Тома (рекомендуется для чувствительных данных) > ENV vars (видно в `describe pod`).
5. **Обновления**: Тома обновляются автоматически, ENV vars — только при рестарте пода.
6. **Immutable**: Помечай критичные секреты как `immutable: true` для безопасности и производительности.
7. **imagePullSecrets**: Для private registry. Можно добавить в ServiceAccount для автоматического применения.
8. **Альтернативы**: Для production рассмотри Vault, AWS Secrets Manager, External Secrets Operator.
