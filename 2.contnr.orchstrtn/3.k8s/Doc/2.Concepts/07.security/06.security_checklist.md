
# Security Checklist — итоговый чек-лист безопасности K8s

> 📌 Это сводный чек-лист, объединяющий все темы безопасности. Используй как **checklist для аудита кластера**. Не все пункты обязательны — выбирай в зависимости от threat model. Чек-лист — это **первый шаг**, а не финальная цель.

---

## 🔹 Как использовать этот чек-лист

- ⬜ — не сделано
- 🟡 — в процессе
- ✅ — сделано
- ⏭️ — не применимо / не требуется

> 💡 **Порядок пунктов не отражает приоритет**. Оценивай каждый пункт по своей threat model.

---

## 🔐 1. Аутентификация и авторизация

### 🔑 Identity & Access

- [ ] **`system:masters` не используется** для повседневной работы (только break-glass)
- [ ] **`kube-controller-manager` запущен с `--use-service-account-credentials`** (каждый контроллер со своим SA)
- [ ] **Root CA защищён** (offline CA или управляемый CA с строгим контролем доступа)
- [ ] **Срок действия сертификатов ≤ 3 года** (для промежуточных и конечных)
- [ ] **Периодический review доступа** (не реже раза в 24 месяца)

### 🛡️ RBAC

- [ ] **Принцип минимальных привилегий** — только необходимые verbs и resources
- [ ] **Нет wildcard (`*`)** в Role/ClusterRole без крайней необходимости
- [ ] **Отдельный SA для каждого приложения** (не `default`)
- [ ] **Namespace-scoped роли** (Role + RoleBinding) вместо cluster-wide где возможно
- [ ] **Запрет `list` на secrets** — только `get` с `resourceNames`
- [ ] **Регулярный аудит RBAC** (`kubectl auth can-i --list`)

> 📖 См. темы: `20_k8s_rbac.md`, `19_k8s_service_accounts.md`

---

## 🌐 2. Сетевая безопасность

### 🔒 Network Policies

- [ ] **CNI-плагин поддерживает NetworkPolicy** (Calico, Cilium — не Flannel)
- [ ] **Default deny в каждом namespace** (запрет всего входящего/исходящего)
- [ ] **Explicit allow policies** для каждой рабочей нагрузки
- [ ] **Egress policies** ограничивают исходящий трафик (особенно к metadata API `169.254.169.254`)

### 🔐 Encryption in Transit

- [ ] **etcd peer communication** — mTLS с отдельным CA
- [ ] **API Server ↔ kubelet** — TLS
- [ ] **Service mesh** (Istio, Linkerd) для mTLS между подами (если требуется)

### 🚪 External Access

- [ ] **API Server не публичен в интернете** (bastion host, VPN, private endpoint)
- [ ] **Kubelet API не публичен** (только изнутри кластера)
- [ ] **etcd не публичен** (только API Server имеет доступ)
- [ ] **Cloud metadata API заблокирован** для подов (если не нужен)
- [ ] **`DenyServiceExternalIPs` admission controller включён** (защита от CVE-2020-8554)
- [ ] **LoadBalancer и ExternalIPs ограничены** через ResourceQuota или admission

> 📖 См. темы: `16_k8s_networking_overview.md`, `17_k8s_service.md`, `18_k8s_ingress.md`

---

## 📦 3. Безопасность подов

### 🛡️ Pod Security Standards

- [ ] **PSA включён** во всех namespaces (кроме system)
- [ ] **`kube-system`, `kube-public`, `kube-node-lease`** → `Privileged`
- [ ] **Обычные приложения** → минимум `Baseline`, лучше `Restricted`
- [ ] **Gradual rollout**: `warn` → `audit` → `enforce`
- [ ] **Версия политики зафиксирована** (например, `v1.28`, не `latest`)

### 🔒 Security Context

- [ ] **`runAsNonRoot: true`** для всех подов
- [ ] **`allowPrivilegeEscalation: false`**
- [ ] **`capabilities.drop: ["ALL"]`** + только необходимые `add`
- [ ] **`seccompProfile.type: RuntimeDefault`** (или custom профиль)
- [ ] **`readOnlyRootFilesystem: true`** где возможно
- [ ] **AppArmor/SELinux** профили включены (для Linux-нод)

### 💾 Resource Limits

- [ ] **Memory limits** установлены (≤ requests для предотвращения OOM узла)
- [ ] **CPU limits** установлены для resource-intensive workloads
- [ ] **LimitRange** в каждом namespace (дефолтные limits)
- [ ] **ResourceQuota** в каждом namespace (общие лимиты)

> 📖 См. темы: `21_k8s_pss.md`, `22_k8s_psa.md`

---

## 📋 4. Журналы событий и аудит

### 📝 Audit Logging

- [ ] **Audit log включён** на API Server
- [ ] **Audit policy настроена** (как минимум — для secrets, pods, RBAC)
- [ ] **Audit log защищён** от несанкционированного доступа
- [ ] **Audit log хранится** в защищённом месте (S3, SIEM)
- [ ] **Алерты на подозрительную активность**:
  - Массовое чтение secrets
  - Создание привилегированных подов
  - Изменения RBAC
  - Доступ из необычных IP

### 📊 Observability

- [ ] **Centralized logging** (EFK, Loki, Fluentd)
- [ ] **Metrics** (Prometheus + Grafana)
- [ ] **Tracing** (Jaeger, Zipkin) — опционально
- [ ] **Алерты** на security-события

## 🏗️ 5. Размещение подов (Scheduling)

### 🎯 Node Isolation

- [ ] **Sensitive workloads** на отдельных нодах (через `nodeSelector`, `affinity`)
- [ ] **Taints/tolerations** для изоляции критичных workloads
- [ ] **RuntimeClass** для sandboxed workloads (gVisor, Kata) где нужно
- [ ] **PodNodeSelector** admission controller для ограничения выбора нод
- [ ] **PodTolerationRestriction** для ограничения tolerations в namespace

### 🚫 Anti-Affinity

- [ ] **Anti-affinity** для реплик критичных приложений (разные ноды/зоны)
- [ ] **TopologySpreadConstraints** для равномерного распределения

> 📖 См. темы: планирование, taints/tolerations (из Блока 3 плана)

---

## 🔑 6. Секреты

### 🔐 Хранение

- [ ] **ConfigMaps НЕ используются** для конфиденциальных данных
- [ ] **Encryption at Rest** включён для secrets в etcd (aescbc или kms)
- [ ] **Существующие секреты перешифрованы** после включения Encryption at Rest
- [ ] **etcd диски шифруются** на уровне ОС (LUKS, eCryptfs)

### 🏢 External Secret Stores

- [ ] **Vault / AWS Secrets Manager / GCP Secret Manager** для production
- [ ] **Secret Store CSI Driver** установлен и настроен
- [ ] **Автоматическая ротация** секретов настроена

### 🚫 Mounting

- [ ] **`automountServiceAccountToken: false`** для подов, которым не нужен API
- [ ] **Projected volume tokens** (не legacy Secret-based tokens)
- [ ] **Bound tokens** с ограниченным audience и expiration
- [ ] **Secrets монтируются в tmpfs** (`emptyDir.medium: Memory`) где возможно
- [ ] **Env vars для секретов** — только если нельзя использовать volume mounts

### 🛡️ Access Control

- [ ] **RBAC на secrets** — минимальные привилегии (только `get` с `resourceNames`)
- [ ] **Namespace isolation** — каждый app в своём namespace
- [ ] **Аудит доступа** к secrets

> 📖 См. темы: `23_k8s_secrets_best_practices.md`

## 🐳 7. Образы контейнеров

### 📦 Minimization

- [ ] **Минимальный базовый образ** (distroless, alpine, scratch)
- [ ] **Нет shell и debug utilities** в production образах
- [ ] **Нет sensitive data** в слоях образа (используй `.dockerignore`)
- [ ] **Multi-stage builds** для уменьшения размера

### 👤 User

- [ ] **`USER` инструкция в Dockerfile** — непривилегированный пользователь
- [ ] **`runAsUser` / `runAsGroup`** в securityContext (не root)
- [ ] **Файлы в образе** имеют правильные permissions

### 🔒 Integrity

- [ ] **Образы ссылаются по SHA256 digest**, а не по тегу (особенно не `latest`)
- [ ] **Image signing** (Cosign, Notary) + проверка подписи при деплое
- [ ] **ImagePolicyWebhook** или **Kyverno/OPA** для проверки политик

### 🔍 Scanning

- [ ] **Сканирование образов** в CI/CD (Trivy, Snyk, Grype)
- [ ] **Блокировка деплоя** образов с critical/high CVE
- [ ] **Регулярное пересканирование** образов в registry
- [ ] **Automated patching** для базовых образов

### 🔄 Updates

- [ ] **`AlwaysPullImages` admission controller** (для multi-tenant)
- [ ] **Image pull secrets** хранятся безопасно (не в Git)

## 🛡️ 8. Admission Controllers

### ✅ Включены по умолчанию (не отключай!)

- [ ] **`CertificateApproval`** — проверка прав на approval сертификатов
- [ ] **`CertificateSigning`** — проверка прав на подписание сертификатов
- [ ] **`CertificateSubjectRestriction`** — запрет `system:masters` в CSR
- [ ] **`LimitRanger`** — применение LimitRange
- [ ] **`PodSecurity`** — применение PSS (PSA)
- [ ] **`ResourceQuota`** — применение ResourceQuota
- [ ] **`MutatingAdmissionWebhook`** — для кастомных мутаций
- [ ] **`ValidatingAdmissionWebhook`** — для кастомных валидаций

### 🟢 Рекомендуется включить дополнительно

- [ ] **`DenyServiceExternalIPs`** — защита от CVE-2020-8554
- [ ] **`NodeRestriction`** — ограничение kubelet (только свои Pod'ы/Node)

### 🔵 Опционально (для специфичных сценариев)

- [ ] **`AlwaysPullImages`** — всегда тянуть образы (для multi-tenant)
- [ ] **`ImagePolicyWebhook`** — кастомные проверки образов
- [ ] **`PodNodeSelector`** — ограничение выбора нод в namespace
- [ ] **`PodTolerationRestriction`** — ограничение tolerations в namespace

### ⚙️ Webhook Security

- [ ] **Webhooks используют TLS** (mutual TLS где возможно)
- [ ] **Webhooks имеют timeouts** (не блокируют API надолго)
- [ ] **`failurePolicy: Fail`** для security-critical webhooks
- [ ] **Webhooks не модифицируют** критичные поля (spec.containers.image) без аудита

## 🎯 9. Дополнительные рекомендации

### 🔄 Supply Chain Security

- [ ] **SBOM (Software Bill of Materials)** генерируется для образов
- [ ] **Provenance** attestations для образов (SLSA)
- [ ] **Dependency scanning** в CI/CD
- [ ] **Signed releases** для всех компонентов

### 🛑 Incident Response

- [ ] **План реагирования на инциденты** документирован
- [ ] **Процедура ротации** всех секретов при компрометации
- [ ] **Процедура отзыва** сертификатов
- [ ] **Backup & restore** etcd протестированы
- [ ] **Disaster recovery** план есть и тестирован

### 📚 Documentation & Training

- [ ] **Security policies** документированы
- [ ] **Developers обучены** security best practices
- [ ] **Onboarding checklist** для новых приложений включает security review
- [ ] **Regular security reviews** (quarterly)

## 🚀 Быстрый аудит кластера

```bash
#!/bin/bash
# Быстрая проверка безопасности кластера

echo "=== 1. PSA Labels ==="
kubectl get ns -o json | jq -r '.items[] | select(.metadata.labels | has("pod-security.kubernetes.io/enforce") | not) | .metadata.name' | grep -v '^kube-' | while read ns; do
  echo "⚠️  Namespace без PSA: $ns"
done

echo ""
echo "=== 2. Privileged Pods ==="
kubectl get pods -A -o json | jq -r '.items[] | select(.spec.containers[].securityContext.privileged == true) | "\(.metadata.namespace)/\(.metadata.name)"'

echo ""
echo "=== 3. Host Network Pods ==="
kubectl get pods -A -o json | jq -r '.items[] | select(.spec.hostNetwork == true) | "\(.metadata.namespace)/\(.metadata.name)"'

echo ""
echo "=== 4. Root Pods ==="
kubectl get pods -A -o json | jq -r '.items[] | select(.spec.containers[].securityContext.runAsUser == 0) | "\(.metadata.namespace)/\(.metadata.name)"'

echo ""
echo "=== 5. Pods without Resource Limits ==="
kubectl get pods -A -o json | jq -r '.items[] | select(.spec.containers[].resources.limits == null) | "\(.metadata.namespace)/\(.metadata.name)"'

echo ""
echo "=== 6. Secrets in etcd (проверка шифрования) ==="
# Требуется доступ к etcd
# ETCDCTL_API=3 etcdctl get /registry/secrets --prefix --keys-only | head -5

echo ""
echo "=== 7. RBAC: кто имеет cluster-admin ==="
kubectl get clusterrolebindings -o json | jq -r '.items[] | select(.roleRef.name == "cluster-admin") | .subjects[] | "\(.kind): \(.name)"'

echo ""
echo "=== 8. ServiceAccounts с automount: true ==="
kubectl get sa -A -o json | jq -r '.items[] | select(.automountServiceAccountToken != false) | "\(.metadata.namespace)/\(.metadata.name)"' | head -10

echo ""
echo "=== 9. Network Policies ==="
kubectl get networkpolicies -A
# Если пусто — NetworkPolicy не используются!

echo ""
echo "=== 10. Admission Controllers ==="
kubectl get --raw /apis/admissionregistration.k8s.io/v1/mutatingwebhookconfigurations
kubectl get --raw /apis/admissionregistration.k8s.io/v1/validatingwebhookconfigurations
```

## 📊 Матрица зрелости безопасности

| Уровень | Описание | Характеристика |
|---------|----------|----------------|
| **🔴 Level 0: None** | Нет security controls | Default K8s, все privileged |
| **🟠 Level 1: Basic** | Базовые controls | RBAC, NetworkPolicy, PSS Baseline |
| **🟡 Level 2: Standard** | Стандартные controls | + Encryption at Rest, Seccomp, Resource Limits, Image Scanning |
| **🟢 Level 3: Advanced** | Продвинутые controls | + PSS Restricted, External Secrets, Service Mesh, Audit Logging |
| **🔵 Level 4: Zero Trust** | Zero Trust Architecture | + mTLS везде, Short-lived tokens, SBOM, Provenance, Automated rotation |

## 🎯 Приоритизация: с чего начать

### 🔴 Критично (сделай сразу)

1. **Включи RBAC** и убери `system:masters` из повседневной работы
2. **Включи Encryption at Rest** для secrets
3. **Примени PSS Baseline** (хотя бы) через PSA
4. **Установи NetworkPolicy** (default deny)
5. **Ограничь доступ к API Server** (не публичный)

### 🟡 Важно (сделай в течение месяца)

6. **Resource Limits** для всех подов
7. **Image Scanning** в CI/CD
8. **Аудит RBAC** и убери лишние права
9. **automountServiceAccountToken: false** где не нужен
10. **Audit logging** включён

### 🟢 Рекомендуется (сделай в течение квартала)

11. **PSS Restricted** для критичных namespaces
12. **External Secret Store** (Vault, AWS SM)
13. **Seccomp/AppArmor** профили
14. **Service Mesh** для mTLS
15. **Image signing** и verification

### 🔵 Идеально (долгосрочная цель)

16. **Zero Trust** архитектура
17. **Automated rotation** всех секретов
18. **SBOM + Provenance** для всех образов
19. **Continuous compliance** (Kyverno/OPA policies)
20. **Chaos engineering** для security

## 💡 Финальные советы

1. **Security — это процесс, не состояние**. Чек-лист — это起点, а не финальная цель.
2. **Threat model first**. Не все пункты нужны всем. Оценивай риски.
3. **Defense in depth**. Несколько слоёв защиты лучше одного сильного.
4. **Automate everything**. Manual security checks не масштабируются.
5. **Shift left**. Встраивай security в CI/CD, не в production.
6. **Least privilege**. Всегда начинай с минимальных прав и добавляй по необходимости.
7. **Assume breach**. Проектируй систему так, чтобы компрометация одного компонента не вела к полному контролю.
8. **Regular reviews**. Security не статичен — пересматривай политики регулярно.
9. **Education**. Обучай разработчиков security best practices.
10. **Monitor & alert**. Если не мониторить — не узнаешь о проблеме.
