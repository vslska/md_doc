## 📚 Теория: Kubernetes кластер

### Что такое Kubernetes кластер?

**Kubernetes кластер** — это группа машин (физических или виртуальных), которые работают вместе как единое целое для запуска контейнеризированных приложений.

```
┌─────────────────────────────────────────────────────────┐
│                   Kubernetes Cluster                    │
│                                                         │
│  ┌──────────────────────────────────────────────────┐   │
│  │              Control Plane                       │   │
│  │  ┌─────────┐ ┌─────────┐ ┌──────────┐ ┌──────┐   │   │
│  │  │API Server││  etcd   │ │ Scheduler│ │  CM  │   │   │
│  │  └─────────┘ └─────────┘ └──────────┘ └──────┘   │   │
│  └──────────────────────────────────────────────────┘   │
│                          │                              │
│                          ▼                              │
│  ┌──────────────────────────────────────────────────┐   │
│  │                   Nodes                          │   │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐           │   │
│  │  │ Node 1  │  │ Node 2  │  │ Node 3  │  ...      │   │
│  │  │┌───────┐│  │┌───────┐│  │┌───────┐│           │   │
│  │  ││ Pod   ││  ││ Pod   ││  ││ Pod   ││           │   │
│  │  │└───────┘│  │└───────┘│  │└───────┘│           │   │
│  │  └─────────┘  └─────────┘  └─────────┘           │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### Два типа ресурсов в кластере

#### 1. Control Plane (Плоскость управления)

**Что делает**: управляет кластером, координирует все действия.

**Компоненты**:
- **API Server** — точка входа для всех запросов (kubectl, контроллеры, ноды)
- **etcd** — распределённое хранилище состояния кластера
- **Scheduler** — решает, на какой ноде запустить Pod
- **Controller Manager** — управляет контроллерами (Deployment, ReplicaSet, etc.)

**Аналогия**: Control Plane — это "мозг" кластера. Он не запускает приложения, но решает, **где** и **как** они должны работать.

#### 2. Nodes (Узлы)

**Что делает**: рабочие машины, на которых запускаются приложения.

**Компоненты на каждой ноде**:
- **kubelet** — агент, который общается с Control Plane и управляет Pod'ами
- **Container Runtime** — программа для запуска контейнеров (containerd, CRI-O, Docker)
- **kube-proxy** — сетевой компонент, который маршрутизирует трафик

**Аналогия**: Nodes — это "руки" кластера. Они выполняют работу — запускают контейнеры.

### Production vs Development

| Характеристика | Production кластер | Minikube (для обучения) |
|----------------|-------------------|------------------------|
| Количество нод | 3+ (для HA) | 1 |
| Control Plane | Отдельные машины/ВМ | Встроен в одну ВМ |
| Масштаб | Сотни/тысячи Pod'ов | Десятки Pod'ов |
| Назначение | Запуск реальных приложений | Обучение, разработка, тестирование |
| Доступность | 99.9%+ | Не критично |

**Почему в production нужно минимум 3 ноды?**

Если одна нода падает:
- Теряется член etcd (etcd требует кворум для работы)
- Теряется экземпляр Control Plane
- Кластер продолжает работать, но теряет отказоустойчивость

С 3 нодами кластер переживёт падение одной ноды без потери данных и управления.

## 🛠️ Что такое Minikube?

**Minikube** — это лёгкая реализация Kubernetes, которая создаёт локальный кластер на твоём компьютере.

### Как это работает?

```
┌─────────────────────────────────────────┐
│         Твой компьютер (Host)           │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │      Minikube VM / Container      │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │     Control Plane           │  │  │
│  │  │  - API Server               │  │  │
│  │  │  - etcd                     │  │  │
│  │  │  - Scheduler                │  │  │
│  │  │  - Controller Manager       │  │  │
│  │  └─────────────────────────────┘  │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │        Node (1 шт)          │  │  │
│  │  │  - kubelet                  │  │  │
│  │  │  - containerd               │  │  │
│  │  │  - kube-proxy               │  │  │
│  │  │  - Pod'ы (твои приложения)  │  │  │
│  │  └─────────────────────────────┘  │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

### Особенности Minikube

✅ **Преимущества**:
- Быстрый старт (1-2 минуты)
- Не требует облака или отдельных серверов
- Идеально для обучения и разработки
- Поддерживает Linux, macOS, Windows

❌ **Ограничения**:
- Только 1 нода (нет HA)
- Ограниченные ресурсы (CPU, RAM)
- Не для production

### Драйверы Minikube

Minikube может использовать разные драйверы для создания кластера:

| Драйвер | Платформа | Особенности |
|---------|-----------|-------------|
| **Docker** | Все | Самый популярный, быстрый, использует контейнеры |
| **VirtualBox** | Все | Классический вариант, использует ВМ |
| **HyperKit** | macOS | Нативный для macOS, быстрый |
| **Hyper-V** | Windows | Нативный для Windows |
| **KVM2** | Linux | Нативный для Linux, быстрый |

**Рекомендация**: используй **Docker** — он самый простой и быстрый.

## 🚀 Практика: Запуск Minikube

### Шаг 1: Проверка установки

#### Проверка minikube

```bash
minikube version
```

**Ожидаемый результат**:
```
minikube version: v1.32.0 (или новее)
commit: ...
```

❌ **Если не установлен**:
```bash
# macOS
brew install minikube

# Linux (amd64)
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube

# Windows (Chocolatey)
choco install minikube
```

#### Проверка kubectl

```bash
kubectl version --client
```

**Ожидаемый результат**:
```
Client Version: v1.29.0 (или новее)
```

❌ **Если не установлен**:
```bash
# macOS
brew install kubectl

# Linux
curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
sudo install kubectl /usr/local/bin/kubectl

# Windows
choco install kubectl
```

#### Проверка Docker

```bash
docker --version
```

**Ожидаемый результат**:
```

```

❌ **Если не установлен**: скачай с [docker.com](https://www.docker.com/products/docker-desktop)

### Шаг 2: Запуск кластера

```bash
minikube start
```

### Что происходит под капотом?

```
1. Проверка драйвера
   └─> Если Docker установлен → используется Docker driver
   └─> Если нет → предлагается установить или выбрать другой драйвер

2. Создание ВМ/контейнера
   └─> Docker: создаётся контейнер с Kubernetes компонентами
   └─> VirtualBox: создаётся виртуальная машина

3. Загрузка базового образа
   └─> Скачивается образ Linux с предустановленными компонентами K8s
   └─> Размер: ~500 MB (кэшируется, второй раз быстрее)

4. Запуск Control Plane
   ├─> Генерация сертификатов (PKI)
   ├─> Запуск etcd
   ├─> Запуск API Server
   ├─> Запуск Controller Manager
   └─> Запуск Scheduler

5. Настройка сети
   └─> Настройка CNI (Container Network Interface)
   └─> Pod'ы смогут общаться друг с другом

6. Запуск Node компонентов
   ├─> Запуск kubelet
   ├─> Запуск kube-proxy
   └─> Запуск container runtime (containerd)

7. Настройка kubectl
   └─> Обновление ~/.kube/config
   └─> Добавление контекста "minikube"

8. Включение addons
   ├─> default-storageclass (для динамического выделения PV)
   └─> storage-provisioner (для локального хранилища)

9. Проверка готовности
   └─> Ожидание, пока все компоненты запустятся
   └─> Проверка здоровья через API
```

### Ожидаемый результат

```
😄  minikube v1.32.0 on Darwin 14.2.1
✨  Using the docker driver based on user configuration
📌  Using Docker Desktop driver with root privileges
👍  Starting control plane node minikube in cluster minikube
🚜  Pulling base image ...
> gcr.io/k8s-minikube/kicbase:...: 372.66 MiB / 372.66 MiB  100.00% 10.23 Mi
🔥  Creating docker container (CPUs=2, Memory=4000MB) ...
🐳  Preparing Kubernetes v1.29.0 on Docker 24.0.7 ...
    ▪ Generating certificates and keys ...
    ▪ Booting up control plane ...
    ▪ Configuring RBAC rules ...
🔗  Configuring bridge CNI (Container Networking Interface) ...
    ▪ Using image gcr.io/k8s-minikube/storage-provisioner:v5
🔎  Verifying Kubernetes components...
🌟  Enabled addons: default-storageclass, storage-provisioner
🏄  Done! kubectl is now configured to use "minikube" cluster
```

### Разбор ключевых строк

| Строка | Что означает |
|--------|--------------|
| `Using the docker driver` | Используется Docker для создания кластера |
| `Starting control plane node` | Запускается Control Plane (API Server, etcd, etc.) |
| `Preparing Kubernetes v1.29.0` | Устанавливается версия Kubernetes |
| `Generating certificates` | Создаются сертификаты для TLS (безопасное общение) |
| `Configuring bridge CNI` | Настраивается сеть для Pod'ов |
| `kubectl is now configured` | kubectl теперь работает с minikube |

### Посмотреть системные Pod'ы

```bash

```

**Ожидаемый результат**:
```

coredns-5644d7b6d9-mh9ll                1/1     Running   0          2m
coredns-5644d7b6d9-pqd2t                1/1     Running   0          2m
etcd-minikube                           1/1     Running   0          2m
kube-apiserver-minikube                 1/1     Running   0          2m
kube-controller-manager-minikube        1/1     Running   0          2m
kube-proxy-rnlps                        1/1     Running   0          2m
kube-scheduler-minikube                 1/1     Running   0          2m
storage-provisioner                     1/1     Running   0          2m
```

### Разбор компонентов

| Pod | Компонент | Что делает |
|-----|-----------|------------|
| `coredns-*` | CoreDNS | DNS для сервисов (service discovery) |
| `etcd-minikube` | etcd | Хранилище состояния кластера |
| `kube-apiserver-minikube` | API Server | Точка входа для всех запросов |
| `kube-controller-manager-minikube` | Controller Manager | Управляет контроллерами |
| `kube-scheduler-minikube` | Scheduler | Планирует Pod'ы на ноды |
| `kube-proxy-*` | kube-proxy | Сетевая маршрутизация |
| `storage-provisioner` | Storage Provisioner | Динамическое выделение PV |

> 💡 **Заметь**: все эти Pod'ы — **статические Pod'ы**. Они управляются kubelet через манифесты в `/etc/kubernetes/manifests/`, а не через API Server.

## 🧪 Практические задания

### Задание 1: Посмотреть конфигурацию kubectl

```bash
kubectl config view
```

**Что ты увидишь**:
- Список кластеров (clusters)
- Список пользователей (users)
- Список контекстов (contexts)
- Текущий контекст (current-context)

**Вопрос**: какой контекст активен? (должен быть `minikube`)

### Задание 2: Переключить контекст (если есть другие кластеры)

```bash
# Посмотреть все контексты
```

```bash
kubectl describe node minikube
```

**Что ты увидишь**:
- Информацию о ноде (IP, ОС, версия kubelet)

```bash
# Остановить

# Проверить статус

# Запустить снова
minikube start
```

## 🐛 Troubleshooting

### Проблема: `minikube start` зависает

**Решение**:
```bash
# Удалить и пересоздать
minikube delete
minikube start --alsologtostderr -v=7
```

Флаг `-v=7` включает подробное логирование, чтобы увидеть, где зависло.

### Проблема: `kubectl` не видит кластер

**Решение**:
```bash
# Проверить текущий контекст
kubectl config current-context

# Если не minikube, переключиться
kubectl config use-context minikube

# Или перенастроить
minikube update-context
```

### Проблема: `minikube status` показывает `Stopped`

**Решение**:
```bash
# Запустить снова
minikube start

# Если не помогает, удалить и пересоздать
minikube delete
minikube start
```

### Проблема: не хватает ресурсов

**Решение**:
```bash
# Увеличить ресурсы при запуске
minikube start --cpus=4 --memory=8192

# Проверить ресурсы
minikube status
```
