## 📚 Теория: Pod и Node

### 1. Pod: Атомарная единица планирования

Pod — это **не просто контейнер**. Это оболочка (wrapper), которая предоставляет общую среду выполнения для одного или нескольких контейнеров.

**Что именно является общим внутри одного Pod'а:**
1. **Сетевое пространство имён (Network Namespace)**: Все контейнеры в Pod'е имеют **один и тот же IP-адрес** и порт. Они общаются друг с другом через `localhost`.
2. **Хранилище (Volumes)**: Контейнеры могут монтировать одни и те же тома для обмена данными.
3. **Контекст безопасности**: Общие настройки `securityContext` (например, от какого пользователя запускаются процессы).

**Важное правило жизненного цикла:**
Pod **жестко привязан** к конкретной Ноде, на которую его запланировал Scheduler. 
- ❌ Pod **не может** "переехать" с одной ноды на другую.
- ✅ Если Нода умирает, Pod умирает вместе с ней. 
- ✅ **ReplicaSet** заметит, что желаемое количество реплик не достигнуто, и создаст **совершенно новый Pod** на другой доступной Ноде.

> 💡 **Когда помещать несколько контейнеров в один Pod?** Только если они tightly coupled (тесно связаны) и должны масштабироваться вместе. Классический пример: паттерн **Sidecar** (основное приложение + контейнер для сбора логогов или прокси).

### 2. Node: Рабочая лошадка кластера

Нода — это физическая или виртуальная машина. За её работу отвечают два ключевых компонента:

| Компонент | Точная роль |
|-----------|-------------|
| **Kubelet** | Агент ноды. Получает спецификации Pod'ов от API Server. **Не запускает контейнеры напрямую**, а говорит Container Runtime, что нужно сделать. Также отправляет heartbeat (статус) в Control Plane. |
| **Container Runtime** | (containerd, CRI-O). Отвечает за загрузку образа из реестра, распаковку и фактический запуск процесса контейнера в изолированном пространстве (cgroups + namespaces). |

### 3. Инструменты kubectl для диагностики

| Команда | Что делает "под капотом" | Когда использовать |
|---------|--------------------------|---------------------|
| `kubectl get` | Делает `GET` запрос к API Server и возвращает краткую сводку. | Быстрая проверка статуса. |
| `kubectl describe` | Делает `GET` ресурса + `GET` связанных с ним `Events`. | Когда Pod не запускается, нужно видеть *почему* (ошибки планирования, образа, проб). |
| `kubectl logs` | Запрашивает у Kubelet логи stdout/stderr конкретного контейнера. | Когда приложение запустилось, но работает не так, как ожидается. |
| `kubectl exec` | Открывает интерактивный поток (SPDY/WebSocket) через API Server → Kubelet → Runtime в конкретный контейнер. | Когда нужно проверить переменные окружения, сеть или файлы внутри. |

## 🚀 Практика: Исследование приложения

*Предполагается, что у тебя уже запущен Deployment `kubernetes-bootcamp` из Практики 2.1.*

### Шаг 1: Базовый осмотр Pod'ов

```bash
kubectl get pods
```

**Ожидаемый результат**:
```text
NAME                                   READY   STATUS    RESTARTS   AGE
kubernetes-bootcamp-7d4f8b7b9c-x2y4z   1/1     Running   0          10m
```

**На что смотреть**:
- `READY 1/1`: 1 контейнер готов из 1 требуемого. Если здесь `0/1`, приложение не прошло readiness probe или ещё не запустилось.
- `RESTARTS`: Если число растёт, контейнер падает и перезапускается (CrashLoopBackOff).

---

### Шаг 2: Глубокая диагностика через `describe`

```bash
# Сохраним имя пода в переменную для удобства
export POD_NAME=$(kubectl get pods -o jsonpath='{.items[0].metadata.name}')

# Получим полную информацию
kubectl describe pod $POD_NAME
```

**Разбор вывода `describe` (самые важные секции):**

1. **Metadata**: Имя, Namespace, Node (на какой ноде физически работает), IP.
2. **Containers**: 
   - `Image`: какой образ используется.
   - `Port`: какие порты декларированы (это информационно, не открывает порты само по себе).
   - `State`: `Running`, `Waiting` или `Terminated`.
   - `Ready`: `True` или `False`.
3. **Events (Самое важное!)**: Хронологический журнал действий Kubelet и Scheduler относительно этого Pod'а.

**Пример Events**:
```text
Events:
  Type    Reason     Age   From               Message
  ----    ------     ----  ----               -------
  Normal  Scheduled  10m   default-scheduler  Successfully assigned default/kubernetes-bootcamp-... to minikube
  Normal  Pulling    10m   kubelet            Pulling image "gcr.io/google-samples/kubernetes-bootcamp:v1"
  Normal  Pulled     10m   kubelet            Successfully pulled image...
  Normal  Created    10m   kubelet            Created container kubernetes-bootcamp
  Normal  Started    10m   kubelet            Started container kubernetes-bootcamp
```
> 💡 **Правило**: Если Pod в статусе `Pending` или `CrashLoopBackOff`, первая команда, которую ты должен выполнить — `kubectl describe pod`. Ответ почти всегда в секции `Events`.

---

### Шаг 3: Доступ через прокси (повторение для контекста)

Открой **второе окно терминала** и запусти:
```bash
kubectl proxy
```
*Оставляем его работать. Он проксирует запросы к API Server на `localhost:8001`.*

В **первом окне** проверим доступ к приложению через API прокси:
```bash
curl http://localhost:8001/api/v1/namespaces/default/pods/$POD_NAME:8080/proxy/
```
**Результат**: `Hello Kubernetes bootcamp! | Running on: <имя_пода> | v=1`

### Шаг 4: Выполнение команд внутри контейнера (`exec`)

Мы можем выполнить команду внутри работающего контейнера, не заходя в него интерактивно:

```bash
kubectl exec $POD_NAME -- env
```
**Что это дает**: Показывает все переменные окружения, которые K8s внедрил в контейнер (например, `KUBERNETES_SERVICE_HOST`, `KUBERNETES_PORT`).

Теперь откроем интерактивную сессию bash:
```bash
kubectl exec -it $POD_NAME -- bash
```
*(Флаг `-i` = interactive, `-t` = tty, выделяет псевдотерминал)*

**Внутри контейнера** ты находишься в изолированной файловой системе этого конкретного образа. Выполни:

```bash
# 1. Посмотрим исходный код (он там есть по условиям туториала)
cat server.js

# 2. Проверим, что приложение слушает порт 8080 ВНУТРИ Pod'а
curl http://localhost:8080

# 3. Выйдем из контейнера
exit
```

> ⚠️ **Важный нюанс**: `curl http://localhost:8080` сработал внутри контейнера, потому что приложение слушает этот порт. Но если бы ты попытался сделать `curl http://localhost:8080` из своего **хост-терминала** (не через `kubectl proxy`), это бы не сработало, так как порт 8080 не проброшен на хост-машину.

## 🧪 Практические задания (для закрепления)

### Задание 1: Исследование Ноды
Узнай, на какой именно ноде работает твой Pod, и изучи её ресурсы.

```bash
# 1. Узнай имя ноды из описания пода
kubectl get pod $POD_NAME -o jsonpath='{.spec.nodeName}'

# 2. Получи детальную информацию об этой ноде (замени <node-name> на полученное имя, скорее всего 'minikube')
kubectl describe node <node-name>
```
**На что смотреть в выводе `describe node`**:
- Секция `Capacity` и `Allocatable`: сколько всего CPU/RAM и сколько доступно для Pod'ов.
- Секция `Conditions`: `Ready`, `MemoryPressure`, `DiskPressure`. Все должны быть `False` (кроме `Ready`, он должен быть `True`).

### Задание 2: Потоковый просмотр логов
Вместо одноразового вывода логов, включи режим следования (follow), как `tail -f`.

```bash
kubectl logs -f $POD_NAME
```
*Сделай несколько `curl` запросов к приложению через прокси в соседнем терминале и посмотри, как в реальном времени появляются записи в логах. Нажми `Ctrl+C` для выхода.*

### Задание 3: Что если в Pod'е нет `bash`?
Многие современные образы (например, на базе `distroless` или `alpine`) не содержат `bash`. Попробуй использовать `sh`:

```bash
kubectl exec -it $POD_NAME -- sh
```
*(Для образа `kubernetes-bootcamp` сработает и `bash`, и `sh`, но `sh` — более универсальный вариант для отладки).*

## 🐛 Troubleshooting

| Симптом | Возможная причина | Команда для диагностики |
|---------|-------------------|--------------------------|
| Pod в статусе `Pending` | Нет нод с достаточными ресурсами или не сработал Taint/Toleration. | `kubectl describe pod $POD_NAME` (смотри Events) |
| Pod в статусе `ImagePullBackOff` | Опечатка в имени образа, нет прав на приватный реестр, нет сети. | `kubectl describe pod $POD_NAME` |
| Pod в статусе `CrashLoopBackOff` | Приложение падает сразу после запуска (ошибка в коде, нет конфига). | `kubectl logs $POD_NAME` и `kubectl logs $POD_NAME --previous` |
| `kubectl exec` выдаёт "OCI runtime exec failed" | В образе нет оболочки (`bash` или `sh`). | Попробуй `-- sh` или проверь Dockerfile образа. |

## 💡 Главный вывод модуля

`kubectl get` дает тебе **что** происходит.

`kubectl exec` позволяет тебе **проверить** это своими руками внутри изолированной среды.
