>[!SUMMARY] Главная цепочка 
>**Dockerfile** (текстовая инструкция) → **Image** (статичный шаблон из слоёв) → **Container** (запущенный процесс с возможностью записи)

## Базовые понятия (с пояснениями)
| Термин             | Что это технически                                                                                    | Зачем нужно                                                                                                         | Аналогия                                                                       |
| ------------------ | ----------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **Dockerfile**     | Текстовый файл с инструкциями сборки. Каждая строка (кроме метаданных) создаёт новый слой образа.     | Чтобы сборка образа была воспроизводимой, версионируемой и автоматизируемой.                                        | Рецепт блюда: ингредиенты + шаги приготовления                                 |
| **Image**          | Набор read-only слоёв (Union FS). Неизменяемый артефакт сборки. Хранится в реестре или локально.      | Чтобы один раз собрать окружение и запускать его где угодно без зависимости от хоста.                               | Замороженный полуфабрикат: достал и приготовил                                 |
| **Container**      | Запущенный экземпляр образа + тонкий writable layer + изолированный процесс (namespaces + cgroups).   | Чтобы запустить приложение в изоляции, с контролем ресурсов и гарантией, что «у меня работает» = «у тебя работает». | Приготовленное блюдо: можно есть, менять подачу, но рецепт (образ) не портится |
| **Layer**          | Дельта изменений файловой системы относительно предыдущего слоя. Кэшируется при сборке.               | Чтобы ускорить сборку (переиспользовать кэш) и сэкономить место (общие слои у разных образов).                      | Прозрачная плёнка: накладываешь новые правки поверх старых, не стирая оригинал |
| **Writable layer** | Слой copy-on-write, создаваемый при запуске контейнера. Все изменения внутри контейнера пишутся сюда. | Чтобы контейнер мог работать (писать логи, временные файлы), не портя исходный образ.                               | Черновик: пишешь поверх чистовика, оригинал остаётся нетронутым                |
>[!NOTE] Важный нюанс 
>Слои образа **неизменяемы**. Если ты удалил файл в контейнере, в слое образа он остаётся, а в writable layer создаётся запись «файл удалён» (whiteout). Поэтому `RUN apt-get install && apt-get clean` должно быть в **одной** инструкции, иначе кэш пакетов останется в нижнем слое.

## Dockerfile: инструкции с примерами

### Базовые инструкции
```dockerfile
# FROM — обязательная первая инструкция. Задаёт базовый образ.
# Совет: используй конкретные теги (python:3.11-slim), а не latest, чтобы сборка не сломалась со временем.
FROM python:3.11-slim # Сразу есть Python, образ легче Ubuntu

# ENV — задаёт переменные окружения. Доступны на этапе сборки и в рантайме.
ENV APP_ENV=production \
    DB_HOST=localhost

# WORKDIR — задаёт рабочую директорию для следующих инструкций (RUN, CMD, COPY).
# Создаёт директорию, если её нет. Лучше абсолютный путь.
WORKDIR /app

# RUN — выполняет команду в оболочке (по умолчанию /bin/sh) и создаёт новый слой.
# Важно: объединяй команды через &&, чтобы уменьшить количество слоёв и очистить кэш в том же слое.
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl && \
    rm -rf /var/lib/apt/lists/*

# COPY / ADD — копируют файлы из контекста сборки в образ.
# COPY — предпочтительнее (проще, предсказуемее).
# ADD — умеет распаковывать архивы и скачивать по URL (но это может быть неочевидно).
COPY src/ ./src/
COPY app.py .
# EXPOSE — документация: какой порт слушает приложение. Не публикует порт на хосте!
# Для публикации используй флаг -p при docker run.
EXPOSE 8080

# CMD / ENTRYPOINT — команда запуска контейнера.
# CMD можно переопределить в docker run, ENTRYPOINT — нет (только с флагом --entrypoint).
# Лучшая практика: ENTRYPOINT фиксирует исполняемый файл, CMD — аргументы по умолчанию.
ENTRYPOINT ["python", "app.py"]
CMD ["--port", "8080"]

```

### Продвинутые инструкции
```dockerfile
# USER — запускает последующие команды и контейнер от указанного пользователя.
# Критично для безопасности: не запускай приложение от root в проде.
USER 1000:1000

# VOLUME — создаёт точку монтирования. Помечает директорию как «внешние данные».
# При запуске можно замапить в volume или bind mount.
VOLUME ["/data", "/logs"]

# ARG — переменная только на этапе сборки (не попадает в рантайм).
# Полезно для передачи версий зависимостей при сборке.
ARG NODE_VERSION=18
FROM node:${NODE_VERSION}-alpine

# HEALTHCHECK — команда проверки здоровья контейнера.
# Docker будет периодически её выполнять и обновлять статус в docker ps.
HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:8080/health || exit 1
```

Порядок инструкций влияет на кэш
```bash
# ❌ Плохо: при изменении кода пересобирается слой с npm install (долго)
COPY . .
RUN npm install

# ✅ Хорошо: слой с зависимостями кэшируется, если package.json не менялся
COPY package*.json ./
RUN npm install
COPY . .
```
## Слои образа и OverlayFS (как это работает)

### Визуализация сборки
```dockerfile
# Dockerfile
FROM alpine:3.18          # Слой 1: базовая ФС (~5 MB)
RUN apk add --no-cache nginx  # Слой 2: +nginx бинарники (+2 MB)
COPY index.html /usr/share/nginx/html/  # Слой 3: +твой файл (+1 KB)
EXPOSE 80                 # Метаданные (0 bytes)
CMD ["nginx", "-g", "daemon off;"]  # Метаданные (0 bytes)
```

```mermaid
graph LR
    A[Слой 1: alpine base] --> B[Слой 2: +nginx]
    B --> C[Слой 3: +index.html]
    C --> D[Image: read-only]
    D --> E[Container: + writable layer]
```

### Принципы работы overlay2
| Принцип                 | Как работает                                                                                              | Практическое следствие                                                                                    |
| ----------------------- | --------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| **Copy-on-Write (CoW)** | При запуске контейнера файлы не копируются. При первой записи файл копируется из образа в writable layer. | Быстрый старт, экономия места. Но первая запись в большой файл может быть медленной.                      |
| **Слои = дельты**       | Каждый `RUN`/`COPY` сохраняет только изменения относительно предыдущего состояния.                        | Удаление файла не уменьшает образ: в нижнем слое файл есть, в верхнем — метка «удалён».                   |
| **Общие слои**          | Если 10 образов основаны на `alpine:3.18`, на диске будет один набор слоёв alpine.                        | Экономия дискового пространства. Обновление базового образа затронет все зависимые образы при пересборке. |
| **Кэш сборки**          | Docker кэширует каждый слой. При повторной сборке проверяет: инструкция + хэш файлов в контексте.         | Меняй редко изменяющиеся файлы (зависимости) раньше часто изменяющихся (исходный код).                    |
**Полезные команды для анализа:**
```bash
# Посмотреть историю слоёв с размерами
docker history myapp:1.0 --human --no-trunc

# Вывод:
# IMAGE          CREATED       CREATED BY                                      SIZE      COMMENT
# abc123         2 hours ago   /bin/sh -c #(nop) COPY file:123...              1.2kB     
# def456         2 hours ago   /bin/sh -c apk add nginx                        2.1MB     
# ghi789         3 days ago    /bin/sh -c #(nop) FROM alpine:3.18              5.6MB     

# Посмотреть, какие слои занимают больше всего места
docker system df -v

# Узнать, какой storage driver используется
docker info | grep "Storage Driver"
# Вывод: Storage Driver: overlay2
```

Типичные ошибки со слоями

1. **Много мелких `RUN`** → много слоёв → медленная сборка и пуш.
```dockerfile
# ❌ 3 слоя
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# ✅ 1 слой
RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*
```

2.  **COPY до установки зависимостей** → при каждом изменении кода сбрасывается кэш `npm install` / `pip install`.
3.  **Секреты в слоях** → `ENV DB_PASSWORD=secret` останется в истории образа даже после удаления. Используй `--secret` при сборке или передавай в рантайме.
## Хранилище данных: полное сравнение

### Типы монтирования

#### 1. Writable layer (по умолчанию)

- **Где**: `/var/lib/docker/overlay2/<id>/diff/`
- **Плюсы**: Не требует настройки, работает «из коробки».
- **Минусы**:
    - Удаляется вместе с контейнером (`docker rm`).
    - Низкая производительность при активной записи (CoW оверхед).
    - Занимает место в `/var`, который может быть маленьким.
- **Когда использовать**: Только для временных файлов, которые не жалко потерять.

#### 2. Bind mounts (привязка к хосту)
```bash
# Синтаксис
docker run --mount type=bind,source=/host/path,target=/container/path image

# Короткая версия (менее явная)
docker run -v /host/path:/container/path image
```
- **Где**: Любое место на хосте, которое укажешь.
- **Плюсы**:
    - Прямой доступ к файлам хоста из контейнера и наоборот.
    - Удобно для разработки: изменил код на хосте → сразу видно в контейнере.
- **Минусы**:
    - Зависит от структуры ФС хоста (не переносимо).
    - Права доступа могут конфликтовать (root в контейнере ≠ твой пользователь на хосте).
    - Ты сам должен создать директорию и управлять её правами.
- **Когда использовать**:
    - Локальная разработка (`./src:/app`).
    - Подключение конфигов хоста (`/etc/nginx/conf.d:/etc/nginx/conf.d:ro`).
    - Доступ к устройствам или сокетам хоста.

#### 3. Volumes (управляется Docker)
```bash
# Создать volume
docker volume create mydata

# Использовать при запуске
docker run --mount source=mydata,target=/data image

# Или одной командой (volume создастся автоматически, если нет)
docker run -v my/data image
```

- **Где**: `/var/lib/docker/volumes/<name>/_data/` (по умолчанию).
- **Плюсы**:
    - Полностью управляется Docker (бэкап, миграция, очистка).
    - Не зависит от структуры ФС хоста (переносимо между хостами).
    - Лучшая производительность для записи (напрямую в ФС хоста, без CoW).
    - Можно использовать драйверы (NFS, CIFS, cloud) для распределённого хранения.
- **Минусы**:
    - Файлы «спрятаны» в `/var/lib/docker` (не так удобно редактировать вручную).
- **Когда использовать**:
    - Базы данных (`/var/lib/postgresql/data`).
    - Персистентные данные приложения (загрузки, кэш, сессии).
    - Продакшен-окружения, где важна переносимость.

#### 4. tmpfs (в оперативной памяти)
```bash
docker run --mount type=tmpfs,target=/tmp,tmpfs-size=100m image
```

- **Где**: ОЗУ хоста (не на диске).
- **Плюсы**: Очень высокая скорость записи/чтения.
- **Минусы**:
    - Данные пропадают при остановке контейнера.
    - Ограничено размером памяти.
- **Когда использовать**:
    - Секреты, которые не должны попасть на диск.
    - Временные файлы с высокой интенсивностью записи.
    - Кэш, который можно пересоздать.


| Критерий               | Writable layer  | Bind mount         | Volume               | tmpfs                 |
| ---------------------- | --------------- | ------------------ | -------------------- | --------------------- |
| **Персистентность**    | Нет             | Да                 | Да                   | Нет                   |
| **Производительность** | Низкая (CoW)    | Высокая            | Высокая              | Очень высокая (RAM)   |
| **Переносимость**      | Да              | Нет (пути хоста)   | Да                   | Зависит от памяти     |
| **Управление**         | Автоматически   | Вручную            | Docker               | Docker                |
| **Безопасность**       | Средняя         | Доступ к хосту     | Высокая              | Высокая (изолировано) |
| **Идеальный сценарий** | Временные файлы | Дев-среда, конфиги | БД, продакшен-данные | Секреты, кэш в памяти |

Почему `--mount` предпочтительнее `-v`

```bash
# ❌ -v: неявный тип, можно ошибиться
docker run -v my/app image  
# Если mydata не существует как volume, Docker создаст... bind mount с таким именем!

# ✅ --mount: явный type=, ошибка вызовет сбой сборки
docker run --mount type=volume,source=mydata,target=/app image
# Если volume нет — ошибка, ты сразу увидишь проблему
```

## Практика: полный цикл работы

### 1. Анализ и оптимизация образа
```bash
# Собрать образ
docker build -t myapp:1.0 .

# Посмотреть, какие слои «тяжёлые»
docker history myapp:1.0 --human --no-trunc

# Пример вывода: ищем слои > 100MB
# Если видишь большой слой от apt-get / npm install — 
# значит, не очищен кэш пакетов в том же слое.

# Посмотреть метаданные образа
docker inspect myapp:1.0 --format '{{json .Config}}' | jq

# Проверить, от какого пользователя запускается
docker inspect myapp:1.0 --format '{{.Config.User}}'
# Если пусто или "root" — добавь USER в Dockerfile
```

### Работа с volumes

```bash
# Создать volume для БД
docker volume create postgres-data

# Запустить контейнер с volume
docker run -d \
  --name db \
  --mount source=postgres-data,target=/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:15

# Написать данные
docker exec db psql -U postgres -c "CREATE TABLE test (id int);"

# Найти данные на хосте (для local driver)
sudo ls /var/lib/docker/volumes/postgres-data/_data/

# Остановить и удалить контейнер
docker stop db && docker rm db

# Volume остался! Данные сохранились
docker volume ls | grep postgres-data

# Запустить новый контейнер с тем же volume — данные на месте
docker run -d --name db2 --mount source=postgres-data,target=/var/lib/postgresql/data postgres:15
docker exec db2 psql -U postgres -c "\dt"  # таблица test будет здесь
```

### 3. Bind mount для разработки

```bash
# Создать тестовый проект
mkdir -p myapp/src
echo "print('Hello from bind mount')" > myapp/src/app.py

# Запустить контейнер с привязкой кода
docker run -it --rm \
  --mount type=bind,source="$(pwd)"/myapp/src,target=/app \
  python:3.11-alpine \
  python /app/app.py

# Вывод: Hello from bind mount

# Теперь измени файл на хосте
echo "print('Updated!')" > myapp/src/app.py

# Запусти снова — изменения подхватятся мгновенно
docker run -it --rm \
  --mount type=bind,source="$(pwd)"/myapp/src,target=/app \
  python:3.11-alpine \
  python /app/app.py
```

### 4. Очистка и обслуживание
```bash
# Что занимает место?
docker system df

# Удалить «висячие» образы (без тегов)
docker image prune

# Удалить все неиспользуемые образы, контейнеры, сети
docker system prune -a  # ⚠️ Осторожно: удалит всё, что не запущено

# Удалить неиспользуемые volumes (данные БД тоже!)
docker volume prune  # ⚠️ Проверь, что не удаляешь нужное

# Полная очистка с подтверждением
docker system prune -a --volumes
```

## Чеклист:хороший Dockerfile
### Перед коммитом Dockerfile
- [ ] Базовый образ с конкретным тегом (`python:3.11-slim`, не `latest`)
- [ ] Зависимости устанавливаются до копирования кода (кэш слоёв)
- [ ] Все `RUN` команды объединены и очищают кэш в том же слое
- [ ] Используется `.dockerignore` (исключены `.git`, `__pycache__`, `node_modules`)
- [ ] Приложение запускается не от root (`USER 1000`)
- [ ] Секреты не зашиты в образ (передаются через env / secrets)
- [ ] Есть `HEALTHCHECK` для продакшена
- [ ] Образ протестирован на уязвимости (`docker scan` / `trivy`)

## Перед запуском в проде
- [ ] Для данных БД/файлов используем volumes, не writable layer
- [ ] Заданы лимиты ресурсов (`--memory`, `--cpus`)
- [ ] Настроена политика перезапуска (`--restart unless-stopped`)
- [ ] Логи пишутся в stdout/stderr (не в файлы внутри контейнера)
- [ ] Порты опубликованы явно (`-p`), не полагаемся на EXPOSE

## Пути bind mounts: (бонус) 

#### `--mount type=bind` (рекомендуемый способ)

**Требует АБСОЛЮТНЫЙ путь.** Относительные пути **не работают**.
```bash
# НЕ сработает (ошибка)
docker run --mount type=bind,source=./config,target=/app/config nginx

# НЕ сработает (ошибка)
docker run --mount type=bind,source=config,target=/app/config nginx

# Сработает (абсолютный путь)
docker run --mount type=bind,source=/home/user/project/config,target=/app/config nginx

# Сработает (с подстановкой переменной)
docker run --mount type=bind,source="$(pwd)"/config,target=/app/config nginx
```

#### Золотое правило

> **Всегда используй абсолютные пути с `--mount type=bind`**  
> Или используй `"$(pwd)"/relative/path` для удобства.

```bash
#  Идеально
docker run --mount type=bind,source="$(pwd)"/config,target=/app/config,readonly nginx

#  Тоже хорошо
docker run -v "$(pwd)"/config:/app/config:ro nginx

#  Никогда так не делай
docker run -v ./config:/app/config nginx
```

#### Правильные паттерны

1. С `--mount` (рекомендуется)
```bash
# Текущая директория
docker run --mount type=bind,source="$(pwd)"/config,target=/app/config nginx

# Домашняя директория
docker run --mount type=bind,source="$HOME"/.ssh,target=/root/.ssh:ro nginx

# Абсолютный путь
docker run --mount type=bind,source=/opt/app/config,target=/app/config nginx
```

2. С `-v` (если используешь, то аккуратно)
```bash
# Всегда абсолютный путь!
docker run -v "$(pwd)"/config:/app/config nginx
docker run -v /opt/app/config:/app/config nginx

# Или явно создай volume, если нужно
docker volume create myconfig
docker run -v myconfig:/app/config nginx
```
