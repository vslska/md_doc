# Docker Build: Сборка образов

> [!SUMMARY] Главная формула **Контекст** + **Dockerfile** + `docker build` = **Image**
> 
> Контекст — это файлы, которые видит сборка. Dockerfile — инструкция, что с ними делать.

## Контекст сборки (Build Context)

### Что это и почему важно
```bash
docker build [OPTIONS] PATH | URL | -
```

| Аргумент           | Что означает                    | Когда использовать                |
| ------------------ | ------------------------------- | --------------------------------- |
| `PATH` (локальный) | Директория с файлами для сборки | **99% случаев**: `docker build .` |
| `URL` (Git)        | Репозиторий с Dockerfile        | CI/CD, сборка из удалённого репо  |
| `-` (stdin)        | Dockerfile и контекст из stdin  | Продвинутые сценарии, пайплайны   |
>[!NOTE] Что такое контекст? 
>Это **все файлы**, которые отправляются в Docker daemon при сборке.
>
>- Если в контексте 10 ГБ, а нужен только `Dockerfile` и `app.py` → сборка будет медленной.
>- Решение: **`.dockerignore`** (как `.gitignore`, но для сборки).

### Пример структуры и контекста
```bash
проект/
├── Dockerfile
├── .dockerignore    # ← Исключает лишнее из контекста
├── src/
│   └──  app.py
├── tests/           # ← Не нужно в образе
│   └──  test_app.py
├── .git/            # ← Точно не нужно в контексте
└──  README.md
```

**.dockerignore** (рекомендуемый минимум):
```
.git
__pycache__
*.pyc
*.pyo
*.pyd
.pytest_cache
.coverage
htmlcov
.env
*.md
tests/
docs/
```

Как проверить, что попадает в контекст?
```bash
# Посмотреть размер контекста перед сборкой
docker build --no-cache -q . 2>&1 | grep -i "sending"

# Или вручную:
tar -czf - . | wc -c  # Размер всех файлов в текущей папке
```

## Команда `docker build`: опции

```bash
docker build [OPTIONS] PATH

# 🔹 Базовые
-t, --tag <name:tag>        # Имя образа: -t myapp:1.0
-f, --file <path>           # Путь к Dockerfile (если не ./Dockerfile)
--no-cache                  # Не использовать кэш слоёв
--pull                      # Всегда тянуть свежий базовый образ

# 🔹 Продвинутые
--build-arg <VAR=value>     # Передать ARG из Dockerfile
--target <stage>            # Собрать только указанный stage (multi-stage)
--platform <os/arch>        # Кросс-компиляция: --platform linux/amd64
--secret <id=...,src=...>   # Передать секрет при сборке (без попадания в слои)

# 🔹 Вывод
--progress <auto|plain|tty> # Формат вывода: plain для логов в CI
-q, --quiet                 # Только вывод образа в конце
```

### Примеры использования
```bash
# Базовая сборка с тегом
docker build -t myapp:1.0 .

# Сборка с кастомным Dockerfile
docker build -f Dockerfile.prod -t myapp:prod .

# Сборка с передачей аргументов
docker build --build-arg VERSION=1.2.3 --build-arg ENV=prod -t myapp:1.2.3 .

# Multi-stage: собрать только финальный stage
docker build --target runtime -t myapp:runtime .

# Кросс-платформенная сборка (требует buildx)
docker buildx build --platform linux/amd64,linux/arm64 -t myapp:latest --push .
```

## Инструкции Dockerfile:
### FROM — базовый образ
```dockerfile
# Синтаксис
FROM [--platform=<platform>] <image>[:<tag>] [AS <name>]

# Примеры
FROM ubuntu:22.04                    # Конкретный тег (✅ хорошо)
FROM python:3.11-slim-bookworm       # Минималистичный образ (✅ лучше)
FROM alpine:3.18 AS builder          # Именованный stage для multi-stage
FROM scratch                         # Пустой образ (только для статических бинарников)
```
>[!NOTE] Почему не `latest`? 
>`FROM nginx:latest` сегодня и через месяц — это разные образы. Сборка станет невоспроизводимой. **Правило**: всегда фиксируй тег или хэш (`nginx:1.25@sha256:abc123...`).

### RUN — выполнение команд при сборке
```bash
# Синтаксис (два формата)
RUN <command>                        # shell form: /bin/sh -c "..."
RUN ["executable", "param1"]        # exec form: напрямую, без оболочки

# Пример: установка пакетов (Debian/Ubuntu)
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
      curl \
      ca-certificates \
      git && \
    rm -rf /var/lib/apt/lists/*

# Пример: установка зависимостей Python
RUN pip install --no-cache-dir -r requirements.txt
```

>[!WARNING] Частые ошибки с RUN
>
>1. **Раздельные команды = лишние слои**
>```dockerfile
># 3 слоя, кэш apt останется в образе
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*
>
># ✅ 1 слой, кэш удалён до фиксации
>RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*
>```
> 2. **`--no-install-recommends`** — не тяни лишние пакеты.
> 3. **`--no-cache-dir` для pip** — не сохраняй кэш в слое.
>

### COPY vs ADD — копирование файлов

| Инструкция | Что умеет                                           | Когда использовать                                      |
| ---------- | --------------------------------------------------- | ------------------------------------------------------- |
| **COPY**   | Копирует файлы/директории из контекста              | **По умолчанию**: копируешь код, конфиги, зависимости   |
| **ADD**    | COPY + авто-распаковка `.tar.*` + скачивание по URL | Только если нужна распаковка архива **на этапе сборки** |
```dockerfile
# COPY: базовый случай
COPY requirements.txt .
COPY src/ ./src/

# COPY с изменением прав (только для файлов, не директорий)
COPY --chown=1000:1000 app.py /app/

# ADD: распаковка архива (редкий кейс)
ADD https://example.com/data.tar.gz /data/  # ⚠️ Не кэшируется, скачивается каждый раз
ADD archive.tar.gz /opt/                     # Распакуется в /opt/
```

>[!TIP] Порядок COPY влияет на кэш
>```
>#  Плохо: при изменении любого файла пересобирается слой с npm install
COPY . .
RUN npm install
>#  Хорошо: слой с зависимостями кэшируется, если package*.json не менялись
>COPY package*.json ./
>RUN npm install
>COPY . .
>```

### WORKDIR — рабочая директория
```dockerfile
# Создаёт директорию и задаёт её для следующих инструкций
WORKDIR /app

# Теперь относительные пути в COPY, RUN, CMD считаются от /app
COPY requirements.txt .          # → /app/requirements.txt
RUN pip install -r requirements.txt
COPY src/ ./src/                 # → /app/src/

# Можно задавать несколько раз (пути накапливаются)
WORKDIR /app
WORKDIR src                      # → /app/src
```

>[!NOTE] Почему WORKDIR лучше `RUN mkdir && cd`?
>
>- Создаёт директорию, если нет (включая промежуточные)
>- Работает в exec-form (CMD/ENTRYPOINT)
>- Чище и предсказуемее

### ENTRYPOINT + CMD — команда запуска

#### Как они работают вместе
```dockerfile
# Формат: exec form (JSON array) — РЕКОМЕНДУЕТСЯ
ENTRYPOINT ["/usr/bin/python"]
CMD ["app.py", "--port", "8080"]

# Итоговая команда при запуске:
# /usr/bin/python app.py --port 8080
```

| Сценарий                             | Что выполнится                                    |
| ------------------------------------ | ------------------------------------------------- |
| `docker run myapp`                   | `ENTRYPOINT + CMD` → `python app.py --port 8080`  |
| `docker run myapp main.py`           | `ENTRYPOINT + аргумент из run` → `python main.py` |
| `docker run --entrypoint bash myapp` | Только `bash` (CMD игнорируется)                  |

#### Shell form vs Exec form
```dockerfile
# ❌ Shell form (не рекомендуется для ENTRYPOINT/CMD)
ENTRYPOINT python app.py
# Проблема: нет обработки сигналов (SIGTERM не дойдёт до Python), 
# переменные окружения могут не подгрузиться

# + Exec form (рекомендуется)
ENTRYPOINT ["/usr/bin/python", "app.py"]
# Python получает сигналы напрямую, поведение предсказуемое
```

>[!TIP] Если нужен shell для подстановки переменных
>```
># Используй exec form + явный shell
>ENTRYPOINT ["/bin/sh", "-c", "python $SCRIPT_NAME"]
># Или лучше: обработай переменные в самом приложении
>```

#### Разница с `ENTRYPOINT`:

- **`ENTRYPOINT`** задаёт команду, которая **всегда** выполняется (её сложнее переопределить)
    
- **`CMD`** задаёт аргументы по умолчанию, которые легко переопределить

### Другие полезные инструкции
```dockerfile
# ENV — переменные окружения (сборка + рантайм)
ENV PYTHONUNBUFFERED=1 \
    APP_ENV=production

# ARG — переменные только для сборки (не попадают в образ)
ARG VERSION=1.0
RUN echo "Building version $VERSION"

# EXPOSE — документация: какой порт слушает приложение
EXPOSE 8080/tcp
# ⚠️ Не публикует порт! Для публикации используй -p при docker run

# USER — запуск от непривилегированного пользователя
RUN useradd -m appuser
USER appuser

# VOLUME — подсказка: эта директория должна быть внешним хранилищем
VOLUME ["/data", "/logs"]

# HEALTHCHECK — проверка здоровья контейнера
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1

# LABEL — метаданные
LABEL maintainer="team@example.com" \
      version="1.2.3" \
      description="My application"
```

### ENV — переменные окружения (сборка + рантайм)

```dockerfile
# Синтаксис
ENV <KEY>=<value> [<KEY>=<value> ...]

# Примеры
ENV PYTHONUNBUFFERED=1 \
    APP_ENV=production \
    DB_HOST=localhost \
    DB_PORT=5432

# Использование в Dockerfile
ENV APP_DIR=/app
WORKDIR $APP_DIR
COPY . $APP_DIR/

# Использование в приложении (рантайм)
# app.py:
import os
db_host = os.getenv('DB_HOST', '127.0.0.1')  # fallback, если не задана
```

| Характеристика      | Описание                                                                       |
| ------------------- | ------------------------------------------------------------------------------ |
| **Доступность**     | В последующих инструкциях Dockerfile + внутри контейнера при запуске           |
| **Переопределение** | `docker run -e VAR=new_value` перебивает значение из Dockerfile                |
| **Видимость**       | Попадает в метаданные образа (`docker inspect`), **не используй для секретов** |
| **Лучшая практика** | Задавай разумные дефолты, а чувствительные данные передавай при запуске        |
```bash
# Переопределить ENV при запуске
docker run -e APP_ENV=staging -e DB_HOST=prod-db.example.com myapp

# Передать из файла (удобно для множества переменных)
docker run --env-file .env.prod myapp

# Посмотреть ENV в образе
docker inspect myapp --format '{{json .Config.Env}}'
```

>[!WARNING] Секреты в ENV — плохая идея
>```
># ❌ Никогда так не делай
ENV DB_PASSWORD=supersecret123
># Пароль останется в истории слоёв и будет виден через `docker history` и `inspect`
># ✅ Правильно: передавай в рантайме
>docker run -e DB_PASSWORD=$(vault get db-pass) myapp
># Или используй Docker secrets / mounted files
>```

### ARG — переменные только для сборки
```dockerfile
# Синтаксис
ARG <name>[=<default value>]

# Пример: версионирование зависимостей
ARG NODE_VERSION=18
ARG APP_VERSION=1.2.3

FROM node:${NODE_VERSION}-alpine
LABEL version=${APP_VERSION}

RUN echo "Building version ${APP_VERSION}"
```

| Характеристика        | Описание                                                                  |
| --------------------- | ------------------------------------------------------------------------- |
| **Доступность**       | Только на этапе сборки (`docker build`), **не попадает в итоговый образ** |
| **Переопределение**   | `docker build --build-arg VAR=value` перебивает дефолт                    |
| **Область видимости** | До первого `FROM` — глобальная, после `FROM` — только в этом stage        |
| **Лучшая практика**   | Используй для версий базовых образов, флагов сборки, но не для секретов   |
```dockerfile
# Multi-stage: ARG нужно объявлять в каждом stage
ARG VERSION=1.0

FROM golang:1.21 AS builder
ARG VERSION
RUN echo "Building v$VERSION"
# ... сборка ...

FROM alpine:3.18 AS runtime
ARG VERSION  # ← нужно объявить снова!
LABEL app.version=$VERSION
```

```bash
# Сборка с переопределением ARG
docker build --build-arg NODE_VERSION=20 --build-arg APP_VERSION=2.0.0 -t myapp:2.0 .

# Посмотреть, какие ARG использовались при сборке
docker history myapp:2.0 --no-trunc | grep "build-arg"
```

>[!TIP] ENV vs ARG — когда что использовать
>```
>Нужна переменная внутри контейнера при запуске? → ENV
>Нужна переменная только при сборке (версия, флаг)? → ARG
>Нужно и там, и там? → Объяви ARG, потом скопируй в ENV:
>
>ARG APP_ENV=production
>ENV APP_ENV=${APP_ENV}
>```

### Версионирование образов (тегирование)

```bash
# Синтаксис тега
docker build -t <name>[:<tag>] .

# Примеры
docker build -t myapp .                    # тег latest (по умолчанию)
docker build -t myapp:1.2.3 .              # семантическое версионирование
docker build -t myapp:prod .               # окружение
docker build -t registry.example.com/team/myapp:1.2.3 .  # с реестром
```
#### Стратегии тегирования
| Тег                               | Когда использовать              | Пример              |
| --------------------------------- | ------------------------------- | ------------------- |
| **Семантический** (`1.2.3`)       | Продакшен, релизы               | `myapp:2.1.0`       |
| **Окружение** (`prod`, `staging`) | Разделение конфигов             | `myapp:prod`        |
| **Git-метка** (`git-abc123`)      | CI/CD, отладка                  | `myapp:git-a1b2c3d` |
| **Дата** (`2024.03.15`)           | Временные билды                 | `myapp:2024.03.15`  |
| **latest**                        | Только для локальной разработки | `myapp:latest` ⚠️   |
>[!WARNING] Почему не стоит полагаться на `latest`
>
>- `latest` — это просто тег, он не обновляется автоматически
>- `docker pull myapp` и `docker pull myapp:latest` — одно и то же
>- В продакшене фиксируй конкретный тег или хэш: `myapp:1.2.3` или `myapp@sha256:abc...`

```bash
# Мульти-тегирование (один образ — несколько имён)
docker build -t myapp:1.2.3 -t myapp:latest -t registry.io/team/myapp:1.2.3 .

# Посмотреть теги образа
docker images myapp --format "table {{.Repository}}\t{{.Tag}}\t{{.Size}}"

# Переименовать/добавить тег существующему образу
docker tag myapp:1.2.3 myapp:stable
```

>[!TIP] Если при сборке Docker-образа не указать тег, 
>Docker автоматически присвоит ему тег **`latest`**.


#### Хэш образа (для максимальной воспроизводимости)

```
# После сборки получить хэш
IMAGE_ID=$(docker images myapp:1.2.3 -q)
DIGEST=$(docker inspect myapp:1.2.3 --format '{{index .RepoDigests 0}}')

# Запуск по хэшу (гарантия, что образ не изменится)
docker run registry.io/team/myapp@sha256:abc123...

# В CI/CD: пуш с хэшем
docker tag myapp:1.2.3 registry.io/team/myapp:$IMAGE_ID
docker push registry.io/team/myapp:$IMAGE_ID
```

### Комбинация ENV + ARG + теги: пример

```dockerfile
# Dockerfile
ARG APP_VERSION=dev
ARG APP_ENV=development

FROM python:3.11-slim

ENV APP_VERSION=${APP_VERSION} \
    APP_ENV=${APP_ENV} \
    PYTHONUNBUFFERED=1

WORKDIR /app
COPY . .

LABEL version=${APP_VERSION} \
      environment=${APP_ENV}

CMD ["python", "app.py"]
```

```bash
# Сборка для продакшена
docker build \
  --build-arg APP_VERSION=2.1.0 \
  --build-arg APP_ENV=production \
  -t myapp:2.1.0 \
  -t myapp:prod \
  .

# Проверка метаданных
docker inspect myapp:2.1.0 --format '{{json .Config.Env}}' | jq
docker inspect myapp:2.1.0 --format '{{json .Config.Labels}}' | jq

# Запуск с переопределением (например, для отладки)
docker run -e APP_ENV=debug myapp:2.1.0
```

## Практика: полный цикл сборки
```bash
# 1. Подготовь проект
mkdir myapp && cd myapp
cat > Dockerfile << 'EOF'
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
USER 1000
CMD ["python", "app.py"]
EOF

cat > .dockerignore << 'EOF'
.git
__pycache__
*.pyc
.env
tests/
EOF

# 2. Собери образ
docker build -t myapp:1.0 .

# 3. Проверь слои
docker history myapp:1.0 --human

# 4. Запусти и протестируй
docker run --rm myapp:1.0 --version

# 5. Запусти в проде-режиме
docker run -d \
  --name myapp \
  --mount source=myapp-data,target=/data \
  -p 8080:8080 \
  --restart unless-stopped \
  myapp:1.0

# 6. Проверь логи и здоровье
docker logs -f myapp
docker inspect myapp --format '{{.State.Health.Status}}'
```

## Чекист: хороший Dockerfile
### Структура и кэш
- [ ] Конкретный тег базового образа (`python:3.11-slim`, не `latest`)
- [ ] Зависимости копируются и устанавливаются до исходного кода
- [ ] Все `RUN` команды объединены и очищают кэш в том же слое
- [ ] Используется `.dockerignore` (исключены `.git`, `__pycache__`, `node_modules`)

####  Безопасность
- [ ] Приложение запускается не от root (`USER 1000`)
- [ ] Секреты не зашиты в образ (передаются через env / secrets / --secret)
- [ ] Используется `--no-install-recommends` и `--no-cache-dir`

### Рантайм
- [ ] Логи пишутся в stdout/stderr (не в файлы внутри контейнера)
- [ ] Есть `HEALTHCHECK` для продакшена
- [ ] `ENTRYPOINT` в exec form (JSON array) для корректной обработки сигналов

### Оптимизация
- [ ] Образ протестирован на уязвимости (`docker scan` / `trivy`)
- [ ] Для компилируемых языков: multi-stage сборка
- [ ] Размер образа адекватен задаче (< 500 MB для веб-приложения)

# Docker Build (кратко)

### Суть

- Контекст = файлы, которые видит сборка → используй `.dockerignore`

- Каждый `RUN`/`COPY` = новый слой → порядок влияет на кэш

- `docker build -t name .` → образ готов

### Шаблон Dockerfile (Python)

```dockerfile

FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install --no-cache-dir -r requirements.txt

COPY . .

USER 1000

EXPOSE 8080

HEALTHCHECK CMD curl -f http://localhost:8080/health || exit 1

CMD ["python", "app.py"]
```


## Быстрые команды
```bash
# Сборка
docker build -t myapp:1.0 .
docker build --no-cache -t myapp:1.1 .

# Отладка
docker history myapp:1.0 --human
docker build --progress=plain -t myapp:1.0 . 2>&1 | grep -i "error"

# Multi-stage
docker build --target runtime -t myapp:runtime .
```
