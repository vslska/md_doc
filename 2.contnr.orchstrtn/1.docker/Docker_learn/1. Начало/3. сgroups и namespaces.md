# Linux Primitives: cgroups + namespaces

> [!SUMMARY] Формула контейнера 
> **Container = namespaces (изоляция «что вижу») + cgroups (лимиты «сколько могу») + rootfs (файловая система)**

---

## Cgroups (Control Groups) — управление ресурсами

**Задача:** Не дать процессу(ам) сожрать всё железо.

### Архитектура   
```bash
cgroups v2 (единая иерархия)
├── /sys/fs/cgroup/
│   ├── user.slice/      # пользовательские процессы
│   ├── system.slice/    # системные сервисы
│   └── docker/          # контейнеры
```
### Контроллеры (cheat sheet)
| Контроллер     | Что лимитирует                   | Практическое применение                         |
| -------------- | -------------------------------- | ----------------------------------------------- |
| **cpu**        | Доля процессорного времени       | `--cpus=1.5`, `--cpu-shares=512`                |
| **memory**     | Потребление RAM                  | `--memory=512m`, OOM killer при превышении      |
| **io** (blkio) | Дисковый ввод/вывод              | `--device-read-bps`, приоритеты доступа к диску |
| **pids**       | Количество процессов             | `--pids-limit=100` — защита от fork-бомб        |
| **cpuset**     | Привязка к конкретным ядрам/NUMA | `--cpuset-cpus=0,1` — для low-latency задач     |
| **devices**    | Доступ к устройствам (`/dev/*`)  | Запрет на доступ к хост-девайсам из контейнера  |
| **hugetlb**    | Огромные страницы памяти         | Для баз данных, требующих HugeTLB               |
Как посмотреть в системе

```bash
# Какие контроллеры доступны
cat /sys/fs/cgroup/cgroup.controllers

# Где находится процесс в иерархии cgroups
cat /proc/<pid>/cgroup

# Лимиты памяти для контейнера Docker
docker inspect <container> | grep -A5 Memory
```

## Namespaces — изоляция окружения

**Задача:** Сделать так, чтобы процесс «не видел» чужие ресурсы.

### 6 типов namespaces
| Namespace | Флаг `unshare` | Что изолирует                                   | Пример использования                                    |
| --------- | -------------- | ----------------------------------------------- | ------------------------------------------------------- |
| **PID**   | `--pid`        | Таблица процессов, видимая внутри               | В контейнере `init` имеет PID=1                         |
| **NET**   | `--net`        | Сетевой стек: интерфейсы, IP, routing, iptables | У контейнера свой `eth0` и `localhost`                  |
| **MOUNT** | `--mount`      | Точки монтирования (`/proc`, `/dev`, volumes)   | Контейнер не видит `/mnt/data` хоста, если не замаппили |
| **USER**  | `--user`       | UID/GID (root внутри ≠ root снаружи)            | Запуск от непривилегированного пользователя             |
| **IPC**   | `--ipc`        | Shared memory, message queues, semaphores       | Изоляция межпроцессного взаимодействия                  |
| **UTS**   | `--uts`        | Hostname и domain name                          | У каждого контейнера своё имя хоста                     |
Как посмотреть в системе
```bash
# Какие namespaces у процесса
ls -la /proc/<pid>/ns/
# Вывод: pid:[4026531836] net:[4026531992] user:[4026531837] ...

# Сравнить с текущим процессом
ls -l /proc/self/ns/

# Войти в сетевой namespace контейнера (если нет curl внутри)
nsenter --net=/proc/<container_pid>/ns/net curl localhost:8080
```

## Практика

### 1. Запуск процесса в изолированном namespace
```bash
# Запустит bash в новом PID и NET namespace
sudo unshare --pid --net --fork --mount-proc bash

# Внутри:
ps aux        # увидишь только процессы внутри namespace
ip addr       # увидишь только loopback (lo)
exit
```
### 2. Ограничение ресурсов через cgroups (v2)
```bash
# Создай группу
sudo mkdir /sys/fs/cgroup/mylimit

# Запиши лимиты
echo "100000" | sudo tee /sys/fs/cgroup/mylimit/cpu.max  # 0.1 CPU
echo "50M" | sudo tee /sys/fs/cgroup/mylimit/memory.max  # 50 MB RAM

# Запусти процесс в этой группе
echo <pid> | sudo tee /sys/fs/cgroup/mylimit/cgroup.procs
```

### 3. Как Docker это использует
```bash
# Запусти контейнер
docker run -d --name test --cpus=0.5 --memory=256m nginx

# Найди PID
docker inspect test --format '{{.State.Pid}}'

# Проверь cgroups
cat /proc/<PID>/cgroup

# Проверь лимиты
cat /sys/fs/cgroup/system.slice/docker-<container_id>.scope/memory.max
```

## Что важно запомнить

- **Cgroups ≠ Namespaces**: первые лимитируют ресурсы, вторые изолируют видимость.
- **Cgroups v2** — единая иерархия, проще в управлении (проверь `mount | grep cgroup`).
- **PID=1 в контейнере** — это не хостовый init, он не умеет «забирать зомби» без proper init.
- **USER namespace** позволяет запускать root внутри контейнера без root-прав снаружи.
- **nsenter** — твой друг для отладки сети/процессов контейнера с хоста.
- **Docker — это просто удобный враппер** над этими примитивами ядра.
