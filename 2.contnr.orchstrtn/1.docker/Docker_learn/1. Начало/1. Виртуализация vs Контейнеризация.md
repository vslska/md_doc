# Виртуализация vs Контейнеризация

> [!SUMMARY] Главная разница 
> **VM** = эмуляция железа + полная ОС (тяжело, медленно, ГБ) **Container** = изоляция процессов на ядре хоста (легко, быстро, МБ)

---

## ⚡ Сравнение
| Критерий            | Контейнер                              | Виртуальная машина          |
| ------------------- | -------------------------------------- | --------------------------- |
| **Размер**          | Мегабайты                              | Гигабайты                   |
| **Запуск**          | Секунды                                | Минуты                      |
| **Ядро**            | Общее с хостом                         | Своё (гостевое)             |
| **Ресурсы**         | Только приложение + механизмы изоляции | Полная ОС + эмуляция железа |
| **Запуск чужой ОС** | Нет                                    | Да                          |
> [!TIP] Почему контейнеры эффективнее 
> Нет накладных расходов на эмуляцию железа и запуск второго ядра. Ресурсы тратятся только на приложение и изоляцию (cgroups/namespaces).

---

## Правила работы с контейнерами

1. **Stateless по умолчанию**
    - Контейнер можно убить в любой момент → данные внутри пропадут.
    - Важные данные → только во **Volumes** или внешние хранилища.
2. **Один процесс = один контейнер**
    - Легче масштабировать, обновлять и отлаживать по отдельности.
3. **Короткий жизненный цикл**
    - Контейнеры не «лечат», а заменяют новыми (immutable infrastructure).
4. **Изоляция ≠ Безопасность на 100%**
    - Изоляция на уровне ядра надежна, но уязвимости ядра затрагивают все контейнеры.
    - Не запускть недоверенный код в продакшене без дополнительной защиты.
5. **Легковесность = скорость**
    - Образы в МБ → быстрый деплой, экономия диска, легче тянуть по сети.

## Когда что выбирать

> [!NOTE] Гибридный подход 
> В облаках часто запускают контейнеры **внутри** виртуальных машин — это дает и изоляцию на уровне железа, и удобство контейнеризации.
