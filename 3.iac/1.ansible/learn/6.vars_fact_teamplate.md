## Переменные в Ansible

Переменные позволяют писать **один playbook**, который работает по-разному в зависимости от хоста, окружения или роли сервера.
### Синтаксис вызова
```yaml
"Текст {{ variable_name }} ещё текст"
# Пример:
msg: "Привет, {{ user_name }}! Твой домашний каталог: {{ home_dir }}"
```

### Источники переменных
| Источник         | Где задаётся                      | Пример                           |
| ---------------- | --------------------------------- | -------------------------------- |
| **Playbook**     | Блок `vars:` внутри play          | `vars: { app_port: 8080 }`       |
| **Inventory**    | Прямо в файле хостов              | `web1 ansible_user=deploy`       |
| **group_vars/**  | YAML-файл по имени группы         | `group_vars/webservers.yml`      |
| **host_vars/**   | YAML-файл по имени хоста          | `host_vars/db1.yml`              |
| **vars_files**   | Внешний YAML, подключаемый в play | `vars_files: ["secrets.yml"]`    |
| **Command line** | Флаг `-e` при запуске             | `ansible-playbook -e "env=prod"` |
| **Facts**        | Автоматически собираются Ansible  | `{{ ansible_distribution }}`     |
>[!TIP] Автоматическая загрузка `group_vars/` и `host_vars/` 
>Если рядом с `inventory` или playbook есть директории `group_vars/` и `host_vars/`, Ansible **автоматически** подгружает файлы с именем группы/хоста. Никаких `include` не нужно!

## Приоритет переменных (упрощённо)

Если одна переменная задана в нескольких местах, Ansible использует значение с **наивысшим приоритетом**:

| Уровень               | Приоритет      | Пример                   |
| --------------------- | -------------- | ------------------------ |
| Командная строка `-e` | Высший         | `-e "version=2.1"`       |
| Playbook `vars:`      | Высокий        | `vars: { version: 2.0 }` |
| 📂`host_vars/`        | Средний        | `host_vars/web1.yml`     |
| 📂`group_vars/`       | Средний-низкий | `group_vars/all.yml`     |
| Inventory файл        | Низкий         | `web1 ansible_user=root` |
| `defaults` (в ролях)  | Самый низкий   | `defaults/main.yml`      |
>[!WARNING] Упрощение для обучения 
В реальности приоритетов ~20. Эта таблица покрывает 95% сценариев, с которыми вы столкнётесь на старте.

## Системные факты (Facts)

**Facts** — это автоматически собираемая информация о целевом хосте при запуске playbook. Аналог `dmidecode`, `ip addr`, `lscpu`, но в удобном JSON-формате.

### Как включить/отключить сбор

```yaml
- name: Play with facts
  hosts: all
  gather_facts: yes  # включено по умолчанию
  tasks: ...

- name: Fast play without facts
  hosts: all
  gather_facts: no   # ускоряет запуск, если факты не нужны
```
### Просмотр фактов вручную
```bash
# Все факты
ansible all -m setup

# Фильтрация по маске (кавычки обязательны в bash!)
ansible all -m setup -a "filter=ansible_mem*"
ansible all -m setup -a "filter=ansible_distribution*"
```
### Использование в playbook
```yaml
- name: Show system info
  ansible.builtin.debug:
    msg: |
      OS: {{ ansible_distribution }} {{ ansible_distribution_version }}
      Arch: {{ ansible_architecture }}
      CPU: {{ ansible_processor_vcpus }} cores
      RAM: {{ ansible_memtotal_mb }} MB
      IP: {{ ansible_default_ipv4.address | default('N/A') }}
```

>[!NOTE] `ansible_*` vs `ansible_facts['*']` 
Современные версии Ansible рекомендуют использовать `ansible_facts['hostname']`, но для обратной совместимости доступны переменные вида `ansible_hostname`. Оба варианта работают.

## Шаблоны Jinja2 и модуль `template`

Модуль `ansible.builtin.template` работает как `copy`, но **обрабатывает файл через шаблонизатор Jinja2** перед копированием на целевой хост.

### Структура шаблона `.j2`
```jinja2
# Файл: templates/motd.j2
Добро пожаловать на сервер {{ inventory_hostname }}!
Операционная система: {{ ansible_distribution }} {{ ansible_distribution_version }}
Компания: {{ company_name | default('Internal Corp') }}
Дата последнего входа: {{ ansible_date_time.iso8601 }}

{% if ansible_memtotal_mb > 4096 %}
⚠️ Внимание: Сервер имеет {{ ansible_memtotal_mb }} МБ RAM. Убедитесь, что настроен swap.
{% endif %}

Администраторы:
{% for admin in admin_users %}
- {{ admin.name }} ({{ admin.email }})
{% endfor %}
```

### Применение в playbook
```yaml
- name: Deploy dynamic MOTD
  ansible.builtin.template:
    src: templates/motd.j2      # Относительный путь к playbook
    dest: /etc/motd
    owner: root
    group: root
    mode: '0644'
    backup: yes
```

>[!TIP] Полезные фильтры Jinja2 в Ansible
>```jinja2
>{{ var | default('fallback') }}     # Значение по умолчанию, если var не задан
>{{ var | lower }}                   # Приведение к нижнему регистру
{{ var | upper }}                   # Приведение к верхнему регистру
{{ var | int }}                     # Преобразование в число
{{ list_var | join(', ') }}         # Объединение списка в строку
{{ path | basename }}               # Только имя файла из пути
>```

## Пример: `vars_files` + `group_vars`

**Структура проекта:**
```
project/
├── playbook.yml
├── group_vars/
│   └── all.yml
├── templates/
│   └── app.conf.j2
└── vars/
    └── defaults.yml
```

**`group_vars/all.yml`** (применяется ко всем хостам):
```yaml
company_name: "ООО Пример"
admin_user: "ivan"
app_port: 8080
```

**`playbook.yml`**:
```yaml
---
- name: Deploy app config
  hosts: all
  vars_files:
    - vars/defaults.yml  # Дополнительные переменные
  tasks:
    - name: Render application config
      ansible.builtin.template:
        src: templates/app.conf.j2
        dest: /etc/myapp/app.conf
```

## Выводы

**Переменные** делают сценарии гибкими и масштабируемыми:

- Один playbook адаптируется под разные хосты, окружения и роли
- Источники переменных чётко иерархичны: `-e` > `playbook vars` > `host_vars` > `group_vars` > `inventory`

**Шаблоны Jinja2 + `template`** позволяют генерировать текстовые файлы динамически:

- Подстановка переменных, условия `{% if %}`, циклы `{% for %}`
- Фильтры (`| default`, `| lower`, `| join`) защищают от ошибок и форматируют вывод
- Идеально для конфигураций, баннеров, скриптов и отчётов

**Системные факты** — основа адаптивной автоматизации:

- Ansible собирает сотни параметров ОС, сети, железа автоматически
- Факты позволяют писать задачи, которые работают по-разному на RHEL и Ubuntu, на слабых и мощных серверах
- Умелое использование `when`, `template`, `group_vars` — залог качественной, безопасной и поддерживаемой автоматизации.

## Лабораторная работа 6: Переменные, факты и шаблоны
### Задания:

#### 1. Подготовка окружения
```
mkdir -p ~/ll105-lab6/{playbooks,templates,group_vars}
cd ~/ll105-lab6
cp ../ll105-lab5/inventory.yml ./
cp ../ll105-lab5/ansible.cfg ./
```
#### 2. Создать переменные окружения
```yaml
# ~/ll105-lab6/group_vars/all.yml
---
company_name: "ООО Пример"
admin_user: "ivan"
support_email: "support@example.com"
```

#### 3. Создать шаблон
```jinja2
# ~/ll105-lab6/templates/motd.j2
****************************************************************
Добро пожаловать на сервер: {{ inventory_hostname }}
Организация: {{ company_name }}
ОС: {{ ansible_distribution }} {{ ansible_distribution_version }}
Доступная RAM: {{ ansible_memtotal_mb }} MB
Текущая дата: {{ ansible_date_time.date }} {{ ansible_date_time.time }}
Поддержка: {{ support_email | default('admin@localhost') }}
****************************************************************
{% if ansible_memtotal_mb < 2048 %}
Внимание: Мало оперативной памяти. Рассмотрите увеличение ресурсов.
{% endif %}
```

#### 4. Создать playbook
```yaml
# ~/ll105-lab6/playbooks/deploy_motd.yml
---
- name: Lab6: Dynamic MOTD deployment
  hosts: all
  become: yes
  gather_facts: yes
  tasks:
    - name: Deploy MOTD from Jinja2 template
      ansible.builtin.template:
        src: ../templates/motd.j2
        dest: /etc/motd
        owner: root
        group: root
        mode: '0644'
        backup: yes

    - name: Verify MOTD content
      ansible.builtin.command: cat /etc/motd
      register: motd_content
      changed_when: false

    - name: Show generated MOTD
      ansible.builtin.debug:
        var: motd_content.stdout_lines
```

#### 5. Проверка и запуск
```bash
# Синтаксис
ansible-playbook --syntax-check playbooks/deploy_motd.yml

# Dry-run
ansible-playbook -i inventory.yml playbooks/deploy_motd.yml --check --diff

# Запуск
ansible-playbook -i inventory.yml playbooks/deploy_motd.yml -v
```

#### 6. Верификация на целевом хосте
```bash
# Проверить содержимое
ansible all -a "cat /etc/motd"

# Убедиться, что права корректны
ansible all -a "ls -l /etc/motd"
```

>[!SUCCESS] Критерии выполнения
>
>- `group_vars/all.yml` создан и содержит `company_name`, `admin_user`
>- Шаблон `motd.j2` использует факты (`ansible_memtotal_mb`, `ansible_distribution`) и условия `{% if %}`
>- Playbook успешно разворачивает `/etc/motd` на всех узлах
>- Вывод содержит имя хоста, дату, объём RAM и название компании
>- Повторный запуск идемпотентен (задача `template` → `ok`, не `changed`)
