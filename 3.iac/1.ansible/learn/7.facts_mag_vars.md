# Модуль 7: Факты и волшебные переменные Ansible

> [!SUMMARY] Цель 
> Научиться использовать автоматически собираемые системные факты, настраивать кастомные метаданные хостов и применять волшебные переменные для создания контекстно-зависимых сценариев.

> [!OBJECTIVES] Задачи
> 
> - Понять архитектуру Ansible-фактов и модуль `setup`
> - Научиться управлять сбором, кэшированием и отключением фактов
> - Создавать статические и динамические `.fact`-файлы
> -  Освоить волшебные переменные (`hostvars`, `groups`, `playbook_dir` и др.)
> -  Корректно работать с датами/временем в шаблонах

## Что такое Ansible-факты?

**Facts** — это автоматически собираемая информация об управляемом хосте (ОС, ядро, CPU, RAM, диски, сеть, переменные окружения и т.д.).

- Собираются модулем `ansible.builtin.setup` в начале выполнения play
- Хранятся в словаре `ansible_facts`
- Доступны в playbook, шаблонах Jinja2 и условиях `when`
```yaml
- name: Показать все собранные факты
  ansible.builtin.debug:
    var: ansible_facts
```

>[!TIP] Где хранить факты? 
`ansible_facts` — рекомендуемый путь в современных версиях Ansible. Переменные вида `ansible_hostname` оставлены для обратной совместимости, но постепенно уходят в legacy.

---
## Часто используемые факты
|Факт|Описание|Пример значения|
|---|---|---|
|`ansible_hostname`|Короткое имя хоста|`web01`|
|`ansible_fqdn`|Полное доменное имя|`web01.example.com`|
|`ansible_distribution`|Дистрибутив ОС|`Ubuntu`, `CentOS Stream`|
|`ansible_distribution_version`|Версия ОС|`22.04`, `9`|
|`ansible_kernel`|Версия ядра|`5.15.0-91-generic`|
|`ansible_processor_vcpus`|Количество vCPU|`4`|
|`ansible_memtotal_mb`|Общая RAM (МБ)|`8192`|
|`ansible_default_ipv4.address`|Основной IP-адрес|`192.168.1.10`|
|`ansible_mounts`|Список точек монтирования|`[{mount: "/", fstype: "ext4", ...}]`|
|`ansible_date_time`|Текущие дата и время хоста|`{"date": "2024-05-12", ...}`|

## Управление сбором фактов

### Ручной сбор и фильтрация
```bash
# Собрать все факты
ansible all -m setup

# Отфильтровать по маске (кавычки обязательны!)
ansible all -m setup -a "filter=ansible_mem*"
ansible all -m setup -a "filter=ansible_distribution*"
```
### Отключение сбора (оптимизация)

По умолчанию `gather_facts: yes`. Отключайте, если факты не нужны:
```yaml
- name: Fast deployment without facts
  hosts: all
  gather_facts: false  # или no
  tasks:
    - name: Quick ping
      ansible.builtin.ping:
```

>[!NOTE] Экономия времени 
На больших инфраструктурах (1000+ хостов) отключение фактов ускоряет запуск playbook на 15–30%.

## Кэширование фактов

Ускоряет повторные запуски за счёт сохранения фактов на диске или в БД.

**Настройка в `ansible.cfg`:**
```bash
[defaults]
fact_caching = jsonfile
fact_caching_connection = /tmp/ansible_facts_cache
fact_caching_timeout = 86400  # TTL в секундах (24 часа)
```

| Бэкенд                | Когда использовать                            |
| --------------------- | --------------------------------------------- |
| `jsonfile`            | Локальные/небольшие среды, отладка            |
| `redis` / `memcached` | CI/CD, большие кластеры, параллельные запуски |
>[!WARNING] Риски кэширования 
Кэшированные факты **не обновляются** при изменении конфигурации хоста (добавление диска, смена IP). Используйте `--flush-cache` или уменьшите `timeout`.

## Пользовательские (Custom) факты

Ansible читает файлы `.fact` из `/etc/ansible/facts.d/` на управляемых хостах. Поддерживает два формата:

### 1. Статические факты (INI)
```ini
# /etc/ansible/facts.d/env.fact
[general]
role = web
environment = production
contact = admin@example.com
```

Доступ в playbook: `{{ ansible_local['env']['general']['role'] }}`
### 2. Динамические факты (исполняемый скрипт)
```bash
#!/bin/bash
# /etc/ansible/facts.d/hardware.fact
# Обязательно: chmod +x /etc/ansible/facts.d/hardware.fact

cat <<EOF
{
  "hw": {
    "serial": "$(cat /sys/class/dmi/id/product_serial 2>/dev/null || echo 'unknown')",
    "model": "$(cat /sys/class/dmi/id/product_name 2>/dev/null || echo 'unknown')"
  }
}
EOF
```

Доступ: `{{ ansible_local['hardware']['hw']['serial'] }}`

>[!IMPORTANT] Правила кастомных фактов
>
>- Директория должна существовать: `mkdir -p /etc/ansible/facts.d/`
>- Ключи **всегда приводятся к нижнему регистру** при парсинге
>- Скрипты должны возвращать **валидный JSON** (без лишних логов в stdout)
>- Проверяйте синтаксис: `python3 -m json.tool < /etc/ansible/facts.d/test.fact`

## Волшебные переменные (Magic Variables)

Magic-переменные встроены в Ansible, **не требуют `gather_facts`** и содержат контекст выполнения. Их нельзя перезаписать.

| Переменная                 | Описание                                    | Пример использования                         |
| -------------------------- | ------------------------------------------- | -------------------------------------------- |
| `inventory_hostname`       | Имя хоста из inventory                      | `{{ inventory_hostname }}`                   |
| `inventory_hostname_short` | Имя до первой точки                         | `web01` из `web01.example.com`               |
| `group_names`              | Список групп текущего хоста                 | `'web' in group_names`                       |
| `groups`                   | Словарь всех групп и их хостов              | `{{ groups['dbservers'] }}`                  |
| `hostvars`                 | Полные переменные всех хостов               | `{{ hostvars['db1']['ansible_host'] }}`      |
| `playbook_dir`             | Абсолютный путь к каталогу playbook         | `{{ playbook_dir }}/templates`               |
| `ansible_playbook_python`  | Python, которым запущен Ansible             | Для отладки сред                             |
| `omit`                     | Специальное значение для пропуска параметра | `mode: "{{ custom_mode \| default(omit) }}"` |
### Практические примеры
```yaml
# 1. Условие на основе группы
when: "'webservers' in group_names"

# 2. Получить IP всех хостов группы (требует собранных фактов!)
{% for host in groups['webservers'] %}
  {{ hostvars[host].ansible_default_ipv4.address | default(hostvars[host].ansible_host, true) }}
{% endfor %}

# 3. Динамический путь к шаблону
src: "{{ playbook_dir }}/templates/{{ ansible_distribution | lower }}_config.j2"
```

>[!TIP] `hostvars` и порядок выполнения 
`hostvars[host]['ansible_facts']` доступен **только если факты уже собраны** для этого хоста в текущем запуске. Если play выполняется только для одной группы, факты других групп будут пустыми. Используйте `ansible_host` из inventory как fallback.

## Работа с датой и временем

`ansible_date_time` собирается один раз при старте play и **не обновляется** в ходе выполнения.

**Рекомендуемый современный подход:**
```jinja2
{# Jinja2 функция now() (Ansible 2.10+) #}
{{ now(utc=true, fmt="%Y-%m-%d %H:%M:%S") }}
{{ now(fmt="%d.%m.%Y") }}
```

>[!NOTE] Почему не `lookup('pipe', 'date ...')`? 
`lookup` выполняет команду на **control node**, а `now()` — чистая Jinja2-функция, быстрее и безопаснее.

## Выводы

**Факты** предоставляют детальную системную информацию об управляемых хостах:

- Автоматически собираются модулем `setup`, доступны через `ansible_facts`
- Позволяют писать адаптивные сценарии, реагирующие на железо, ОС и сеть
- Кэширование ускоряет работу, но требует контроля актуальности данных

**Волшебные переменные** дают доступ к контексту выполнения playbook:

- `inventory_hostname`, `group_names`, `groups`, `hostvars`, `playbook_dir`
- Работают без сбора фактов, незаменимы для кросс-хостовой логики и динамических путей
- `omit` и `hostvars` расширяют возможности условной передачи параметров

**Пользовательские факты** расширяют стандартный набор метаданных:

- Статические (`.fact` INI) и динамические (скрипты → JSON) форматы
- Позволяют интегрировать внешние системы, CMDB, аппаратные метки
- Эти механизмы вместе позволяют писать адаптивные, универсальные и самодокументируемые сценарии автоматизации.

## Лабораторная работа 7: Факты и волшебные переменные
### Задания :

#### 1. Подготовка
```bash
mkdir -p ~/ll105-lab7/{playbooks,templates,group_vars,facts.d}
cd ~/ll105-lab7
cp ../ll105-lab6/inventory.yml ./
cp ../ll105-lab6/ansible.cfg ./
```
#### 2. Создать кастомный факт
```ini
# ~/ll105-lab7/facts.d/role.fact (будет скопирован на хосты)
[general]
role = web
environment = lab
```
#### 3. Создать шаблон `/etc/issue`
```jinja2
# ~/ll105-lab7/templates/issue.j2
{{ ansible_distribution }} {{ ansible_distribution_version }} - {{ inventory_hostname }}
Kernel {{ ansible_kernel }} on an {{ ansible_architecture }} ({{ ansible_processor_vcpus }} CPUs, {{ ansible_memtotal_mb }} MB RAM)
Role: {{ ansible_local['role']['general']['role'] | default('undefined') }}
Generated: {{ now(utc=true, fmt="%Y-%m-%d %H:%M:%S") }}
```
#### 4. Создать playbook
```yaml
# ~/ll105-lab7/playbooks/facts_lab.yml
---
- name: Lab7: Facts & Magic Variables
  hosts: all
  become: yes
  gather_facts: yes
  tasks:
    - name: Show RAM and IP
      ansible.builtin.debug:
        msg: "Host: {{ inventory_hostname }} | RAM: {{ ansible_memtotal_mb }}MB | IP: {{ ansible_default_ipv4.address | default('N/A') }}"

    - name: Deploy custom facts directory
      ansible.builtin.file:
        path: /etc/ansible/facts.d
        state: directory
        mode: '0755'

    - name: Copy custom fact file
      ansible.builtin.copy:
        src: ../facts.d/role.fact
        dest: /etc/ansible/facts.d/role.fact
        mode: '0644'

    - name: Re-gather facts to include custom ones
      ansible.builtin.setup:
        filter: ansible_local

    - name: Deploy /etc/issue from template
      ansible.builtin.template:
        src: ../templates/issue.j2
        dest: /etc/issue
        owner: root
        group: root
        mode: '0644'
        backup: yes
      when: ansible_memtotal_mb > 1024  # Условие: RAM > 1GB

    - name: Show generated /etc/issue
      ansible.builtin.command: cat /etc/issue
      register: issue_content
      changed_when: false

    - name: Display result
      ansible.builtin.debug:
        var: issue_content.stdout_lines
```

#### 5. Проверка и запуск
```bash
# Синтаксис
ansible-playbook --syntax-check playbooks/facts_lab.yml

# Запуск
ansible-playbook -i inventory.yml playbooks/facts_lab.yml -v

# Проверка на хостах
ansible all -a "cat /etc/issue"
ansible all -a "cat /etc/ansible/facts.d/role.fact"
```

>[!SUCCESS] Критерии выполнения
>
>- Факты RAM и IP выводятся в `debug`
>- Кастомный факт `role.fact` скопирован и распознан (повторный `setup` с фильтром)
>- Шаблон `/etc/issue` развернут только на хостах с RAM > 1024 МБ
>- В `/etc/issue` присутствуют `inventory_hostname`, `ansible_distribution`, дата через `now()`
>- Кастомная роль отображается корректно или показывает `undefined` (если факт не применился)
