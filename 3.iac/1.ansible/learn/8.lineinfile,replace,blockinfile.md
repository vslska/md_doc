## `ansible.builtin.lineinfile`: механика и параметры

### Алгоритм выполнения
```
1. Прочитать файл по `path`
2. Найти строку, соответствующую `regexp` (если задан)
3. Если найдена:
   - Сравнить строку с `line`
   - Если совпадает → `changed: false`
   - Если не совпадает → заменить строку → `changed: true`
4. Если не найдена и `state: present`:
   - Вставить `line` в конец файла или по `insertafter`/`insertbefore`
   - `changed: true`
5. Если `state: absent` и строка найдена → удалить → `changed: true`
6. Если `backup: yes` → создать копию перед изменением
7. Если `validate` задан → выполнить команду с `%s` = путь к временному файлу
```
### Ключевые параметры с техническими деталями

|Параметр|Тип|По умолчанию|Описание и особенности|
|---|---|---|---|
|`path`|str|—|Путь к файлу. Поддерживает `~` для домашней директории.|
|`line`|str|—|Строка для вставки/замены. **Обязателен** при `state: present`.|
|`regexp`|str|`None`|RegEx для поиска. Если не задан → поиск по точному совпадению `line`.|
|`state`|str|`present`|`present` = гарантировать наличие; `absent` = гарантировать отсутствие.|
|`create`|bool|`false`|Создать файл, если не существует. Работает только с `state: present`.|
|`backup`|bool|`false`|Создать резервную копию с суффиксом `.<timestamp>~`.|
|`insertafter`|str|`EOF`|Вставить после первой строки, соответствующей паттерну. `EOF` = конец файла.|
|`insertbefore`|str|`BOF`|Вставить перед первой строкой, соответствующей паттерну. `BOF` = начало файла.|
|`firstmatch`|bool|`false`|При множественных совпадениях `regexp` использовать только первое.|
|`validate`|str|`None`|Команда для проверки файла. `%s` подставляется путь к временному файлу.|
### Примеры с пояснением механики
```yaml
#  Замена порта SSH: regexp находит строку, line подставляет новое значение
- name: Change SSH port to 2222
  ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    regexp: '^#?Port\s+\d+'  # Находит "Port 22" или "#Port 22"
    line: 'Port 2222'         # Заменяет на эту строку
    backup: yes
    validate: 'sshd -t -f %s' # Проверяет синтаксис перед применением

#  Добавление строки, если её нет (идемпотентно)
- name: Ensure AllowUsers directive
  ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    line: 'AllowUsers root ansible'
    insertafter: '^# Authentication'  # Вставить после секции аутентификации
    backup: yes

#  Удаление строки по паттерну
- name: Remove legacy banner
  ansible.builtin.lineinfile:
    path: /etc/motd
    regexp: '^This system is for authorized use only$'
    state: absent
    backup: yes
```

>[!WARNING] RegEx в YAML: экранирование
>```yaml
># ❌ Опасно: \b интерпретируется как backspace в YAML
>regexp: "\broot\b"
>
># ✅ Безопасно: используйте одинарные кавычки или двойное экранирование
regexp: '\broot\b'      # одинарные кавычки: без интерпретации
regexp: "\\broot\\b"    # двойные кавычки: экранирование обратного слеша
>```

## `ansible.builtin.replace`: массовая замена с контекстом
### Алгоритм выполнения
```
1. Прочитать файл целиком в память
2. Если заданы `after`/`before`:
   - Найти позицию первого совпадения `after` (или начало файла)
   - Найти позицию первого совпадения `before` после `after` (или конец файла)
   - Ограничить область замены между этими позициями
3. Применить `re.sub(regexp, replace, content)` к целевой области
4. Сравнить результат с оригиналом:
   - Если есть изменения → записать файл → `changed: true`
   - Если нет → `changed: false`
5. Если `backup: yes` → сохранить копию перед записью
```
### Ключевые параметры
|Параметр|Тип|Описание|
|---|---|---|
|`path`|str|Путь к файлу|
|`regexp`|str|RegEx для поиска **всех** вхождений (поддерживает группы `()`)|
|`replace`|str|Текст замены; поддерживает обратные ссылки `\1`, `\2`, ...|
|`after`|str|Ограничить замену: начинать после первого совпадения этого паттерна|
|`before`|str|Ограничить замену: заканчивать перед первым совпадением этого паттерна|
|`backup`|bool|Создать резервную копию|
### Обратные ссылки и группы захвата
```yaml
# Пример: заменить "ServerName old.example.com" на "ServerName prod.example.com"
- name: Update server name
  ansible.builtin.replace:
    path: /etc/apache2/apache2.conf
    regexp: '(ServerName\s+)\S+'  # Группа 1: "ServerName ", группа 2: старое имя
    replace: '\1prod.example.com'  # \1 = "ServerName ", + новое имя
    backup: yes
```
### Ограничение области: `after`/`before`
```yaml
# Закомментировать все строки внутри секции <VirtualHost>
- name: Comment out VirtualHost block
  ansible.builtin.replace:
    path: /etc/apache2/sites-available/default.conf
    after: '<VirtualHost'    # Начать замену после этой строки
    before: '</VirtualHost>' # Закончить замену перед этой строкой
    regexp: '^(?!#)(.+)$'    # Заменить все строки, не начинающиеся с #
    replace: '# \1'          # Добавить # в начало
    backup: yes
```
>[!NOTE] Ограничения `after`/`before`
>
>- `after` и `before` находят **первое** совпадение в файле. Для работы с множественными секциями используйте `blockinfile` или `template`.
>- Паттерны `after`/`before` не поддерживают группы захвата — только поиск позиции.

## `ansible.builtin.blockinfile`: управление блоками с маркерами

### Механика маркеров
```
По умолчанию маркеры имеют вид:
# {mark} ANSIBLE MANAGED BLOCK

Где {mark} заменяется на:
- "BEGIN" для открывающего маркера
- "END" для закрывающего маркера

Пример в файле:
# BEGIN ANSIBLE MANAGED BLOCK
net.ipv4.ip_forward = 1
net.core.somaxconn = 1024
# END ANSIBLE MANAGED BLOCK
```
### Алгоритм выполнения
```
1. Прочитать файл
2. Поиск пары маркеров (BEGIN/END) с заданным шаблоном `marker`
3. Если маркеры найдены:
   - Извлечь содержимое между ними
   - Сравнить с новым `block` (с учётом `block_start`/`block_end`)
   - Если совпадает → `changed: false`
   - Если не совпадает → заменить содержимое → `changed: true`
4. Если маркеры не найдены и `state: present`:
   - Вставить блок с маркерами в позицию `insertafter`/`insertbefore`
   - `changed: true`
5. Если `state: absent` и маркеры найдены → удалить блок с маркерами → `changed: true`
```
### Ключевые параметры

|Параметр|Тип|По умолчанию|Описание|
|---|---|---|---|
|`path`|str|—|Путь к файлу|
|`block`|str|—|Многострочный текст (используйте YAML `|
|`marker`|str|`# {mark} ANSIBLE MANAGED BLOCK`|Шаблон маркеров; `{mark}` заменяется на `BEGIN`/`END`|
|`insertafter`|str|`EOF`|Позиция вставки, если блок ещё не существует|
|`insertbefore`|str|`BOF`|Позиция вставки перед паттерном|
|`state`|str|`present`|`present` = гарантировать наличие блока; `absent` = удалить|
|`backup`|bool|`false`|Резервная копия перед изменением|
|`create`|bool|`false`|Создать файл, если не существует|
### Пример: добавление блока sysctl
```yaml
- name: Add kernel tuning parameters
  ansible.builtin.blockinfile:
    path: /etc/sysctl.conf
    marker: "# {mark} ANSIBLE KERNEL TUNING"
    block: |
      net.ipv4.ip_forward = 1
      net.core.somaxconn = 1024
      vm.swappiness = 10
      fs.file-max = 2097152
    insertafter: EOF
    backup: yes
```

### Кастомизация маркеров (для не-комментарных файлов)
```yaml
# Для HTML/XML файлов используйте комментарии соответствующего формата
- name: Add analytics script to HTML
  ansible.builtin.blockinfile:
    path: /var/www/html/index.html
    marker: "<!-- {mark} ANSIBLE MANAGED BLOCK -->"
    block: |
      <script>
        console.log('Analytics loaded');
      </script>
    insertafter: '</head>'
```

>[!WARNING] Ручное редактирование между маркерами 
Если пользователь вручную изменит текст между маркерами, следующий запуск playbook **перезапишет** эти изменения. Маркеры — это контракт: «этим блоком управляет Ansible».

## Регулярные выражения в контексте Ansible

### Синтаксис RegEx: шпаргалка для конфигурационных файлов

|Паттерн|Значение|Пример использования|
|---|---|---|
|`^`|Начало строки|`^Port\s` — найти директиву порта|
|`$`|Конец строки|`enabled$` — найти строки, заканчивающиеся на "enabled"|
|`\s`|Пробельный символ (пробел, таб, новая строка)|`Port\s+\d+` — "Port" + пробелы + цифры|
|`\d`|Цифра `[0-9]`|`\d{1,3}(\.\d{1,3}){3}` — простой паттерн IPv4|
|`.*`|Жадное совпадение: 0+ любых символов|`^#.*$` — закомментированные строки|
|`.*?`|Нежадное совпадение|`^Listen(.*?):` — захватить минимальное содержимое до ":"|
|`()`|Группа захвата (для `\1`, `\2` в `replace`)|`(ServerName\s+)(\S+)`|
|`(?:...)`|Не захватывающая группа|`Port\s+(?:22\|2222)` — найти порт 22 или 2222, но не захватывать|
|`(?=...)`|Позитивный просмотр вперёд|`root(?=\s|
|`\b`|Граница слова|`\broot\b` — слово "root", но не "rooted"|

### Безопасные паттерны для типичных задач
```yaml
# Найти директиву с возможным комментарием и пробелами
regexp: '^#?\s*Port\s+\d+'

# Найти IP-адрес в конфиге (упрощённо)
regexp: '\b(?:\d{1,3}\.){3}\d{1,3}\b'

# Найти строку с переменной в shell-скрипте
regexp: '^export\s+MY_VAR='

# Опасно: жадный паттерн может захватить лишнее
regexp: 'password.*'  # Захватит до конца строки или файла!

# Безопасно: ограничить конец строки
regexp: '^password\s*=\s*.+$'
```

### Тестирование RegEx перед применением
```bash
# Проверить, какие строки совпадут, без внесения изменений
ansible all -m lineinfile \
  -a "path=/etc/ssh/sshd_config regexp='^#?Port' state=absent" \
  --check -vvv

# Использовать grep для локальной проверки паттерна
grep -nE '^#?Port\s+\d+' /etc/ssh/sshd_config
```

## Безопасность и отладка: технические рекомендации
### Обязательные параметры для продакшена
```bash
# 1. Резервная копия для критичных файлов
backup: yes  # Создаёт файл с суффиксом .<timestamp>~

# 2. Валидация перед применением
validate: 'nginx -t -c %s'  # %s = путь к временному файлу с изменениями

# 3. Запуск в check-mode перед применением
# В CLI:
ansible-playbook playbook.yml --check --diff

# 4. Логирование изменений
# В ansible.cfg:
[defaults]
log_path = /var/log/ansible/ansible.log
```
### Отладка: как понять, что изменилось
```bash
# Показать различия в файлах (работает с --check)
ansible-playbook playbook.yml --check --diff -v

# Детальный вывод выполнения модуля
ansible-playbook playbook.yml -vvv | grep -A10 "lineinfile\|replace\|blockinfile"

# Проверить резервные копии
ansible all -a "ls -la /etc/ssh/*~ 2>/dev/null || echo 'No backups'"
```

### Частые ошибки и решения

|Ошибка|Причина|Решение|
|---|---|---|
|`changed` при каждом запуске, хотя файл не меняется|`regexp` не точно соответствует строке; `line` имеет лишние пробелы|Используйте `--diff` для просмотра различий; проверьте точное совпадение строки|
|`regexp` не находит строку|Экранирование в YAML; неверный синтаксис RegEx|Используйте одинарные кавычки; тестируйте паттерн через `grep -E`|
|`blockinfile` дублирует блок|Маркеры не совпадают с шаблоном; файл был отредактирован вручную|Проверьте параметр `marker`; используйте уникальные маркеры для разных блоков|
|`validate` падает, хотя конфиг валиден|`%s` не подставляется; команда требует абсолютный путь|Убедитесь, что `validate` содержит `%s`; используйте абсолютные пути в команде|

## Выводы

**Механика модулей**:

- `lineinfile`: поиск по `regexp` → замена/добавление одной строки; идемпотентен за счёт точного сравнения.
- `replace`: массовая замена всех вхождений `regexp`; поддержка `after`/`before` для ограничения области; обратные ссылки `\1` требуют групп `()`.
- `blockinfile`: управление блоками через маркеры; автоматическая замена содержимого между маркерами; ручное редактирование между маркерами перезаписывается.

**RegEx в YAML**:

- Всегда заключайте паттерны в одинарные кавычки для избежания интерпретации `\` и `#`.
- Используйте нежадные квантификаторы `.*?` и якоря `^`/`$` для точного совпадения.
- Тестируйте паттерны через `grep -E` перед использованием в playbook.

**Безопасность изменений**:

- `backup: yes` создаёт резервные копии с временным суффиксом — обязательно для продакшена.
- `validate` с `%s` проверяет синтаксис файла перед применением — критично для конфигов сервисов.
- `--check --diff` показывает предполагаемые изменения без применения — обязательный этап перед запуском.

**Выбор модуля**:

- Точечная правка строки → `lineinfile`.
- Массовая замена по паттерну → `replace`.
- Добавление/замена многострочного блока → `blockinfile`.
- Полная генерация файла из переменных → `template`.

## Лабораторная работа 8: Управление конфигурационными файлами

### Последовательность выполнения

#### 0: Подготовка окружения
```bash
mkdir -p ~/lab8/{playbooks, inventory}
cd ~/lab8
cp ../lab7/inventory.yml ./inventory/
cp ../lab7/ansible.cfg ./
```
#### 1: Создать playbook `text_ops.yml`
```yaml
# ~/lab8/playbooks/text_ops.yml
---
- name: Text and Config Management
  hosts: all
  become: true
  gather_facts: false
  tasks:
    # 1. Добавить/обновить AllowUsers в sshd_config
    - name: Ensure AllowUsers directive
