## Управление пользователями и группами
### Модуль `ansible.builtin.group`
| Параметр | Описание                                 | Пример       |
| -------- | ---------------------------------------- | ------------ |
| `name`   | Имя группы                               | `developers` |
| `state`  | `present` (создать) / `absent` (удалить) | `present`    |
| `gid`    | Числовой идентификатор группы            | `2000`       |
| `system` | Создать системную группу                 | `yes`        |
```yaml
- name: Create developers group
  ansible.builtin.group:
    name: developers
    state: present
    gid: 2000
```

### Модуль `ansible.builtin.user`
| Параметр        | Описание                                                 | Пример                                          |
| --------------- | -------------------------------------------------------- | ----------------------------------------------- |
| `name`          | Логин пользователя                                       | `devops`                                        |
| `comment`       | Полное имя / описание                                    | `"DevOps Engineer"`                             |
| `shell`         | Оболочка входа                                           | `/bin/bash`                                     |
| `groups`        | Дополнительные группы                                    | `wheel,developers`                              |
| `append`        | Добавлять к текущим группам (`yes`), или заменить (`no`) | `yes`                                           |
| `password`      | **Хэш** пароля (не plain text!)                          | `"{{ 'Pa$$w0rd' \| password_hash('sha512') }}"` |
| `password_lock` | Заблокировать вход по паролю                             | `yes`                                           |
| `create_home`   | Создать домашнюю директорию                              | `yes` (по умолчанию)                            |
>[!WARNING] Безопасность паролей 
Никогда не храните plain-text пароли в playbook! Используйте фильтр `password_hash` или `ansible-vault`.
>```bash
># Генерация хэша вручную (если нужно)
openssl passwd -6 'Pa$$w0rd'
>```

```yaml
- name: Create devops user with sudo rights
  ansible.builtin.user:
    name: devops
    comment: "DevOps Engineer"
    shell: /bin/bash
    groups: wheel
    append: yes
    password: "{{ 'Pa$$w0rd' | password_hash('sha512') }}"
    update_password: on_create  # не менять пароль при повторных запусках, если пользователь уже есть
    create_home: yes
```

## Управление пакетами
### RHEL/CentOS/Fedora: `ansible.builtin.dnf`
| Параметр            | Описание                                   | Пример             |
| ------------------- | ------------------------------------------ | ------------------ |
| `name`              | Пакет или список `['pkg1', 'pkg2']`        | `['vim', 'mc']`    |
| `state`             | `present`, `latest`, `absent`              | `present`          |
| `update_cache`      | Выполнить `dnf makecache` перед установкой | `yes`              |
| `disable_gpg_check` | Отключить проверку подписи                 | Только для тестов! |
```yaml
- name: Install essential utilities
  ansible.builtin.dnf:
    name: ['mc', 'htop', 'wget', 'git']
    state: present
    update_cache: yes

# Подключение репозитория (GPG-ключи добавятся автоматически)
- name: Enable EPEL repository
  ansible.builtin.dnf:
    name: epel-release
    state: present
```
### Debian/Ubuntu: `ansible.builtin.apt`
```yaml
- name: Install utilities on Debian/Ubuntu
  ansible.builtin.apt:
    name: ['mc', 'htop', 'wget', 'curl']
    state: present
    update_cache: yes  # аналог apt update
    # upgrade: safe   # раскомментировать для обновления всех пакетов
    # autoremove: yes # удалить неиспользуемые зависимости
```
### Кроссплатформенная альтернатива: `ansible.builtin.package`  
```yaml
- name: Install packages (OS-agnostic)
  ansible.builtin.package:
    name: ['vim', 'htop']
    state: present
```

>[!TIP] Когда использовать `package`? 
Если один playbook должен работать и на RHEL, и на Ubuntu без условий `when`. Ansible сам выберет `dnf` или `apt`.

## Управление службами: `systemd` vs `service`
>[!NOTE] systemd and service
>- `ansible.builtin.systemd` — современный стандарт для систем с `systemd`. 
>- `ansible.builtin.service` — устаревший, но обратно совместимый модуль. 
>- Рекомендуется использовать `systemd`.

```yaml
- name: Ensure sshd is running and enabled at boot
  ansible.builtin.systemd:
    name: sshd
    state: started      # started, stopped, restarted, reloaded
    enabled: yes        # автозагрузка
    masked: no          # разблокировать, если был masked
```

|Параметр|Значения|Описание|
|---|---|---|
|`state`|`started`, `stopped`, `restarted`, `reloaded`|Текущее состояние службы|
|`enabled`|`yes` / `no`|Автозапуск при загрузке ОС|
|`daemon_reload`|`yes`|Выполнить `systemctl daemon-reload` (при изменении `.service` файлов)|

## Настройка брандмауэра: `ansible.posix.firewalld`
>[!IMPORTANT] Требует коллекции
>```bash
ansible-galaxy collection install ansible.posix
>```

| Параметр    | Описание                            | Пример                  |
| ----------- | ----------------------------------- | ----------------------- |
| `service`   | Имя предопределённого сервиса       | `http`, `ssh`, `https`  |
| `port`      | Порт/протокол                       | `8080/tcp`              |
| `zone`      | Зона firewalld                      | `public` (по умолчанию) |
| `state`     | `enabled` / `disabled`              | `enabled`               |
| `permanent` | Сохранить в конфигурации            | `yes`                   |
| `immediate` | Применить к runtime без перезапуска | `yes`                   |
```yaml
- name: Open HTTP, HTTPS and custom port
  ansible.posix.firewalld:
    service: "{{ item }}"
    permanent: yes
    immediate: yes
    state: enabled
  loop:
    - http
    - https

- name: Open custom application port
  ansible.posix.firewalld:
    port: 8080/tcp
    permanent: yes
    immediate: yes
    state: enabled

# Перезапуск firewalld НЕ требуется при использовании immediate: yes
# Но если нужно сбросить правила:
- name: Reload firewalld to apply permanent rules cleanly
  ansible.builtin.systemd:
    name: firewalld
    state: reloaded
```

>[!TIP] `permanent` + `immediate`
>
>- `permanent: yes` → правило сохраняется после ребута
>- `immediate: yes` → правило применяется сразу в runtime Используйте оба параметра вместе для предсказуемого результата.

## Комбинированный Playbook: базовая настройка хоста
```yaml
# base_setup.yml
---
- name: Initial host configuration
  hosts: all
  become: yes
  tasks:
    # 1. Пользователь
    - name: Create admin user
      ansible.builtin.user:
        name: admin
        comment: "System Administrator"
        shell: /bin/bash
        groups: wheel
        append: yes
        password: "{{ 'SecureP@ss2024' | password_hash('sha512') }}"
        update_password: on_create

    # 2. Пакеты
    - name: Install monitoring tools
      ansible.builtin.package:
        name: ['mc', 'htop', 'net-tools', 'curl']
        state: present

    # 3. Службы
    - name: Ensure sshd and firewalld are running
      ansible.builtin.systemd:
        name: "{{ item }}"
        state: started
        enabled: yes
      loop:
        - sshd
        - firewalld

    # 4. Брандмауэр
    - name: Allow SSH, HTTP, HTTPS in firewalld
      ansible.posix.firewalld:
        service: "{{ item }}"
        permanent: yes
        immediate: yes
        state: enabled
      loop:
        - ssh
        - http
        - https
```

## Выводы

**Централизованное управление**:

- Ansible позволяет декларативно описывать желаемое состояние пользователей, пакетов, служб и правил брандмауэра
- Все изменения идемпотентны: безопасный повторный запуск без дублирования или поломки

**Безопасность и лучшие практики**:

- Никогда не храните пароли в открытом виде — используйте `password_hash` или `ansible-vault`
- `permanent: yes` + `immediate: yes` в `firewalld` гарантирует сохранение и мгновенное применение
- Для кроссплатформенных сценариев используйте `ansible.builtin.package` вместо `dnf`/`apt`

**Архитектура административных задач**:

- Группировка задач по логическим блокам улучшает читаемость
- `become: yes` на уровне play упрощает управление привилегиями
- `update_password: on_create` предотвращает случайный сброс паролей при повторных запусках

#### 1. Подготовка
```bash
mkdir -p ~/lab5/{playbooks,inventory}

# Скопируйте inventory.yml и ansible.cfg из предыдущих лаб

```
#### 2. Генерация хэша пароля
```bash
openssl passwd -6 'Pa$$w0rd'
# Скопируйте вывод вида: $6$rounds=5000$salt$hash...
```

#### 3. Создать `playbooks/admin_tasks.yml`
```yaml
# ~/lab5/playbooks/admin_tasks.yml
---
- name: Admin tasks automation
  hosts: all
  become: true
  vars:
    admin_user: asnible
    admin_password_hash: "$6$..." # ← Вставьте ваш хэш сюда или используйте password_hash фильтр

  tasks:
    - name: Create stepan user with administrator group
      ansible.builtin.user:
        name: "{{ admin_user }}"
        comment: "administrator"
        shell: /bin/bash
        groups: ansible
        append: true
        password: "{{ admin_password_hash }}"
        update_password: on_create

    - name: Install required packages
      ansible.builtin.package:
        name: ['mc', 'firewalld', 'wget']
        state: present
    
    - name: Install pyton firewall lib for deb
      ansible.builtin.apt:
        name: python3-firewall
        state: present
      when: ansible_os_family == "Debian"
    
    - name: Ensure sshd is running and enabled
      ansible.builtin.systemd:
        name: firewalld
        state: started
        enabled: true

    - name: Open ports 22, 80, 443 in firewalld
      ansible.posix.firewalld:
        port: "{{ item }}"
        permanent: true
        immediate: true
        state: enabled
      loop:
        - 22/tcp
        - 80/tcp
        - 443/tcp
```

#### 4. Проверка и запуск
```bash
# Проверка синтаксиса
ansible-playbook --syntax-check playbooks/admin_tasks.yml

# Dry-run
ansible-playbook -i inventory/inventory.yml playbooks/admin_tasks.yml --check

# Запуск
ansible-playbook -i inventory/inventory.yml playbooks/admin_tasks.yml -v
```
#### 5. Верификация на целевом хосте
```bash
# Проверка пользователя
ansible all -a "id stepan"
ansible all -a -b "grep stepan /etc/shadow"

# Проверка пакетов
ansible all -a "rpm -q mc wget"  # RHEL
ansible ansible_deb -m shell -a "dpkg -l | grep -E 'mc|wget'"  # Debian

# Проверка служб и firewall
ansible all -a "systemctl is-active firewalld"
ansible all -b -a "firewall-cmd --list-all"
```

>
>- Пользователь `stepan` создан, состоит в группе `wheel`, имеет заданный хэш пароля

>- `sshd` в статусе `active (running)` и `enabled`
>- В `firewalld` открыты порты `22/tcp`, `80/tcp`, `443/tcp` с флагами `permanent` и `runtime`
>- Playbook выполняется идемпотентно (повторный запуск → все задачи `ok`, не `changed`)
