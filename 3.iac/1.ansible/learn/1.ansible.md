# Модуль 1 Ansible
## Что такое Ansible?

- Ansible — инструмент автоматизации ИТ-задач (конфигурация, развертывание, управление).
- **Agentless-подход**: не требует установки агентов на управляемые узлы.
- Использует **SSH** как основной транспорт (для Linux) или **WinRM** (для Windows).
- Язык описания задач — **YAML** (читаемый и человекоспонятный).
- Идемпотентность: повторный запуск задачи не ломает систему, если желаемое состояние уже достигнуто.

> [!TIP] Почему agentless — это важно?
> 
> - Меньше накладных расходов на управляемых узлах
> - Проще безопасность (только стандартный SSH)
> - Быстрое подключение новых хостов

## Архитектура Ansible
```text
┌─────────────────┐
│  Control Node   │ ← Здесь установлен Ansible, запускаются           playbooks
│    ( сервер )   │
└────────┬────────┘
         │ SSH / WinRM
         ▼
┌─────────────────┐
│  Managed Nodes  │ ← Целевые серверы (без установки Ansible!)
│ (веб, БД, сеть) │
└─────────────────┘
```

### Ключевые компоненты:

- **Control Node** — машина, с которой запускается Ansible (требуется Python)
- **Managed Nodes** (Hosts) — целевые системы, которыми управляют
- **Inventory** — файл/источник со списком хостов и их группами
- **Modules** — единицы кода, выполняющие конкретные действия (`yum`, `copy`, `service`)
- **Collections** — пакеты с модулями, плагинами и ролями (распространение через Ansible Galaxy)
- **Tasks** — отдельные шаги (вызов модуля с параметрами)
- **Playbooks** — YAML-файлы с упорядоченными задачами для воспроизводимого выполнения.

### Инвентаризация (Inventory)

Инвентаризация — это «карта» управляемых узлов. Поддерживает:

- Статические файлы (INI или YAML формат)
- Динамические источники (AWS EC2, Kubernetes, скрипты)
- Группы и вложенные группы для масштабирования
- Переменные на уровне хоста/группы.

## Установка Ansible
```bash
# RHEL/CentOS/Fedora
sudo dnf install -y ansible-core

# Debian/Ubuntu
sudo apt update && sudo apt install -y ansible-core

# Arch Linux
sudo pacman -S ansible-core 

# macOS (через Homebrew)
brew install ansible

# Проверка установки
ansible --version
ansible-core --version
```

> [!NOTE] Версии
> 
> - `ansible` — полный пакет (включает зависимости)
> - `ansible-core` — минимальная версия (рекомендуется для обучения)

## Где Ansible ищет конфигурацию? (приоритет по убыванию)

1. Переменная окружения: `ANSIBLE_CONFIG=/path/to/ansible.cfg`
2. Файл в текущей директории: `./ansible.cfg`
3. Домашняя директория: `~/.ansible.cfg`
4. Глобальный файл: `/etc/ansible/ansible.cfg`

> [!TIP] Для начала можно не создавать конфиг — Ansible работает с настройками по умолчанию.

## Инвентаризация: примеры и паттерны

### Базовый INI-формат (`/etc/ansible/hosts` или свой файл)

```ini
# Простой список хостов
[web]
192.168.1.10
server1.example.com

[db]
192.168.1.20
192.168.1.21

# Хосты с переменными (пользователь, порт, ключ)
[production]
web01 ansible_host=10.0.0.5 ansible_user=deploy ansible_port=2222
db01 ansible_host=10.0.0.10 ansible_user=root

# Группы групп
[all_services:children]
web
db

# Диапазоны и паттерны
[servers]
node[1:5].example.com  # node1, node2, ..., node5
192.168.1.[10:20]      # 192.168.1.10 - 192.168.1.20
```

### Использование кастомного inventory-файла
```bash
# Запуск с указанием своего файла инвентаризации
ansible all -i ./my_inventory.ini -m ping

# Проверка списка хостов в группе
ansible web -i ./my_inventory.ini --list-hosts
```

## Подключение к удалённым узлам

Ansible использует SSH. Перед началом работы убедитесь:

```bash
# 1. Проверка SSH-доступа вручную
ssh user@192.168.1.10

# 2. Настройка SSH-ключей (рекомендуется)
ssh-copy-id user@192.168.1.10

# 3. Быстрая проверка через Ansible
ansible all -i ./inventory.ini -m ping
```

> [!WARNING] Если используется пароль (не ключи)
>
>- Добавьте `--ask-pass` для запроса пароля при запуске
>- Для `become` (sudo) используйте `--ask-become-pass`
>- Но лучше настроить аутентификацию по ключам!

## Запуск команд: Ad-Hoc режим

Ad-hoc команды — способ выполнить одну задачу без создания playbook.

### Синтаксис:
```bash
# Запуск модуля
ansible <группа_или_хост> -i <inventory> -m <модуль> -a "<аргументы>"

# Выполнение произвольной shell-команды
ansible <группа_или_хост> -i <inventory> -a "<shell-команда>"
```
### Практические примеры:

```bash
# Проверка подключения (модуль ping)
ansible all -i ./inventory.ini -m ping

# Сбор фактов о системе (аналог uname, ifconfig и т.д.)
ansible web -i ./inventory.ini -m setup -a "filter=ansible_distribution*"

# Выполнение shell-команды
ansible all -a "uptime"

# Команда с повышенными привилегиями (sudo)
ansible db -b -a "systemctl status postgresql"

# Копирование файла (модуль copy)
ansible web -m copy -a "src=/local/file.txt dest=/remote/file.txt mode=0644"

# Установка пакета (для Debian/Ubuntu)
ansible web -m apt -a "name=nginx state=present" -b

# Проверка без внесения изменений (check mode)
ansible web -m apt -a "name=nginx state=present" -b --check
```

### Полезные флаги:
|Флаг|Описание|
|---|---|
|`-i <file>`|Указать путь к inventory-файлу|
|`-m <module>`|Выбрать модуль (по умолчанию: `command`)|
|`-a "<args>"`|Аргументы для модуля или команды|
|`-b`|Запуск с `become` (sudo)|
|`--check`|Dry-run: показать, что будет изменено, без применения|
|`--diff`|Показать различия в файлах (работает с `--check`)|
|`-v`, `-vv`, `-vvv`|Уровень детализации вывода (отладка)|

## Первый Playbook: от простого к сложному

### Пример 1: Минимальный рабочий playbook
```yaml
# hello.yml
---
- name: Say Hello to All Hosts
  hosts: all
  gather_facts: no  # не собирать системные факты для скорости
  tasks:
    - name: Print greeting message
      ansible.builtin.debug:
        msg: "Hello from Ansible!"
```

**Запуск:**
```bash
ansible-playbook -i ./inventory.ini hello.yml
```

### Пример 2: Чуть сложнее — работа с фактами и условиями
```yaml
# system_info.yml
---
- name: Gather and display system info
  hosts: web
  gather_facts: yes  # собираем факты о системе
  tasks:
    - name: Show OS distribution
      ansible.builtin.debug:
        msg: "OS: {{ ansible_distribution }} {{ ansible_distribution_version }}"

    - name: Show available memory
      ansible.builtin.debug:
        msg: "Memory: {{ ansible_memtotal_mb }} MB"
```

>[!TIP] Факты (facts)
>
>- Автоматически собираемые данные о хосте (ОС, CPU, сеть и т.д.)
>- Доступны как переменные: `{{ ansible_hostname }}`, `{{ ansible_default_ipv4.address }}`
>- Отключайте `gather_facts: no`, если они не нужны — это ускоряет выполнение

### Пример 3: Идемпотентная задача (управление сервисом)
```yaml
# nginx_basic.yml
---
- name: Ensure Nginx is installed and running
  hosts: web
  become: yes  # использовать sudo
  tasks:
    - name: Install nginx package
      ansible.builtin.apt:  # или yum/dnf для RHEL-семейства
        name: nginx
        state: present
        update_cache: yes

    - name: Start and enable nginx service
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: yes
```

>Этот плейбук можно запускать многократно: если nginx уже установлен и запущен — ничего не изменится.

## Советы по написанию качественных Playbook
```yaml
---
# 1. Всегда начинайте с --- (явное указание YAML)
# 2. Используйте понятные имена
- name: [Что делает?] [На каких хостах?]
  hosts: <group>
  become: yes  # если нужны права root
  vars:        # локальные переменные (опционально)
    app_port: 8080

  tasks:
    # 3. Каждая задача должна иметь name
    - name: Step 1: Describe what this task does
      ansible.builtin.<module>:
        key: value
      # 4. Комментируйте сложные или нетривиальные шаги
      # Это поможет вам и команде через месяц

    # 5. Группируйте логически связанные задачи
    - name: Step 2: Next logical block
      ...
```

>[!CHECKLIST] Перед запуском плейбука
>
>- Проверен синтаксис: `ansible-playbook --syntax-check playbook.yml`
>- Протестирован в check-mode: `--check --diff`
>- Указан правильный inventory: `-i ./my_hosts.ini`
>- Понятны имена задач (вывод будет читаемым)

## Подерживаемые ОС (управляющие и управляемые узлы)

**Control Node** (где установлен Ansible):

- Linux: RHEL, CentOS, Fedora, Ubuntu, Debian, SUSE, Alpine
- macOS (через Homebrew)
- WSL2 на Windows

**Managed Nodes** (целевые системы):

- Любые системы с SSH-доступом и Python (2.7 или 3.5+)
- Сетевые устройства (через специальные коллекции: `ansible.netcommon`)
- Windows (через WinRM и коллекцию `ansible.windows`)

> [!NOTE] Для Windows-узлов потребуется дополнительная настройка подключения (тема следующих уроков).

## Мини-практика

Попробуйте самостоятельно:

1. Создать простой inventory-файл с двумя группами: `web` и `db`
2. Выполнить `ping` ко всем хостам: `ansible all -i ./hosts -m ping`
3. Узнать версию ОС на хостах группы `web`: `ansible web -m setup -a "filter=ansible_distribution*"`
4. Написать и запустить playbook, который выводит имя хоста и количество CPU

```yaml
# practice.yml
---
- name: Practice task
  hosts: web
  gather_facts: yes
  tasks:
    - name: Show hostname and CPU count
      ansible.builtin.debug:
        msg: "Host: {{ ansible_hostname }}, CPUs: {{ ansible_processor_vcpus }}"
```
