## Архитектура выполнения

### Компоненты и их роль в движке Ansible

| Компонент        | Техническая роль                                                                                               | Требования                                                        |
| ---------------- | -------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| **Control Node** | Инициатор выполнения: парсит YAML, строит executor, управляет очередью задач                                   | Python 3.8+, `ansible-core`, доступ к целевым хостам по SSH/WinRM |
| **Managed Node** | Целевая система: получает модуль через SSH, выполняет его в изолированном процессе, возвращает JSON-результат  | SSH-сервер (OpenSSH), Python 2.7/3.5+ (для выполнения модулей)    |
| **Inventory**    | Источник метаданных хостов: разрешает имена хостов, подставляет переменные, группирует цели                    | Файл (INI/YAML), плагин (AWS, K8s), скрипт (динамический)         |
| **Module**       | Единица исполнения: Python-скрипт, передаётся на managed node, выполняется, возвращает `changed`/`failed`/`ok` | Зависит от модуля (большинство — только stdlib Python)            |
| **Plugin**       | Расширение движка: callback, connection, inventory, lookup — загружаются динамически при необходимости         | Соответствие интерфейсу плагина, регистрация в `ansible.cfg`      |
| **Playbook**     | Декларативное описание желаемого состояния: компилируется в исполняемый план задач                             | Валидный YAML, корректные FQCN модулей, доступные переменные      |

### Механика agentless-выполнения
```
1. Контрольный узел устанавливает SSH-соединение с managed node
2. Создаёт временную директорию (~/.ansible/tmp/)
3. Копирует модуль + аргументы (в JSON) на удалённый хост
4. Запускает модуль через Python с параметрами
5. Модуль выполняет логику, возвращает JSON-результат
6. Контрольный узел парсит результат, обновляет статус задачи
7. Очищает временные файлы (если не задано ANSIBLE_KEEP_REMOTE_FILES=1)
```

>[!NOTE] Идемпотентность на уровне модуля 
Каждый модуль самостоятельно проверяет текущее состояние перед внесением изменений. Если желаемое состояние уже достигнуто → возвращает `changed: false`. Это свойство модуля, а не движка.

## Установка и проверка окружения
### Установка `ansible-core` (2026)
```bash
# RHEL/CentOS Stream 9+
sudo dnf install -y ansible-core

# Debian 12 / Ubuntu 22.04+
sudo apt update && sudo apt install -y ansible-core

# Arch Linux
sudo pacman -S ansible-core

# macOS (Homebrew)
brew install ansible-core

# Проверка версии и зависимостей
ansible-core --version
python3 --version
ssh -V
```

### Проверка минимальных требований
```bash
# Проверка доступности управляемых хостов
ansible all -i inventory.ini -m ping -vvv

# Проверка версии Python на удалённом хосте
ansible all -i inventory.ini -m setup -a "filter=ansible_python_version"

# Проверка прав на выполнение модулей (временно отключить requiretty)
ansible all -i inventory.ini -m shell -a "echo test" --become
```


>[!WARNING] Версии пакетов
>
>- `ansible` (метапакет) → включает `ansible-core` + зависимости (устаревший способ)
>- `ansible-core` → минимальная версия, рекомендуется для обучения и производства
>- Убедитесь, что версия `ansible-core` ≥ 2.14 для поддержки актуальных коллекций

## Конфигурация: приоритеты и параметры

### Порядок поиска `ansible.cfg` (по убыванию приоритета)
```bash
1. $ANSIBLE_CONFIG                    # переменная окружения
2. ./ansible.cfg                      # текущая директория (проект)
3. ~/.ansible.cfg                     # домашняя директория пользователя
4. /etc/ansible/ansible.cfg           # глобальный системный конфиг
```

### Проверка активного конфига
```bash
ansible --version | grep "config file"
# Вывод: config file = /home/user/project/ansible.cfg
```
### Ключевые параметры `[defaults]` (для начала)
```ini
[defaults]
# Путь к инвентарю (можно несколько через запятую)
inventory = ./inventory.yml

# Пользователь для SSH-подключения
remote_user = ansible

# Приватный SSH-ключ
private_key_file = ~/.ssh/id_ed25519

# Таймаут подключения (сек)
timeout = 30

# Количество параллельных форков
forks = 10

# Отключить проверку хост-ключей (только для лаб!)
host_key_checking = False

# Логирование
log_path = /var/log/ansible/ansible.log

# Интерпретатор Python на managed node (если не /usr/bin/python)
# interpreter_python = /usr/bin/python3
```

>[!NOTE] Параметры `become`
>```ini
>[privilege_escalation]
>become = True
>become_method = sudo
>become_user = root
>become_ask_pass = False
>```

## Инвентаризация: механика и синтаксис

### Форматы и парсинг
| Формат | Парсер                 | Поддержка переменных            | Динамические источники |
| ------ | ---------------------- | ------------------------------- | ---------------------- |
| INI    | `ansible.builtin.ini`  | Да, через `:vars` и `key=value` | Нет                    |
| YAML   | `ansible.builtin.yaml` | Да, нативная структура          | Нет                    |
| Plugin | Специфичный плагин     | Зависит от плагина              | Да (AWS, K8s, скрипты) |
### INI-формат: синтаксис переменных
```ini
[webservers]
# Хост с переменными подключения
web01 ansible_host=10.0.0.10 ansible_user=deploy ansible_port=2222

# Переменные для всей группы
[webservers:vars]
ansible_user=nginx
http_port=80
ntp_server=pool.ntp.org

# Группа групп (мета-группа)
[prod:children]
webservers
db

# Паттерны и диапазоны
[batch]
node[01:10].example.com      # node01...node10
192.168.1.[100:110]          # 192.168.1.100-110
```
---
### YAML-формат: структура данных
```yaml
# inventory.yml
---
all:
  children:
    webservers:
      hosts:
        web01.example.com:
          ansible_host: 10.0.0.10
          ansible_port: 22
        web02.example.com:
          ansible_host: 10.0.0.11
      vars:
        ansible_user: nginx
        http_port: 80

    db:
      hosts:
        db01.example.com:
          ansible_host: 10.0.0.20
          ansible_user: postgres
          ansible_port: 2222

    prod:
      children:
        webservers:
        db
```
### Использование кастомного inventory
```bash
# Указать файл явно
ansible all -i ./inventory.yml -m ping

# Проверить разрешённые хосты для шаблона
ansible 'webservers:&prod' --list-hosts -i ./inventory.yml

# Показать переменные для хоста
ansible-inventory --host web01.example.com -i ./inventory.yml
```

## SSH-подключение: параметры и отладка

### Механика подключения

- Ansible использует `ssh` CLI-утилиту (не библиотеку) для установки соединения
- Параметры передачи: `ControlPersist`, `pipelining`, `ssh_args` из `ansible.cfg`
- Аутентификация: ключи (рекомендуется), пароль (`--ask-pass`), `ssh-agent`

### Проверка и настройка
```bash
# 1. Ручная проверка SSH
ssh -i ~/.ssh/id_ed25519 ansible@192.168.1.10 "echo OK"

# 2. Копирование ключа (если не настроено)
ssh-copy-id -i ~/.ssh/id_ed25519.pub ansible@192.168.1.10

# 3. Проверка через Ansible с детализацией
ansible all -i inventory.yml -m ping -vvv
```

```bash
# 1. Ручная проверка SSH
ssh -i ~/.ssh/id_ed25519 ansible@192.168.1.10 "echo OK"

# 2. Копирование ключа (если не настроено)
ssh-copy-id -i ~/.ssh/id_ed25519.pub ansible@192.168.1.10

# 3. Проверка через Ansible с детализацией
ansible all -i inventory.yml -m ping -vvv
```

>[!WARNING] `host_key_checking = False` 
Отключайте только в изолированных лабораторных средах. В production используйте `known_hosts` или доверенные сертификаты.

## Ad-hoc команды: синтаксис и параметры

### Общая форма
```bash
ansible <host_pattern> -i <inventory> [flags] -m <module> -a "<module_args>"
```

### Ключевые флаги CLI

|Флаг|Описание|Пример|
|---|---|---|
|`-i <file>`|Путь к inventory|`-i ./hosts.yml`|
|`-m <module>`|Модуль для выполнения|`-m ping`|
|`-a "<args>"`|Аргументы модуля (строка)|`-a "name=nginx state=present"`|
|`-b`|Включить `become` (sudo)|`-b -a "systemctl restart nginx"`|
|`--check`|Dry-run: не применять изменения|`--check`|
|`--diff`|Показать различия в файлах|`--check --diff`|
|`-v`/`-vv`/`-vvv`|Уровень детализации вывода|`-vvv` для отладки модуля|
|`--limit <pattern>`|Ограничить выполнение подмножеством хостов|`--limit "web01:web02"`|
### Практические примеры
```bash
# Проверка подключения (модуль ping)
ansible all -i inventory.yml -m ping

# Сбор фактов с фильтрацией
ansible web -i inventory.yml -m setup -a "filter=ansible_*_address"

# Выполнение shell-команды
ansible all -a "uptime"

# Команда с sudo
ansible db -b -a "systemctl status postgresql"

# Копирование файла
ansible web -m copy -a "src=/tmp/config.txt dest=/etc/app/config.txt mode=0644"

# Установка пакета (Debian)
ansible web -m apt -a "name=nginx state=present update_cache=yes" -b

# Проверка без применения (идемпотентность)
ansible web -m apt -a "name=nginx state=present" -b --check
```


>[!NOTE] Модуль `command` vs `shell`
>
>- `command`: выполняет команду напрямую, без оболочки. Безопаснее, но не поддерживает пайпы, перенаправления, переменные оболочки.
>- `shell`: выполняет команду через `/bin/sh`. Мощнее, но требует экранирования и осторожности с пользовательским вводом.

## Playbook: структура и выполнение

### Минимальный валидный playbook
```yaml
# hello.yml
---
- name: Say Hello to All Hosts
  hosts: all
  gather_facts: false
  tasks:
    - name: Print greeting message
      ansible.builtin.debug:
        msg: "Hello from Ansible!"
```

### Ключевые директивы play
|Директива|Уровень|Описание|Пример|
|---|---|---|---|
|`name`|play / task|Человекочитаемое описание|`name: Install nginx`|
|`hosts`|play|Целевые хосты/группы из inventory|`hosts: webservers`|
|`gather_facts`|play|Собирать ли `ansible_facts`|`gather_facts: false`|
|`become`|play / task|Повышение привилегий (sudo)|`become: true`|
|`vars`|play / task|Локальные переменные|`vars: { port: 80 }`|
|`tasks`|play|Список задач (обязательно)|`tasks: [{...}]`|
### Пример с фактами и условиями
```yaml
# system_info.yml
---
- name: Gather and display system info
  hosts: web
  gather_facts: true
  tasks:
    - name: Show OS distribution
      ansible.builtin.debug:
        msg: "OS: {{ ansible_distribution }} {{ ansible_distribution_version }}"

    - name: Show available memory
      ansible.builtin.debug:
        msg: "Memory: {{ ansible_memtotal_mb }} MB"
      when: ansible_memtotal_mb is defined
```

### Идемпотентная задача: управление сервисом
```yaml
# nginx_basic.yml
---
- name: Ensure Nginx is installed and running
  hosts: web
  become: true
  tasks:
    - name: Install nginx package
      ansible.builtin.apt:
        name: nginx
        state: present
        update_cache: true

    - name: Start and enable nginx service
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: true
```

>[!NOTE] Порядок выполнения задач 
Задачи выполняются последовательно сверху вниз. При ошибке в задаче (без `ignore_errors`) выполнение play останавливается для данного хоста.


## Отладка и проверка playbook

### Предварительные проверки
```bash
# Проверка синтаксиса YAML и структуры
ansible-playbook --syntax-check playbook.yml

# Показать, какие хосты будут затронуты
ansible-playbook playbook.yml --list-hosts

# Показать список задач без выполнения
ansible-playbook playbook.yml --list-tasks
```

### Безопасное тестирование
```bash
# Check mode: показать, что ИЗМЕНИЛОСЬ бы, без применения
ansible-playbook playbook.yml --check

# + diff: показать различия в текстовых файлах
ansible-playbook playbook.yml --check --diff

# Запустить с детализацией вывода
ansible-playbook playbook.yml -v      # кратко
ansible-playbook playbook.yml -vvv    # подробно (отладка)
```
### Отладка во время выполнения
```bash
# Начать выполнение с конкретной задачи
ansible-playbook playbook.yml --start-at-task "Install nginx"

# Пошаговое выполнение (подтверждение перед каждой задачей)
ansible-playbook playbook.yml --step

# Выполнить только задачи с определённым тегом
ansible-playbook playbook.yml --tags "install,config"
```

## Выводы

**Архитектура**:

- Ansible использует agentless-модель: модули передаются и выполняются на managed node через SSH, результат возвращается в виде JSON.
- Control Node требует Python и `ansible-core`; managed node — только SSH и Python для выполнения модулей.

**Конфигурация**:

- Приоритет поиска `ansible.cfg`: env → ./ → ~/. → /etc/.
- Параметры `inventory`, `remote_user`, `private_key_file`, `host_key_checking` критичны для начальной настройки.

**Инвентаризация**:

- Поддерживает INI и YAML форматы, группы, вложенные группы, переменные на уровне хоста/группы.
- Динамические источники требуют плагинов и настраиваются отдельно.

**Ad-hoc и Playbooks**:

- Ad-hoc команды — для одноразовых задач; playbooks — для воспроизводимых сценариев.
- Идемпотентность обеспечивается на уровне модулей: `changed: false` при достижении желаемого состояния.

**Отладка**:

- `--check`, `--diff`, `-vvv` — основные инструменты безопасного тестирования.
- `ansible-playbook --syntax-check` и `--list-tasks` помогают валидировать playbook до выполнения.

## Лабораторная работа 1: Базовое выполнение задач
###  Последовательность выполнения

#### 0: Подготовка окружения
```bash
# Создать каталог проекта
mkdir -p ~/lab1/{inventory,playbooks}
cd ~/lab1

# Создать базовый inventory (INI-формат)
cat > inventory/hosts << 'EOF'
[web]
192.168.1.10 ansible_user=ansible
192.168.1.11 ansible_user=ansible

[db]
192.168.1.20 ansible_user=ansible
EOF
```
#### 1: Проверка подключения
```bash
# Проверить доступность всех хостов
ansible all -i inventory/hosts -m ping -v

# Проверить сбор фактов для группы web
ansible web -i inventory/hosts -m setup -a "filter=ansible_distribution*" -v
```
#### 2: Создать минимальный playbook
```yaml
# playbooks/hello.yml
---
- name: Basic execution
  hosts: all
  gather_facts: false
  tasks:
    - name: Print greeting
      ansible.builtin.debug:
        msg: "Hello from {{ inventory_hostname }}!"
```

#### 3: Проверка и запуск
```bash
# Проверка синтаксиса
ansible-playbook --syntax-check playbooks/hello.yml

# Запуск с детализацией
ansible-playbook -i inventory/hosts playbooks/hello.yml -v
```

#### 4: Создать playbook с фактами
```yaml
# playbooks/system_info.yml
---
- name: System information
  hosts: web
  gather_facts: true
  tasks:
    - name: Show hostname and CPU count
      ansible.builtin.debug:
        msg: "Host: {{ ansible_hostname }}, CPUs: {{ ansible_processor_vcpus }}"
```

#### Шаг 5: Верификация
```bash
# Запустить playbook с фактами
ansible-playbook -i inventory/hosts playbooks/system_info.yml -v

# Проверить идемпотентность (повторный запуск → changed=0)
ansible-playbook -i inventory/hosts playbooks/system_info.yml | grep "changed="
```

>[!CHECK] Критерии выполнения
>
>- `ansible all -m ping` возвращает `SUCCESS` для всех хостов в `inventory/hosts`
>- `hello.yml` выводит сообщение с `inventory_hostname` для каждого хоста
>- `system_info.yml` корректно подставляет `ansible_hostname` и `ansible_processor_vcpus`
>- Повторный запуск `system_info.yml` → `changed=0` (идемпотентность)
>- Все команды выполняются без ошибок аутентификации или синтаксиса
