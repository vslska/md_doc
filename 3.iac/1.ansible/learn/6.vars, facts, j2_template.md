## Механика разрешения переменных

### Алгоритм поиска значения переменной
```
При обращении к {{ var_name }} в задаче или шаблоне:
1. Проверить переменные командной строки (-e)
2. Проверить vars: в текущем play
3. Проверить host_vars/<inventory_hostname>.yml
4. Проверить group_vars/<group_name>.yml для всех групп хоста
5. Проверить переменные из inventory (INI/YAML)
6. Проверить ansible_facts (если gather_facts: true)
7. Проверить vars: внутри роли
8. Проверить defaults/main.yml внутри роли
9. Если не найдено → ошибка "undefined variable" (если не использован фильтр default)
```

### Источники переменных: техническая детализация
| Источник                   | Время подстановки     | Область видимости           | Переопределяет                       | Пример синтаксиса                    |
| -------------------------- | --------------------- | --------------------------- | ------------------------------------ | ------------------------------------ |
| `-e` / `--extra-vars`      | До парсинга playbook  | Глобально (все plays/hosts) | Всё                                  | `-e "env=prod"`                      |
| `vars:` в play             | Парсинг play          | В рамках одного play        | `host_vars`, `group_vars`, inventory | `vars: { port: 80 }`                 |
| `host_vars/<host>.yml`     | Загрузка inventory    | Конкретный хост             | `group_vars`, inventory              | `web01.yml: ansible_user=deploy`     |
| `group_vars/<group>.yml`   | Загрузка inventory    | Все хосты группы            | `inventory` переменные               | `webservers.yml: http_port=80`       |
| `vars_files:`              | Парсинг play          | В рамках play               | `group_vars`, inventory              | `vars_files: [secrets.yml]`          |
| Inventory файл             | Загрузка inventory    | Хост/группа                 | Только `defaults` ролей              | `web01 ansible_port=2222`            |
| `ansible_facts`            | Runtime (после setup) | Хост, где собраны факты     | Только `defaults`                    | `{{ ansible_default_ipv4.address }}` |
| `defaults/main.yml` (role) | Загрузка роли         | Внутри роли                 | Ничего (базовые значения)            | `app_port: 8080`                     |

>[!NOTE] Автоматическая загрузка `group_vars/` и `host_vars/` 
Ansible ищет эти директории относительно:
>
>1. Каталога с inventory-файлом
>2. Каталога с playbook (если inventory не указан явно) Файлы загружаются по точному совпадению имени: `group_vars/webservers.yml` применяется ко всем хостам в группе `webservers`.

## Приоритет переменных: полная иерархия

Упрощённая таблица покрывает большинство сценариев, но для отладки сложных конфликтов полезна полная последовательность (по убыванию приоритета):
```
1.  командная строка: -e, --extra-vars
2.  vars_prompt в play
3.  vars_files в play
4.  vars в play
5.  set_facts / registered vars (runtime)
6.  role vars (role_name/vars/main.yml)
7.  block vars (только для задач внутри block)
8.  task vars (внутри задачи)
9.  host_vars / host_vars/*.yml
10. group_vars / group_vars/*.yml
11. inventory переменные (груповые: [group:vars])
12. inventory переменные (хостовые: host key=value)
13. role defaults (role_name/defaults/main.yml)
14. модульные значения по умолчанию
```

### Практическое правило
```bash
# Если переменная ведёт себя «неожиданно»:
# 1. Проверь, не переопределена ли она в более приоритетном источнике
# 2. Используй debug для трассировки:
- name: Debug variable resolution
  ansible.builtin.debug:
    msg: "port={{ app_port }} (source: {{ app_port | default('undefined', true) }})"
  verbosity: 2  # Покажет только при -vv или выше
```

>[!WARNING] `host_vars` vs `group_vars` 
Если хост входит в несколько групп, переменные из `group_vars/` применяются в порядке объявления групп в inventory. При конфликте значений выигрывает последняя группа в списке.

## Системные факты (Facts): механика и управление

### Процесс сбора фактов
```
1. При старте play с gather_facts: true Ansible выполняет модуль setup на каждом хосте
2. setup собирает данные через:
   - /proc, /sys, lscpu, ip, df и другие системные интерфейсы
3. Результат (JSON) сохраняется в:
   - Переменная ansible_facts (словарь)
   - Плоские переменные ansible_* (для обратной совместимости)
4. Факты доступны во всех последующих задачах play
```
### Управление сбором фактов
| Параметр                     | Значение             | Эффект                                                     |
| ---------------------------- | -------------------- | ---------------------------------------------------------- |
| `gather_facts: true`         | По умолчанию         | Полный сбор всех фактов (~1-3 сек на хост)                 |
| `gather_facts: false`        | Отключение           | Ускорение запуска, но недоступны `ansible_*` переменные    |
| `gather_subset: min`         | Минимальный набор    | Только базовые факты: hostname, distribution, architecture |
| `gather_subset: network`     | Только сеть          | Факты об интерфейсах, IP, маршрутах                        |
| `gather_subset: '!hardware'` | Исключение категории | Собрать всё, кроме данных о железе                         |
### Фильтрация и выборка фактов
```shell
# Просмотр всех фактов (вывод может быть большим)
ansible all -m setup

# Фильтрация по маске (кавычки обязательны для bash globbing)
ansible all -m setup -a "filter=ansible_mem*"
ansible all -m setup -a "filter=ansible_*_address"

# Просмотр конкретного факта в структуре
ansible all -m setup -a "filter=ansible_default_ipv4"
```
### Использование в playbook
```yaml
- name: Show system info with safe defaults
  ansible.builtin.debug:
    msg: |
      Host: {{ inventory_hostname }}
      OS: {{ ansible_distribution }} {{ ansible_distribution_version }}
      Kernel: {{ ansible_kernel }}
      CPUs: {{ ansible_processor_vcpus | default('N/A') }}
      RAM: {{ ansible_memtotal_mb | int }} MB
      Primary IP: {{ ansible_default_ipv4.address | default('N/A') }}
```

>[!NOTE] `ansible_facts['key']` vs `ansible_key`
>
>- `ansible_facts['distribution']` — рекомендуемый способ (явный доступ к словарю)
>- `ansible_distribution` — legacy-переменная, автоматически создаётся из фактов
>- Оба работают, но `ansible_facts` предпочтительнее для нового кода

### Кэширование фактов
```ini
# ansible.cfg — ускорение повторных запусков
[defaults]
fact_caching = jsonfile
fact_caching_connection = /tmp/ansible_facts_cache
fact_caching_timeout = 3600  # TTL в секундах
```

```yaml
# Принудительное обновление фактов при использовании кэша
- name: Refresh facts for critical hosts
  ansible.builtin.setup:
    gather_subset: all
  when: inventory_hostname in groups['critical']
```

## Jinja2 в Ansible: синтаксис и контекст выполнения

### Типы конструкций

|Конструкция|Синтаксис|Назначение|Пример|
|---|---|---|---|
|Выражение|`{{ ... }}`|Подстановка значения|`{{ ansible_hostname }}`|
|Утверждение|`{% ... %}`|Логика: if/for/set|`{% if ansible_os_family == "RedHat" %}`|
|Комментарий|`{# ... #}`|Не выводится в результат|`{# Generated by Ansible #}`|
|Raw-блок|`{% raw %}...{% endraw %}`|Экранирование `{{ }}` в шаблоне|`{% raw %}{{ var }}{% endraw %}`|

### Контекст переменных в шаблоне

Шаблон `.j2` получает доступ ко **всем** переменным, доступным в момент вызова модуля `template`:

- Переменные из `vars:`, `vars_files:`, `group_vars/`, `host_vars/`
- Системные факты (`ansible_facts`, `ansible_*`)
- Магические переменные (`inventory_hostname`, `groups`, `hostvars`, `playbook_dir`)
- Результаты предыдущих задач (через `register`)

```yaml
# Пример: передача зарегистрированной переменной в шаблон
- name: Gather disk info
  ansible.builtin.command: df -h /
  register: disk_usage

- name: Deploy report from template
  ansible.builtin.template:
    src: report.j2
    dest: /tmp/report.txt
  # disk_usage.stdout доступен в шаблоне как {{ disk_usage.stdout }}
```

## Фильтры Jinja2: категории и примеры

### Таблица часто используемых фильтров

|Категория|Фильтр|Описание|Пример|Результат|
|---|---|---|---|---|
|**Fallback**|`default(val, true)`|Значение если var не определён **или пуст**|`{{ port \| default(80, true) }}`|`80` если `port` нет или `""`|
|**Текст**|`upper` / `lower`|Регистр строки|`{{ "Text" \| lower }}`|`"text"`|
|**Текст**|`replace(old, new)`|Замена подстроки|`{{ "a-b" \| replace("-", "_") }}`|`"a_b"`|
|**Текст**|`trim`|Убрать пробелы по краям|`{{ " text " \| trim }}`|`"text"`|
|**Числа**|`int` / `float`|Преобразование типа|`{{ "42" \| int }}`|`42` (integer)|
|**Числа**|`round(p, method)`|Округление|`{{ 3.14159 \| round(2) }}`|`3.14`|
|**Коллекции**|`length`|Количество элементов|`{{ [1,2,3] \| length }}`|`3`|
|**Коллекции**|`join(sep)`|Объединение списка|`{{ [1,2,3] \| join(",") }}`|`"1,2,3"`|
|**Коллекции**|`sort` / `reverse`|Сортировка|`{{ [3,1,2] \| sort }}`|`[1, 2, 3]`|
|**Коллекции**|`unique`|Убрать дубликаты|`{{ [1,2,2] \| unique }}`|`[1, 2]`|
|**Коллекции**|`selectattr(key, op, val)`|Фильтрация словарей|`{{ users \| selectattr('active', 'equalto', true) }}`|список активных|
|**Дата/время**|`to_datetime`|Строка → datetime-объект|`{{ "2024-01-01" \| to_datetime }}`|datetime object|
|**Дата/время**|`strftime(fmt)`|Форматирование даты|`{{ now() \| strftime('%Y-%m-%d') }}`|`"2024-01-15"`|
|**Ansible**|`to_nice_json` / `to_yaml`|Сериализация|`{{ data \| to_nice_json }}`|отформатированный JSON|
|**Ansible**|`password_hash(algo, salt)`|Хэширование пароля|`{{ "pass" \| password_hash('sha512') }}`|хэш для модуля `user`|
|**Ansible**|`ansible.builtin.regex_replace`|RegEx-замена (Ansible-фильтр)|`{{ "abc123" \| regex_replace('\d+', 'NUM') }}`|`"abcNUM"`|

### Цепочки фильтров

Фильтры выполняются слева направо:
```jinja2
{{ company_name | lower | replace('corp', 'company') | default('Internal') }}
{{ [3,1,2] | sort | reverse | join(', ') }}  # → "3, 2, 1"
```
>[!TIP] Фильтр `default` с вторым аргументом
>```jinja2
{{ var | default('fallback') }}      # Только если var не определён
{{ var | default('fallback', true) }} # Если var не определён ИЛИ пустая строка/список/словарь
>```

## Тесты Jinja2: проверка свойств переменных

Тесты используются в условиях `{% if %}` для проверки типа, значения или состояния переменной.

### Часто используемые тесты
| Тест                                       | Описание               | Пример                                                   |
| ------------------------------------------ | ---------------------- | -------------------------------------------------------- |
| `defined` / `undefined`                    | Переменная существует  | `{% if var is defined %}`                                |
| `none`                                     | Значение `None` / null | `{% if var is none %}`                                   |
| `equalto(val)` / `eq`                      | Равенство              | `{% if count is eq(10) %}`                               |
| `gt(val)`, `ge(val)`, `lt(val)`, `le(val)` | Числовое сравнение     | `{% if mem is ge(2048) %}`                               |
| `in sequence`                              | Вхождение в коллекцию  | `{% if role in ['web', 'api'] %}`                        |
| `string`, `number`, `sequence`, `mapping`  | Проверка типа          | `{% if data is sequence %}`                              |
| `match(pattern)` / `search(pattern)`       | RegEx-проверка         | `{% if name is match('^web\d+$') %}`                     |
| `version(ver, op)`                         | Сравнение версий       | `{% if ansible_version.full is version('2.14', '>=') %}` |
### Пример условия с тестами
```jinja2
{% if ansible_os_family == "RedHat" and ansible_distribution_major_version | int >= 8 %}
# RHEL 8+ specific config
{% elif ansible_os_family == "Debian" %}
# Debian/Ubuntu config
{% else %}
# Fallback for unknown OS
{% endif %}
```

## Модуль `ansible.builtin.template`: параметры и механика

### Алгоритм рендеринга
```
1. Прочитать шаблон .j2 с control node
2. Обработать через движок Jinja2 в контексте текущих переменных
3. Сравнить результат с существующим файлом на managed node
4. Если содержимое отличается → записать новый файл → changed: true
5. Если содержимое идентично → ничего не делать → changed: false
6. Если задан validate → выполнить команду с %s = путь к временному файлу
7. Если задан backup → создать копию оригинала перед заменой
```

| Параметр                   | Тип  | По умолчанию | Описание                                                               |
| -------------------------- | ---- | ------------ | ---------------------------------------------------------------------- |
| `src`                      | str  | —            | Путь к шаблону на control node (относительно playbook)                 |
| `dest`                     | str  | —            | Путь назначения на managed node (обязателен)                           |
| `owner` / `group` / `mode` | str  | —            | Права итогового файла (применяются после записи)                       |
| `backup`                   | bool | `false`      | Создать резервную копию с суффиксом `.<timestamp>~`                    |
| `validate`                 | str  | `None`       | Команда для проверки файла: `%s` подставляется путь к временному файлу |
| `trim_blocks`              | bool | `true`       | Убрать первый перенос строки после `{% %}` (улучшает форматирование)   |
| `lstrip_blocks`            | bool | `false`      | Убрать пробелы в начале строки перед `{% %}`                           |
| `newline_sequence`         | str  | `\n`         | Последовательность переноса строки: `\n`, `\r`, `\r\n`                 |
| `follow`                   | bool | `false`      | Следовать за символическими ссылками при проверке `dest`               |
### Пример с валидацией и форматированием
```yaml
- name: Deploy nginx config with validation
  ansible.builtin.template:
    src: templates/nginx.conf.j2
    dest: /etc/nginx/nginx.conf
    owner: root
    group: root
    mode: '0644'
    backup: yes
    validate: 'nginx -t -c %s'  # %s = путь к временному файлу с отрендеренным контентом
    trim_blocks: yes
    lstrip_blocks: yes
```
>[!WARNING] Переменная `%s` в `validate` 
Ansible подставляет **абсолютный путь** к временному файлу. Убедитесь, что команда валидации принимает абсолютный путь и не зависит от текущего каталога.

## Отладка и типичные ошибки

### Частые проблемы и решения

| Ошибка                                       | Причина                                             | Решение                                                                                      |
| -------------------------------------------- | --------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| `undefined variable: X`                      | Переменная не определена в контексте шаблона        | Использовать `\| default('fallback')` или проверить источник переменной                      |
| Шаблон рендерится, но файл не меняется       | Содержимое идентично существующему                  | Это ожидаемое поведение идемпотентности; проверь `--diff` для подтверждения                  |
| `validate` падает, хотя конфиг валиден       | `%s` не подставлен; команда требует абсолютный путь | Убедись, что `validate` содержит `%s` и команда работает с абсолютными путями                |
| Фильтр `int` возвращает 0 для строки         | Строка не является числом                           | Использовать `\| int(default=0)` или предварительно валидировать входные данные              |
| Условия `{% if %}` не работают как ожидается | Неправильный тест или тип переменной                | Использовать `debug` для вывода типа: `{{ var \| type_debug }}`                              |
| Переменные из `host_vars` не применяются     | Файл назван не точно по имени хоста в inventory     | Проверить точное совпадение: `host_vars/web01.example.com.yml` для хоста `web01.example.com` |

### Инструменты отладки
```shell
# Показать, какие переменные доступны в шаблоне
- name: Dump all variables (use with caution: large output)
  ansible.builtin.debug:
    var: vars
  verbosity: 3  # Только при -vvv

# Проверить рендеринг шаблона без записи на хост
ansible-playbook playbook.yml --check --diff

# Просмотреть итоговое содержимое переменной
ansible all -m debug -a "var=ansible_facts" --limit web01
```

## Выводы

 **Разрешение переменных**:

- Ansible использует строгий порядок поиска: `-e` → `play vars` → `host_vars` → `group_vars` → `inventory` → `role defaults`
- `group_vars/` и `host_vars/` загружаются автоматически при совпадении имён с группами/хостами в inventory
- Для отладки конфликтов используйте `debug` с `verbosity` или `ansible-playbook -vvv`

 **Системные факты**:

- Собираются модулем `setup` при `gather_facts: true`; можно фильтровать через `gather_subset` или `-a "filter=..."`
- `ansible_facts['key']` — рекомендуемый способ доступа; `ansible_key` — legacy-алиас
- Кэширование (`fact_caching`) ускоряет повторные запуски, но требует контроля актуальности данных

 **Jinja2 в Ansible**:

- Контекст шаблона включает все переменные, доступные на момент вызова `template`
- Фильтры (`default`, `join`, `password_hash`) и тесты (`defined`, `version`, `match`) расширяют возможности логики
- Цепочки фильтров выполняются слева направо; `default(val, true)` обрабатывает и `undefined`, и пустые значения

 **Модуль `template`**:

- Рендерит шаблон на control node, сравнивает с целевым файлом, записывает только при изменениях (идемпотентность)
- `validate` с `%s` проверяет синтаксис до применения; `backup: yes` сохраняет оригинал
- `trim_blocks`/`lstrip_blocks` улучшают форматирование итоговых конфигов

## Лабораторная работа 6: Переменные, факты и шаблоны
### Последовательность выполнения

#### Шаг 0: Подготовка окружения
```shell
mkdir -p ~/lab6/{playbooks,templates,group_vars,inventory}
cd ~/lab6
cp ../lab5/inventory.yml ./
cp ../lab5/ansible.cfg ./
```
#### Шаг 1: Создать переменные окружения
```yaml
# ~/lab6/group_vars/all.yml
---
company_name: "ООО Пример"
admin_user: "ivan"
support_email: "support@example.com"
min_ram_mb: 2048
```
#### Шаг 2: Создать шаблон `/etc/motd`
```jinja2
# ~/lab6/templates/motd.j2
****************************************************************
 Добро пожаловать: {{ inventory_hostname | upper }}
 Организация: {{ company_name | default('Internal') }}
 ОС: {{ ansible_distribution }} {{ ansible_distribution_version }}
 RAM: {{ ansible_memtotal_mb | int }} MB
 Дата: {{ ansible_date_time.date }} {{ ansible_date_time.time }}
 Поддержка: {{ support_email | default('admin@localhost', true) }}
****************************************************************
{% if ansible_memtotal_mb | int < min_ram_mb | default(2048) %}
⚠️ Warning: Low memory ({{ ansible_memtotal_mb }} MB < {{ min_ram_mb }} MB)
{% endif %}
```

#### Шаг 3: Создать playbook
```yaml
# ~/lab6/playbooks/deploy_motd.yml
---
- name: Dynamic MOTD deployment
  hosts: all
  become: true
  gather_facts: true
  tasks:
    - name: Deploy MOTD from Jinja2 template
      ansible.builtin.template:
        src: ../templates/motd.j2
        dest: /etc/motd
        owner: root
        group: root
        mode: '0644'
        backup: yes
        trim_blocks: yes
        lstrip_blocks: yes

    - name: Verify MOTD content (idempotent read)
      ansible.builtin.command: cat /etc/motd
      register: motd_content
      changed_when: false

    - name: Show generated MOTD
      ansible.builtin.debug:
        var: motd_content.stdout_lines
      when: motd_content.stdout_lines is defined
```
