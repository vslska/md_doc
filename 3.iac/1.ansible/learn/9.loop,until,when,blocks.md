# Модуль 9: Циклы, блоки и условные конструкции

> [!SUMMARY] Цель  
> Освоить механизмы управления потоком выполнения задач в Ansible: итеративную обработку (`loop`), повторные попытки (`until`), условное выполнение (`when`) и группировку с обработкой ошибок (`block`/`rescue`/`always`).

---

## Циклы `loop`

### Механика работы

- `loop` последовательно выполняет задачу для каждого элемента переданной последовательности.
- На каждой итерации значение элемента доступно в переменной `item`.
- По умолчанию циклы выполняются **синхронно и последовательно**. Параллельная итерация внутри одной задачи не поддерживается.
- Каждая итерация логируется отдельно. Статус задачи (`ok`/`changed`/`failed`) вычисляется для каждого элемента независимо.

### Базовый синтаксис
```yaml
- name: Install multiple packages
  ansible.builtin.dnf:
    name: "{{ item }}"
    state: present
  loop:
    - nginx
    - firewalld
    - htop
```
### Управление выводом и поведением: `loop_control`
|Параметр|Назначение|Пример|
|---|---|---|
|`label`|Переопределяет отображаемое имя итерации в логе|`label: "{{ item.name }}"`|
|`pause`|Задержка (в секундах) между итерациями|`pause: 2`|
|`index_var`|Имя переменной, содержащей текущий индекс (начиная с 0)|`index_var: idx`|
|`loop_var`|Заменяет имя переменной `item` (полезно при вложенных циклах)|`loop_var: pkg`|
```yaml
- name: Create users with explicit loop control
  ansible.builtin.user:
    name: "{{ user.name }}"
    shell: "{{ user.shell }}"
  loop:
    - { name: alice, shell: /bin/bash }
    - { name: bob, shell: /bin/zsh }
  loop_control:
    label: "{{ user.name }}"
    loop_var: user
```
### Вложенные циклы

Для корректной обработки вложенных структур используется модуль `ansible.builtin.subelements` или конструкция `loop: "{{ dict_list | subelements('key') }}"`. В большинстве сценариев предпочтительно предварительное преобразование данных через фильтры `dict2items` / `items2dict`.

### Идемпотентность

`loop` сам по себе не обеспечивает идемпотентность. Идемпотентность определяется **модулем**, вызываемым внутри цикла. Если модуль проверяет состояние перед изменением, цикл будет идемпотентным.

## Повторные попытки: `until`

### Механика работы

- Задача выполняется повторно, пока выражение `until` не вернёт `true`.
- Результат последней попытки сохраняется в переменной, указанной в `register`.
- По умолчанию: `retries: 3`, `delay: 5`.
- Условия `changed_when` и `failed_when` применяются **только к последней попытке**. Промежуточные ошибки не останавливают playbook, если не задано `ignore_errors: yes` или `failed_when`.

### Синтаксис и пример
```yaml
- name: Wait for service to accept connections
  ansible.builtin.command: systemctl is-active --quiet nginx
  register: svc_result
  until: svc_result.rc == 0
  retries: 10
  delay: 3
  ignore_errors: yes
```

### Ограничения

- `until` работает только с задачами, которые возвращают статус завершения. Асинхронные задачи (`async`/`poll`) обрабатываются отдельно.
- Переменная `register` перезаписывается на каждой итерации. Доступны только результаты последней попытки.
- Не рекомендуется использовать `until` для ожидания длительных процессов (>5 мин). В таких случаях предпочтительны `async`/`poll` или внешние оркестраторы.

## Условное выполнение: `when`

### Контекст выполнения

- Выражение `when` вычисляется движком Jinja2 **перед запуском задачи**.
- Имеет доступ ко всем фактам, магическим переменным, переменным инвентаря и зарегистрированным результатам.
- Если выражение возвращает `false` или не может быть вычислено, задача получает статус `skipped`.

### Операторы и функции Jinja2
|Оператор/Функция|Описание|Пример|
|---|---|---|
|`==`, `!=`|Строгое сравнение|`ansible_distribution == "Ubuntu"`|
|`>`, `>=`, `<`, `<=`|Числовое сравнение|`ansible_memtotal_mb >= 4096`|
|`in`, `not in`|Проверка вхождения|`'web' in group_names`|
|`is defined`, `is not defined`|Проверка существования переменной|`custom_var is defined`|
|`match()`, `search()`|RegEx-тесты (только полные/частичные совпадения)|`ansible_hostname is match('^web\d+$')`|
|`version()`, `version_compare()`|Сравнение версий|`ansible_distribution_version is version('20.04', '>=')`|
### Группировка условий
```yaml
# Неявный AND (список)
when:
  - ansible_os_family == "RedHat"
  - ansible_distribution_major_version | int >= 9

# Явная логика
when: (ansible_distribution == "Ubuntu") or (ansible_distribution == "Debian")
```

### Приоритет и вычисление

- `when` вычисляется в контексте выполнения задачи. Переменные, созданные `set_fact` в предыдущих задачах, доступны.
- Переменные из `loop` доступны только внутри итерации. Использование `item` вне цикла вызовет `undefined variable`.
- Для отладки: `ansible-playbook playbook.yml -vvv` покажет вычисленное значение условия в строке `skipping: [host] (condition: ...)`.

## Блоки `block` / `rescue` / `always`

### Поток выполнения

1. `block` → выполняется последовательно. При первом `failed` (без `ignore_errors`) управление передаётся в `rescue`.
2. `rescue` → выполняется только при ошибке в `block`. Если `rescue` также завершается с ошибкой, playbook завершается с `failed`.
3. `always` → выполняется в любом случае: после успешного `block`, после `rescue`, или при прерывании playbook (Ctrl+C, сигнал SIGTERM).


### Переменные контекста ошибки (доступны только в `rescue`)

| Переменная              | Тип  | Содержимое                                           |
| ----------------------- | ---- | ---------------------------------------------------- |
| `ansible_failed_task`   | dict | `name`, `action`, `args`, `path`, `when`             |
| `ansible_failed_result` | dict | `rc`, `stdout`, `stderr`, `msg`, `changed`, `failed` |

```yaml
rescue:
  - name: Log failure details
    ansible.builtin.debug:
      msg: |
        Task: {{ ansible_failed_task.name }}
        Module: {{ ansible_failed_task.action }}
        RC: {{ ansible_failed_result.rc | default('N/A') }}
        Stderr: {{ ansible_failed_result.stderr | default('N/A') | trim }}
```

### Область видимости параметров

Параметры, указанные на уровне `block` (`become`, `vars`, `when`, `environment`), применяются ко всем задачам внутри `block`. Это позволяет избежать дублирования:
```yaml
- name: Configure application
  become: yes
  environment:
    APP_ENV: production
  block:
    - name: Deploy config
      template: ...
    - name: Restart service
      systemd: ...
```
### Важные особенности

- `rescue` не отменяет изменения, внесённые задачами `block` до момента ошибки. Для отката требуется явная логика (удаление файлов, возврат конфигов, вызов rollback-плейбука).
- `always` не перехватывает исключения Python или критические ошибки движка Ansible (например, синтаксический YAML-ошибка, недоступность инвентаря).
- Вложенные блоки поддерживаются. Ошибка во внутреннем `block` передаётся в ближайший внешний `rescue`.

## Выводы
 **`loop`** — стандартный механизм итерации с Ansible 2.5+. Заменяет `with_items`/`with_dict`. Поддерживает `loop_control` для управления выводом, задержками и именами переменных. Идемпотентность зависит от вызываемого модуля.

 **`until`** — механизм повторного выполнения задачи до достижения условия. Требует `register` для проверки результата. Параметры `retries`/`delay` задают лимиты. Промежуточные ошибки не прерывают playbook по умолчанию.

 **`when`** — условный фильтр на уровне задачи. Вычисляется движком Jinja2 перед выполнением. Поддерживает логические операторы, тесты типов/версий, RegEx-проверки. Список условий работает как неявный `AND`.

 **`block`/`rescue`/`always`** — структура управления потоком выполнения. `block` группирует задачи и наследует параметры (`become`, `vars`, `when`). `rescue` активируется при ошибке и предоставляет контекст через `ansible_failed_*`. `always` выполняется всегда, включая прерывания. Не выполняет автоматический откат изменений.

## Лабораторная работа 9

### Задания и реализация
```yaml
# ~/lab9/playbooks/logic_ops.yml
---
- name: Lab9: Loops, Conditions & Error Handling
  hosts: all
  become: yes
  gather_facts: yes
  tasks:
    # 1. Цикл с условием фильтрации
    - name: Create non-admin users
      ansible.builtin.user:
        name: "{{ item }}"
        state: present
        shell: /bin/bash
      loop: [alice, bob, carol, admin_user, devops_admin]
      when: "'admin' not in item"

    # 2. Блок установки с обработкой ошибок
    - name: Nginx deployment block
      block:
        - name: Install nginx
          ansible.builtin.package:
            name: nginx
            state: present

        - name: Enable and start service
          ansible.builtin.systemd:
            name: nginx
            state: started
            enabled: yes

        # Раскомментируйте для тестирования rescue:
        # - name: Trigger failure
        #   ansible.builtin.command: /bin/false

      rescue:
        - name: Handle deployment failure
          ansible.builtin.debug:
            msg: "Failed: {{ ansible_failed_task.name }} | RC: {{ ansible_failed_result.rc | default('N/A') }}"

        - name: Clean broken state
          ansible.builtin.package:
            name: nginx
            state: absent

      always:
        - name: Remove temp marker
          ansible.builtin.file:
            path: "/tmp/.nginx_lab9_{{ inventory_hostname }}"
            state: absent
```

### Верификация
```bash
# Проверка созданных пользователей (admin_user должен отсутствовать)
ansible all -a "getent passwd | grep -E 'alice|bob|carol|admin'"

# Проверка статуса nginx
ansible all -a "systemctl is-active nginx"

# Проверка выполнения always (файл должен отсутствовать)
ansible all -a "test -f /tmp/.nginx_lab9_$(hostname) && echo 'EXISTS' || echo 'REMOVED'"
```

>[!CHECK] Критерии выполнения
>
>- `loop` обрабатывает 5 элементов, 2 пропускаются по `when`
>- В логе видны статусы `ok`/`skipping` для каждой итерации
>- `block` выполняется полностью или передаёт управление в `rescue` при ошибке
>- `rescue` имеет доступ к `ansible_failed_task` и `ansible_failed_result`
>- `always` выполняется независимо от исхода `block`/`rescue`
