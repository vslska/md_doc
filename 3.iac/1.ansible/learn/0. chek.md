## Типичные пробелы и как их закрыть

###  Пробел 1: Отладка и логирование

**Что часто упускают**: как читать вывод Ansible, использовать `-v`/`-vvv`, фильтровать логи.

**Как закрыть**:
```bash
# Уровни детализации
ansible-playbook playbook.yml -v      # Кратко
ansible-playbook playbook.yml -vv     # Детали задач
ansible-playbook playbook.yml -vvv    # Внутренности модулей, переменные

# Фильтрация вывода
ansible-playbook playbook.yml -vvv 2>&1 | grep -E "changed|failed|ok"

# Логирование в файл
# ansible.cfg:
[defaults]
log_path = /var/log/ansible/ansible.log
```

### Пробел 2: Тестирование идемпотентности

**Что часто упускают**: проверка, что повторный запуск не вносит изменений.

**Как закрыть**:
```bash
# Первый запуск: ожидаем changed > 0
ansible-playbook playbook.yml | grep "changed="

# Второй запуск: ожидаем changed=0
ansible-playbook playbook.yml | grep "changed="

# Автоматизация проверки в CI
if ansible-playbook playbook.yml --check | grep -q "changed=0"; then
  echo "✅ Idempotent"
else
  echo "❌ Not idempotent"
  exit 1
fi
```

### Пробел 3: Безопасность секретов

**Что часто упускают**: секреты в логах, переменные окружения, `no_log`.

**Как закрыть**:
```yaml
# Скрыть вывод задачи с секретами
- name: Set database password
  ansible.builtin.command: "set_db_pass {{ db_password }}"
  no_log: true  # Не логировать аргументы и вывод

# Не показывать переменную в отладке
- name: Debug secret
  ansible.builtin.debug:
    var: db_password
  no_log: true

# Использовать ansible-vault для файлов
ansible-vault encrypt vars/secrets.yml
ansible-playbook playbook.yml --ask-vault-pass
```

### Пробел 4: Кроссплатформенность

**Что часто упускают**: playbook работает только на одной ОС.

**Как закрыть**:
```yaml
# Использовать ansible.builtin.package вместо dnf/apt
- name: Install nginx
  ansible.builtin.package:
    name: nginx
    state: present

# Или условная логика
- name: Install on RHEL
  ansible.builtin.dnf:
    name: nginx
    state: present
  when: ansible_os_family == "RedHat"

- name: Install on Debian
  ansible.builtin.apt:
    name: nginx
    state: present
  when: ansible_os_family == "Debian"
```

### Пробел 5: Документирование кода

**Что часто упускают**: playbook'и и роли без описания, сложно поддерживать.

**Как закрыть**:
```yaml
# В задачях: всегда указывать name
- name: Ensure nginx is installed and running
  ansible.builtin.package:
    name: nginx
    state: present
  # Комментарий для сложных шагов
  # Это необходимо, потому что в старых образах пакет называется nginx1.18

# В ролях: заполнить README.md и мета-данные
# roles/myrole/README.md
## My Role
Configures XYZ service with support for A, B, C.

### Variables
| Name | Default | Description |
|------|---------|-------------|
| xyz_port | 8080 | Port to listen on |

# В коллекциях: документация через antsibull
# После добавления DOCUMENTATION в модули:
antsibull-docs collection --use-html-brief mycompany.infra_utils
```

## Рекомендации (2026)

1. **Всегда используйте коллекции** для распространения кода — это стандарт де-факто.
2. **FQCN для всех модулей** — избегайте конфликтов имён и неявных зависимостей.
3. **SemVer для версионирования** — `major.minor.patch` с чёткой политикой изменений.
4. **Тестируйте идемпотентность** в CI — это основа надёжной автоматизации.
5. **Шифруйте секреты** через `ansible-vault` или внешние Secret Manager.
6. **Документируйте переменные** в `defaults/main.yml` и `README.md` ролей.
7. **Используйте `--check --diff`** перед применением изменений в production.
8. **Мониторьте устаревание** — `ansible-playbook --syntax-check` и `ansible-lint` в CI.
