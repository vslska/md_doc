## Архитектура коллекций (Collections)

**Коллекция** — это упакованный набор артефактов Ansible: роли, модули, плагины, документы, тесты.

### Ключевые компоненты
|Компонент|Описание|Пример|
|---|---|---|
|**Namespace**|Логическое пространство имён (организация/пользователь)|`mycompany`, `community`|
|**Collection name**|Имя коллекции внутри namespace|`infra_utils`, `rhel_system_roles`|
|**Version**|Семантическая версия (SemVer)|`1.2.3`, `2.0.0-beta.1`|
|**FQCN**|Fully Qualified Collection Name|`mycompany.infra_utils.server_setup`|

### Формат поставки

- Коллекция распространяется как `.tar.gz`-архив с фиксированной структурой
- Внутри: `galaxy.yml`, `MANIFEST.json`, `FILES.json`, каталоги `plugins/`, `roles/`, `meta/`
- Установка: `ansible-galaxy collection install namespace.collection[:version]`

>[!NOTE] FQCN — обязательный стандарт (2026) 
При использовании модулей/ролей из коллекции **требуется** указывать полное имя:
>```yaml
># ❌ Устарело (работает только для builtin)
>- name: Install package
  package: name=nginx state=present
>
># ✅ Современный стандарт
>- name: Install package
  ansible.builtin.package: name=nginx state=present
>
># ✅ Для сторонних коллекций
>- name: Configure server
>  mycompany.infra_utils.server_setup:
>  packages: [nginx, firewalld]
>```

## Структура коллекции

### Базовая структура каталогов
```
mycollection/
├── galaxy.yml                 # Метаданные коллекции (обязательно)
├── MANIFEST.json              # Генерируется при сборке (не редактировать вручную)
├── FILES.json                 # Список файлов для упаковки (авто)
├── README.md
├── CHANGELOG.rst
├── meta/
│   └── runtime.yml           # Минимальная версия Ansible, зависимости
├── plugins/
│   ├── modules/              # Кастомные модули
│   ├── filter/               # Jinja2-фильтры
│   ├── test/                 # Jinja2-тесты
│   ├── callback/             # Плагины вывода
│   └── ...
├── roles/                    # Роли, встроенные в коллекцию
├── playbooks/                # Примеры playbook'ов
├── docs/                     # Документация (авто-генерация через antsibull)
└── tests/                    # Интеграционные тесты
```

### Файл `galaxy.yml` (обязательные поля)
```yaml
---
namespace: mycompany          # Имя namespace (должно существовать на Galaxy)
name: infra_utils             # Имя коллекции
version: 1.0.0                # SemVer: major.minor.patch
readme: README.md
authors:
  - Your Name <you@example.com>
description: Infrastructure utility modules and roles
license:
  - MIT                       # Список лицензий
# Зависимости от других коллекций
dependencies:
  ansible.utils: ">=2.6.0"
  community.general: ">=7.0.0"
# Теги для поиска на Galaxy
tags: [infrastructure, utils, rhel, networking]
# Репозиторий исходного кода
repository: https://github.com/mycompany/ansible-infra-utils
# Документация
documentation: https://mycompany.github.io/ansible-infra-utils
# Требуемая версия Ansible
requires_ansible: ">=2.14.0"
# Файлы, которые НЕ включать в сборку (игнорируются при ansible-galaxy collection build)
build_ignore:
  - "*.log"
  - ".git/"
  - "tests/integration/targets/*/files/large_test_data"
```
>[!WARNING] Версионирование зависимостей Используйте семантические ограничения:
>
>- `">=1.2.0"` — любая версия новее 1.2.0
>- `"~1.2.0"` — совместимые версии в рамках 1.2.x (не 1.3.0)
>- `"^1.2.0"` — совместимые версии в рамках 1.x (не 2.0.0)
>- `"1.2.3"` — точная версия

## Создание и сборка коллекции

### Шаг 1: Инициализация структуры
```bash
# Создать базовую структуру
ansible-galaxy collection init mycompany.infra_utils

# Результат:
mycompany-infra_utils/
├── galaxy.yml
├── README.md
├── meta/
│   └── runtime.yml
├── plugins/
│   └── modules/
└── roles/
```
### Шаг 2: Добавление модуля
```python
# plugins/modules/server_info.py
#!/usr/bin/python
# -*- coding: utf-8 -*-

from ansible.module_utils.basic import AnsibleModule

DOCUMENTATION = r'''
---
module: server_info
short_description: Gather basic server information
description:
  - Returns hostname, OS, and IP address as facts.
options:
  include_network:
    description: Include detailed network interfaces
    type: bool
    default: false
author:
  - Your Name (@yourhandle)
'''

EXAMPLES = r'''
- name: Get basic server info
  mycompany.infra_utils.server_info:
  register: result

- name: Get with network details
  mycompany.infra_utils.server_info:
    include_network: true
'''

def main():
    module = AnsibleModule(
        argument_spec=dict(
            include_network=dict(type='bool', default=False),
        ),
        supports_check_mode=True,
    )
    
    result = dict(
        changed=False,
        hostname=module.run_command(['hostname'], check_rc=True)[1].strip(),
        os_release=module.run_command(['cat', '/etc/os-release'], check_rc=True)[1].strip(),
    )
    
    if module.params['include_network']:
        # Упрощённый пример: в реальности использовать ansible_facts
        result['interfaces'] = module.run_command(['ip', '-br', 'addr'], check_rc=True)[1].strip()
    
    module.exit_json(**result)

if __name__ == '__main__':
    main()
```

### Шаг 3: Документирование модуля

Добавьте `DOCUMENTATION`, `EXAMPLES`, `RETURN` в модуль — они будут использованы для авто-генерации документации через `antsibull-docs`.

### Шаг 4: Сборка коллекции
```bash
cd mycompany-infra_utils

# Проверка структуры перед сборкой
ansible-galaxy collection build --dry-run

# Сборка .tar.gz-архива
ansible-galaxy collection build

# Результат: mycompany-infra_utils-1.0.0.tar.gz в текущем каталоге
```
### Шаг 5: Публикация (опционально)
```bash
# Публикация на Ansible Galaxy (требуется API-ключ)
ansible-galaxy collection publish mycompany-infra_utils-1.0.0.tar.gz \
  --api-key=your_galaxy_api_key

# Публикация в приватный Automation Hub (RH)
ansible-galaxy collection publish mycompany-infra_utils-1.0.0.tar.gz \
  --server=automation_hub \
  --api-key=your_ah_token
```
>[!NOTE] Тестирование перед публикацией
>```bash
># Установить локально для тестов
>ansible-galaxy collection install ./mycompany-infra_utils-1.0.0.tar.gz
>
># Проверить доступность модуля
>ansible-doc -t module mycompany.infra_utils.server_info
>
># Протестировать в playbook
ansible localhost -m mycompany.infra_utils.server_info -a "include_network=true"
>```

## Управление зависимостями: `requirements.yml`

### Формат файла
```yaml
# requirements.yml
---
collections:
  # Установка с указанием версии
  - name: community.general
    version: ">=7.0.0,<8.0.0"
  
  # Установка конкретной версии
  - name: ansible.posix
    version: "1.5.4"
  
  # Установка из локального файла
  - name: mycompany.infra_utils
    source: ./mycompany-infra_utils-1.0.0.tar.gz
    type: file
  
  # Установка из Git-репозитория
  - name: mycompany.experimental
    source: https://github.com/mycompany/ansible-experimental.git
    type: git
    version: main
  
  # Установка с указанием namespace отдельно
  - name: infra_utils
    namespace: mycompany
    version: "1.0.0"

roles:
  # Роли устанавливаются отдельно
  - name: geerlingguy.nginx
    version: "3.2.1"
```

### Установка зависимостей
```bash
# Установить все зависимости из requirements.yml
ansible-galaxy install -r requirements.yml

# Установить только коллекции
ansible-galaxy collection install -r requirements.yml

# Установить только роли
ansible-galaxy role install -r requirements.yml

# Принудительно переустановить
ansible-galaxy install -r requirements.yml --force
```
### Использование в CI/CD
```yaml
# .github/workflows/ansible-test.yml (пример)
- name: Install collection dependencies
  run: ansible-galaxy collection install -r requirements.yml --requirements-file requirements.yml

- name: Run playbook
  run: ansible-playbook -i inventory/site.yml playbooks/deploy.yml
```
## Использование коллекций в playbook

### Способ 1: Глобальное подключение на уровне play
```yaml
---
- name: Deploy infrastructure
  hosts: all
  collections:
    - mycompany.infra_utils
    - community.general
  tasks:
    # Модули из подключённых коллекций доступны без FQCN
    - name: Get server info
      server_info:  # Авто-резолвится в mycompany.infra_utils.server_info
        include_network: true
```
### Способ 2: Явное указание FQCN (рекомендуется)
```yaml
---
- name: Deploy infrastructure
  hosts: all
  tasks:
    - name: Get server info
      mycompany.infra_utils.server_info:
        include_network: true
      
    - name: Format output
      ansible.builtin.debug:
        msg: "{{ result.hostname }} running {{ result.os_release }}"
```
### Способ 3: Динамическое подключение через `include_role`
```yaml
- name: Apply collection role
  ansible.builtin.include_role:
    name: mycompany.infra_utils.server_setup
    # collections:  # Опционально, если роль использует модули из других коллекций
    #   - community.general
```
>[!WARNING] Конфликты имён модулей Если две коллекции содержат модуль с одинаковым именем (например, `config`), Ansible использует первый найденный в порядке:
>
>1. `collections:` в play
>2. `collections:` в role `meta/runtime.yml`
>3. Глобальные настройки `collections_paths`
>
**Решение**: всегда используйте FQCN для избежания неоднозначностей.

# plugins/modules/disk_report.py
#!/usr/bin/python
# -*- coding: utf-8 -*-

from ansible.module_utils.basic import AnsibleModule

DOCUMENTATION = r'''
---
module: disk_report
short_description: Report disk usage for a path
description:
  - Returns disk usage statistics for the specified path.
options:
  path:
    description: Path to check
    type: str
    required: true
author:
  - Your Name
'''

EXAMPLES = r'''
- name: Check root partition
  mycompany.lab_utils.disk_report:
    path: /
  register: result

- name: Show usage
  debug:
    msg: "Used: {{ result.used_percent }}%"
'''

RETURN = r'''
used_percent:
  description: Percentage of disk used
  type: float
  returned: always
  sample: 45.2
available_gb:
  description: Available space in GB
  type: float
  returned: always
  sample: 120.5
'''

def main():
    module = AnsibleModule(
        argument_spec=dict(
            path=dict(type='str', required=True),
        ),
        supports_check_mode=True,
    )
    
    path = module.params['path']
    
    # Получение статистики через statvfs (кроссплатформенно)
    stat = os.statvfs(path)
    total = stat.f_blocks * stat.f_frsize
    free = stat.f_bavail * stat.f_frsize
    used = total - free
    used_percent = round((used / total) * 100, 2) if total > 0 else 0.0
    available_gb = round(free / (1024**3), 2)
    
    module.exit_json(
        changed=False,
        used_percent=used_percent,
        available_gb=available_gb,
    )

if __name__ == '__main__':
    import os
    main()
```
