# Модуль 10: Автоматизация управления хранилищем

> [!SUMMARY] Цель 
> Научиться декларативно управлять блочными устройствами, разделами, LVM, файловыми системами и точками монтирования с использованием специализированных модулей Ansible.

> [!OBJECTIVES] Задачи
> 
> -  Освоить `community.general.parted` для управления таблицами разделов
> -  Научиться создавать ФС через `community.general.filesystem`
> -  Управлять LVM-структурой: `lvg` (группы томов) и `lvol` (логические тома)
> -  Декларативно монтировать устройства через `ansible.posix.mount` / `ansible.builtin.mount`

## Подготовка: коллекции модулей

Модули `parted`, `filesystem`, `lvg`, `lvol` входят в коллекцию `community.general`. `mount` historically в `ansible.posix`, в современных версиях перенесён в `ansible.builtin`.
```bash
# Установка зависимостей перед выполнением playbook
ansible-galaxy collection install community.general ansible.posix
```
>[!NOTE] Пространства имён 
В playbook необходимо указывать полное имя модуля: `community.general.parted`, `community.general.lvg` и т.д. Сокращённые имена (`parted`, `lvg`) работают только если коллекция явно указана в `collections:` блоке play.

## Управление разделами: `community.general.parted`

### Механика работы

- Вызывает утилиту `parted` на целевом хосте.
- Проверяет текущую таблицу разделов перед внесением изменений.
- **Идемпотентность**: гарантирует создание раздела с заданными границами. Изменение существующего раздела (ресайз, сдвиг) **не поддерживается** из-за риска потери данных. Для таких операций используется `state: absent` + повторное создание или ручное вмешательство.

### Ключевые параметры

|Параметр|Тип|Описание|
|---|---|---|
|`device`|str|Блочное устройство (`/dev/sdb`)|
|`number`|int|Номер раздела (1, 2, ...)|
|`state`|str|`present` / `absent`|
|`part_start`|str|Начальная граница. Поддерживает `%`, `B`, `KiB`, `MiB`, `GiB`, `TiB`. Для оптимального выравнивания рекомендуется `1MiB`|
|`part_end`|str|Конечная граница. `-1` означает «до конца диска»|
|`fs_type`|str|Тип файловой системы (метка, не форматирует!)|
|`label`|str|Тип таблицы разделов (`msdos` / `gpt`)|

```yaml
- name: Create primary partition on /dev/sdb
  community.general.parted:
    device: /dev/sdb
    number: 1
    state: present
    part_start: 1MiB
    part_end: 100%
    fs_type: ext4
```

## Создание файловых систем: `community.general.filesystem`

### Механика работы

- Выполняет `mkfs.<fstype>` на целевом устройстве.
- Проверяет наличие суперблока ФС перед запуском. Если ФС уже существует → `ok`.
- **Идемпотентность**: не перезаписывает существующую ФС, если не задан `force: yes`.
### Ключевые параметры

| Параметр | Тип  | Описание                                                     |
| -------- | ---- | ------------------------------------------------------------ |
| `fstype` | str  | Тип ФС: `ext2`, `ext3`, `ext4`, `xfs`, `btrfs`, `vfat` и др. |
| `dev`    | str  | Целевое устройство или логический том                        |
| `opts`   | str  | Дополнительные флаги для `mkfs`                              |
| `force`  | bool | Принудительное создание поверх существующей ФС (`yes`)       |
```yaml
# Стандартное создание ext4
- name: Create ext4 filesystem
  community.general.filesystem:
    fstype: ext4
    dev: /dev/sdb1

# Создание с проверкой блоков (опционально, значительно увеличивает время)
- name: Create ext4 with bad blocks check
  community.general.filesystem:
    fstype: ext4
    dev: /dev/sdb1
    opts: -cc
```

>[!WARNING] `opts: -cc` 
Флаг `-cc` запускает чтение-запись по всему диску для поиска bad-блоков. На современных NVMe/SSD и дисках >1ТБ выполнение может занять часы. В production обычно достаточно `-c` (только чтение) или пропуска флага.

## Управление LVM: `lvg` и `lvol`

### `community.general.lvg` (Volume Groups)

Аналог `vgcreate` / `vgextend` / `vgreduce`. Автоматически выполняет `pvcreate`, если переданные устройства не являются физическими томами.
```yaml
- name: Create/extend volume group
  community.general.lvg:
    vg: vgdata
    pvs: /dev/sdb1
    # pesize: 4M  # Опционально: размер физического экстента
```

### `community.general.lvol` (Logical Volumes)

Аналог `lvcreate` / `lvextend` / `lvreduce`. Поддерживает динамическое изменение размера.
```yaml
- name: Create logical volume
  community.general.lvol:
    vg: vgdata
    lv: lvdata
    size: 10g
    # size: "50%VG"  # Альтернативный синтаксис
    # resizefs: yes  # Работает ТОЛЬКО если ФС уже существует на LV
```

### Техническая особенность `resizefs`

- `resizefs: yes` вызывает `resize2fs` (ext*) или `xfs_growfs` (XFS) **после** изменения размера LV.
- При создании нового LV (`state: present`) ФС ещё не существует → `resizefs` игнорируется.
- При увеличении существующего LV (`size` больше текущего) → автоматически расширяет ФС без downtime.
- Уменьшение (`size` меньше текущего) требует `shrink: yes` и предварительно запущенной проверки ФС (`e2fsck`). Ansible **не выполняет** `e2fsck` автоматически.

## Монтирование: `ansible.builtin.mount`
### Поведение параметра `state`

| Значение `state` | Действие с ФС                   | Запись в `/etc/fstab`     |
| ---------------- | ------------------------------- | ------------------------- |
| `mounted`        | Монтирует немедленно            | Да                        |
| `unmounted`      | Отмонтирует немедленно          | Сохраняет запись          |
| `present`        | Не монтирует/не отмонтирует     | Только обновляет fstab    |
| `absent`         | Отмонтирует (если смонтирована) | ❌ Удаляет запись из fstab |
```yaml
- name: Mount logical volume
  ansible.builtin.mount:
    path: /mnt/data
    src: /dev/mapper/vgdata-lvdata
    fstype: ext4
    opts: defaults,noatime
    state: mounted
    dump: 0
    passno: 2
```

>[!NOTE] Пути в `src` 
Для LVM рекомендуется использовать `/dev/mapper/<vg>-<lv>` или UUID. Прямые пути `/dev/sdX` нестабильны при изменении порядка обнаружения дисков.

## Выводы по модулю

 **Управление дисками**: `community.general.parted` обеспечивает декларативное создание разделов. Изменение существующих разделов не поддерживается из-за рисков целостности данных.

 **Файловые системы**: `community.general.filesystem` проверяет наличие суперблока перед вызовом `mkfs`. Идемпотентно, не разрушает данные без `force: yes`.

 **LVM-структура**: `lvg` и `lvol` автоматизируют `pvcreate`/`vgcreate`/`lvcreate`. `resizefs: yes` расширяет ФС при увеличении LV, но требует предварительно существующей ФС. Уменьшение (`shrink`) требует ручной проверки `e2fsck`.

 **Монтирование**: `ansible.builtin.mount` декларирует желаемое состояние через `state`. `mounted` = активное монтирование + `fstab`, `present` = только `fstab`. Это критично для идемпотентного управления точками подключения.

 **Автоматизация хранилищ**: Ansible позволяет полностью автоматизировать развёртывание storage-стека от блочного уровня до точек монтирования, сохраняя воспроизводимость и контроль изменений.

## Лаба 10: Автоматизация управления хранилищем

### Последовательный playbook

> P.S. нужно докинуть стоража на вирт, на примере sdb   
```yaml
# ~/lab10/playbooks/storage.yml
---
- name: Storage Provisioning
  hosts: ansible_db
  become: true
  gather_facts: false
  tasks:
    # 1. Раздел
    - name: Create partition on /dev/vdb
      community.general.parted:
        device: /dev/vdb
        number: 1
        state: present
        part_start: 1MiB
        part_end: 100%
        fs_type: xfs

    # 2. LVM: Volume Group
    - name: Create volume group vgdata
      community.general.lvg:
        vg: vgdata
        pvs: /dev/vdb1

    # 3. LVM: Logical Volume
    - name: Create logical volume lvdata
      community.general.lvol:
        vg: vgdata
        lv: lvdata
        size: "100%FREE"

    # 4. Файловая система
    - name: Create ext4 filesystem
      community.general.filesystem:
        fstype: ext4
        dev: /dev/mapper/vgdata-lvdata

    # 5. Монтирование + fstab
    - name: Mount lvdata to /mnt/data
      ansible.builtin.mount:
        path: /mnt/data
        src: /dev/mapper/vgdata-lvdata
        fstype: ext4
        opts: defaults,noatime
        state: mounted
```

### Верификация
```bash
# Проверка раздела
ansible ansible_db -a "parted /dev/vdb print | grep 1"

# Проверка LVM
ansible all -a "vgs vgdata"
ansible all -a "lvs vgdata"

# Проверка ФС
ansible all -a "blkid /dev/mapper/vgdata-lvdata"

# Проверка монтирования и fstab
ansible all -a "df -h /mnt/data"
ansible all -a "grep vgdata-lvdata /etc/fstab"
```

>[!CHECK] Критерии выполнения
>
>- Раздел `/dev/sdb1` создан с выравниванием `1MiB`
>- VG `vgdata` и LV `lvdata` присутствуют (`vgs`/`lvs` возвращают статус)
>- ФС `ext4` отформатирована (уникальный UUID в `blkid`)
>- `/mnt/data` смонтирован, запись в `/etc/fstab` содержит `defaults,noatime`
>- Повторный запуск playbook → все задачи `ok`, без изменений (`changed: 0`)
