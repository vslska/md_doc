> [!SUMMARY] Цель
> Научиться управлять поведением Ansible через конфигурационные файлы и эффективно организовывать список управляемых хостов (инвентаризацию).

## Что такое конфигурация Ansible?

Ansible поддерживает **несколько источников настроек**, которые применяются в определённом порядке приоритета:

| Источник                | Пример                           | Приоритет |
| ----------------------- | -------------------------------- | --------- |
| Переменные окружения    | `ANSIBLE_CONFIG=./prod.cfg`      | Highest   |
| CLI-опции               | `ansible -c local ...`           | ⬇️        |
| Ключевые слова Playbook | `gather_facts: no`               | ⬇️        |
| Файл `ansible.cfg`      | `[defaults] remote_user=ansible` | ⬇️        |
| Значения по умолчанию   | Встроенные в Ansible             |  Lowest   |

> [!TIP] Правило «последний переписывает первый» Если параметр задан в нескольких местах — используется значение с **наивысшим приоритетом**.

## Где Ansible ищет `ansible.cfg`? (приоритет по убыванию)
```bash
️1. $ANSIBLE_CONFIG          # переменная окружения
2.  /ansible.cfg            # текущая директория (проект)
3. ~/.ansible.cfg           # домашняя директория пользователя
4. /etc/ansible/ansible.cfg # глобальная системная настройка
```

>[!CHECK] Как узнать, какой конфиг используется?
>```bash
>ansible --version
># Строка "config file = ..." покажет активный путь
>```
>

## Формат конфигурационного файла

`ansible.cfg` использует **INI-формат**:

```ini
# Это комментарий (решётка)
[defaults]
inventory = ./hosts
remote_user = ansible
host_key_checking = False

; Это тоже комментарий (точка с запятой)
; Но только если комментарий не в середине строки!
log_path = /var/log/ansible.log ; путь к лог-файлу
```

>[!WARNING] Важные нюансы синтаксиса
>
>- Секции указываются в квадратных скобках: `[defaults]`, `[privilege_escalation]`
>- Ключи чувствительны к регистру: `remote_user`, а не `Remote_User`
>- Значения булевых типов: `True/False`, `yes/no`, `1/0`

## Утилита `ansible-config`: справочник по настройкам


## Ключевые параметры в `[defaults]`

```ini
[defaults]
# Путь к инвентаризации (можно несколько через запятую)
inventory = ./inventory.yml,./static_hosts.ini

# Пользователь для подключения по SSH
remote_user = ansible

# Приватный SSH-ключ
private_key_file = ~/.ssh/id_ed25519

# Отключить проверку хост-ключей (только для тестовых сред!)
host_key_checking = False

# Таймаут подключения (секунды)
timeout = 30

# Количество параллельных процессов (форков)
forks = 10

# Отключить создание .retry-файлов
retry_files_enabled = False

# Логирование
log_path = /var/log/ansible/ansible.log

# Интерпретатор Python на удалённых узлах (если не /usr/bin/python)
# interpreter_python = /usr/bin/python3
```

>[!WARNING] Безопасность Никогда не отключайте `host_key_checking = False` в production! Это открывает возможность MITM-атак.

## Инвентаризация (Inventory): основы

**Inventory** — это «карта» управляемых хостов. Ansible поддерживает:

| Тип              | Формат                             | Когда использовать                               |
| ---------------- | ---------------------------------- | ------------------------------------------------ |
| **Статическая**  | INI, YAML, JSON                    | Фиксированный парк серверов, лабораторные стенды |
| **Динамическая** | Скрипты, плагины (AWS, Azure, K8s) | Облачные среды, автоскейлинг, контейнеры         |

---
## Статическая инвентаризация: INI vs YAML

### INI-формат (классический)
```ini
# Простые хосты
[webservers]
web1.example.com
web2.example.com

# Хост с переменными
[db]
db1.example.com ansible_user=postgres ansible_port=2222

# Переменные для всей группы
[webservers:vars]
ansible_user=nginx
http_port=80

# Группа групп (мета-группа)
[prod:children]
webservers
db

# Диапазоны и паттерны
[batch]
node[01:10].example.com      # node01...node10
192.168.1.[100:110]          # 192.168.1.100-110
```

### YAML-формат (рекомендуемый для новых проектов)
```yaml
# inventory.yml
---
all:
  children:
    webservers:
      hosts:
        web1.example.com:
          ansible_host: 10.0.0.10
          ansible_port: 22
        web2.example.com:
          ansible_host: 10.0.0.11
      vars:
        ansible_user: nginx
        http_port: 80

    db:
      hosts:
        db1.example.com:
          ansible_host: 10.0.0.20
          ansible_user: postgres
          ansible_port: 2222

    prod:
      children:
        webservers:
        db
```

>[!TIP] Почему YAML предпочтительнее?
>
>- Лучшая читаемость при вложенных структурах
>- Поддержка комментариев (в INI тоже есть, но в YAML удобнее)
>- Единый формат с Playbook'ами
>- Легче парсится внешними инструментами

## Шаблоны хостов (Host Patterns)

Шаблоны позволяют гибко выбирать целевые хосты в командах и playbook'ах.

### Синтаксис в CLI:
```bash
ansible <ШАБЛОН> -i <inventory> -m <module> -a "<args>"
```
### Синтаксис в Playbook:
```yaml
- name: My playbook
  hosts: <ШАБЛОН>
  tasks: ...
```

### Практические примеры шаблонов:
| Шаблон             | Что выбирает                        |
| ------------------ | ----------------------------------- |
| `all` или `*`      | Все хосты в инвентаре               |
| `webservers`       | Все хосты группы `webservers`       |
| `web1.example.com` | Конкретный хост                     |
| `webservers:db`    | Объединение групп (ИЛИ)             |
| `webservers:&prod` | Пересечение: веб-серверы в prod (И) |
| `webservers:!prod` | Веб-серверы, кроме группы prod (НЕ) |
| `*.example.com`    | Хосты, соответствующие шаблону      |
| `webservers[0]`    | Первый хост в группе (по алфавиту)  |
| `webservers[-1]`   | Последний хост в группе             |
>[!EXAMPLE] Комбинации шаблонов
>``` bash
># Запустить только на веб-серверах из Новосибирска
>ansible 'webservers:&nsk' -m ping
>
># Все хосты, кроме базы данных
>ansible 'all:!db' -m setup
>
># Регулярные выражения (требует кавычек!)
>ansible '~web\d+\.example\.com' -m ping
>```

## Выводы

**Конфигурация**:

- Ansible использует иерархию источников настроек (env → CLI → playbook → cfg → defaults)
- `ansible.cfg` в корне проекта — лучший способ изолировать настройки
- Утилита `ansible-config` помогает отлаживать и документировать конфигурацию

 **Инвентаризация**:

- Inventory — центральный механизм указания хостов и групп
- Поддерживаются INI и YAML (YAML предпочтительнее для новых проектов)
- Группы, подгруппы и шаблоны дают гибкость в выборе целей
- Переменные можно задавать на уровне хоста, группы или группы групп

**Практика**:

- Один проект = один `ansible.cfg` + один `inventory.yml`
- Всегда проверяйте подключение через `ansible ... -m ping` перед запуском плейбуков
- Используйте `--check` и `--diff` для безопасного тестирования изменений

## Лабораторная работа 2: Настройка конфигурации и инвентаризации
### Задания:

#### 1. Подготовка окружения
```bash
# Создать каталог проекта
mkdir -p ~/ll105-lab2
cd ~/ll105-lab2

# Создать базовую структуру
mkdir -p group_vars host_vars playbooks
```

#### 2. Локальный `ansible.cfg`

```ini
# ~/ll105-lab2/ansible.cfg
[defaults]
inventory = ./inventory.yml
remote_user = ansible
host_key_checking = False
retry_files_enabled = False
forks = 5
gathering = smart

[privilege_escalation]
become = True
become_method = sudo
become_user = root
become_ask_pass = False
```

#### 3. Инвентаризация `inventory.yml`
```yaml
# ~/ll105-lab2/inventory.yml
---
all:
  children:
    servers:
      hosts:
        c9-server02:
          ansible_host: 192.168.1.10
          ansible_user: student
    clients:
      hosts:
        client01:
          ansible_host: 192.168.1.20
          ansible_user: student
    lab:
      children:
        servers:
        clients
      vars:
        ansible_port: 22
        ansible_ssh_common_args: '-o StrictHostKeyChecking=no'
```

#### 4. Проверка подключения
```bash
# Проверить синтаксис инвентаря
ansible-inventory --graph

# Протестировать подключение ко всем хостам
ansible all -m ping

# Проверить только группу servers
ansible servers -m ping
```

#### 5. Тестовый Playbook `check.yml`
```bash
# ~/ll105-lab2/playbooks/check.yml
---
- name: Lab2: Check host information
  hosts: all
  gather_facts: yes
  tasks:
    - name: Show hostname and OS
      ansible.builtin.debug:
        msg: "Host: {{ ansible_hostname }}, OS: {{ ansible_distribution }} {{ ansible_distribution_version }}"

    - name: Show IP address
      ansible.builtin.debug:
        msg: "IP: {{ ansible_default_ipv4.address | default('N/A') }}"
```

#### 6. Запуск и проверка
```bash
# Проверить синтаксис плейбука
ansible-playbook --syntax-check playbooks/check.yml

# Запустить в режиме проверки (dry-run)
ansible-playbook -i ./inventory.yml playbooks/check.yml --check

# Запустить полноценно
ansible-playbook -i ./inventory.yml playbooks/check.yml
```

>[!SUCCESS] Критерии выполнения
>
>- `ansible.cfg` находится в корне проекта и переопределяет глобальные настройки
?- `inventory.yml` содержит группы `servers` и `clients` с корректными переменными
>- Команда `ansible all -m ping` возвращает `SUCCESS` для всех хостов
>- `check.yml` выводит имя хоста, ОС и IP-адрес
>- Используется локальный inventory через `-i ./inventory.yml`
