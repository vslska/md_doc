# Модуль 14: Использование системных ролей (RHEL System Roles)

> [!SUMMARY] Цель
> Научиться применять преднастроенные системные роли Ansible для автоматизации административных задач на RHEL-совместимых системах, используя официальные пакеты и коллекции.

> [!OBJECTIVES] Задачи
> 
> -  Понять архитектуру RHEL System Roles и отличие от пользовательских ролей
> -  Освоить установку через `dnf` (пакет) и `ansible-galaxy` (коллекция)
> -  Научиться применять роли `timesync`, `network`, `selinux`, `firewall` с корректными переменными
> -  Использовать `group_vars/` для централизованного управления конфигурацией ролей

## Что такое системные роли (System Roles)

**RHEL System Roles** — это набор преднастроенных ролей Ansible, предоставляемых Red Hat для автоматизации типовых задач администрирования.

### Технические характеристики

|Характеристика|Описание|
|---|---|
|**Источник**|Пакет `rhel-system-roles` (DNF) или коллекция `fedora.linux_system_roles` (Galaxy)|
|**Пространство имён**|`linux-system-roles.*` (legacy) → `fedora.linux_system_roles.*` (2026)|
|**Идемпотентность**|Гарантирована: роли проверяют текущее состояние перед применением изменений|
|**Поддержка бэкендов**|Автоматический выбор: `NetworkManager` vs `network-scripts`, `chrony` vs `ntp`|
|**Тестирование**|Все роли проходят CI/CD в рамках стандартов Red Hat Enterprise Linux|

> [!NOTE] Эволюция доставки (2026)
> 
> - **Раньше**: роли устанавливались через `dnf install rhel-system-roles` → `/usr/share/ansible/roles/`
> - **Сейчас**: предпочтительный способ — коллекции через Galaxy: `ansible-galaxy collection install fedora.linux_system_roles`
> - Пакет `rhel-system-roles` в RHEL 9+ может устанавливать коллекцию как зависимость

## Установка и подключение системных ролей

### Способ 1: Через пакет (традиционный)
```bash
# RHEL/CentOS Stream
sudo dnf install rhel-system-roles

# Роли появятся в системном каталоге
ls /usr/share/ansible/roles/ | grep linux-system-roles
```
### Способ 2: Через коллекцию (рекомендуемый, 2026)
```bash
# Установка коллекции из Ansible Galaxy
ansible-galaxy collection install fedora.linux_system_roles

# Добавление в requirements.yml для проекта
# collections/requirements.yml
---
collections:
  - name: fedora.linux_system_roles
    version: ">=1.0.0"
```

### Подключение в playbook
```yaml
# Вариант А: с указанием коллекции на уровне play
- name: Configure system time
  hosts: all
  collections:
    - fedora.linux_system_roles
  roles:
    - timesync

# Вариант Б: с FQCN (fully qualified collection name)
- name: Configure system time
  hosts: all
  roles:
    - role: fedora.linux_system_roles.timesync
```

>[!WARNING] Совместимость имён 
При использовании коллекции переменные роли **не меняют имена**. Например, `timesync_ntp_servers` остаётся прежним, независимо от способа подключения роли.

## Обзор доступных системных ролей
|Роль (FQCN)|Назначение|Ключевые переменные|
|---|---|---|
|`fedora.linux_system_roles.timesync`|Синхронизация времени (chrony/ntp/ptp)|`timesync_ntp_servers`, `timesync_ntp_provider`|
|`fedora.linux_system_roles.network`|Настройка сетевых интерфейсов|`network_connections`, `network_provider`|
|`fedora.linux_system_roles.firewall`|Управление firewalld|`firewall`, `firewall_purge`|
|`fedora.linux_system_roles.selinux`|Политики SELinux, контексты, булевы|`selinux_state`, `selinux_booleans`, `selinux_fcontexts`|
|`fedora.linux_system_roles.storage`|Управление LVM, разделами, ФС|`storage_pools`, `storage_volumes`|
|`fedora.linux_system_roles.logging`|Настройка rsyslog/journald|`logging_outputs`, `logging_inputs`|
|`fedora.linux_system_roles.kernel_settings`|Параметры ядра через sysctl/tuned|`kernel_settings_sysctl`, `kernel_settings_tuned_profile`|

## Роль `timesync`: синхронизация времени

### Механика работы

- Автоматически выбирает бэкенд: `chrony` (по умолчанию в RHEL 8+), `ntp` или `linuxptp`
- Устанавливает и настраивает выбранный сервис
- Применяет конфигурацию идемпотентно
### Ключевые переменные
```yaml
# Выбор провайдера (опционально)
timesync_ntp_provider: chrony  # chrony, ntp, linuxptp

# Список серверов (обязательно)
timesync_ntp_servers:
  - hostname: pool.ntp.org
    iburst: yes
  - hostname: time.google.com
    iburst: yes
    prefer: yes  # Предпочтительный сервер

# Дополнительные настройки (для chrony)
timesync_chrony_makestep: "1.0 3"  # Шаг синхронизации
timesync_chrony_maxupdateskew: 100.0
```
### Пример playbook
```yaml
---
- name: Configure NTP via system role
  hosts: all
  become: yes
  collections:
    - fedora.linux_system_roles
  roles:
    - role: timesync
      vars:
        timesync_ntp_provider: chrony
        timesync_ntp_servers:
          - hostname: pool.ntp.org
            iburst: yes
```

>[!NOTE] Переменная `iburst` 
Ускоряет начальную синхронизацию, отправляя пакет запросов при старте сервиса. Рекомендуется для всех серверов.

## Роль `network`: настройка сетевых интерфейсов

### Механика работы

- Использует `NetworkManager` (по умолчанию) или `network-scripts` (legacy)
- Конфигурация описывается через список `network_connections`
- Поддерживает: Ethernet, bridge, bond, VLAN, MacVLAN, Wi-Fi, 802.1X

### Ключевые параметры `network_connections`
```yaml
network_connections:
  - name: eth1-profile          # Имя профиля NetworkManager
    interface_name: eth1        # Физический интерфейс
    type: ethernet              # Тип: ethernet, bridge, bond, vlan
    autoconnect: yes            # Автоподключение при загрузке
    ip:
      dhcp4: no                 # Отключить DHCPv4
      auto6: no                 # Отключить SLAAC
      address:                  # Статические адреса
        - 192.0.2.10/24
      gateway4: 192.0.2.1       # Шлюз (устаревший, но работает)
      # Или современный синтаксис:
      # route: [{ destination: "0.0.0.0/0", gateway: "192.0.2.1" }]
      dns:
        - 8.8.8.8
        - 8.8.4.4
      dns_search: example.com
```
### Пример playbook
```yaml
---
- name: Configure static network
  hosts: targets
  become: yes
  collections:
    - fedora.linux_system_roles
  roles:
    - role: network
      vars:
        network_provider: nm  # nm = NetworkManager, initscripts = legacy
        network_connections:
          - name: prod-eth1
            interface_name: eth1
            type: ethernet
            autoconnect: yes
            ip:
              dhcp4: no
              auto6: no
              address:
                - 192.0.2.10/24
              gateway4: 192.0.2.1
              dns:
                - 192.0.2.1
```

>[!WARNING] Применение сетевых изменений 
Роль применяет конфигурацию немедленно. При ошибке в настройках (`gateway`, `address`) возможна потеря связи с хостом. Всегда тестируйте в `--check` режиме и имейте консольный доступ.

## Роль `selinux`: управление политиками

### Механика работы

- Управляет режимом SELinux (`enforcing`, `permissive`, `disabled`)
- Настраивает булевы переменные, контексты файлов, порты, логины
- По умолчанию применяет изменения **инкрементально** (не удаляет существующие настройки)

### Ключевые переменные
```yaml
# Режим и политика
selinux_state: enforcing          # enforcing, permissive, disabled
selinux_policy: targeted          # targeted, minimum, mls

# Булевы переменные (инкрементально)
selinux_booleans:
  - name: samba_enable_home_dirs
    state: on
    persistent: yes               # Сохранить после перезагрузки

# Контексты файлов
selinux_fcontexts:
  - target: "/srv/data(/.*)?"
    setype: httpd_sys_content_t
    ftype: d                      # d=directory, f=file, - = любой
    state: present                # present/absent

# Порты
selinux_ports:
  - ports: 8080
    proto: tcp
    setype: http_port_t
    state: present

# Очистка локальных изменений перед применением
selinux_all_purge: yes            # Удалить все кастомные настройки перед применением новых
# Или точечно:
selinux_booleans_purge: yes
selinux_fcontexts_purge: yes
```

### Пример playbook
```yaml
---
- name: Harden SELinux configuration
  hosts: all
  become: yes
  collections:
    - fedora.linux_system_roles
  roles:
    - role: selinux
      vars:
        selinux_state: enforcing
        selinux_policy: targeted
        selinux_booleans:
          - name: httpd_can_network_connect
            state: on
            persistent: yes
        selinux_fcontexts:
          - target: "/opt/app/logs(/.*)?"
            setype: httpd_log_t
            ftype: d
            state: present
        selinux_restore_dirs:
          - /opt/app/logs  # Запустить restorecon после изменения контекстов
```

>[!NOTE] `selinux_restore_dirs` 
После изменения контекстов через `selinux_fcontexts` необходимо явно указать каталоги для `restorecon`, иначе изменения не применятся к существующим файлам.

## Роль `firewall`: управление firewalld

### Механика работы

- Абстракция над `firewalld`: управление зонами, сервисами, портами, правилами
- Поддерживает `permanent` + `immediate` применение (аналог модуля `firewalld`)
- Идемпотентна: не дублирует правила

### Ключевые переменные
```yaml
# Базовая конфигурация
firewall:
  - name: public              # Зона firewalld
    state: enabled            # enabled/disabled
    service:
      - name: ssh
        state: enabled
      - name: http
        state: enabled
    port:
      - port: 8080/tcp
        state: enabled
    rich_rule:
      - rule: "family ipv4 source address 192.0.2.0/24 service name ssh accept"
        state: enabled

# Очистка правил перед применением
firewall_purge: yes           # Удалить все правила зоны перед применением новых
```

### Пример playbook
```yaml
---
- name: Configure firewall via system role
  hosts: all
  become: yes
  collections:
    - fedora.linux_system_roles
  roles:
    - role: firewall
      vars:
        firewall:
          - name: public
            service:
              - name: ssh
                state: enabled
              - name: http
                state: enabled
            port:
              - port: 8080/tcp
                state: enabled
```

>[!TIP] Проверка правил 
После применения роли проверьте результат:
>```bash
firewall-cmd --zone=public --list-all>
>```

## Выводы по модулю

 **Системные роли — стандартизация автоматизации**:

- Преднастроенные, протестированные роли от Red Hat снижают риск ошибок в критичных подсистемах (сеть, SELinux, время).
- Идемпотентность и проверка состояния встроены на уровне роли, а не playbook.

 **Два способа доставки**:

- Пакет `rhel-system-roles` (DNF) → системные роли в `/usr/share/ansible/roles/`
- Коллекция `fedora.linux_system_roles` (Galaxy) → современный, версионируемый способ
- Переменные ролей **не зависят** от способа установки: `timesync_ntp_servers` работает одинаково.

 **Ключевые роли и их особенности**:

- `timesync`: автоматический выбор `chrony`/`ntp`, переменная `iburst` для быстрой синхронизации.
- `network`: описание через `network_connections`, поддержка `NetworkManager`, риск потери связи при ошибке.
- `selinux`: инкрементальное применение, `selinux_restore_dirs` для применения контекстов, `*_purge` для сброса.
- `firewall`: абстракция над `firewalld`, поддержка `rich_rule`, идемпотентное управление правилами.

 **Организация конфигурации**:

- Переменные ролей удобно размещать в `group_vars/` или `host_vars/` для централизованного управления.
- Использование `collections:` на уровне play или FQCN (`fedora.linux_system_roles.timesync`) обеспечивает явность зависимостей.

 **Безопасность и тестирование**:

- Всегда используйте `--check` перед применением сетевых или SELinux-изменений.
- Имейте консольный доступ к хостам на случай потери сети.
- Повторный запуск должен подтверждать идемпотентность (`changed=0`).

## Лабораторная работа 14: Автоматизация административных задач

###  Чёткая последовательность выполнения

#### 0: Подготовка окружения
```bash
mkdir -p ~/ll105-lab14/{playbooks,group_vars,inventory}
cd ~/ll105-lab14
cp ../ll105-lab13/inventory.yml ./inventory/
cp ../ll105-lab13/ansible.cfg ./

# Установка коллекции (если не установлена)
ansible-galaxy collection install fedora.linux_system_roles
```
#### 1: Настройка переменных в `group_vars/`
```yaml
# group_vars/all.yml
---
# Timesync
timesync_ntp_provider: chrony
timesync_ntp_servers:
  - hostname: pool.ntp.org
    iburst: yes

# Firewall
firewall:
  - name: public
    service:
      - name: ssh
        state: enabled
      - name: http
        state: enabled
    port:
      - port: 8080/tcp
        state: enabled
```
#### 2: Создать playbook `system_roles.yml`
```yaml
# playbooks/system_roles.yml
---
- name: Lab14: Apply RHEL System Roles
  hosts: all
  become: yes
  gather_facts: yes
  collections:
    - fedora.linux_system_roles
  roles:
    - role: timesync
    - role: firewall
```
#### 3: Проверка и запуск
```bash
# Проверка синтаксиса
ansible-playbook --syntax-check playbooks/system_roles.yml

# Dry-run (проверка без применения)
ansible-playbook -i inventory/inventory.yml playbooks/system_roles.yml --check -v

# Запуск
ansible-playbook -i inventory/inventory.yml playbooks/system_roles.yml -v
```
#### 4: Верификация
```bash
# Проверка синхронизации времени
ansible all -a "chronyc sources -v"  # или "ntpq -p" для ntp

# Проверка firewalld
ansible all -a "firewall-cmd --zone=public --list-services"
ansible all -a "firewall-cmd --zone=public --list-ports"

# Проверка, что роль идемпотентна
ansible-playbook -i inventory/inventory.yml playbooks/system_roles.yml | grep "changed="
# Ожидается: changed=0 при повторном запуске
```

#### 5: Дополнительное задание (SELinux)
```yaml
# group_vars/all.yml (добавить)
selinux_state: enforcing
selinux_booleans:
  - name: httpd_can_network_connect
    state: on
    persistent: yes

# playbooks/system_roles.yml (добавить роль)
  roles:
    - role: timesync
    - role: firewall
    - role: selinux  # Требуется перезагрузка при смене режима с disabled
```

>[!CHECK] Критерии выполнения
>
>- Коллекция `fedora.linux_system_roles` установлена и доступна
>- `group_vars/all.yml` содержит переменные для `timesync` и `firewall`
>- Playbook применяет обе роли без ошибок
>- `chronyc sources` показывает активные серверы синхронизации
>- `firewall-cmd --list-all` отображает сервисы `ssh`, `http` и порт `8080/tcp`
>- Повторный запуск → `changed=0` (идемпотентность)
>- (Бонус) `getenforce` возвращает `Enforcing`, `semanage boolean -l | grep httpd_can_network_connect` → `on`
