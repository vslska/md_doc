# Модуль 12: Комплексные проекты автоматизации

> [!SUMMARY] Цель
> Освоить механизмы организации масштабируемых проектов в Ansible: обработчики (`handlers`), модульное включение задач (`import_*/include_*`) и шифрование конфиденциальных данных (`ansible-vault`).

> [!OBJECTIVES] Задачи
> 
> -  Настроить механизм уведомлений и обработчиков с управлением выполнением (`flush_handlers`)
> -  Различать статический (`import_*`) и динамический (`include_*`) импорт задач и плейбуков
> -  Освоить шифрование переменных и файлов через `ansible-vault` и интеграцию с CI/CD
> -  Структурировать проект с выносом переменных, задач и обработчиков в отдельные артефакты

## Обработчики (Handlers) и уведомления

### Механика выполнения

- **Обработчик** — это задача, которая выполняется **только** при получении уведомления (`notify`) от другой задачи, завершившейся со статусом `changed`.
- Обработчики размещаются в секции `handlers:` на уровне play.
- **Порядок выполнения**: все обработчики, получившие уведомление в ходе выполнения play, запускаются **единым блоком после завершения всех задач** (но до завершения play).
- **Идемпотентность**: если задача, отправляющая `notify`, не внесла изменений (`ok`), уведомление не отправляется, и обработчик не выполняется.

### Базовый синтаксис
```yaml
- name: Deploy web server
  hosts: webservers
  become: yes
  tasks:
    - name: Install nginx package
      ansible.builtin.apt:
        name: nginx
        state: latest
      notify: Restart nginx service  # Уведомление отправляется только при changed

    - name: Deploy config
      ansible.builtin.template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: Reload nginx config

  handlers:
    - name: Restart nginx service
      ansible.builtin.systemd:
        name: nginx
        state: restarted

    - name: Reload nginx config
      ansible.builtin.systemd:
        name: nginx
        state: reloaded
```

### Управление выполнением: `meta: flush_handlers`

По умолчанию обработчики выполняются в конце play. Для принудительного запуска в нужной точке используется `meta: flush_handlers`.
```yaml
- name: Install and configure app
  tasks:
    - name: Update packages
      ansible.builtin.apt:
        name: '*'
        state: latest
      notify: Reboot if kernel updated

    - name: Flush handlers now
      ansible.builtin.meta: flush_handlers  # Обработчики выполняются здесь

    - name: Post-reboot validation
      ansible.builtin.command: uname -r
      register: kernel_version
```

### Группировка через `listen`

Директива `listen` позволяет нескольким обработчикам реагировать на одно имя уведомления или одному обработчику — на несколько событий.
```yaml
handlers:
  - name: Restart web services
    ansible.builtin.systemd:
      name: "{{ item }}"
      state: restarted
    loop: [nginx, php-fpm]
    listen: "web stack restart"  # Алиас для уведомления

# В задаче:
- name: Update PHP config
  ansible.builtin.template:
    src: php.ini.j2
    dest: /etc/php/8.2/fpm/php.ini
  notify: "web stack restart"  # Запустит оба сервиса из loop выше
```

>[!WARNING] Ограничения имен 
Имя обработчика (или значение `listen`) должно быть **уникальным в рамках play**. Переменные в имени обработчика (`name: "Restart {{ service }}"`) **не поддерживаются** — имя вычисляется на этапе парсинга, до подстановки переменных.

## Модульность: Include vs Import

Ansible предоставляет два механизма повторного использования кода: **статический** (`import_*`) и **динамический** (`include_*`).

### Сравнительная таблица (актуально для Ansible 2.15+)

|Характеристика|`import_*` (статический)|`include_*` (динамический)|
|---|---|---|
|**Время обработки**|Парсинг (до выполнения playbook)|Runtime (во время выполнения)|
|**Видимость переменных**|Видны на этапе парсинга (можно использовать в путях)|Видны только в момент выполнения|
|**Условия (`when`)**|Применяются ко всему импортируемому блоку|Проверяются для каждого включения|
|**Циклы (`loop`)**|❌ Не поддерживаются|✅ Поддерживаются|
|**Изменение пути файла**|Невозможно (путь фиксирован)|Возможен (путь может быть переменной)|
|**Отладка**|Сложнее (ошибки синтаксиса видны до запуска)|Проще (ошибки видны при выполнении)|
|**Производительность**|Быстрее (препроцессинг)|Медленнее (проверка на лету)|

### Доступные модули (2026)

|Модуль|Тип|Назначение|
|---|---|---|
|`ansible.builtin.import_tasks`|Статический|Включение файла задач в текущий play|
|`ansible.builtin.include_tasks`|Динамический|Динамическое включение файла задач|
|`ansible.builtin.import_playbook`|Статический|Включение другого playbook (только статически)|
|`ansible.builtin.import_role`|Статический|Включение роли на этапе парсинга|
|`ansible.builtin.include_role`|Динамический|Динамическое включение роли|
|`ansible.builtin.include_vars`|Динамический|Загрузка переменных из файла во время выполнения|

> [!NOTE] Устаревшие конструкции 
> Модуль `ansible.builtin.include` (без суффикса) и ключевое слово `roles:` на уровне play **помечены как deprecated**. Используйте `import_role` / `include_role` и явные `import_tasks` / `include_tasks`.

### Пример: статический импорт задач
```yaml
# main.yml
- name: Configure server
  hosts: all
  tasks:
    - ansible.builtin.import_tasks: tasks/prep.yml
    - ansible.builtin.import_tasks: tasks/install.yml
    - ansible.builtin.import_tasks: tasks/config.yml
```
### Пример: динамическое включение с условием
```yaml
- name: Install OS-specific packages
  ansible.builtin.include_tasks: "tasks/{{ ansible_os_family }}.yml"
  when: ansible_os_family in ['RedHat', 'Debian']
```

## Ansible Vault: шифрование конфиденциальных данных

### Механика работы

- `ansible-vault` шифрует файлы/переменные с использованием AES-256.
- Зашифрованный контент помечается тегом `$ANSIBLE_VAULT;1.1;AES256` в начале файла.
- При выполнении playbook расшифровка происходит «на лету» при наличии пароля.

### Управление зашифрованным контентом

|Команда|Назначение|Пример|
|---|---|---|
|`ansible-vault create <file>`|Создать новый зашифрованный файл|`ansible-vault create secrets.yml`|
|`ansible-vault encrypt <file>`|Зашифровать существующий файл|`ansible-vault encrypt vars/prod.yml`|
|`ansible-vault view <file>`|Просмотреть содержимое (без расшифровки на диск)|`ansible-vault view secrets.yml`|
|`ansible-vault edit <file>`|Редактировать в зашифрованном виде|`ansible-vault edit secrets.yml`|
|`ansible-vault rekey <file>`|Сменить пароль шифрования|`ansible-vault rekey secrets.yml`|
|`ansible-vault decrypt <file>`|Полностью расшифровать файл (снять защиту)|`ansible-vault decrypt secrets.yml`|
|`ansible-vault encrypt_string`|Зашифровать строку для вставки в YAML|См. ниже|

### Шифрование строки для вставки в playbook
```bash
ansible-vault encrypt_string 'MySecretPassword' --name 'db_password'
```

**Вывод:**
```yaml
db_password: !vault |
          $ANSIBLE_VAULT;1.1;AES256
          33363235313136373130393... (зашифрованный блок)
```
Этот блок можно напрямую вставить в playbook или vars-файл.

### Использование паролей при запуске

|Способ|Синтаксис|Применение|
|---|---|---|
|Интерактивный запрос|`ansible-playbook site.yml --ask-vault-pass`|Ручной запуск, отладка|
|Файл с паролем|`ansible-playbook site.yml --vault-password-file ~/.vault_pass`|CI/CD, автоматизация|
|Скрипт для пароля|`ansible-playbook site.yml --vault-password-file ~/.vault_script.sh`|Интеграция с Secret Manager|
|Несколько паролей|`--vault-id dev@~/.dev_pass --vault-id prod@~/.prod_pass`|Мульти-окружения|

>[!WARNING] Безопасность файла с паролем 
Файл, указанный в `--vault-password-file`, должен иметь права `0600`. Скрипт (`--vault-password-file script.sh`) должен быть исполняемым и возвращать пароль в stdout без лишних символов.

## Выводы

 **Обработчики и уведомления**:

- Механизм `notify` → `handler` обеспечивает реакцию на изменения (`changed`) без избыточных действий.
- Обработчики выполняются единым блоком после задач play; `meta: flush_handlers` позволяет управлять точкой выполнения.
- `listen` упрощает группировку уведомлений и снижает дублирование имен.

 **Модульность (`import_*` / `include_*`)**:

- `import_*` (статический) — для предсказуемой структуры, когда пути и переменные известны на этапе парсинга.
- `include_*` (динамический) — для гибких сценариев с условиями, циклами и переменными путями.
- `import_playbook` — единственный способ включения плейбуков (всегда статический).

 **Ansible Vault**:

- Шифрование на уровне файлов (`encrypt`) или строк (`encrypt_string`) защищает секреты в репозитории.
- Интеграция с `--vault-password-file` или скриптами позволяет использовать Vault в CI/CD без интерактивного ввода.
- Зашифрованный контент прозрачно расшифровывается при выполнении, но не попадает в логи.

 **Организация комплексных проектов**:

- Разделение на `tasks/`, `handlers/`, `vars/`, `templates/` улучшает читаемость и переиспользование.
- Сочетание статического импорта (для структуры) и динамического включения (для гибкости) даёт баланс между предсказуемостью и адаптивностью.
- Шифрование секретов + модульность + обработчики = масштабируемый, безопасный и поддерживаемый код автоматизации.

## Лабо работа 12: Комплексный проект
### Последовательность выполнения
#### 0: Подготовка структуры
```bash
mkdir -p ~/ll105-lab12/{playbooks,tasks,handlers,vars,templates}
cd ~/ll105-lab12
cp ../ll105-lab11/inventory.yml ./inventory/
cp ../ll105-lab11/ansible.cfg ./
```
#### 1: Создать зашифрованные переменные
```bash
# Создать файл с секретами и зашифровать его
cat > vars/secrets.yml << 'EOF'
db_password: "SuperSecret123!"
api_key: "sk-proj-abc123xyz"
EOF

# Зашифровать файл (задайте пароль при запросе)
ansible-vault encrypt vars/secrets.yml

# Альтернатива: зашифровать строку для вставки
ansible-vault encrypt_string 'SuperSecret123!' --name 'db_password'
```

#### 2: Создать выносные артефакты

**`tasks/install.yml`** (задачи установки):
```yaml
---
- name: Install required packages
  ansible.builtin.package:
    name: ['nginx', 'python3-psycopg2']
    state: present
  notify: Restart nginx service
```

**`handlers/main.yml`** (обработчики):
```yaml
---
- name: Restart nginx service
  ansible.builtin.systemd:
    name: nginx
    state: restarted
  listen: "web services restart"
```

**`vars/app_config.yml`** (открытые переменные):
```yaml
---
app_name: "MyApp"
nginx_port: 8080
db_host: "db.internal"
# db_password загружается из secrets.yml
```

#### 3: Создать основной playbook
```yaml
# playbooks/site.yml
---
- name: Lab12: Complex Project Setup
  hosts: all
  become: yes
  gather_facts: yes

  # Загрузка переменных
  vars_files:
    - ../vars/app_config.yml
    - ../vars/secrets.yml  # Зашифрованный файл

  # Подключение обработчиков
  handlers:
    - ansible.builtin.import_tasks: ../handlers/main.yml

  tasks:
    # Статический импорт задач установки
    - ansible.builtin.import_tasks: ../tasks/install.yml

    # Динамическое включение конфигурации с условием
    - name: Configure OS-specific settings
      ansible.builtin.include_tasks: "../tasks/config_{{ ansible_os_family }}.yml"
      when: ansible_os_family in ['RedHat', 'Debian']

    # Задача с использованием секрета
    - name: Deploy app config with credentials
      ansible.builtin.template:
        src: ../templates/app.conf.j2
        dest: /etc/myapp/app.conf
        mode: '0640'
      notify: "web services restart"  # Использование listen-алиаса

    # Принудительный запуск обработчиков перед валидацией
    - name: Flush handlers before validation
      ansible.builtin.meta: flush_handlers

    # Валидация после перезапуска
    - name: Verify service is running
      ansible.builtin.command: systemctl is-active nginx
      register: nginx_status
      changed_when: false
      failed_when: nginx_status.rc != 0
```

#### 4: Создать шаблон с использованием секрета

```jinja2
# templates/app.conf.j2
[app]
name = {{ app_name }}
port = {{ nginx_port }}

[database]
host = {{ db_host }}
password = {{ db_password }}  # Секрет из vault
```
#### 5: Проверка и запуск
```bash
# 1. Проверка синтаксиса (без расшифровки)
ansible-playbook --syntax-check playbooks/site.yml

# 2. Запуск с интерактивным вводом пароля
ansible-playbook -i inventory/inventory.yml playbooks/site.yml --ask-vault-pass -v

# 3. Запуск с файлом пароля (для CI/CD)
echo "MyVaultPassword" > ~/.vault_pass
chmod 600 ~/.vault_pass
ansible-playbook -i inventory/inventory.yml playbooks/site.yml --vault-password-file ~/.vault_pass -v
```

#### 6: Верификация
```bash
# Проверка, что обработчик сработал (перезапуск только при changed)
ansible all -a "systemctl status nginx --no-pager | grep Active"

# Проверка, что секрет не попал в лог в открытом виде
ansible-playbook ... -vvv 2>&1 | grep -i "SuperSecret"  # Должно быть пусто

# Проверка содержимого конфига (пароль должен быть подставлен)
ansible all -a "grep password /etc/myapp/app.conf"
```

>[!CHECK] Критерии выполнения
>
>- `vars/secrets.yml` зашифрован (содержит `$ANSIBLE_VAULT` в начале)
>- Обработчики вынесены в `handlers/main.yml` и подключены через `import_tasks`
>- Задачи установки вынесены в `tasks/install.yml` и импортированы статически
>- Условное включение `include_tasks` работает только для подходящих ОС
>- Секрет `db_password` подставляется в шаблон, но не виден в логах даже при `-vvv`
>- `meta: flush_handlers` гарантирует выполнение перезапуска до задачи валидации
>- Повторный запуск без изменений → обработчики не выполняются (идемпотентность)
