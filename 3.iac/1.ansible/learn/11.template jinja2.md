
# Модуль 11: Шаблоны Jinja2 в Ansible

> [!SUMMARY] Цель
> Научиться генерировать конфигурационные файлы и динамическое содержимое с использованием шаблонизатора Jinja2, фильтров, тестов и управляющих структур в контексте выполнения Ansible.

> [!OBJECTIVES] Задачи
> 
> -  Освоить синтаксис разделителей Jinja2: `{{ }}`, `{% %}`, `{# #}`
> -  Применять фильтры и тесты для преобразования и проверки данных
> -  Использовать управляющие структуры (`if`/`for`) в шаблонах
> -  Настроить модуль `template` с параметрами `backup`, `validate`, `trim_blocks`

---
## Синтаксис шаблонов Jinja2
| Разделитель                | Назначение                                | Пример                                   |
| -------------------------- | ----------------------------------------- | ---------------------------------------- |
| `{{ ... }}`                | Выражения: подстановка значений           | `{{ inventory_hostname }}`               |
| `{% ... %}`                | Утверждения: условия, циклы, присваивания | `{% if ansible_os_family == "RedHat" %}` |
| `{# ... #}`                | Комментарии (не попадают в итоговый файл) | `{# Generated by Ansible #}`             |
| `{% raw %}...{% endraw %}` | Экранирование: вывод литералов `{{ }}`    | `{% raw %}{{ var }}{% endraw %}`         |
>[!NOTE] Контекст переменных в шаблоне 
Шаблон получает доступ ко всем переменным, доступным в момент выполнения задачи: `vars`, `hostvars`, `group_vars`, `ansible_facts`, magic-переменные (`inventory_hostname`, `groups` и др.). Переменные из `loop` доступны только если шаблон вызывается внутри цикла.

## Подстановка переменных

### Базовый синтаксис
```jinja2
# В шаблоне (.j2):
ServerName {{ inventory_hostname }}
Listen {{ web_port | default(80) }}

# В условии задачи (YAML) — кавычки обязательны:
- name: Use variable
  debug:
    msg: "Value: {{ my_var }}"

# В условии when (YAML) — кавычки и {{ }} не нужны:
- name: Conditional task
  debug:
    msg: "Executed"
  when: my_var is defined
```

>[!WARNING] Разница контекстов
>
>- В шаблонах (`.j2`): `{{ var }}` — всегда выражение, кавычки не требуются
>- В YAML-задачах: `"{{ var }}"` — строка с подстановкой, кавычки обязательны
>- В `when`: `var is defined` — чистое выражение Jinja2, без `{{ }}` и кавычек

## Фильтры (Filters)

Фильтр — функция преобразования данных. Применяется через `|`, возможна цепочка.

### Категории и примеры

| Категория      | Фильтр                                           | Пример                                                                   | Результат                                     |
| -------------- | ------------------------------------------------ | ------------------------------------------------------------------------ | --------------------------------------------- |
| **Текст**      | `upper`, `lower`, `capitalize`                   | `{{ "text" \| upper }}`                                                  | `TEXT`                                        |
| **Текст**      | `replace(old, new)`                              | `{{ "a-b-c" \| replace("-", "_") }}`                                     | `a_b_c`                                       |
| **Числа**      | `int`, `float`, `round(digits)`                  | `{{ 3.14159 \| round(2) }}`                                              | `3.14`                                        |
| **Коллекции**  | `length`, `sum`, `sort`, `unique`                | `{{ [1,2,2,3] \| unique }}`                                              | `[1, 2, 3]`                                   |
| **Коллекции**  | `selectattr(key, op, value)`                     | `{{ users \| selectattr('uid', '>=', 1000) }}`                           | отфильтрованный список                        |
| **Дата/время** | `to_datetime`, `strftime(fmt)`                   | `{{ ansible_date_time.iso8601 \| to_datetime \| strftime('%Y-%m-%d') }}` | `2024-01-15`                                  |
| **Ansible**    | `to_nice_json`, `to_yaml`, `password_hash(algo)` | `{{ "pass" \| password_hash('sha512') }}`                                | хэш для модуля `user`                         |
| **Fallback**   | `default(value, true)`                           | `{{ custom_port \| default(80, true) }}`                                 | `80` если `custom_port` не определён или пуст |

>[!NOTE] Параметр `true` в `default` 
`default(val, true)` подставляет значение, если переменная **не определена ИЛИ пустая**. Без второго аргумента — только если не определена.

### Комбинирование фильтров
```jinja2
{{ company_name | lower | replace('corp', 'company') | default('Internal') }}
{{ [3,1,2] | sort | reverse | join(', ') }}  → "3, 2, 1"
```

## Тесты (Tests)

Тест проверяет свойство переменной. Синтаксис: `{{ var is test_name(args) }}`.

| Тест                                       | Описание               | Пример                            |
| ------------------------------------------ | ---------------------- | --------------------------------- |
| `defined` / `undefined`                    | Проверка существования | `{% if var is defined %}`         |
| `none`                                     | Проверка на `None`     | `{% if var is none %}`            |
| `equalto(val)` / `eq`                      | Равенство              | `{% if count is eq(10) %}`        |
| `gt(val)`, `ge(val)`, `lt(val)`, `le(val)` | Сравнение чисел        | `{% if mem is ge(2048) %}`        |
| `in sequence`                              | Вхождение в коллекцию  | `{% if role in ['web', 'api'] %}` |
| `string`, `number`, `sequence`, `mapping`  | Проверка типа          | `{% if data is sequence %}`       |
| `lower`, `upper`                           | Регистр строки         | `{% if name is lower %}`          |
>[!TIP] Псевдонимы операторов
`==` ≡ `is equalto()`, `>=` ≡ `is ge()`, `!=` ≡ `is ne()` — выбирайте стиль для читаемости.

## Управляющие структуры

### Условия `if` / `elif` / `else`
```jinja2
{% if ansible_os_family == "RedHat" %}
PackageMgr: dnf
{% elif ansible_os_family == "Debian" %}
PackageMgr: apt
{% else %}
PackageMgr: unknown
{% endif %}
```

### Циклы `for`
```jinja2
{% for iface in ansible_interfaces if iface != 'lo' %}
Interface: {{ iface }}
  IP: {{ hostvars[inventory_hostname]['ansible_' + iface]['ipv4']['address'] | default('N/A') }}
{% endfor %}
```

### Переменные цикла

| Переменная                 | Описание                   |
| -------------------------- | -------------------------- |
| `loop.index`               | Текущий индекс (с 1)       |
| `loop.index0`              | Текущий индекс (с 0)       |
| `loop.first` / `loop.last` | Первая/последняя итерация  |
| `loop.length`              | Общее количество элементов |
```jinja2
{% for pkg in packages %}
{{ loop.index }}. {{ pkg }}{% if not loop.last %},{% endif %}
{% endfor %}
```

## Операторы в выражениях

### Математические
```jinja2
{{ 10 + 5 }}      # сложение → 15
{{ 10 // 4 }}     # целочисленное деление → 2
{{ 15 % 7 }}      # остаток → 1
{{ 2 ** 3 }}      # степень → 8
```
### Сравнения и логика
```jinja2
{{ 10 == 10 and 5 != 2 }}   # true
{{ not (a > b or c is defined) }}  # группировка скобками
{{ "web" in group_names }}  # проверка вхождения
```

## Модуль `ansible.builtin.template`

### Механика работы

- Читает шаблон `.j2` с control node
- Обрабатывает через движок Jinja2 в контексте текущих переменных
- Записывает результат на managed node
- Сравнивает с существующим файлом → `changed` только при различии

### Ключевые параметры
| Параметр                   | Описание                                      | Пример                            |
| -------------------------- | --------------------------------------------- | --------------------------------- |
| `src`                      | Путь к шаблону на control node                | `templates/nginx.conf.j2`         |
| `dest`                     | Путь на managed node                          | `/etc/nginx/nginx.conf`           |
| `owner` / `group` / `mode` | Права итогового файла                         | `mode: '0640'`                    |
| `backup`                   | Резервная копия перед заменой                 | `backup: yes`                     |
| `validate`                 | Команда проверки файла перед применением      | `validate: 'nginx -t -c %s'`      |
| `trim_blocks`              | Убирать первый перенос после `{% %}`          | `trim_blocks: yes` (по умолчанию) |
| `lstrip_blocks`            | Убирать пробелы в начале строки перед `{% %}` | `lstrip_blocks: yes`              |
```yaml
- name: Deploy nginx config from template
  ansible.builtin.template:
    src: templates/nginx.conf.j2
    dest: /etc/nginx/nginx.conf
    owner: root
    group: root
    mode: '0644'
    backup: yes
    validate: 'nginx -t -c %s'
    trim_blocks: yes
    lstrip_blocks: yes
```

>[!NOTE] Переменная `%s` в `validate` 
Ansible подставит путь к временному файлу с отрендеренным содержимым. Это позволяет проверить синтаксис до замены продакшен-файла.

## Выводы

 **Jinja2 в Ansible** — шаблонизатор для генерации текстового контента с подстановкой переменных, логикой и преобразованиями.

 **Синтаксис**: `{{ }}` для выражений, `{% %}` для управляющих конструкций, `{# #}` для комментариев. Контекст выполнения включает все переменные, доступные в момент вызова задачи.

 **Фильтры** преобразуют данные: `default`, `join`, `sort`, `password_hash`, `strftime` и др. Цепочки фильтров выполняются слева направо.

 **Тесты** проверяют свойства переменных: `defined`, `in`, `number`, `equalto`. Используются в условиях `{% if %}`.

 **Управляющие структуры** (`if`/`for`) позволяют генерировать динамическое содержимое: конфигурации под разные ОС, списки сервисов, условные блоки.

 **Модуль `template`** обеспечивает идемпотентную доставку отрендеренных файлов. Параметры `validate`, `backup`, `trim_blocks` повышают надёжность и читаемость результата.

 **Чистые и структурированные шаблоны** облегчают сопровождение инфраструктуры: логика инкапсулирована в `.j2`, данные — в переменных, изменения — в `git`.

## Лабораторная работа 11: Шаблоны Jinja2

### Последовательность

#### Шаг 0: Подготовка
```bash
mkdir -p ~/ll105-lab11/{playbooks,templates,inventory}
cd ~/ll105-lab11
cp ../ll105-lab10/inventory.yml ./inventory/
cp ../ll105-lab10/ansible.cfg ./
```
#### 1: Создать шаблон отчёта
```jinja2
# ~/ll105-lab11/templates/node_report.j2
{# Node Report - Generated by Ansible #}
Hostname: {{ inventory_hostname | upper }}
IP Address: {{ ansible_default_ipv4.address | default('N/A') }}
Interfaces: {{ ansible_interfaces | reject('equalto', 'lo') | sort | join(', ') }}
CPU Cores: {{ ansible_processor_vcpus }}
Groups: {{ group_names | sort | join(', ') }}
Admin: {{ admin_user | capitalize | default('Not Specified', true) }}
Timestamp: {{ now(utc=true, fmt='%Y-%m-%d %H:%M:%S') }}
```

#### 2: Создать playbook
```yaml
# ~/ll105-lab11/playbooks/jinja_report.yml
---
- name: Lab11: Jinja2 Template Report
  hosts: all
  become: yes
  gather_facts: yes
  vars:
    admin_user: "{{ lookup('env', 'USER') }}"  # или задать в group_vars
  tasks:
    - name: Generate node report
      ansible.builtin.template:
        src: ../templates/node_report.j2
        dest: /etc/node_report.txt
        owner: root
        group: root
        mode: '0644'
        backup: yes
        trim_blocks: yes
        lstrip_blocks: yes
```

#### 3: Проверка и запуск
```bash
# Проверка синтаксиса
ansible-playbook --syntax-check playbooks/jinja_report.yml

# Dry-run с показом различий
ansible-playbook -i inventory/inventory.yml playbooks/jinja_report.yml --check --diff -v

# Запуск
ansible-playbook -i inventory/inventory.yml playbooks/jinja_report.yml -v
```

#### 4: Верификация результата
```bash
# Просмотр сгенерированного отчёта на всех хостах
ansible all -a "cat /etc/node_report.txt"

# Проверка наличия резервной копии (при повторном запуске)
ansible all -a "ls -la /etc/node_report.txt*"

# Валидация содержимого (примеры)
# 1. Имя хоста в верхнем регистре
ansible all -a "grep '^Hostname:' /etc/node_report.txt | grep -E '[A-Z]+'"

# 2. Список интерфейсов через запятую (без lo)
ansible all -a "grep '^Interfaces:' /etc/node_report.txt"

# 3. Admin с заглавной буквы или дефолт
ansible all -a "grep '^Admin:' /etc/node_report.txt"
```

>[!CHECK] Критерии выполнения
>
>- `inventory_hostname` выводится в верхнем регистре (`upper`)
>- `ansible_interfaces` отфильтрован (`reject('equalto', 'lo')`), отсортирован, объединён через `,`
>- `ansible_processor_vcpus` подставлен без преобразований
>- `group_names` отсортирован и выведен списком
>- `admin_user` применён `capitalize`, с `default('Not Specified', true)` для пустых значений
>- Файл `/etc/node_report.txt` создан с правами `0644`, резервная копия сохраняется при изменениях
>- Повторный запуск → задача `ok`, без `changed` (идемпотентность)
