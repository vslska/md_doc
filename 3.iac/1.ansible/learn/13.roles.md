## Структура роли: стандартные каталоги

Роль — это предопределённая структура каталогов, содержащая связанные артефакты автоматизации. Ansible автоматически загружает файлы `main.yml` из каждого каталога при вызове роли.

|Каталог|Файл по умолчанию|Назначение|Приоритет переменных|
|---|---|---|---|
|`tasks/`|`main.yml`|Основные задачи роли|—|
|`handlers/`|`main.yml`|Обработчики, вызываемые через `notify`|—|
|`templates/`|—|Jinja2-шаблоны для модуля `template`|—|
|`files/`|—|Статические файлы для модуля `copy`|—|
|`vars/`|`main.yml`|Переменные с **высоким** приоритетом|Высокий (переопределяют `defaults`)|
|`defaults/`|`main.yml`|Переменные с **низким** приоритетом (значения по умолчанию)|Низкий (легко переопределяются)|
|`meta/`|`main.yml`|Метаданные: зависимости (`dependencies`), автор, лицензия, `allow_duplicates`|—|
|`library/`|—|Кастомные модули/плагины, доступные только в этой роли|—|

> [!NOTE] Минимальная роль 
> Роль считается валидной, если содержит хотя бы один каталог с файлом `main.yml`. Пустые каталоги можно удалять без последствий.

## Создание роли: `ansible-galaxy init`

### Инициализация структуры
```bash
# Создать каталог для ролей в проекте
mkdir -p roles
cd roles

# Инициализировать роль с полной структурой
ansible-galaxy init rsyncserver

# Результат:
roles/
└── rsyncserver/
    ├── README.md
    ├── defaults/
    │   └── main.yml
    ├── tasks/
    │   └── main.yml
    ├── handlers/
    │   └── main.yml
    ├── templates/
    ├── files/
    ├── vars/
    ├── meta/
    │   └── main.yml
    └── tests/
        ├── inventory
        └── test.yml
```

### Наполнение роли (пример: `rsyncserver`)

**`defaults/main.yml`** (переменные по умолчанию):
```yaml
---
rsync_port: 873
rsync_user: rsync
rsync_group: rsync
rsync_max_connections: 10
rsync_log_file: /var/log/rsyncd.log
rsync_module_name: backup
rsync_path: /srv/backup
```

**`tasks/main.yml`** (основная логика):
```yaml
---
- name: Install rsync package
  ansible.builtin.package:
    name: rsync
    state: present

- name: Create rsync group
  ansible.builtin.group:
    name: "{{ rsync_group }}"
    state: present

- name: Create rsync user
  ansible.builtin.user:
    name: "{{ rsync_user }}"
    group: "{{ rsync_group }}"
    shell: /usr/sbin/nologin
    create_home: no
    state: present

- name: Create backup directory
  ansible.builtin.file:
    path: "{{ rsync_path }}"
    state: directory
    owner: "{{ rsync_user }}"
    group: "{{ rsync_group }}"
    mode: '0755'

- name: Deploy rsyncd.conf from template
  ansible.builtin.template:
    src: rsyncd.conf.j2
    dest: /etc/rsyncd.conf
    owner: root
    group: root
    mode: '0644'
    backup: yes
  notify: Restart rsyncd service

- name: Ensure rsyncd is enabled and running
  ansible.builtin.systemd:
    name: rsyncd
    state: started
    enabled: yes
```

**`templates/rsyncd.conf.j2`**:
```jinja2
# Managed by Ansible - do not edit manually
uid = {{ rsync_user }}
gid = {{ rsync_group }}
port = {{ rsync_port }}
max connections = {{ rsync_max_connections }}
log file = {{ rsync_log_file }}
pid file = /var/run/rsyncd.pid

[{{ rsync_module_name }}]
path = {{ rsync_path }}
comment = Backup Module
read only = no
list = yes
auth users = {{ rsync_auth_users | default('') }}
secrets file = {{ rsync_secrets_file | default('') }}
```

**`handlers/main.yml`**:
```yaml
---
- name: Restart rsyncd service
  ansible.builtin.systemd:
    name: rsyncd
    state: restarted
```

**`meta/main.yml`** (метаданные и зависимости):
```yaml
---
galaxy_info:
  author: Your Name
  description: Rsync server role for backup management
  license: MIT
  min_ansible_version: "2.14"

dependencies: []
  # Пример зависимости:
  # - role: common
  #   tags: ['common']

allow_duplicates: yes  # Разрешить многократный вызов роли с разными параметрами
```

## Подключение ролей в playbook

### Способ 1: Ключевое слово `roles:` (статический, на уровне play)
```yaml
# site.yml
---
- name: Deploy rsync server
  hosts: backup_servers
  become: yes
  roles:
    - rsyncserver  # Простой вызов
    - role: rsyncserver
      rsync_port: 9000  # Переопределение переменной для этого хоста
      rsync_module_name: prod_backup
```

### Способ 2: `include_role` (динамический, на уровне tasks)
```yaml
- name: Configure backup infrastructure
  hosts: all
  tasks:
    - name: Prepare storage
      ansible.builtin.file:
        path: /srv/backup
        state: directory

    - name: Include rsync role dynamically
      ansible.builtin.include_role:
        name: rsyncserver
      vars:
        rsync_port: 9000
      when: "'backup' in group_names"
```

### Способ 3: `import_role` (статический, на уровне tasks)
```yaml
- name: Configure backup infrastructure
  hosts: all
  tasks:
    - name: Import rsync role statically
      ansible.builtin.import_role:
        name: rsyncserver
        tasks_from: main.yml  # Опционально: указать другой файл задач
        handlers_from: main.yml  # Опционально: указать другой файл обработчиков
```

### Сравнение методов подключения
| Характеристика         | `roles:`                 | `include_role`             | `import_role`                        |
| ---------------------- | ------------------------ | -------------------------- | ------------------------------------ |
| **Уровень**            | Play                     | Tasks                      | Tasks                                |
| **Время обработки**    | Парсинг (статический)    | Runtime (динамический)     | Парсинг (статический)                |
| **Условия `when`**     | Применяются ко всей роли | Проверяются при включении  | Применяются ко всей роли             |
| **Циклы `loop`**       | ❌ Не поддерживаются      |  Поддерживаются            | ❌ Не поддерживаются                  |
| **Переменные пути**    | ❌ Фиксированное имя роли | Путь может быть переменной | ❌ Фиксированное имя роли             |
| **Порядок выполнения** | До задач play            | В точке вызова             | В точке вызова (но парсится заранее) |

>[!NOTE] Приоритет переменных в ролях
>```
>1. Переменные командной строки (-e)
>2. Переменные play / host_vars / group_vars
>3. Переменные, переданные при вызове роли (в roles: или include_role)
>4. role_name/vars/main.yml (высокий приоритет)
>5. role_name/defaults/main.yml (низкий приоритет, значения по умолчанию)
>```

## Поиск и расположение ролей
### Порядок поиска (по умолчанию)
```
1. ./roles/                  # Относительно текущего playbook
2. ~/.ansible/roles/         # Домашняя директория пользователя
3. /etc/ansible/roles/       # Системный каталог
4. /usr/share/ansible/roles/ # Глобальный каталог (после установки из Galaxy)
```

### Настройка `roles_path`
```ini
# ansible.cfg
[defaults]
roles_path = ./roles:/opt/shared-roles:~/.ansible/roles
```
### Использование абсолютного пути
```yaml
- hosts: webservers
  roles:
    - role: /opt/custom-roles/common
```

## Многократное выполнение роли

По умолчанию роль выполняется **один раз** на хост, даже если указана несколько раз в `roles:`.

### Способ 1: Разные параметры при вызове
```yaml
- hosts: all
  roles:
    - { role: rsyncserver, rsync_module_name: "backup_daily", rsync_path: "/srv/daily" }
    - { role: rsyncserver, rsync_module_name: "backup_weekly", rsync_path: "/srv/weekly" }
```
Ansible выполнит роль дважды, так как наборы переменных различаются.
### Способ 2: Метаданные `allow_duplicates`
```yaml
# meta/main.yml внутри роли
---
allow_duplicates: yes
```
Разрешает повторный вызов роли даже с идентичными параметрами.

> [!WARNING] `include_role` / `import_role` и параметры 
> При использовании `include_role` или `import_role` параметры передаются через блок `vars:` рядом с вызовом, а не как аргументы роли.

## Выводы

 **Роли обеспечивают модульность**:

- Стандартизированная структура каталогов (`tasks/`, `defaults/`, `templates/` и др.) упрощает навигацию и повторное использование.
- Автоматическая загрузка `main.yml` из каждого каталога снижает явное подключение файлов.

 **Управление переменными**:

- `defaults/main.yml` — значения по умолчанию (низкий приоритет, легко переопределяются).
- `vars/main.yml` — переменные роли с высоким приоритетом (переопределяют `defaults`, но уступают `host_vars`/`play vars`).
- Передача параметров при вызове роли (`roles: - {role: x, var: value}`) позволяет адаптировать поведение без изменения кода роли.

 **Способы подключения**:

- `roles:` — статический вызов на уровне play, простой и предсказуемый.
- `include_role` — динамический вызов в `tasks:`, поддерживает условия и циклы.
- `import_role` — статический вызов в `tasks:`, парсится заранее, быстрее, но менее гибкий.

 **Повторное использование**:

- `allow_duplicates: yes` в `meta/main.yml` разрешает многократный вызов роли.
- Разные наборы параметров при вызове автоматически приводят к повторному выполнению.
- Зависимости в `meta/main.yml` (`dependencies:`) позволяют автоматически подключать другие роли.

 **Поиск и организация**:

- `roles_path` в `ansible.cfg` централизует расположение общих ролей.
- Абсолютные пути позволяют подключать роли вне стандартных каталогов.
- Эти механизмы позволяют создавать масштабируемые, поддерживаемые и безопасные проекты автоматизации.
