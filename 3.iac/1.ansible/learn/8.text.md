# Модуль 8: Работа с текстом и конфигурационными файлами

> [!SUMMARY] Цель  
> Научиться безопасно и идемпотентно изменять текстовые файлы на управляемых хостах с помощью специализированных модулей Ansible и регулярных выражений.

> [!OBJECTIVES] Задачи
> 
> - Освоить `lineinfile`: точечное добавление/замена/удаление строк
> - Научиться массово заменять текст через `replace` (в том числе в ограниченных секциях)
> - Управлять многострочными блоками через `blockinfile` с маркерами
> - Применять RegEx безопасно в YAML-синтаксисе
> - Понять, когда использовать каждый модуль, а когда предпочесть `template`

---
## Сравнение модулей работы с текстом
| Модуль                        | Что делает                                           | Идемпотентен? | Когда применять                                             |
| ----------------------------- | ---------------------------------------------------- | ------------- | ----------------------------------------------------------- |
| `ansible.builtin.lineinfile`  | Гарантирует наличие/отсутствие **одной строки**      | Да            | Точечные правки конфигов (порты, флаги, пользователи)       |
| `ansible.builtin.replace`     | Заменяет **все вхождения** RegEx-шаблона             | Да            | Массовая замена, рефакторинг конфигов, поиск/замена в логах |
| `ansible.builtin.blockinfile` | Вставляет/удаляет **многострочный блок** с маркерами |  Да           | Добавление секций конфигов, блоков кода, HTML-фрагментов    |

>[!TIP] Правило
Если конфигурационный файл сложный или требует условий/переменных — используйте `template`. Текстовые модули идеальны для быстрых правок, где генерация всего файла избыточна.

## `lineinfile`: управление одной строкой

### Ключевые параметры
| Параметр                       | Описание                                             | Пример                 |
| ------------------------------ | ---------------------------------------------------- | ---------------------- |
| `path`                         | Путь к файлу                                         | `/etc/ssh/sshd_config` |
| `line`                         | Строка для вставки или замены                        | `Port 2222`            |
| `regexp`                       | RegEx для поиска строки, которую нужно заменить      | `^#?Port\s`            |
| `state`                        | `present` (добавить/обновить) или `absent` (удалить) | `present`              |
| `create`                       | Создать файл, если не существует                     | `yes`                  |
| `backup`                       | Резервная копия перед изменением                     | `yes`                  |
| `insertafter` / `insertbefore` | Позиция вставки относительно шаблона                 | `^# Authentication`    |
| `validate`                     | Команда проверки файла перед применением             | `sshd -t -f %s`        |
### Как работает `regexp` + `line`

1. Ansible ищет строку, соответствующую `regexp`
2. Если найдена → заменяет её на `line`
3. Если не найдена → добавляет `line` в конец (или по `insertafter/insertbefore`)
### Примеры
```yaml
# Изменить порт SSH (если строка есть → заменит, если нет → добавит)
- name: Change SSH port to 2222
  ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    regexp: '^#?Port\s'
    line: 'Port 2222'
    backup: yes
    validate: 'sshd -t -f %s'

# Добавить строку, если её нет
- name: Allow root login via SSH
  ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    line: 'AllowUsers root ansible'
    insertafter: '^# Authentication'
    backup: yes

# Удалить строку по шаблону
- name: Remove legacy banner
  ansible.builtin.lineinfile:
    path: /etc/motd
    regexp: 'This system is for authorized use only'
    state: absent
```

>[!WARNING] `regexp` в YAML 
Всегда заключайте RegEx в кавычки: `regexp: '^Listen '` Без кавычек YAML может интерпретировать `#` как комментарий, а `\` как escape-символ.

## `replace`: массовая замена по RegEx

### Ключевые параметры
| Параметр           | Описание                                               |
| ------------------ | ------------------------------------------------------ |
| `path`             | Путь к файлу                                           |
| `regexp`           | Шаблон для поиска **всех** вхождений                   |
| `replace`          | Текст замены (поддерживает обратные ссылки `\1`, `\2`) |
| `after` / `before` | Ограничить замену секцией между/после шаблонов         |
| `backup`           | Резервная копия                                        |
### Примеры
```yaml
# Глобальная замена хоста
- name: Update hostname in hosts file
  ansible.builtin.replace:
    path: /etc/hosts
    regexp: 'old\.server\.local'
    replace: 'new.server.local'
    backup: yes

# Замена только внутри секции (после VirtualHost до конца файла)
- name: Comment out lines after VirtualHost
  ansible.builtin.replace:
    path: /etc/apache2/sites-available/default.conf
    after: '<VirtualHost'
    regexp: '^(.+)$'
    replace: '# \1'
    backup: yes

# Замена между маркерами
- name: Redact passwords in config section
  ansible.builtin.replace:
    path: /opt/app/settings.ini
    after: '^# START CREDENTIALS'
    before: '^# END CREDENTIALS'
    regexp: 'password\s*=\s*.+'
    replace: 'password = ********'
```

>[!NOTE] Обратные ссылки 
`\1`, `\2` работают в `replace`, если в `regexp` есть захватывающие группы `()`. Пример: `regexp: '(ServerName\s+)(\S+)'` → `replace: '\1prod.example.com'`

## `blockinfile`: управление многострочными блоками

### Как это работает?

- При первом запуске вставляет блок с **маркерами начала и конца**
- При повторных запусках **находит маркеры и заменяет содержимое блока**
- Если маркеры удалены вручную → блок вставляется заново

### Ключевые параметры

| Параметр                       | Описание                                                         | Пример                   |
| ------------------------------ | ---------------------------------------------------------------- | ------------------------ |
| `path`                         | Путь к файлу                                                     | `/etc/sysctl.conf`       |
| `block`                        | Многострочный текст (YAML `                                      | `или`>`)                 |
| `marker`                       | Шаблон маркеров (по умолчанию: `# {mark} ANSIBLE MANAGED BLOCK`) | `<!-- {mark} CUSTOM -->` |
| `insertafter` / `insertbefore` | Позиция вставки                                                  | `</head>`                |
| `state`                        | `present` / `absent`                                             | `present`                |
| `backup`                       | Резервная копия                                                  | `yes`                    |
### Пример
```yaml
- name: Add sysctl parameters block
  ansible.builtin.blockinfile:
    path: /etc/sysctl.conf
    marker: "# {mark} ANSIBLE NETWORK BLOCK"
    block: |
      net.ipv4.ip_forward = 1
      net.core.somaxconn = 1024
      vm.swappiness = 10
    backup: yes
```

> [!IMPORTANT] Никогда не редактируйте вручную текст между маркерами! 
> При следующем запуске playbook ваши правки будут **перезаписаны**. Маркеры сигнализируют: «этим управляет Ansible».

## Основы RegEx для Ansible
| Конструкция | Значение                   | Пример в Ansible                               |
| ----------- | -------------------------- | ---------------------------------------------- |
| `^`         | Начало строки              | `^Listen`                                      |
| `$`         | Конец строки               | `enabled$`                                     |
| `.`         | Любой символ               | `file.log`                                     |
| `.*`        | 0 или более любых символов | `^#.*$` (пустые или закомментированные строки) |
| `\d`        | Цифра                      | `\d+`                                          |
| `[a-z]`     | Буква в нижнем регистре    | `[a-z]+`                                       |
| `\s`        | Пробельный символ          | `Port\s+22`                                    |
| `\b`        | Граница слова              | `\broot\b`                                     |
| `()`        | Захватывающая группа       | `(Port)\s+(\d+)`                               |
>[!TIP] Тестирование RegEx
>```bash
># Проверить, какие строки совпадут, без внесения изменений:
ansible all -m lineinfile -a "path=/etc/ssh/sshd_config regexp='^Port' state=absent" --check -v
>```

## Рекомендации по безопасной работе
```yaml
# 1. Всегда используйте backup для критичных файлов
backup: yes

# 2. Проверяйте валидность после правки
validate: 'nginx -t -c %s'   # %s подставит путь к временному файлу

# 3. Работайте в check-mode перед применением
ansible-playbook playbook.yml --check --diff

# 4. Избегайте "широких" RegEx в replace
regexp: 'password.*'     # ⚠️ Опасно: захватит всю строку до конца файла
regexp: 'password\s*=\s*.*' # ✅ Безопасно: ограничено логическим концом
```

## Выводы

 **Ansible предоставляет мощные средства управления текстом**:

- `lineinfile` — точечные, безопасные правки строк с валидацией
- `replace` — массовая замена с поддержкой контекста и обратных ссылок
- `blockinfile` — управляемые многострочные секции с автоматической заменой

 **Использование регулярных выражений позволяет гибко настраивать файлы**:

- RegEx ограничивает область изменений, предотвращая случайные поломки
- В YAML RegEx всегда заключайте в кавычки, избегайте "жадных" паттернов
- `--check --diff` — обязательный этап перед применением к production

 **Правильный выбор модуля — залог стабильности**:

- `lineinfile` — для строк, `blockinfile` — для фрагментов, `replace` — для замены
- Эти модули особенно полезны при настройке конфигураций, где полная генерация через `template` избыточна или невозможна
- Для сложных, динамических конфигов всегда предпочтительнее Jinja2-шаблоны

## Лабо 8: Управление конфигурационными файлами

### Задания :

#### 1. Подготовка
```bash
mkdir -p ~/ll105-lab8/{playbooks,inventory}
cd ~/ll105-lab8
cp ../ll105-lab7/inventory.yml ./
cp ../ll105-lab7/ansible.cfg ./
```
#### 2. Создать playbook `text_ops.yml`
```yaml
# ~/ll105-lab8/playbooks/text_ops.yml
---
- name: Lab8: Text and Config Management
  hosts: all
  become: yes
  gather_facts: no  # факты не нужны для этих задач
  tasks:
    # 1. Добавить AllowUsers root (идемпотентно, с проверкой)
    - name: Ensure AllowUsers directive in sshd_config
      ansible.builtin.lineinfile:
        path: /etc/ssh/sshd_config
        regexp: '^AllowUsers\s'
        line: 'AllowUsers root'
        backup: yes
        validate: 'sshd -t -f %s'

    # 2. Добавить блок переменных в /etc/profile
    - name: Add environment variables block
      ansible.builtin.blockinfile:
        path: /etc/profile
        marker: "# {mark} LAB8 CUSTOM ENV"
        block: |
          export EDITOR=vim
          export HISTTIMEFORMAT="%F %T "
        backup: yes

    # 3. Заменить приветствие в /etc/motd
    - name: Update MOTD greeting
      ansible.builtin.replace:
        path: /etc/motd
        regexp: '(Welcome to )(\S+)'
        replace: '\1{{ inventory_hostname }} (managed by Ansible)'
        backup: yes
```
#### 3. Проверка и запуск
```bash
# Синтаксис
ansible-playbook --syntax-check playbooks/text_ops.yml

# Dry-run с diff (покажет, какие строки изменятся)
ansible-playbook -i inventory.yml playbooks/text_ops.yml --check --diff -v

# Запуск
ansible-playbook -i inventory.yml playbooks/text_ops.yml -v
```
#### 4. Верификация
```bash
# Проверка sshd_config
ansible all -a "grep AllowUsers /etc/ssh/sshd_config"

# Проверка /etc/profile
ansible all -a "sed -n '/LAB8 CUSTOM ENV/,/ANSIBLE MANAGED BLOCK/p' /etc/profile"

# Проверка motd
ansible all -a "cat /etc/motd"
```

>[!SUCCESS] Критерии выполнения
>
>- `AllowUsers root` добавлена в `sshd_config`, валидация `sshd -t` прошла
>- Блок переменных в `/etc/profile` окружён маркерами `# BEGIN/LAB8 CUSTOM ENV`
>- `/etc/motd` содержит имя хоста, старые приветствия заменены
>- Все задачи идемпотентны: повторный запуск → `ok`, не `changed`
>- Резервные копии созданы (проверка: `ls /etc/*~` или `ls /etc/*.backup`)
