# Модуль 4: Работа с файлами и каталогами в Ansible
>[!SUMMARY] Цель
Научиться управлять файлами, каталогами и правами доступа на управляемых хостах с помощью специализированных модулей Ansible.

>[!OBJECTIVES] Задачи урока
>
>- Освоить модули: `file`, `copy`, `fetch`, `template`, `assemble`, `unarchive`
>- Научиться управлять правами (`mode`), владельцами (`owner`/`group`)
>- Работать с символическими и жёсткими ссылками
>- Архивировать и распаковывать файлы
>- Понимать систему прав доступа Unix (octal/binary/mask)

## Ключевые модули: обзор
| Модуль                       | Назначение                                               | Идемпотентен? | Пример использования                           |
| ---------------------------- | -------------------------------------------------------- | ------------- | ---------------------------------------------- |
| `ansible.builtin.file`       | Управление файлами, каталогами, ссылками, правами        | Да            | Создать директорию, изменить права             |
| `ansible.builtin.copy`       | Копирование файла **с** control node **на** managed node | Да            | Деплой конфигурационного файла                 |
| `ansible.builtin.fetch`      | Копирование файла **с** managed node **на** control node | Да            | Сбор логов, бэкап конфигов                     |
| `ansible.builtin.template`   | Копирование + шаблонизация через Jinja2                  | Да            | Генерация конфигов с переменными               |
| `ansible.builtin.assemble`   | Сборка файла из нескольких фрагментов                    | Да            | Конфигурация из частей (как `/etc/sudoers.d/`) |
| `ansible.builtin.unarchive`  | Распаковка архивов (tar, zip, gz)                        | Да            | Развёртывание приложения из архива             |
| `ansible.builtin.lineinfile` | Гарантия наличия/отсутствия строки в файле               | Да            | Быстрое редактирование конфигов                |

>[!TIP] Почему идемпотентность важна? 
>Все модули выше проверяют текущее состояние перед изменением. Если файл уже имеет нужные права/содержимое — задача завершится как `ok`, без изменений. Это позволяет безопасно запускать плейбуки многократно.

## Модуль `file`: управление файловой системой
### Ключевые параметры:
|Параметр|Описание|Пример|
|---|---|---|
|`path` / `dest`|Путь к файлу, каталогу или ссылке|`/etc/myapp/config.conf`|
|`state`|Желаемое состояние (см. ниже)|`directory`, `link`, `absent`|
|`mode`|Права доступа (octal или symbolic)|`'0755'`, `'u+rwx,g+rx,o=rx'`|
|`owner`|Пользователь-владелец|`nginx`, `root`|
|`group`|Группа-владелец|`www-data`, `admins`|
|`src`|Источник для ссылок|`/actual/path` (для `state: link`)|
|`force`|Принудительное создание (перезапись)|`yes` для ссылок|
### Значения `state`:
```yaml
state: file      # Обычный файл (проверяет существование)
state: directory # Каталог (создаёт, если нет)
state: link      # Символическая ссылка (требует src)
state: hard      # Жёсткая ссылка (требует src)
state: touch     # Создать пустой файл или обновить mtime
state: absent    # Удалить файл/каталог/ссылку (рекурсивно для каталогов)
```
### Практические примеры:
```yaml
#  Создать каталог с правами
- name: Create backup directory
  ansible.builtin.file:
    path: /opt/backup
    state: directory
    mode: '0750'        # rwxr-x---
    owner: backup-user
    group: backup-group

# Создать символическую ссылку
- name: Link config to shared location
  ansible.builtin.file:
    src: /opt/app/config.yml
    dest: /etc/app/config.yml
    state: link

#  Удалить файл, если существует
- name: Remove temporary file
  ansible.builtin.file:
    path: /tmp/deploy.lock
    state: absent

#  Создать пустой файл (touch)
- name: Create marker file
  ansible.builtin.file:
    path: /var/run/myapp.ready
    state: touch
    mode: '0644'

#  Изменить права на существующий файл (без изменения содержимого)
- name: Secure private key
  ansible.builtin.file:
    path: /etc/ssl/private/app.key
    mode: '0600'
    owner: root
    group: root
```

>[!WARNING] Рекурсивное удаление 
При `state: absent` для каталога удаляется **всё содержимое**. Всегда проверяйте путь!

## Модуль `copy`: копирование файлов на удалённый хост

### Ключевые параметры:
| Параметр          | Описание                                                      | Пример                          |
| ----------------- | ------------------------------------------------------------- | ------------------------------- |
| `src`             | Путь на **control node** (относительно playbook или `files/`) | `files/nginx.conf`              |
| `content`         | Прямое содержимое файла (альтернатива `src`)                  | `"server { listen 80; }"`       |
| `dest`            | Путь на **managed node** (обязательно)                        | `/etc/nginx/nginx.conf`         |
| `mode`            | Права доступа после копирования                               | `'0644'`                        |
| `owner` / `group` | Владелец и группа                                             | `root`, `nginx`                 |
| `backup`          | Создать бэкап при изменении                                   | `yes` → `.YYYY-MM-DD@HH:MM:SS~` |
| `validate`        | Команда для проверки файла перед заменой                      | `nginx -t -c %s`                |
### Практические примеры:
```yaml
# Копирование файла из каталога playbook
- name: Deploy SSH config
  ansible.builtin.copy:
    src: files/sshd_config
    dest: /etc/ssh/sshd_config
    owner: root
    group: root
    mode: '0600'
    backup: yes
    validate: 'sshd -t -f %s'  # Проверка перед применением

# Создание файла из переменной (content)
- name: Create environment file
  ansible.builtin.copy:
    content: |
      APP_ENV=production
      DB_HOST={{ db_host }}
      LOG_LEVEL=info
    dest: /opt/app/.env
    mode: '0640'
    owner: app-user
    group: app-group

# Копирование с проверкой валидности (для конфигов)
- name: Deploy nginx config with validation
  ansible.builtin.copy:
    src: templates/nginx.conf
    dest: /etc/nginx/nginx.conf
    validate: 'nginx -t -c %s'  # %s подставится путь к файлу
    backup: yes
```

>[!TIP] Пути в `src`
>
>- Относительный путь ищется относительно каталога с playbook
>- Специальная директория `files/` рядом с playbook — удобное место для статики
>- Абсолютные пути на control node тоже работают, но менее переносимы

## Модуль `fetch`: скачивание файлов с управляемых хостов

### Когда использовать:

- Сбор логов для аудита
- Бэкап конфигурационных файлов
- Диагностика: получение дампов, статусов
### Ключевые параметры:
| Параметр          | Описание                               | Пример                       |
| ----------------- | -------------------------------------- | ---------------------------- |
| `src`             | Путь на **managed node**               | `/var/log/secure`            |
| `dest`            | Путь на **control node** (директория!) | `./logs/`                    |
| `flat`            | Убрать структуру `hostname/` в пути    | `yes` для одного хоста       |
| `fail_on_missing` | Ошибка, если файл не найден            | `no` для опциональных файлов |
### Практические примеры:
```yaml
# Сбор логов с сохранением структуры по хостам
- name: Collect security logs
  ansible.builtin.fetch:
    src: /var/log/secure
    dest: ./logs/secure/
    flat: no  # Результат: ./logs/secure/hostname/secure

# Бэкап конфига с одного хоста (flat)
- name: Backup nginx config
  ansible.builtin.fetch:
    src: /etc/nginx/nginx.conf
    dest: ./backups/nginx.conf
    flat: yes  # Результат: ./backups/nginx.conf (без подкаталогов)

# Опциональный файл: не ошибаться, если отсутствует
- name: Fetch optional debug log
  ansible.builtin.fetch:
    src: /tmp/debug.log
    dest: ./debug_logs/
    fail_on_missing: no
```

>[!WARNING] `dest` — это всегда **директория** (если `flat: no`) Ansible автоматически создаст подкаталог с именем хоста, чтобы избежать конфликтов при сборе с нескольких узлов.

## Модуль `unarchive`: распаковка архивов

### Ключевые параметры:
| Параметр          | Описание                            | Пример                      |
| ----------------- | ----------------------------------- | --------------------------- |
| `src`             | Путь к архиву                       | `files/app.tar.gz`          |
| `dest`            | Каталог назначения на managed node  | `/opt/app`                  |
| `remote_src`      | Архив уже на удалённом хосте?       | `yes` / `no` (по умолчанию) |
| `extra_opts`      | Доп. опции для утилиты распаковки   | `['--strip-components=1']`  |
| `list_files: yes` | Вернуть список распакованных файлов | Для отладки                 |
### Практические примеры:
```yaml
# Распаковка архива с control node
- name: Deploy application archive
  ansible.builtin.unarchive:
    src: files/myapp-v1.2.tar.gz
    dest: /opt/myapp
    remote_src: no  # Архив на управляющем узле
    owner: app-user
    group: app-group

# Распаковка локального архива на managed node
- name: Extract downloaded package
  ansible.builtin.unarchive:
    src: /tmp/package.zip
    dest: /opt/packages
    remote_src: yes  # Архив уже на целевом хосте

# Распаковка с очисткой вложенности
- name: Extract and flatten structure
  ansible.builtin.unarchive:
    src: files/source.tar.gz
    dest: /usr/local/src/myapp
    extra_opts: [--strip-components=1]  # Убрать верхний уровень каталога
```
