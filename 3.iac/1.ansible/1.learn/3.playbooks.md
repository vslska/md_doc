## Что такое Playbook?

**Playbook** — это сценарий автоматизации в формате **YAML**, который описывает:

- **Какие хосты** затронуть (`hosts`)
- **Какие задачи** выполнить (`tasks`)
- **С какими параметрами** (переменные, модули, условия)

```yaml
# Минимальный рабочий playbook
---
- name: Hello Play
  hosts: all
  gather_facts: no
  tasks:
    - name: Print greeting
      ansible.builtin.debug:
        msg: "Привет, {{ inventory_hostname }}! 👋"
```

>[!TIP] Почему YAML?
>
>- Человекочитаемый формат (меньше ошибок при написании)
>- Поддерживает комментарии (`#`)
>- Единый стандарт для всей экосистемы Ansible
>- Легко версионировать в Git

## Общая структура Playbook
```yaml
---
# 1. Начало YAML-документа (обязательно!)

- name: <Описание сценария>          # Название play (отображается в выводе)
  hosts: <группа_или_хост>           # Целевые узлы из inventory
  become: yes                        # Повышение привилегий (опционально)
  gather_facts: yes                  # Собирать ли системные факты (по умолчанию: yes)
  
  vars:                              # Локальные переменные (опционально)
    app_port: 8080
    package_name: nginx

  tasks:                             # Список задач (обязательно)
    - name: <Что делает задача>
      <модуль>:
        <параметр>: <значение>
      
    - name: <Следующая задача>
      <модуль>:
        ...
```

>[!WARNING] Критично для YAML
>
>- Всегда начинайте файл с `---`
>- Используйте **пробелы**, а не табы (2 пробела — стандарт Ansible)
>- Сохраняйте единую глубину отступов внутри блоков
>- Ключи `name`, `hosts`, `tasks` должны быть на одном уровне

---
## Ключевые элементы:

| Ключ            | Уровень     | Описание                    | Пример                  |
| --------------- | ----------- | --------------------------- | ----------------------- |
| `name`          | play / task | Человекочитаемое описание   | `name: Install nginx`   |
| `hosts`         | play        | Целевые хосты/группы        | `hosts: webservers`     |
| `tasks`         | play        | Список действий             | `tasks: [{...}, {...}]` |
| `become`        | play / task | Повышение до root (sudo)    | `become: yes`           |
| `vars`          | play / task | Локальные переменные        | `vars: {port: 80}`      |
| `gather_facts`  | play        | Собирать ли факты о системе | `gather_facts: no`      |
| `ignore_errors` | task        | Продолжать при ошибке       | `ignore_errors: yes`    |

## Работа с модулями:примеры

### Установка пакета (идемпотентная задача)
```yaml
- name: Ensure nginx is installed
  ansible.builtin.dnf:        # для RHEL/CentOS/Fedora
  # ansible.builtin.apt:     # для Debian/Ubuntu
    name: nginx
    state: present            # present/absent/latest
    update_cache: yes         # обновить кэш пакетов (только для apt)
```
**Идемпотентность**: если пакет уже установлен — задача пройдёт как `ok`, без изменений.
### Управление сервисом
```yaml
- name: Start and enable nginx
  ansible.builtin.service:
    name: nginx
    state: started            # started/stopped/restarted
    enabled: yes              # автозагрузка при загрузке ОС
```
---
### Работа с файлами
```yaml
- name: Deploy configuration file
  ansible.builtin.copy:
    content: |
      server {
        listen 80;
        server_name _;
      }
    dest: /etc/nginx/conf.d/default.conf
    owner: root
    group: root
    mode: '0644'
    backup: yes               # создать бэкап при изменении
```
>[!TIP] Модуль `copy` vs `template`
>
>- `copy` — копирует файл «как есть»
>- `template` — обрабатывает файл через Jinja2 (переменные, условия) — тема следующего модуля

## Модули `command` и `shell`: когда и как использовать
| Модуль    | Когда использовать                               | Безопасность          | Пример                            |
| --------- | ------------------------------------------------ | --------------------- | --------------------------------- |
| `command` | Простые команды без оболочек                     | Высокая               | `cmd: systemctl status nginx`     |
| `shell`   | Команды с пайпами, перенаправлением, переменными |  Требует осторожности | `cmd: echo $HOME > /tmp/test.txt` |
### Примеры:
```yaml
#  Безопасно: команда без оболочки
- name: Check running processes
  ansible.builtin.command:
    cmd: ps aux --sort=-%mem | head -10
  #  Но: пайпы НЕ работают в command! Используйте shell:

#  С оболочкой: когда нужны пайпы, переменные, редиректы
- name: Count logged users
  ansible.builtin.shell:
    cmd: who | wc -l
  args:
    executable: /bin/bash     # явное указание оболочки

#  Регистрация вывода для дальнейшего использования
- name: Get system uptime
  ansible.builtin.command: uptime
  register: uptime_result     # сохраняем вывод в переменную

- name: Show captured output
  ansible.builtin.debug:
    var: uptime_result.stdout
```

>[!WARNING] Безопасность при использовании `shell`
>
>- Никогда не подставляйте непроверенные переменные напрямую в команду
>- Используйте `args: {warn: false}` только если уверены в команде
>- Предпочитайте специализированные модули (`yum`, `service`, `copy`) вместо `shell`

## Повышение привилегий (`become`)
```yaml
---
- name: System update with sudo
  hosts: all
  become: yes                  # включить sudo для всех задач в play
  become_method: sudo          # метод: sudo, su, pbrun и др.
  become_user: root            # от чьего имени выполнять (по умолчанию: root)
  
  tasks:
    - name: Update all packages
      ansible.builtin.dnf:
        name: "*"
        state: latest
```

---
### Локальное повышение (только для одной задачи):
```yaml
tasks:
  - name: Read a file (без sudo)
    ansible.builtin.command: cat /etc/hostname
  
  - name: Edit system config (с sudo)
    ansible.builtin.lineinfile:
      path: /etc/ssh/sshd_config
      line: "PermitRootLogin no"
    become: yes                # только эта задача выполняется с sudo
```

>[!TIP] Запрос пароля при запуске
>```bash
># Если sudo требует пароль:
>ansible-playbook playbook.yml --ask-become-pass
>
># Для отладки: показать, какие команды выполняются с sudo:
ansible-playbook playbook.yml -vvv
>```

## Несколько `play` в одном Playbook

Playbook может содержать несколько независимых блоков `play`, выполняемых последовательно:
```yaml
---
# Play 1: Подготовка веб-серверов
- name: Configure web servers
  hosts: webservers
  become: yes
  tasks:
    - name: Install nginx
      ansible.builtin.dnf:
        name: nginx
        state: present

# Play 2: Настройка базы данных
- name: Configure database servers
  hosts: db
  become: yes
  vars:
    db_port: 5432
  tasks:
    - name: Install PostgreSQL
      ansible.builtin.dnf:
        name: postgresql-server
        state: present

# Play 3: Финальная проверка (без sudo)
- name: Verify deployment
  hosts: all
  gather_facts: no
  tasks:
    - name: Show service status
      ansible.builtin.command: systemctl is-active nginx
      register: result
      changed_when: false      # не помечать задачу как "changed"
      
    - name: Display result
      ansible.builtin.debug:
        msg: "nginx is {{ 'running' if result.rc == 0 else 'stopped' }}"
```

>[!NOTE] Порядок выполнения
>
>- Play выполняются **строго сверху вниз**
>- Переменные из одного play **не видны** в другом (если не использовать `set_fact` или `host_vars`)

## Инструменты отладки и проверки Перед запуском:
```bash
# Проверка синтаксиса YAML и структуры
ansible-playbook --syntax-check playbook.yml

# Показать, какие хосты будут затронуты
ansible-playbook playbook.yml --list-hosts

# Показать список задач без выполнения
ansible-playbook playbook.yml --list-tasks
```

### Безопасное тестирование:
```bash
# Check mode: показать, что ИЗМЕНИЛОСЬ бы, без применения
ansible-playbook playbook.yml --check

# + diff: показать различия в текстовых файлах
ansible-playbook playbook.yml --check --diff

# Запустить с детализацией вывода
ansible-playbook playbook.yml -v      # кратко
ansible-playbook playbook.yml -vvv    # подробно (отладка)
```
### Отладка во время выполнения:
```bash
# Начать выполнение с конкретной задачи
ansible-playbook playbook.yml --start-at-task "Install nginx"

# Пошаговое выполнение (подтверждение перед каждой задачей)
ansible-playbook playbook.yml --step

# Выполнить только задачи с определённым тегом
ansible-playbook playbook.yml --tags "install,config"
```

>[!TIP] Теги (tags) — базовое использование

```yaml
tasks:
  - name: Install package
    ansible.builtin.dnf:
      name: nginx
    tags: install
  
  - name: Configure service
    ansible.builtin.template:
      src: nginx.conf.j2
      dest: /etc/nginx/nginx.conf
    tags: config
```

```bash
# Запустить только задачи с тегом "install"
ansible-playbook playbook.yml --tags install
```

## Обработка ошибок и продолжение выполнения
```bash
tasks:
  # Игнорировать ошибку (задача помечается как "failed", но выполнение продолжается)
  - name: Try to stop non-critical service
    ansible.builtin.service:
      name: legacy-app
      state: stopped
    ignore_errors: yes

  # Продолжить, даже если команда вернёт код ошибки ≠ 0
  - name: Check optional file
    ansible.builtin.command: cat /opt/optional/config.txt
    register: result
    failed_when: result.rc not in [0, 1]  # 0 = OK, 1 = файл не найден (допустимо)
```

>[!WARNING] Используйте `ignore_errors` с осторожностью
>
>- Скрывает реальные проблемы
>- Лучше явно указывать допустимые коды возврата через `failed_when`

## Шпаргалка: частые модули для начала
| Модуль                        | Назначение                          | Ключевые параметры               |
| ----------------------------- | ----------------------------------- | -------------------------------- |
| `ansible.builtin.debug`       | Вывод отладочной информации         | `msg`, `var`                     |
| `ansible.builtin.command`     | Выполнение команды (без shell)      | `cmd`, `creates`, `removes`      |
| `ansible.builtin.shell`       | Выполнение команды в shell          | `cmd`, `executable`, `creates`   |
| `ansible.builtin.copy`        | Копирование файла на удалённый хост | `content`, `src`, `dest`, `mode` |
| `ansible.builtin.file`        | Управление файлами/директориями     | `path`, `state`, `owner`, `mode` |
| `ansible.builtin.dnf` / `apt` | Управление пакетами                 | `name`, `state`, `update_cache`  |
| `ansible.builtin.service`     | Управление сервисами                | `name`, `state`, `enabled`       |
| `ansible.builtin.ping`        | Проверка подключения                | (нет параметров)                 |

>
># Только параметры (кратко)

## Выводы

 **Playbook — основа автоматизации**:

- Декларативный YAML-формат: описываем **что** нужно, а не **как**
- Идемпотентность: безопасный повторный запуск без побочных эффектов
- Читаемость: плейбуки = документация + код

 **Структура и лучшие практики**:

- Всегда используйте `name` для play и каждой задачи
- Начинайте файлы с `---`, используйте 2-пробельные отступы
- Проверяйте синтаксис перед запуском: `--syntax-check`
- Тестируйте в `--check` режиме перед применением

**Безопасность и отладка**:

- `become` — контролируемое повышение привилегий
- `--check --diff` — безопасное предпросмотр изменений
- `-vvv` и `--step` — мощные инструменты отладки

#### 2. Создать `playbooks/sysinfo.yml`
```yamk
# ~/ll105-lab3/playbooks/sysinfo.yml
---
- name: Lab3: Collect system information
  hosts: all
  gather_facts: yes
  tasks:
    - name: Display basic host info
      ansible.builtin.debug:
        msg: |
          Host: {{ ansible_hostname }}
          OS: {{ ansible_distribution }} {{ ansible_distribution_version }}
          Kernel: {{ ansible_kernel }}
          CPUs: {{ ansible_processor_vcpus }}
          Memory: {{ ansible_memtotal_mb }} MB

    - name: Show network interfaces
      ansible.builtin.debug:
        var: ansible_default_ipv4.address
      when: ansible_default_ipv4.address is defined

    - name: Check disk usage on root
      ansible.builtin.command: df -h /
      register: disk_info
      changed_when: false

    - name: Display disk usage
      ansible.builtin.debug:
        var: disk_info.stdout_lines
```
#### 3. Проверить и выполнить
```bash
# 1. Проверка синтаксиса
ansible-playbook --syntax-check playbooks/sysinfo.yml

# 2. Проверка в dry-run режиме
ansible-playbook -i inventory/inventory.yml playbooks/sysinfo.yml --check

# 3. Полноценный запуск
ansible-playbook -i inventory/inventory.yml playbooks/sysinfo.yml

# 4. Запуск с детализацией (если что-то пошло не так)
ansible-playbook -i inventory/inventory.yml playbooks/sysinfo.yml -vv
```

#### 4. Дополнительное задание (по желанию)
```yaml
# Создать playbook для установки и проверки nginx
# playbooks/nginx_basic.yml
---
- name: Lab3 Bonus: Basic nginx setup
  hosts: webservers
  become: yes
  tasks:
    - name: Install nginx
      ansible.builtin.dnf:
        name: nginx
        state: present
        update_cache: yes

    - name: Ensure nginx is running
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: yes

    - name: Verify nginx is listening on port 80
      ansible.builtin.command: ss -tlnp | grep :80
      register: nginx_port
      changed_when: false
      failed_when: nginx_port.rc != 0

    - name: Show confirmation
      ansible.builtin.debug:
        msg: "nginx is active and listening on port 80"
```

>[!SUCCESS] Критерии выполнения лабы
>
>- `sysinfo.yml` корректно собирается и выполняется без ошибок
>- Вывод содержит имя хоста, ОС, количество CPU и информацию о диске
>- Использованы `gather_facts: yes` и переменные `ansible_*`
>- Запуск с `--check` не вносит изменений (все задачи `ok`, не `changed`)
>- (Бонус) `nginx_basic.yml` устанавливает и проверяет веб-сервер
