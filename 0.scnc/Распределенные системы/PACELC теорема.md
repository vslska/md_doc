# 📐 PACELC теорема

**Теорема PACELC** — это расширение классической `[[Теорема CAP|теоремы CAP]]`, сформулированное профессором Дэниелом Абади в 2012 году. Она устраняет главный недостаток теоремы CAP: описание компромиссов системы **в штатном режиме работы**, когда в сети нет аварий и разделений.

---

## 🧩 Расшифровка аббревиатуры

Формула теоремы строится на логическом условии **IF... ELSE...** (Если... Иначе...):

$$\text{If } \mathbf{P} \text{ (Partition), then choice between } \mathbf{A} \text{ and } \mathbf{C}$$
$$\text{Else } (\mathbf{E}), \text{ choice between } \mathbf{L} \text{ and } \mathbf{C}$$

*   **P (Partition)**: Если произошел сетевой сбой (Разрыв сети)...
*   **A (Availability)** / **C (Consistency)**: ...система должна выбрать между **Доступностью** или **Согласованностью** (это классическая CAP).
*   **E (Else)**: Иначе, если сеть работает абсолютно нормально (Штатный режим)...
*   **L (Latency)** / **C (Consistency)**: ...система должна выбрать между **Задержкой (Скоростью)** или **Согласованностью**.

---

## ⚖️ Почему возникает выбор в штатном режиме (E -> L vs C)?

Даже если сеть исправна, данные не могут перемещаться со скоростью выше скорости света. Если у вас реплицированная база данных (несколько серверов), при записи новых данных у инженера есть два пути:

1.  **Выбор Latency (Низкая задержка / Скорость)**: Сервер принимает запись от клиента, мгновенно отвечает «Успешно!» и только потом, в фоновом режиме, отправляет копию на другие сервера. 
    *   *Плюс*: Клиент получает ответ мгновенно.
    *   *Минус*: Если другой клиент в этот же миг обратится к соседнему серверу, он прочитает старые данные. Согласованность нарушена.
2.  **Выбор Consistency (Высокая согласованность)**: Сервер принимает запись, блокирует клиента, отправляет данные на все остальные сервера, ждет от них подтверждения и только после этого отвечает клиенту «Успешно!».
    *   *Плюс*: Данные на всех серверах гарантированно одинаковы в любой момент.
    *   *Минус*: Клиент долго ждет ответ (высокий Latency) из-за накладных расходов на сетевые запросы между серверами.

## 📊 Классификация реальных баз данных по PACELC

В зависимости от настроек по умолчанию, СУБД делятся на 4 основные категории:

### 1. PC/EC (Строгие максималисты)
*   **При сбое (PC)**: Выбирают Согласованность данных, блокируя ответы.
*   **В норме (EC)**: Жертвуют скоростью, чтобы всегда отдавать самые свежие данные.
*   **Примеры**: `[[etcd]]`, `[[Apache ZooKeeper]]`, [[VoltDB]], [[Google Spanner]].

### 2. PA/EL (Максимально быстрые / Прогрессивные)
*   **При сбое (PA)**: Остаются доступными, отдавая старые данные.
*   **В норме (EL)**: Работают на максимальной скорости, обновляя реплики асинхронно.
*   **Примеры**: `[[Cassandra]]`, [[DynamoDB]], [[CouchDB]], [[Riak]].

### 3. PC/EL (Строгие при аварии, быстрые в норме)
*   **При сбое (PC)**: Не допускают рассинхронизации данных при разрыве сети.
*   **В норме (EL)**: Предпочитают отвечать клиенту быстро, допуская временный лаг репликации при нормальной работе.
*   **Примеры**: [[MongoDB]] (в стандартной конфигурации).

### 4. PA/EC (Доступные при аварии, строгие в норме)
*   **При сбое (PA)**: Позволяют читать и писать данные в изолированные сегменты сети.
*   **В норме (EC)**: Требуют синхронного подтверждения от всех узлов, замедляя обычные транзакции.
*   **Примеры**: Редкая конфигурация. Встречается в некоторых реляционных БД (например, [[PostgreSQL]] или [[MySQL]]) при специфической настройке полусинхронной репликации.

## 📝 Краткая шпаргалка для архитектора



| Тип системы | Как ведет себя при разрыве сети? | Как ведет себя, когда всё хорошо? | Основной приоритет |
| :--- | :--- | :--- | :--- |
| **PC/EC** | Выдает ошибку, спасая данные | Ждет все сервера, увеличивая пинг | **Точность данных** |
| **PA/EL** | Отвечает старыми данными | Отвечает мгновенно, пишет в фоне | **Скорость работы** |
