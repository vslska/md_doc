
> Это фундаментальные концепции, лежащие в основе всех систем — от embedded до distributed cloud.

##  Параллелизм vs Конкуренция (Concurrency)

| Термин | Определение | Ключевая идея |
|--------|-------------|---------------|
| **Concurrency** (Конкуренция) | Способность выполнять **несколько задач в перекрывающееся время** → *логический* параллелизм. Может быть реализован даже на одном ядре (через переключение контекста). | **Работоспособность при ожидании** (I/O, сеть, диск). Пример: async/await, event loop. |
| **Parallelism** (Параллелизм) | Фактическое **одновременное выполнение задач** на нескольких вычислительных единицах (ядрах, CPU, машинах). | **Ускорение вычислений** (CPU-bound). Пример: multiprocessing, GPU kernels. |

>  *Concurrency ≠ Parallelism* — но они совместимы:  
> — Асинхронный код может быть запущен в нескольких процессах → и конкурентность, и параллелизм.

## Модели конкуренции и параллелизма

| Модель | Уровень | Особенности | Примеры |
|--------|---------|-------------|---------|
| **Многопроцессность** | OS | Изоляция, полный параллелизм (GIL bypass), высокая память/накладные расходы | `multiprocessing`, `Gunicorn sync workers` |
| **Многопоточность** | OS / Runtime | Лёгкие, но в CPython — ограничены GIL → эффективны только при I/O | `threading`, `Celery + threads` (редко) |
| **Асинхронность (event-driven)** | Приложение / Runtime | Однопоточная кооперативная многозадачность. Требует `await`-совместимых операций. | `asyncio`, `gevent`, `Gunicorn + gevent workers` |
| **Распределённые вычисления** | Сеть | Компоненты на разных машинах. Требуется сериализация, fault tolerance, сетевая устойчивость | `Celery`, `Apache Spark`, `Ray` |
| **Гибридные модели** | — | Комбинация вышеуказанных (например: процессы + async воркеры) | `Gunicorn + 4 async workers` |

##  Масштабируемость (Scalability)

### Виды:
| Вид | Описание | Примеры |
|-----|----------|---------|
| **Вертикальное масштабирование** (Scale-up) | Увеличение ресурсов одной машины (CPU, RAM, SSD) | Более мощный сервер в облаке |
| **Горизонтальное масштабирование** (Scale-out) | Добавление новых узлов (серверов) и распределение нагрузки | Кластер Celery workers + балансировщик |
| **Функциональное шардирование** | Разделение по *семантике* (сервисы, домены) | Микросервисы: auth-service, payment-service |
| **Данные шардирование** | Разделение данных по ключу (user_id, region) | PostgreSQL + Citus, MongoDB shards |

### Законы и ограничения:
- **Закон Амдала**: Максимальное ускорение ограничено *последовательной* частью программы.  
  → Чтобы достичь 10× ускорения, >90% кода должно быть параллельным.
- **Закон Густавсона**: При увеличении размера задачи — можно достичь почти линейного ускорения.
- **CAP-теорема** (для распределённых систем):  
  В условиях сетевого разделения можно выбрать **только два** из:  
  🔹 **C**onsistency, 🔹 **A**vailability, 🔹 **P**artition tolerance

## Практические рекомендации

1. **Прежде чем масштабировать — профилируй**  
   — 90% узких мест — в БД, неоптимальных запросах, блокировках.
2. **Асинхронность ≠ ускорение CPU-задач**  
   — Она экономит *ресурс ожидания*, а не ускоряет вычисления.
3. **Цена координации растёт с масштабом**  
   — Распределённые транзакции, distributed locks — дорого.
4. **Масштабируемость — это trade-off**  
   — Децентрализация → сложность; Кэш → риск неактуальности; Репликация → задержка.

> 📚 Важные темы для углубления:  
> — Race conditions, deadlocks, livelocks  
> — Memory consistency models (sequential, relaxed)  
> — Actor model (Erlang/Elixir, Akka)  
> — CSP (Communicating Sequential Processes — Go channels, Python `asyncio.Queue`)  
> — CRDTs (Conflict-Free Replicated Data Types) для eventual consistency
